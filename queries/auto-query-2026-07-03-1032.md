## 1. 설계자 전결정의 구조: [[entities/adema.md|adema]]와 [[entities/agora-opt.md|agora opt]]가 보여주는 한계

[[entities/adema.md|adema]]는 "Knowledge-State Orchestration Architecture"라는 명칭 자체가 문제의 본질을 드러냅니다. **오케스트레이션**이란 악보(설계자가 정한 규칙)가 이미 존재하고, 시스템은 그에 맞춰 연주(실행)한다는 은유입니다. 기억 상태를 어떻게 구성하고, 언제 어떤 지식 상태로 전이할지에 대한 정책이 아키텍처 수준에서 하드코딩된다는 뜻입니다.

[[entities/agora-opt.md|agora opt]] 역시 이 패턴을 확인시킵니다. 해당 논문("From Soliloquy to Agora")이 런타임 메모리와 토론 구조를 통해 다중 에이전트 정렬을 달성한다고 하더라도, **메모리에 접근하는 방식 자체**는 런타임 이전에 결정된 것입니다. [[entities/agora-opt.md|agora opt]]의 설명에서도 MASPO가 "프롬프트 설계 시점에서 사전 정렬을 보장"한다는 점이 강조되는데, 이는 정렬의 책임을 설계 시점으로 분산시킨다는 긍정적 측면과 동시에, **런타임에서 메모리 정책 자체를 학습하지 않는다**는 부정적 측면을 동시에 드러냅니다.

## 2. 의미론적 정보의 평탄화: [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]과의 구조적 동형성

이 한계는 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]이 [[concepts/adaptive-inference.md|adaptive inference]] 맥락에서 이미 밝혀낸 패턴과 정확히 동형입니다.

Crab 논문이 지적하듯, "에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율"이 간극의 본질입니다. 이를 메모리 아키텍처에 대입하면:

> **에이전트의 의미론적 기억 요구(지금 무엇을 기억해야 하는가, 무엇을 잊어야 하는가)를 아키텍처가 사전 결정한 검색/저장 메커니즘(정적 인덱스, 고정된 유사도 임계값 등)으로 번역할 때 발생하는 비효율**

기억 전략이라는 풍부한 의미론적 결정 공간이, 고정된 파이프라인 안에서 평탄화(flattening)되는 것입니다.

## 3. AutoMem의 패러다임: 메타메모리를 모델 액션으로

이에 대해 AutoMem("Automated Learning of Memory as a Cognitive Skill")은 인지과학의 **metamemory** 개념—"무엇을 인코딩할지, 언제 검색할지, 지식을 어떻게 조직할지 아는 능력"—를 LLM에 도입합니다. 읽기·쓰기·잊기를 RL로 학습하는 "일급 모델 액션"으로 정의한다는 점이 핵심인데, 이는:

- **무엇을 기억할지**(what to encode) → 설계자가 정한 스키마가 아니라 환경 상호작용 통해 학습
- **언제 검색할지**(when to retrieve) → 고정된 트리거가 아니라 가치 함수 기반 결정
- **어떻게 조직할지**(how to organize) → 사전 정의된 인덱스 구조가 아니라 학습된 정책

## 4. [[concepts/adaptive-inference.md|adaptive inference]]와의 병렬: 적응의 차원 확장

[[concepts/adaptive-inference.md|adaptive inference]]에서 SpecKV가 보여준 패턴이 여기서도 적용됩니다. SpecKV의 설명에 따르면:

> "기존 컨텍스트 기반 적응(CADENCE 등)이 외부 환경에 반응했다면, SpecKV는 **압축이라는 내부 시스템 상태**에 반응하는 적응의 새로운 유형을 제시한다."

이와 정확히 병렬적으로:
- **기존 메모리 아키텍처** → 외부 환경(쿼리, 컨텍스트)에 반응하는 검색 (적응의 대상이 외부 입력에 한정)
- **AutoMem** → 에이전트 **자신의 내부 인지 상태**(현재 기억의 유용성, 충돌 여부 등)에 반응하는 메모리 정책 (적응의 대상이 내부 상태로 확장)

즉, 적응의 결정 차원이 "검색 쿼리 최적화"에서 "기억 전략 자체의 최적화"로 이동하는 것입니다.

## 5. [[concepts/adaptive-forgetting.md|adaptive forgetting]]의 구조적 결여

Wiki 슬러그 목록에 [[concepts/adaptive-forgetting.md|adaptive forgetting]]과 [[concepts/adaptive-forgetting-as-function.md|adaptive forgetting as function]]이 독립 엔티티로 존재한다는 사실 자체가, 연구 커뮤니티가 "잊기"를 적응적 으로 만들어야 한다는 문제 의식을 이미 보유하고 있음을 보여줍니다. 그러나 기존 아키텍처들에서 이 적응적 망각은 **함수로서 하드코딩**될 뿐, **학습 대상**이 되지 못했습니다. AutoMem은 이를 RL 학습 대상으로 끌어올림으로써, [[concepts/adaptive-forgetting-as-function.md|adaptive forgetting as function]]이라는 엔티티가 실제로 함수형 정의를 넘어 학습 가능한 정책으로 진화할 수 있는 경로를 제공합니다.

## 결론

기존 메모리 아키텍처들이 메모리 접근 정책을 학습 대상에서 제외한 것은 단순한 설계 선택이 아니라, [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]이 [[entities/adema.md|adema]]와 [[entities/agora-opt.md|agora opt]]의 형태로 메모리 도메인에 현현된 **구조적 한계**입니다. 에이전트가 환경 변화에 맞춰 기억 전략을 수정하려면, 기억 정책이 아키텍처의 하드코딩된 구성요소가 아니라 [[concepts/adaptive-inference.md|adaptive inference]]에서 SpecKV가 추측 길이(γ)를 학습한 것처럼, **모델 자체가 최적화하는 일급 결정 변수**가 되어야 합니다. AutoMem의 메타메모리 접근은 이 간극을 메우는 최초의 구체적 경로를 제시합니다.

## 핵심 통찰

기존 메모리 아키텍처는 기억 전략이라는 풍부한 의미론적 결정 공간을 고정된 파이프라인으로 평탄화하는 반면, AutoMem은 메타메모리를 일급 모델 액션으로 정의하여 적응의 대상을 외부 환경에서 내부 인지 상태로 확장한다.
