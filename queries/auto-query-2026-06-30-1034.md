## 2. 위험의 실제 발생 지점: 공유 저장소 생태계

최근 논문 *"Govern the Repository, Not the Agent: Measuring Ecosystem-Level Risk in AI-Native Software"*는 이 전제가 정확히 어디서 붕괴하는지 보여줍니다:

> 자율 코딩 에이전트가 이제 공유 저장소에서 대규모로 PR을 열고 병합하지만, 학계는 여전히 **에이전트 하나씩, 고립된 벤치마크에서** 평가한다.

여기서 위험의 발생 지점이 이동합니다:

| 기존 가정 | 실제 위험 발생 지점 |
|---|---|
| 개별 에이전트의 단일 실행 | 다수 에이전트가 생산한 아티팩트의 **저장소 내 누적·상호작용** |
| 에이전트→환경 단방향 영향 | 에이전트 A의 정상 출력이 에이전트 B의 컨텍스트가 되어 **코드베이스 수준 의존성 오염** |
| 샌드박스 내 격리 가능성 | 공유 저장소는 본질적으로 **격리 불가능한 공용 상태(shared mutable state)** |

이는 [[algorithm-system-translation-gap]]이 묘사하는 간극의 확장판입니다. *Crab* 연구가 "에이전트의 의미론적 의존 그래프를 OS의 전체 스냅샷 메커니즘으로 번역할 때 비효율이 발생한다"고 진단한 것과 동일한 구조적 문제가, 여기서는 **"에이전트 수준 안전 증명을 저장소 수준 위험 평가로 번역할 때 검증 공백이 발생"**하는 형태로 나타납니다.

## 3. 근본적 오지정이 의미하는 것

이 오지정이 "근본적"인 이유는, 단순히 검사 범위가 좁은 것을 넘어 **위험의 물리적 성질 자체를 오인**하고 있기 때문입니다.

### 3-1. 위험의 척도 오류
[[acceptable-risk-quantification]]와 연관된 *When No Benchmark Exists*가 벤치마크 없이 비교 안전성을 평가하려는 시도이긴 하지만, 여전히 비교 단위를 개별 모델 간 대비에 둡니다. 그러나 저장소 생태계에서 위험은 **개별 에이전트의 최악 출력이 아니라, 정상 출력들의 조합이 만들어내는 emergent hazard**입니다.

### 3-2. [[co-failure-ceiling]]의 생태계적 적용
*"When Does Combining Language Models Help?"*가 다중 모델 시스템(라우팅, 투표, MoA)의 이득이 **공동 실패 한계(co-failure ceiling)**에 의해 상한된다고 밝힌 발견은, 에이전트 안전성에 직접 적용됩니다. 개별 에이전트가 각각 독립 테스트를 통과(낮은 개별 실패율)하더라도, 저장소에서 공동으로 작업할 때 **공동 실패 한계가 생태계 수준에서 훨씬 낮게 형성**될 수 있습니다.

### 3-3. [[agentic-harness-engineering]]의 한계
*Crab*이 C/R 런타임을 하네스의 핵심으로 삼아 [[agentic-harness-engineering]]의 범위를 "애플리케이션 로직에서 실행 환경까지" 확장했다는 점은 진일보합니다. 그러나 이 확장도 여전히 **단일 에이전트의 실행 환경**을 기준으로 합니다. 저장소 수준 위험은 하네스가 아예 다루지 않는 차원입니다.
