# Auto Query: 현재 Wiki에서 에이전트 하네스 최적화는 스키마 축적 병목(aggregate-pipeline-serving)과 체크포인트/복원(agentic-harness-engineering) 등 '동기적 실행을 전제한' 인프라 차원에서 분석되는데, future-based 비동기 함수 호출이 LLM 디코딩과 도구 실행을 분리할 수 있다면, 하네스의 근본적 병목이 인프라가 아니라 동기성 가정 자체에 있었던 것은 아닌가?

**날짜**: 2026-05-16
**대상 논문**: Concurrency without Model Changes: Future-based Asynchronous Function Calling for LLMs
**선택 이유**: 기존 Wiki의 에이전트 런타임 분석이 스키마 레이턴시와 상태 복원에 집중하면서 함수 호출의 동기적 실행 의미론을 불가변의 전제로 삼고 있는 점을 정면으로 도전하는 논문이기 때문이다.

## Wiki 답변

질문자의 통찰에 전적으로 동의합니다. 현재 Wiki에 축적된 지식을 관통해 보면, 기존의 하네스 최적화 논의는 결국 **'동기적 실행(Synchronous Execution)'이라는 보이지 않는 상한선 아래에서의 국소적 최적화**에 머물러 있었음을 명확히 알 수 있습니다.

### 1. 동기성 가정 하의 인프라 차원 최적화와 그 한계
기존 Wiki에서 하네스의 병목은 인프라 관점에서 분석되었습니다. 

먼저 [[aggregate-pipeline-serving]]와 관련된 *"Tool Attention Is All You Need"* 논문은 다중 서버 MCP 배포 환경에서 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 **'스키마 축적'**에 의해 결정된다고 지적했습니다. 그러나 이 병목 역시 도구 호출이 일어날 때마다 스키마를 동기적으로 로드하고 파이프라인을 정지시키는 구조적 특성에서 기인한 2차적 증상입니다.

또한 [[agentic-harness-engineering]]와 관련된 *"Crab: A Semantics-Aware Checkpoint/Restore Runtime"* 논문은 C/R(Checkpoint/Restore) 런타임을 통해 하네스의 결함 내성, 롤백, 분기 등을 인프라 계층에서 직접 구현했습니다. 이는 에이전트 실행의 실패를 대비한 훌륭한 인프라적 안전망이지만, 근본적으로 '실행이 멈춘 상태(블로킹)'에서 상태를 스냅샷하고 복원한다는 동기적 실행의 틀을 벗어나지는 못합니다.

### 2. Future-based 비동기 호출이 드러내는 '진짜' 병목
최근 논문인 *"Concurrency without Model Changes: Future-based Asynchronous Function Calling for LLMs"*는 이 동기성 가정을 정면으로 깨뜨립니다. 해당 논문은 기존 함수 호출(도구 사용)이 **동기적 실행 의미론(Synchronous execution semantics)**에 구속되어 있어, LLM 디코딩이 도구 실행이 완료될 때까지 차단된다는 구조적 모순을 지적합니다.

즉, 스키마 축적 병목이든 체크포인트 오버헤드든, 하네스가 아무리 인프라를 최적화하더라도 **"도구를 부르는 순간 LLM의 사고(디코딩)가 멈춰야 한다"**는 동기성 가정 자체가 병목의 근원이었습니다. Future 기반 비동기 호출은 모델 아키텍처 변경 없이도 디코딩과 도구 실행을 분리하여, 하네스의 역할을 '동기적 실행의 관리자'에서 '비동기적 컨텍스트의 조율자'로 전환시킵니다.

### 3. 새로운 개념의 발견: 비동기 에이전트 작업 공간으로의 패러다임 전환
이러한 맥락에서 기존의 [[agentic-harness-engineering]]은 동기성 가정이라는 한계를 인프라로 감싸안은 '컴펜세이션(Compensation)' 설계였다고 볼 수 있습니다. 비동기 함수 호출이 도입되면, 하네스가 더 이상 병목을 관리하는 인프라가 아니라 병목 자체가 해소되는 구조로 진화해야 합니다.

이때 Wiki 슬러그 목록에 존재하는 **[[asynchronous-agent-workspace]]**라는 개념이 이 논의의 핵심 빌딩 블록으로 부각됩니다. LLM이 디코딩을 멈추지 않고 도구의 반환값을 Future 객체 등으로 비동기적으로 수신한다면, 하네스는 정적 스냅샷(Crab 방식)을 찍는 대신, 흘러가는 비동기 컨텍스트와 미해결 Promise들을 안전하게 유지하는 **[[asynchronous-agent-workspace]]**로 그 형태를 근본적으로 재설계되어야 합니다.

### 결론
질문자의 지적대로 하네스의 근본적 병목은 인프라가 아니라 **'동기성 가정 자체'**에 있었습니다. [[aggregate-pipeline-serving]]가 해결하려 했던 스키마 병목과 [[agentic-harness-engineering]]가 보완하려 했던 상태 관리 비효율은 모두 동기적 블로킹이라는 공통 분모를 가진 하위 문제들이었습니다. Future-based 비동기 호출은 이 공통 분모를 제거함으로써, 하네스 최적화의 패러다임을 '동기적 인프라 관리'에서 '[[asynchronous-agent-workspace]]**에 기반한 비동기적 의존성 조율'로 전환해야 함을 시사합니다.
