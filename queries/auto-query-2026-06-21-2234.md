# 컨텍스트 위임이 도메인 정책 준수에 초래하는 근원적 한계

## 1. 문제의 진단: 구조적 상태 진공

현재 Wiki가 포착하는 에이전트 메모리 아키텍처는 명확히 양극화되어 있다:

- **지식 축적 층**: [[adema]]가 장기 지식 상태(knowledge-state)를 오케스트레이션하며, 에이전트가 "무엇을 알고 있는가"를 관리
- **실행 직렬화 층**: [[agentic-harness-engineering]]의 관점에서 [[algorithm-system-translation-gap]]을 현현하는 Crab이 에이전트의 의미론적 의존 그래프를 OS 수준의 체크포인트 스냅샷으로 번역하여 "어떤 실행 상태였는가"를 보존

그러나 이 두 극단 사이에는 **"현재 작업의 정책적 상태를 어떻게 구조적으로 유지하는가"**라는 계층이 존재하지 않는다. LedgerAgent가 최근 논문에서 명시하는 **사실(facts)·식별자(identifiers)·제약조건(constraints)**으로 구성된 구조적 작업 상태가 바로 이 진공을 메우는 독립 계층이다. 기존 아키텍처는 이를 암묵적으로 컨텍스트(프롬프트)에 위임함으로써, 도메인 정책 준수에 세 가지 근원적 한계를 초래한다.

## 2. 첫 번째 한계: 의미론적 구조의 평탄화와 재해석 비용

[[algorithm-system-translation-gap]]이 Crab의 맥락에서 시사하는 바는 정확히 이 문제의 실행 측 패러다임이다. Crab은 "에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율"을 간극의 본질로 규정한다.

컨텍스트 위임은 이와 **동형(isomorphic)인 문제**를 정책 층에서 재생산한다:

- 정책 제약조건은 본래 **구조적 그래프**(어떤 식별자가 어떤 제약에 묶이고, 어떤 사실이 어떤 제약을 활성화하는가)이지만
- 컨텍스트에 위임되면 **평탄한 텍스트 시퀀스**로 변환되어
- 매 턴마다 모델이 이를 **재해석·재구성**해야 한다

이는 [[agent-execution-semantic-opacity]]과 직결된다. 실행 상태의 의미론적 불투명성이 이미 문제로 식별되어 있으나, 컨텍스트 위임은 이 불투명성을 정책 준수 층까지 확장시킨다. 모델이 특정 턴에서 "고객 계정이 잠금 상태이므로 비밀번호 재설정 도구를 호출할 수 없다"는 제약을 컨텍스트에서 재해석하지 못하면, 이는 단순한 추론 오류가 아니라 **아키텍처적 결함**이다 — 구조적 상태가 존재하지 않아 발생한 것이다.

## 3. 두 번째 한계: 상태 전이의 추적 불가능성과 감사 공백

[[ai-risk-regulation]]의 맥락에서, EU AI Act·NIST RMF·CoE Convention이 요구하는 사전 안전 증명은 "절차적 검사에서 증거 기반 인증으로의 전환"을 요구한다. [[ai-governance]]는 이를 통계적 인증으로 정량화하려는 시도를 포착한다.

그러나 컨텍스트 위임 방식에서는 **상태 전이 자체가 관찰 불가능**하다:

- ADEMA는 지식 상태의 변화를 추적하지만, 이는 세계 지식의 갱신이지 작업 제약의 전이가 아니다
- Crab은 실행 스냅샷을 보존하지만, 스냅샷 내에서 "어떤 정책 제약이 활성/비활성화되었는가"를 구분하지 않는다
- 컨텍스트에 잠긴 작업 상태는 턴 간에 모델의 주의(attention)에 의해 암묵적으로 선택·누락되며, 이 과정에 대한 **감사 가능한 기록(auditable trail)**이 존재하지 않는다

[[auditable-gap]]이 이미 식별된 개념이라면, 컨텍스트 위임은 이 간극을 정책 준수 도메인에서 구체화한다. LedgerAgent의 구조적 상태가 독립 계층으로 존재할 때만 "제약조건 X가 턴 T에서 사실 Y의 도입으로 활성화되었고, 이로 인해 도구 Z의 호출이 차단되었다"는 전이를 기계적으로 검증할 수 있다.

## 4. 세 번째 한계: [[adaptive-forgetting]]과 정책 일관성의 충돌

[[adaptive-forgetting]]이 모델의 적응적 망각을 함수로서 다룬다면, 컨텍스트 위임은 이 망각 메커니즘이 **정책 제약에 직접 작용**하도록 허용한다.

컨텍스트 윈도우 내에서 작업 상태가 유지되는 방식은 본질적으로 **소프트 제약(soft constraint)**이다:

- 긴 대화에서 초기에 확립된 제약조건(예: "이 고객은 프리미엄 등급이므로 무료 재설정이 가능하다")이 컨텍스트 후반부에서 주의 가중치에 밀려 **효과적으로 망각**될 수 있다
- 이는 [[belief-state-as-constraint]]가 암시하는 바와 충돌한다 — 믿음 상태가 제약으로 기능하려면 그 상태가 **안정적으로 유지**되어야 하지만, 컨텍스트 위임은 안정성을 모델의 주의 메커니즘에 의존시킨다
- [[adaptive-forgetting-as-function]]의 관점에서, 망각이 적응적 기능일 수 있으나, 정책 제약의 망각은 **비적응적이고 위험한** 결과를 낳는다

LedgerAgent의 구조적 상태 계층은 이를 하드 제약(hard constraint)으로 전환한다. 상태가 컨텍스트가 아닌 명시적 데이터 구조에 존재하므로, 모델의 망각 메커니즘이 정책 일관성을 침해할 경로가 원천적으로 차단된다.

## 5. 새로운 연결: [[agent-knowledge-separation]]의 재해석과 확장

이 분석에서 **새로운 연결**이 발견된다. [[agent-knowledge-separation]]이 기존에 지식의 분리를 다룬다면, LedgerAgent의 등장은 이 개념을 **세 가지 축**으로 확장해야 함을 시사한다:

1. **세계 지식** (ADEMA가 관리) — "무엇을 아는가"
2. **실행 상태** (Crab이 직렬화) — "어디까지 실행되었는가"  
3. **정책 작업 상태** (LedgerAgent가 구조화) — "현재 작업에서 어떤 제약이 활성인가"

기존 아키텍처는 축 1과 2만 명시적 계층으로 인식했으며, 축 3을 축 1(지식)이나 축 2(실행)에 암묵적으로 흡수시켰다. 그러나 정책 작업 상태는 지식도 실행 아티팩트도 아닌 **고유한 유형의 상태**이다 — 턴 간에 진화하지만 세계 지식처럼 영속적이지 않고, 실행 스냅샷처럼 전체 시스템 상태를 담지도 않는다.

이 세 축의 분리가 명시적으로 아키텍처에 반영되지 않으면, [[agent-os-semantic-gap]]은 단일 간극이 아니라 **다중 간극의 중첩**으로 확장되며, 각 간극은 서로 다른 유형의 상태 손실을 초래한다.

## 결론

컨텍스트 위임 방식의 근원적 한계는 **"정책 준수를 모델의 추론 능력에 종속시킨다"**는 데 있다. ADEMA가 지식을, Crab이 실행을 각각 구조적으로 관리하면서도 정책 작업 상태만 컨텍스트에 맡기는 것은, 마치 데이터베이스에 스키마 없이 모든 것을 단일 텍스트 필드에 저장하고 매번 쿼리로 파싱하도록 하는 것과 같다. LedgerAgent가 제시하는 구조적 상태 계층은 이를 "정책 준수가 아키텍처적 보장이 아니라 추론적 재생산에 달려있는" 상태에서, **"정책 준수가 데이터 구조적 불변량으로 보장되는"** 상태로 전환시키는 필수적인 중간 계층임이 명확하다.

## 핵심 통찰

기존 agent-knowledge-separation이 지식과 실행의 이분법만 다루었다면, 정책 작업 상태(policy-task-state)를 제3의 독립 축으로 인식하지 못한 구조적 상태 진공이 컨텍스트 위임이라는 근원적 아키텍처 결함을 낳는다.
