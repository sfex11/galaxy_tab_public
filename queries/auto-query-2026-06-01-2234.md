# 상류 요구사항 추론이 에이전틱 SWE의 진정한 병목인가?

질문하신 thesis는 Wiki의 여러 논문들이 암시하는 패턴과 강하게 공명합니다. 구체적으로 분해해보겠습니다.

## 1. 하류 병목 재정의의 선례: 우리는 이미 병목을 잘못 짚은 적이 있다

[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 핵심 발견은 이 논의의 출발점이 됩니다. 다중 서버 MCP 배포에서 *"파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 **스키마 축적**에 의해 결정된다"*는 실증은, 커뮤니티가 '당연히 LLM 호출이 병목일 것'이라 가정했다가 틀렸던 사례입니다. 이는 질문자의 주장과 구조적으로 동일한 패턴입니다: **"당연히 실행 효율이 병목일 것"이라는 가정도 재검토 대상**이 됩니다.

## 2. 하류 인프라 최적화의 근본적 한계

[[entities/agentic-harness-engineering.md|agentic harness engineering]]와 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]이 보여주는 것은, 하류 최적화가 결국 **번역 비효율의 문제**에 부딪힌다는 점입니다. Crab 런타임의 분석에 따르면, 에이전트의 의미론적 의존 그래프를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율이 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]의 본질입니다. C/R 런타임이 아무리 정교해져도, 이것은 **상위 의미론을 하위 시스템 메커니즘으로 손실 없이 번역하는 문제**이며, 정보이론적으로 손실이 불가피한 영역입니다.

즉, 하류 인프라 최적화는 **체감 레이턴시를 줄일 수는 있어도, 에이전트가 '잘못된 것을 빠르게' 실행하는 문제는 해결하지 못합니다.**

## 3. 상류 품질 부재가 하류에서 폭발하는 메커니즘: 구성적 비일관성

가장 결정적인 증거는 최근 논문에서 밝혀진 [[concepts/compositional-incoherence.md|compositional incoherence]]입니다:

> *Multi-component LLM agents assemble probabilistic claims from components that each see only part of a joint problem; the composition can violate basic probability axioms even when every component is locally correct.*

이것이 바로 SpecBench가 포착하려는 병목의 정확한 메커니즘입니다. 사양 설계 단계에서 요구사항이 불완전하게 정의되면:
- 각 컴포넌트는 **자신이 본 부분에 대해서는 정상적으로 동작**하지만
- 조합되었을 때 **기본 확률 공리를 위반하는 결과**가 산출됩니다

하류 C/R 최적화가 이 문제를 해결할 수 없는 이유는 명확합니다: 롤백은 '실패한 실행'을 되돌릴 뿐, '잘못 명세된 요구사항'을 교정하지 못합니다.

## 4. 병목의 위계 구조 재구성

Wiki 지식을 기반으로 에이전틱 SWE의 병목을 3층으로 재구성하면:

| 층위 | 병목 유형 | Wiki 증거 | 최적화 가능성 |
|------|-----------|-----------|---------------|
| **상류** | 요구사항 추론 품질 | SpecBench, [[concepts/compositional-incoherence.md|compositional incoherence]] | 근본적·난해 |
| **중류** | 의미론-시스템 번역 | [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]], [[concepts/agent-os-semantic-gap.md|agent os semantic gap]] | 정보손실 불가피 |
| **하류** | 실행 파이프라인 효율 | [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]], [[entities/agentic-harness-engineering.md|agentic harness engineering]] | 공학적 최적화 가능 |

하류 최적화가 가장 '납득하기 쉬운' 엔지니어링 문제이기 때문에 연구가 집중되었지만, [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 교훈을 상위 계층으로 확장하면: **가시적인 병목을 고치는 것과, 실제 시스템의 성능을 결정하는 병목을 고치는 것은 다르다**는 결론에 도달합니다.

## 5. 새로운 개념적 연결

이 분석에서 다음 연결이 드러납니다:

- **[[concepts/benchmark-specification-gap.md|benchmark specification gap]]** ← 이 슬러그가 존재한다는 것 자체가, 벤치마크가 측정하려는 것과 실제 소프트웨어 개발이 요구하는 것 사이에 간극이 있음을 이미 Wiki가 인식하고 있었음을 보여줍니다. SpecBench는 이 간극을 '사양 설계'라는 구체적 축으로 좁힙니다.
- **[[concepts/compositional-incoherence.md|compositional incoherence]]** → SpecBench의 동기: 구성적 비일관성의 근인을 상류 사양 단계에서 추적하려는 시도로 읽을 수 있습니다.

## 결론

Wiki의 논문들이 누적적으로 보여주는 패턴은 질문자의 thesis를 지지합니다. [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]이 실행 파이프라인 내에서조차 병목의 위치를 재정의해야 했듯이, 에이전틱 SWE 전체에서도 **진정한 병목은 상류 요구사항 추론 품질**에 있을 가능성이 높습니다. 하류 인프라 최적화([[entities/agentic-harness-engineering.md|agentic harness engineering]], [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]])는 필수적이지만 보완적 역할에 머물며, [[concepts/compositional-incoherence.md|compositional incoherence]]가 보여주듯 상류에서의 결함은 하류에서 복원 불가능한 형태로 폭발합니다.

SpecBench가 '사양 설계'를 독립 평가 축으로 분리한 것은, 이 병목을 정식화하려는 첫 번째 조치로 해석할 수 있습니다.
