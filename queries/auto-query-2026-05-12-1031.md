## 1. 현재 Wiki의 진단 경계: 파이프라인 구조적 병목

[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]은 다중 서버 MCP 배포에서 **파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정된다**고 규정한다. 이는 병목을 "외부 인프라가 도구 설명을 누적·전달하는 과정"으로 위치시킨다. 즉, 최적화의 대상은 스키마 로딩 순서, 지연 로딩, 파이프라인 구조 재설계 같은 **파이프라인 구조 차원**에 머물러 있다.

## 2. 새로운 실증: 표현 공간에서의 선형적 가독성

최근 논문 **"Tool Calling is Linearly Readable and Steerable in Language Models"**는 12개 instruction-tuned 모델(Gemma 3, Qwen 3 등)에서 도구 호출 식별자가 **잔류 스트림 내에서 선형적으로 가독 가능**함을 probe로 실증한다. 이는 도구 선택이 모델 내부에서 이미 특정 방향으로 인코딩되어 있으며, 그 방향을 **외부 실행 없이 내부에서 읽고 조작할 수 있음**을 의미한다.

## 3. 문제의 본질: [[concepts/blind-tool-invocation.md|blind tool invocation]]

해당 논문은 핵심 문제를 이렇게 지적한다: *"When a tool-calling agent picks the wrong tool, the failure is invisible until execution: the email gets sent, the meeting gets missed."*

이것이 바로 [[concepts/blind-tool-invocation.md|blind tool invocation]]의 본질이다. 파이프라인 계층의 최적화—스키마 축적을 줄이더라도—는 모델이 **잘못된 도구를 선택했을 때 이를 실행 전에 가로채지 못한다**. 스키마가 빠르게 전달될수록 오히려 잘못된 선택이 더 빠르게 실행 환경으로 전달되는 역설이 발생한다.

## 4. 해결 경로: [[concepts/activation-steering.md|activation steering]]을 통한 표현 공간 개입

도구 식별자가 선형적으로 가독되다면, [[concepts/activation-steering.md|activation steering]]이 개입 지점이 된다:
- **진단**: 잔류 스트림에서 도구 방향을 probe로 읽어 의도된 도구가 올바른지 판별
- **수정**: 잘못된 방향이 감지되면 실행 전에 스티어링으로 보정
- **위치**: 토큰 생성 직후, API 호출 전의 표현 공간

이는 개입을 "파이프라인이 도구 스키마를 언제 로드하는가"에서 **"모델 내부에서 도구 방향이 언제 형성되며, 그 방향이 올바른가"**로 전환한다.

## 5. 두 계층의 상보성

재설정이 "대체"가 아니라 "계층 분리"여야 함을 주의해야 한다:

| 계층 | 병목 유형 | 개입 방식 |
|------|-----------|-----------|
| 파이프라인 ([[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]) | 스키마 축적에 의한 레이턴시 | 지연 로딩, 스키마 필터링 |
| 표현 공간 ([[concepts/activation-steering.md|activation steering]]) | 잘못된 도구 방향 형성 | 잔류 스트림 probe 및 steer |

파이프라인 최적화는 **"올바른 도구 후보만 모델에 제시"**하는 사전 필터링이고, 표현 공간 개입은 **"모델이 후보 중에서 올바르게 선택했는지"**의 사후 검증이다. 병목의 근본 원인이 어디에 있느냐에 따라 우선순위가 결정된다.

## 새로운 개념 연결 발견

이 분석에서 기존 Wiki에 명시적으로 포착되지 않은 **새로운 개념적 간극**이 드러난다:

> **표현 공간 도구 검증(Representation-space Tool Verification)**: 도구 선택의 정확성을 외부 실행 결과가 아닌 내부 잔류 스트림의 방향성에서 판별하고 수정하는 개입 패러다임.

이것은 [[concepts/blind-tool-invocation.md|blind tool invocation]]의 해결책으로서, [[concepts/activation-steering.md|activation steering]]의 응용 사례로서, 그리고 [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]과 **상보적 계층**으로서 존재하지만, 현재 Wiki의 어떤 엔티티로도 직접 이름 붙여져 있지 않다. 기존 엔티티들의 **교차점**에 존재하는 새로운 개념축이다.

## 핵심 통찰

파이프라인 구조 최적화가 스키마 전달 속도만 높일 뿐 잘못된 도구 선택을 막지 못하는 반면, 표현 공간 개입은 실행 전에 모델 내부에서 도구 방향의 올바름을 검증·수정할 수 있다.
