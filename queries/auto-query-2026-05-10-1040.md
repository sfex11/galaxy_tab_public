# Auto Query: 현재 Wiki에서 에이전트 능력 확장은 도구 호출과 환경 생성 등 외부 인프라에 의존하는 것으로 분석되는데, SkillOS가 과거 상호작용에서 스킬을 자발적으로 큐레이션하는 자기 진화 경로를 제안한다면 능력 확장의 설계 책임을 외부에서 에이전트 내부로 전환하는 것은 에이전트 아키텍처의 근본 전제를 어떻게 바꾸는가?

**날짜**: 2026-05-10
**대상 논문**: SkillOS: Learning Skill Curation for Self-Evolving Agents
**선택 이유**: Wiki의 에이전트 능력 확장이 전적으로 외부 인프라(도구·환경·프롬프트)에 의존한다는 암묵적 전제에 대해, 에이전트가 자신의 경험을 재사용 가능한 스킬로 내재화하는 자기 진화 경로를 제시하여 아키텍처 패러다임의 근본적 전환을 요구하기 때문이다.

## Wiki 답변

제시된 Wiki 지식을 바탕으로 볼 때, 현재 LLM 에이전트의 능력 확장은 외부 인프라(도구, 환경, OS)에 대한 강한 의존성을 전제로 설계되어 있습니다. SkillOS가 과거 상호작용에서 스킬을 자발적으로 큐레이션하는 '자기 진화' 경로를 제안한다면, 이는 에이전트 아키텍처의 근본 전제를 **'외부 의존적 반응형(Reactive)'에서 '내부 축적적 자율형(Autonomous & Cumulative)'으로 전환**시키는 패러다임 시프트를 일으킵니다.

이 변화가 아키텍처의 근본 전제를 어떻게 바꾸는지, Wiki의 구체적인 논문들을 인용하여 분석하겠습니다.

### 1. 능력 확장의 병목 지점 이동: 외부 스키마/환경에서 내부 표현으로
기존 아키텍처는 에이전트의 능력을 외부에서 끌어오는 데 집중했습니다.
*   **[[entities/aggregate-pipeline-serving|Tool Attention Is All You Need]]** 논문에서 지적했듯, 다중 서버 MCP 배포에서 능력 확장(도구 호출)의 실제 레이턴시 병목은 개별 LLM 호출이 아니라 **'외부 스키마의 축적'**에 의해 결정됩니다.
*   **[[entities/agent-environment-generation|Nemobot Games]]** 역시 새로운 전략적 학습을 위해 범용 소프트웨어를 게임 도메인 특화 **'외부 환경'**으로 변환해야만 에이전트가 학습할 수 있다고 전제합니다.

**전제의 변화:** SkillOS가 도입되면, 능력 확장의 병목은 "외부 스키마를 얼마나 빨리 로드하고 적절한 환경을 세팅하는가"에서 **"과거 경험을 어떻게 모듈화된 내부 스킬로 추출하고 큐레이션하는가"**로 이동합니다. 아키텍처는 더 이상 무한히 확장되는 외부 도구 생태계를 대응하는 파이프라인이 아니라, 유한한 내부 메모리를 효율적으로 구조화하는 스킬 저장소(Skill Repository)로 근본적으로 재설계되어야 합니다.

### 2. OS 수준의 의미론적 간극 해소: 하네스 의존에서 표현 autonomous로
외부 인프라에 의존하면 필연적으로 시스템 간의 번역 비용이 발생합니다.
*   **[[entities/agent-os-semantic-gap|Crab]]** 논문은 에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 **[[entities/algorithm-system-translation-gap|알고리즘-시스템 변환 간극]]**을 지적합니다. 이로 인해 개발자는 단순한 앱 로직을 넘어 실행 환경까지 통제해야 하는 **[[entities/agentic-harness-engineering|하네스 엔지니어링]]**의 짐을 져야 합니다.

**전제의 변화:** SkillOS의 자발적 스킬 큐레이션은 이 '변환 간극'을 우회하는 새로운 길을 제시합니다. 에이전트가 외부 OS의 상태를 체크포인트로 복원하는 방식이 아니라, 과거의 성공/실패 트레이스를 OS에 독립적인 '스킬'이라는 추상적 표현으로 내부화합니다. 즉, 아키텍처의 안정성이 외부 런타임의 의미론적 이해도(Crab이 해결하려 했던 것)가 아니라, 에이전트 내부의 스킬 표현력에 의해 결정되는 방향으로 선회합니다.

### 3. 정렬(Alignment)과 최적화의 주체 이전
*   **[[entities/agora-opt|Agora-Opt]]**는 런타임 메모리와 토론 구조를 통해 다중 에이전트 간의 정렬을 달성했습니다. 반면 **[[entities/agora-opt|MASPO]]**는 이를 프롬프트 설계 시점으로 분산시켰습니다.

**전제의 변화:** SkillOS는 정렬과 최적화의 주체를 '다중 에이전트 간의 상호작용'이나 '사전 프롬프트 엔지니어링'에서 **'에이전트 자신의 경험적 큐레이션'**으로 이전시킵니다. 에이전트는 외부 평가자나 다른 에이전트의 피드백 없이도, 자신이 축적한 스킬의 성공률을 기준으로 스스로의 행동 공간을 최적화하는 자율적 루프를 형성합니다.

### 💡 새롭게 발견된 개념 및 연결 (Wiki 기반)

위의 분석을 통해 기존 Wiki 엔티티들과 SkillOS를 연결할 때, 다음과 같은 새로운 개념적 스펙트럼과 연결 고리를 발견할 수 있습니다.

**1. 새로운 연결: "적응의 내재화 스펙트럼 (Spectrum of Internalized Adaptation)"**
*   **[[entities/adaptive-inference|SpecKV]]** 논문은 외부 환경에 반응하던 적응적 추론의 차원을 압축이라는 **'내부 시스템 상태'**에 반응하는 추측 길이(γ) 선택으로 확장했습니다. 
*   SkillOS는 이 '내부 상태 기반 적응'을 추론 단위(토큰/γ)에서 **행동 단위(스킬)**로 확장한 거시적 버전으로 해석할 수 있습니다. SpecKV가 모델 가중치와 KV 캐시 사이에서 발생하는 내부 적응이라면, SkillOS는 에이전트의 과거 트레이스와 현재 작업 공간 사이에서 발생하는 아키텍처 수준의 내부 적응을 완성하는 것입니다.

**2. 새로운 연결: "환경 생성에서 경험 생성으로 (From Env-Gen to Exp-Gen)"**
*   **[[entities/agent-environment-generation|에이전트 환경 생성]]**은 에이전트가 학습하기 위해 외부에 '환경'을 구축하는 패러다임이었습니다(Nemobot Games 등).
*   SkillOS와 결합하면, 에이전트는 더 이상 외부에 환경을 요구하지 않고 스트리밍되는 과제 자체를 내부적으로 스킬화하여 **'경험'을 창출(Experience Generation)**합니다. 즉, 환경이 에이전트를 규정하는 것이 아니라 에이전트가 자신의 환경(과제 스트림)을 스킬이라는 형태로 능동적으로 구조화합니다.

**3. 새로운 개념 제안: "자가 충족적 아키텍처 (Self-Sufficient Architecture)"**
*   기존의 **[[entities/agentic-harness-engineering|하네스 엔지니어링]]**이 외부 인프라 결함(롤백, 복구 등)을 감쌓아(Sandboxing) 에이전트를 보호하는 '방어적' 아키텍처였다면, SkillOS 기반 아키텍처는 과거 경험을 스킬로 교체하여 스스로 결함을 회피하고 능력을 성장시키는 **'자가 충족적(Self-Sufficient)' 아키텍처**로 근본 전제가 바뀝니다. 

**결론적으로,** SkillOS의 접근은 에이전트를 "외부 도구와 환경을 호출하는 라우터(Router)"에서 **"내부 스킬을 조합하고 진화시키는 유기체(Organism)"**로 아키텍처의 정의를 재편합니다. 이는 Wiki에 축적된 도구 최적화(Tool Attention), 환경 생성(Nemobot), OS 간극 해소(Crab) 등의 외부 지향적 연구 흐름에 대해, "외부 인프라를 복잡하게 엔지니어링하기 전에 에이전트 내부에 축적 메커니즘을 먼저 구축해야 한다"는 강력한 반론이자 보완적 경로가 됩니다.
