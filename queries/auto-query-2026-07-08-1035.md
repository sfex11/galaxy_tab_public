# 개별 에이전트 행동 제어 패러다임의 한계: PiSAs가 노출하는 구조적 누출 문제

## 결론부터 말하면: 불가능하다.

PiSAs가 지적하는 문맥적 무결성(Contextual Integrity) 위반은 **외부 공격자의 침투 경로**가 아니라 **다중 에이전트 아키텍처의 정상 작동 모드 자체**에서 발생한다. 기존의 인라인 하네스나 런타임 거버넌스는 이를 탐지·방어할 수 없는데, 그 이유를 Wiki의 논문들을 통해 세 가지 차원에서 구체화할 수 있다.

---

## 1. 병목 재정의: 개별 에이전트가 아닌 파이프라인 구조가 문제다

[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 핵심 발견은 이 문제의 본질을 직격한다:

> 다중 서버 MCP 배포에서 파이프라인 전체의 레이턴시 병목이 **개별 LLM 호출이 아닌 스키마 축적**에 의해 결정됨

이는 단순히 성능 문제가 아니다. 스키마가 파이프라인을 따라 축적된다는 것은, **에이전트 A가 호출한 도구의 스키마(구조, 파라미터, 반환값 형식)가 에이전트 B의 컨텍스트에 정상적으로 유입된다**는 뜻이다. PiSAs의 "공유 메모리와 에이전트 간 메시지를 통한 구조적 누출"이 정확히 이 경로를 가리키는 것인데, 개별 에이전트의 입력/출력을 모니터링하는 인라인 하네스는 이 스키마 축적 자체를 **정상 동작**으로 분류한다. 방어해야 할 대상이 방어 체계의 정의상 '정상'에 포함되어 있다.

[[concepts/bottleneck-misattribution.md|bottleneck misattribution]]이라는 개념이 이를 뒷받침한다—우리가 문제의 원인을 잘못 지정하면, 그에 기반한 방어 체계는 근본적으로 엉뚱한 곳을 감시하게 된다.

## 2. 의미론적 간극: OS 수준의 상태 관리는 에이전트 간 정보 흐름을 볼 수 없다

[[concepts/agent-os-semantic-gap.md|agent os semantic gap]]과 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]이 제시하는 간극은 하네스 기반 방어의 두 번째 한계다:

> Crab이 이 간극을 구체적 런타임 문제로 현현한다: 에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율이 간극의 본질이다.

이를 PiSAs 문맥으로 번역하면: OS 수준의 [[concepts/agent-sandbox-state-management.md|agent sandbox state management]]는 **상태의 존재 여부**는 알지만 **그 상태가 어떤 사용자의 문맥적 무결성 영역에 속하는지**를 모른다. Crab의 C/R 런타임이 아무리 정교하게 체크포인트/복원을 수행해도, 스냅샷 내에 "사용자 A의 정보가 사용자 B의 에이전트 메모리에 정상적으로 복사되어 있다"는 것을 **의미론적 위반**으로 인식할 수 없다.

[[concepts/agent-execution-semantic-opacity.md|agent execution semantic opacity]]이 정확히 이 상태를 지칭한다—에이전트 실행의 의미론적 내용(어떤 정보가 누구의 문맥에서 와서 누구에게로 흐르는가)이 실행 인프라 수준에서 불투명하다.

## 3. 하네스의 확장 한계: 애플리케이션→실행 환경까지 갔으나, 에이전트 간 경계까지는 못 갔다

[[entities/agentic-harness-engineering.md|agentic harness engineering]]의 진화 경로를 보면 문제가 선명해진다:

> C/R 런타임이 하네스의 핵심 구성요소(결함 내성·롤백·분기)를 직접 구현하는 인프라 계층으로, 하네스 엔지니어링의 범위를 **애플리케이션 로직에서 실행 환경까지** 확장함을 시사한다.

Crab은 하네스의 범위를 에이전트 **내부**의 실행 환경까지 확장하는 데 성공했다. 그러나 PiSAs의 문제는 에이전트 **간** 경계에서 발생한다. 이는 [[concepts/agent-boundary-semantic-collapse.md|agent boundary semantic collapse]]라는 개념으로 포착되어야 할 영역인데, 기존 하네스 패러다임은 에이전트를 **경계가 명확한 단위**로 전제하고 그 내부를 제어한다. 에이전트 간 경계가 의미론적으로 붕괴되는 현상 자체를 감지하려면, 제어 대상이 "개별 에이전트의 행동"이 아니라 "에이전트 간 정보 흐름의 문맥적 적합성"이어야 한다.

## 대안적 패러다임: 무엇이 필요한가?

Wiki의 슬러그 목록에서 다음 개념들이 PiSAs 문제에 대한 대안적 접근을 시사한다:

| 기존 패러다임 | 필요한 전환 | 관련 Wiki 개념 |

| 인라인 하네스 (per-agent) | 파이프라인 수준의 구조적 누출 탐지 | [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]], [[concepts/adaptive-bottleneck-migration.md|adaptive bottleneck migration]] |
| OS 수준 샌드박스 격리 | 의미론적 경계 보존 | [[concepts/agent-os-semantic-preservation-failure.md|agent os semantic preservation failure]], [[concepts/agent-native-immune-system.md|agent native immune system]] |
| 런타임 거버넌스 (행동 제어) | 설계 시점의 아키텍처 수준 분리 | [[entities/agora-opt.md|agora opt]] (런타임 정렬의 한계 → 설계 시점 분산) |

특히 [[entities/agora-opt.md|agora opt]]의 관련 논문에서 MASPO가 제시한 통찰은 여기서도 적용된다:

> Agora-Opt가 런타임 메모리·토론 구조를 통해 다중 에이전트 정렬을 달성했다면, MASPO는 **프롬프트 설계 시점에서 사전 정렬을 보장**하는 상보적 경로를 제공하여, 런타임 정렬 메커니즘의 부담을 설계 시점으로 분산시킨다.

PiSAs의 구조적 누출도 마찬가지다—런타임에 탐지하려는 시도 자체가 [[concepts/agent-execution-semantic-opacity.md|agent execution semantic opacity]]에 부딪힌다. 해결은 **아키텍처 설계 시점**에서 에이전트 간 [[concepts/agent-knowledge-separation.md|agent knowledge separation]]을 강제하고, [[concepts/agent-native-serving.md|agent native serving]] 계층에서 문맥적 무결성을 일급(first-class) 제약으로 처리하는 방향으로 이동해야 한다.

## 새로운 개념적 연결 발견

이 분석을 통해 다음 연결이 명시적으로 필요함을 확인한다:

1. **[[concepts/agent-boundary-semantic-collapse.md|agent boundary semantic collapse]] ↔ [[concepts/agent-os-semantic-preservation-failure.md|agent os semantic preservation failure]]**: 에이전트 간 경계의 의미론적 붕괴가 OS 수준의 상태 보존 메커니즘(스냅샷, 체크포인트)에 의해 오히려 **강화·지속**된다는 역설적 관계. Crab이 상태를 '잘' 보존할수록, 사용자 간 정보 혼 contamination도 '잘' 보존된다.

2. **[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]] ↔ [[concepts/agent-knowledge-separation.md|agent knowledge separation]]**: 파이프라인의 스키마 축적 메커니즘이 지식 분리 원칙과 구조적으로 충돌한다—스키마 공유가 파이프라인 효율을 위해 설계된 정상 경로이기 때문에, 분리를 강제하려면 파이프라인 아키텍처 자체를 재설계해야 한다.

3. **[[concepts/agent-native-immune-system.md|agent native immune system]]**의 실제 필요성 구체화: 바이러스(외부 공격)가 아니라 자가면역질환(정상 작동 속 자기 조직의 경계 붕괴)에 대응해야 하므로, 방화벽이 아닌 **면역 관용(immune tolerance)** 메커니즘—즉, "이 정보는 이 문맥에서 허용된 것인가"를 판별하는 능력—이 필요하다.

**요약**: PiSAs가 노출하는 문맥적 무결성 위반은 개별 에이전트의 행동을 제어하는 것으로는 잡을 수 없다. 그것은 에이전트 **간**의 구조적 정보 흐름에서 발생하며, [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]이 보여주듯 파이프라인의 정상 동작 모드이기 때문이다. 방어 패러다임은 "에이전트가 무엇을 하는가"에서 **"에이전트 간에 무엇이 흐르는가, 그것이 문맥적으로 정당한가"**로 전환되어야 한다.

## 핵심 통찰

PiSAs가 노출하는 한계는 개별 에이전트 제어의 실패가 아니라, 파이프라인 정상 작동 모드 자체가 방어 대상인 구조적 역설이다.
