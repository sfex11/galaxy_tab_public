## Wiki 답변

제시해주신 관점에 전적으로 동의합니다. 확률적-결정론적 경계(SDB)를 4역할 계약으로 명명하는 접근은, 현재 Wiki에 파편화되어 기록된 런타임 최적화 시도들이 사실 **단일 경계 계약의 서로 다른 차원(상태, 제어, 데이터)에서 발생하는 부분적 증상들에 불과함**을 구조적으로 증명합니다.

Wiki에 축적된 논문들의 구체적인 인용을 통해, 기존의 분산된 런타임 접근들이 어떻게 단일 SDB 계약의 조합 패턴으로 환원되는지 분석할 수 있습니다.

### 1. 상태 계약의 누출: [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]과 [[concepts/agent-os-semantic-gap.md|agent os semantic gap]]
Crab 논문이 다루는 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]과 [[concepts/agent-os-semantic-gap.md|agent os semantic gap]]은 SDB의 **'상태 유지(State)' 역할**이 명시되지 않아 발생한 전형적인 충돌입니다. 에이전트의 의미론적 의존 그래프(확률적 상태)를 OS의 체크포인트 메커니즘(결정론적 전체 스냅샷)으로 강제 번역할 때 발생하는 비효율은, 상태 경계를 교차할 때의 직렬화/역직렬화 계약이 없었기 때문에 나타난 현상입니다.

### 2. 제어 계약의 하부 이동: [[entities/agentic-harness-engineering.md|agentic harness engineering]]
[[entities/agentic-harness-engineering.md|agentic harness engineering]]이 애플리케이션 로직에서 실행 환경(C/R 런타임)까지 범위를 확장하여 결함 내성·롤백·분기를 구현한 현상은, SDB의 **'실행 제어(Control)' 역할**을 상위 계층에서 하위 인프라 계층으로 강제로 끌어내린 시도로 해석됩니다. SDB가 일급 아키텍처 객체로 추상화되지 않았기 때문에, 하네스가 제어 흐름을 감당하기 위해 자신의 경계를 인프라까지 팽창시킨 파편화 패턴입니다.

### 3. 데이터 계약의 병목 외부화: [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]
Tool Attention 논문이 밝혀낸 [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 병목 재정의("개별 LLM 호출이 아닌 스키마 축적이 병목")는 SDB의 **'데이터 교환(Data)' 역할**의 부재를 적나라하게 보여줍니다. 결정론적 도구의 스키마가 확률적 모델로 파이프라인을 통해 전달될 때, 이 경계에서의 데이터 포맷팅과 라우팅에 대한 계약이 없었기에 병목이 파이프라인 전체로 누출된 것입니다.

### 💡 새로운 개념 및 연결 발견: **런타임 파편화의 SDB 투영 (SDB-Projection of Runtime Fragmentation)**

이 분석을 통해 위 지식 베이스에서 새롭게 도출되는 연결은, 기존의 파편화된 엔티티들이 사실 **단일 경계의 4역할 중 일부만을 과보상(Over-compensation)하려는 대칭적 실수**를 공유하고 있다는 점입니다. 

즉, [[concepts/agent-os-semantic-gap.md|agent os semantic gap]]은 '상태 계약'의 부재를 OS 레벨의 스냅샷으로 덮으려 했고, [[entities/agentic-harness-engineering.md|agentic harness engineering]]은 '제어 계약'의 부재를 런타임 팽창으로 덮으려 했으며, [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]은 '데이터 계약'의 부재를 파이프라인 레이턴시 최적화로 덮으려 했습니다. 

결론적으로 본 논문이 SDB를 4역할 계약의 일급 객체로 명명한다면, Wiki에 산재한 이 세 가지 런타임 최적화 접근법들은 각기 독립된 문제가 아니라 **"SDB라는 단일 경계 계약을 구성하는 4개의 슬롯 중 특정 슬롯에만 집중하여 발생한 불완전한 조합 패턴"**으로 완벽하게 통합 및 환원됩니다. 이는 에이전트 네이티브 아키텍처 연구에서 하드웨어/OS/파이프라인 단위의 쪼개진 최적화를 멈추고, SDB 계약 설계로 패러다임을 전환해야 한다는 강력한 근거가 됩니다.
