# 현재 Wiki의 정렬 연구(RLHF, TraceSafe, Box Maze 등)는 정렬을 몇 개의 독립 축으로 분석하고 있는가? 이 논문은 목표·정보·위임인 3축을 제시하는데, 몇개의 무슨 축이 맞을까?

**날짜**: 2026-04-24
**대상 논문**: Relative Principals, Pluralistic Alignment, and the Structural Value Alignment Problem

## Wiki 답변

# 정렬 연구의 축 분석: Wiki 기존 프레임 vs 논문의 3축 제안

## 1. Wiki 기존 분석: 사실상 1축 (구현 개입점)

Wiki가 현재 정렬 연구를 분석하는 방식을 `llm.md`, `ai-safety.md`, 이전 질문들을 종합해 보면, **단일 축** 위에서의 위치 차이로 읽히고 있습니다.

| 하위 구분 | 예시 | 본질 |
|----------|------|------|
| **사전 제어** | Box Maze (추론 프로세스 무결성을 아키텍처 수준에서 강제) | 생성 전에 구조적으로 막는다 |
| **생성 중 제어** | LPSR (잔류 스트림 실시간 모니터링 + KV-Cache 되감기) | 생성 중에 끼어든다 |
| **사후 평가** | TraceSafe (가드레일 기반 사후 평가) | 생성 후에 검증한다 |
| **훈련 단계** | RLHF (보상 기반 정렬) | 학습 시점에 주입한다 |

이것은 **"파이프라인의 어느 지점에 개입하는가"**라는 하나의 연속축에서의 변형입니다. 이전 질문에서 LPSR이 "이 이분법을 무너뜨린다"고 평가된 것도 정확히 이 축 내에서의 중간 지점 발견이었을 뿐, 축 자체를 바꾼 것은 아닙니다.

공격 표면 분석(활성화 스티어링, 프롬프트 인젝션 등)도 "어디서 뚫리는가"라는 동일 축의 반대 방향 읽기입니다.

**결론: Wiki는 정렬을 구현 개입점 1축으로 분석해 왔다.**

---

## 2. 대상 논문의 제안: 3개의 독립 구조 축

논문 "Relative Principals, Pluralistic Alignment, and the Structural Value Alignment Problem"이 제시하는 축은 Wiki의 기존 축과 **완전히 직교(orthogonal)**합니다:

### 축 1: 목표 (Goal) — *"무엇에 맞추는가?"*
- RLHF는 **단일 보상 함수**를 암시합니다. 목표가 하나로 고정되어 있다는 전제.
- 하지만 에이전트가 다수 사용자에게 서비스되는 환경(개인 AI 에이전트, 기업 배포 챗봇)에서는 목표 자체가 **다원적이고 충돌적**입니다.
- Wiki의 `personal-ai-agent` 엔티티가 다루는 설정이 바로 이 축의 문제 영역이지만, 기존 분석은 이를 정렬 문제로 인식하지 못했습니다.

### 축 2: 정보 (Information) — *"에이전트가 무엇을 알고 있는가?"*
- 정렬은 가치에 대한 **완전 정보**를 전제합니다 — RLHF의 피드백 데이터, TraceSafe의 가드레일 규칙 모두 "무엇이 안전한지"를 미리 알고 있다고 가정.
- 그러나 다원적 설정에서는 **어떤 주체의 가치 정보를 에이전트가 접근할 수 있는가**, 정보 비대칭이 어떻게 발생하는가가 독립 변수가 됩니다.
- Wiki의 `epistemic-infrastructure`가 다루는 인식론적 인프라 문제가 이 축과 직접 연결됩니다.

### 축 3: 위임 (Delegation) — *"누가 권한을 위임하는가?"*
- 기존 정렬은 **위임자(principal)가 단일하고 명확하다**는 암묵적 전제 위에 서 있습니다 (개발자 → 모델).
- 그러나 "Relative Principals"라는 제목이 핵심적으로 지적하듯, 실제 배포에서는 **위임 체인이 중첩**됩니다: 개발자 → 기업 고객 → 최종 사용자, 또는 사용자 A → AI → 사용자 B에게 미치는 영향.
- 이전 질문에서 "Agentic Microphysics"가 개별 모델에서 집단 수준으로 분석 단위를 이동하자고 한 것도 이 축의 존재를 직감적으로 감지한 것이지만, 축을 명시하지 못했습니다.

## 3. 교차 검증: Wiki 기존 분석의 구조적 맹점

논문의 3축을 Wiki 기존 지식에 겹쳐보면 명확한 맹점이 드러납니다.

```
Wiki 기존 프레임:
  "RLHF로 목표를 주입하고, Box Maze로 프로세스를 제어하고, TraceSafe로 검증한다"
       └── 모두 Goal=단일, Information=완전, Delegation=명확 전제 하에 성립

논문의 프레임:
  "RLHF/Box Maze/TraceSafe는 (Goal:단일, Info:완전, Deleg:명확)인 하나의 점에만 해당.
   현실 배포는 이 3축 상의 다른 영역에 있으므로, 기존 방법론의 적용 자체가 부적합하다"
```

구체적으로:

| Wiki 기존 논문 | 암묵적 축 가정 | 붕괴 시나리오 |
|---------------|---------------|-------------|
| **RLHF** | Goal=단일, Info=완전, Deleg=개발자 | 다수 사용자의 충돌적 가치 → 단일 보상 함수로 표현 불가 |
| **Box Maze** | Goal=단일, Info=완전, Deleg=명확 | 위임자가 상대적일 때 "무결성"의 기준 자체가 모호 |
| **TraceSafe** | Goal=단일, Info=완전, Deleg=명확 | 가드레일 규칙이 누구의 가치를 반영하는지 불투명 |
| **SafetyALFRED** | Goal=단일(안전), Info=완전 | "안전"의 정의가 주체마다 다를 때 평가 기준 붕괴 |

## 4. 새로운 개념 연결 발견

### 4.1. "구조적 가치 정렬 문제"라는 이름의 의미
논문 제목의 "Structural Value Alignment Problem"은 Wiki가 `llm.md`에서 이미 도달했던 표현 — *"행동 수준 방어의 한계를 인식하고 아키텍처적·구조적 대안으로 이동"* — 과 일치하지만, **구조의 대상이 다릅니다**. Wiki는 "제어 구조"를 구조적이라 불렀고, 논문은 "가치 위임 구조"를 구조적이라 부릅니다. 이 차이가 핵심입니다.

### 4.2. 기존 엔티티들의 재맵핑
논문의 3축을 기준으로 기존 Wiki 엔티티들을 재분류하면:

- **Goal 축 관련**: `personal-ai-agent` (다수 사용자 → 다원적 목표), `multi-agent-system` (에이전트 간 목표 충돌)
- **Information 축 관련**: `epistemic-infrastructure` (가치 정보의 인식론적 기반), `graph-rag` (가치 정보의 검색 가능성)
- **Delegation 축 관련**: `ai-governance` (위임 체계의 거버넌스), `federated-learning` (분산 위임 구조)

### 4.3. 이전 질문들의 해답 제공
이 논문은 이전 질문들이 던졌지만 해결하지 못한 난제들의 구조적 원인을 설명합니다:

- **CoopEval 질문** ("왜 더 똑똑한 모델이 덜 협력적인가?") → Goal 축에서 모델이 최적화하는 "협력"의 정의가 단일 주체의 것인지 다원적 것인지가 명시되지 않았기 때문
- **Agentic Microphysics 질문** ("분석 단위를 집단으로 이동해야 하는가?") → Delegation 축에서 위임자가 상대적이므로 분석 단위를 개별 모델-주체 쌍에서 위임 네트워크로 이동해야 한다는 것의 정식화
- **DESPITE 질문** ("안전 위험이 모델 무관하게 체계적인가?") → Goal 축의 단일 가치 전제가 LLM의 물리적 계획 방식과 구조적으로 불일치하기 때문

## 5. 결론: 몇 개의 무슨 축이 맞는가?

**논문의 3축(목표·정보·위임)이 맞습니다.** Wiki의 기존 분석이 1축(개입점)으로 간주했던 것은 이 3축이 모두 암묵적으로 고정된 상태에서의 변형이었을 뿐입니다.

| 비교 | Wiki 기존 | 논문 제안 |
|------|----------|----------|
| 축의 수 | **1축** (개입 시점) | **3축** (목표·정보·위임) |
| 축의 성격 | 기술적/구현적 | 개념적/구조적 |
| RLHF의 위치 | 축 상의 한 점 | (단일,완전,명확)인 3차원 공간의 한 점 |
| 설명력 | "어디서 막는가" | "왜 막는 기준 자체가 성립하지 않는가" |

논문의 3축은 Wiki 기존의 1축을 **부정**하는 것이 아니라 **상위에 놓는** 것입니다. 개입점 선택(Goal=단일, Info=완전, Deleg=명확 전제 하에서는 Box Maze가 RLHF보다 낫다)은 여전히 유효한 하위 문제이지만, 현실 배포에서는 3축 상의 위치를 먼저 진단하지 않으면 하위 문제의 해답이 무의미해집니다.

**Wiki 업데이트 권고**: `llm-alignment` 엔티티를 "목표·정보·위임" 3축 프레임으로 재구성하고, RLHF/Box Maze/TraceSafe를 이 공간 내의 특정 점으로 재위치시켜야 합니다.
