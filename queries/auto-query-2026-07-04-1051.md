## 1. 암묵적 전제의 해체: 두 엔티티가 숨기고 있는 모순

Wiki의 두 핵심 엔티티는 실제로 각자의 논문 내에서 **자기 모순적인 증거**를 이미 포함하고 있습니다.

### [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 자가 부정

이 엔티티의 핵심 논문 "Tool Attention Is All You Need"는 다중 서버 MCP 배포에서 **파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정된다**고 밝힙니다. 이것은 곧 "더 많은 도구를 연결할수록 스키마가 축적되고, 이것이 오히려 파이프라인의 실제 병목을 악화시킨다"는 뜻입니다. 도구를 늘리는 것이 성능 향상이 아니라 **자기 저해(self-undermining)**로 작동할 수 있다는 증거가 이미 이 엔티티 안에 내재되어 있었던 것입니다.

### [[entities/agentic-harness-engineering.md|agentic harness engineering]]의 범위 확장의 역설

Crab 논문은 C/R 런타임이 결함 내성·롤백·분기를 직접 구현하여 하네스 엔지니어링의 범위를 "애플리케이션 로직에서 실행 환경까지" 확장한다고 평가합니다. 그러나 여기서 핵심 질문이 발생합니다: **롤백과 재시도 인프라를 정교화하는 것이 왜 필요한가?** 그것은 첫 시도가 실패하기 때문입니다. 만약 90회 실행에서 추론 노력이 일차 시도 신뢰성의 결정적 요인이라면, 롤백/재시도 인프라에 대한 투자는 **근본 원인(추론 부족)이 아닌 증상(실패 후 복구)을 치료**하는 것에 해당합니다.

## 2. 새로운 연결: [[concepts/adaptive-inference.md|adaptive inference]]가 제시하는 대안 경로

여기서 [[concepts/adaptive-inference.md|adaptive inference]]가 결정적 매개체로 등장합니다. SpecKV의 기여를 다시 읽어보면:

> "적응적 추론의 결정 차원을 모델/라우트 선택에서 **추측 길이(γ) 선택**으로 확장한다. 기존 컨텍스트 기반 적응이 외부 환경에 반응했다면, SpecKV는 **압축이라는 내부 시스템 상태**에 반응하는 적응의 새로운 유형을 제시한다."

이 패턴을 추론 예산 배분 문제에 적용하면 완벽히 대응됩니다:

| 기존 패러다임 | 대안 패러다임 |
|---|---|
| 외부 도구 선택(모델/라우트) | 내부 추론 깊이 조절(γ) |
| 스키마 축적에 따른 병목 | 압축 상태에 따른 적응 |
| 더 많은 도구 → 더 나은 결과 | 더 많은 추론 → 더 나른 첫 시도 |

[[concepts/adaptive-inference.md|adaptive inference]]는 이미 "외부 환경 반응 → 내부 상태 반응"이라는 전환을 보여준 선례입니다. 질문에서 제시된 연구 결과는 이 전환을 추론 도메인까지 확장해야 함을 주장하는 것입니다.

## 3. 근본적 방향 전환의 논리 구조

### 발견된 새로운 연결: [[concepts/bottleneck-misattribution.md|bottleneck misattribution]]

Wiki 슬러그 목록에 존재하는 이 엔티티가 여기서 정확히 빛을 발합니다. 우리가 최적화하려는 병목이 실제 병목이 아닐 때 발생하는 오류입니다:

- **[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 귀인**: 병목 = LLM 호출 성능 → 실제 병목 = 스키마 축적
- **[[entities/agentic-harness-engineering.md|agentic harness engineering]]의 귀인**: 병목 = 인프라 결함 내성 부족 → 실제 병목 = 추론 노력 부족
- **관찰 연구의 귀인 정정**: 병목 = 도구 접근성 부족 → 실제 병목 = 추론 예산 부족

세 경우 모두 [[concepts/bottleneck-misattribution.md|bottleneck misattribution]]의 동일한 구조를 보입니다: **해결 가능해 보이는 인프라 문제로 귀인함으로써, 해결하기 어려운 인지적 한계 문제를 회피하는 패턴**입니다.

### 발견된 새로운 연결: [[concepts/capability-cooperation-paradox.md|capability cooperation paradox]]

이 엔티티 역시 슬러그 목록에 존재합니다. 더 많은 도구(역량)가 추가될수록 에이전트의 일차 시도 신뢰성이 오히려 하락하는 현상은 정확히 이 역설의 현현입니다:

- 도구가 늘어남 → 스키마 축적([[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]) → 인지적 부하 증가 → 추론 예산이 도구 선택에 소모됨 → 핵심 문제 해결을 위한 추론 예산 잔여 감소 → 일차 시도 신뢰성 하락

## 4. 그러나 단순 전환은 위험하다: [[concepts/adaptive-forgetting.md|adaptive forgetting]]의 경고

전환이 필요하다는 주장에도 불구하고, Wiki는 반대 방향의 위험도 시사합니다. 추론 예산을 극대화하는 것이 항상 정답이 아닐 수 있는데, 이는 [[concepts/adaptive-forgetting.md|adaptive forgetting]]의 존재가 암시합니다:

긴 추론 사슬은 이전 단계의 중간 결론을 망각할 위험을 내포하며, 이것이 도구 호출의 외부 기억 보완 기능과 trade-off를 형성할 수 있습니다. 즉, **"추론 예산만 늘리면 된다"는 것도 또 다른 극단적 단순화**입니다.

## 5. 결론: 균형점의 재정의, not 단순 전환

질문에 대한 답은 **"예, 하지만 단순한 대체가 아닌 계층적 재구성이어야 한다"**입니다.

구체적으로 제안되는 아키텍처 최적화의 새로운 방향:

1. **1차 최적화 축**: 추론 예산 배분 — [[concepts/adaptive-inference.md|adaptive inference]]의 SpecKV가 내부 상태 기반 적응을 보여주듯, 문제 복잡도에 따른 추론 깊이(γ)를 동적으로 조절하는 메커니즘을 도구 선택보다 우선

2. **2차 최적화 축**: 도구 인프라의 **감축** — [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]이 스키마 축적의 역효과를 보여주듯, 도구 수를 줄이고 각 도구의 스키마를 경량화하여 추론 예산의 도구 선택 오버헤드를 최소화

3. **3차 최적화 축**: [[entities/agentic-harness-engineering.md|agentic harness engineering]]의 재정의 — 롤백/재시도를 "실패 복구"가 아닌 "추론 예산의 점진적 투입" 메커니즘으로 재해석. 첫 시도에 적은 예산을 투입하고, 실패 시 복구가 아닌 **추론 깊이를 늘려 재시도**하는 구조

### 최종적으로 발견된 새로운 연결 맵

```
[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]] (스키마 축적 역효과)
        ↓ 귀인 오류
[[concepts/bottleneck-misattribution.md|bottleneck misattribution]] (인프라 병목으로의 오귀인)
        ↓ 실제 병목
추론 예산 부족 (관찰 연구 발견)
        ↓ 기술적 구현 경로
[[concepts/adaptive-inference.md|adaptive inference]] (내부 상태 기반 γ 조절)
        ↓ 부작용 경계
[[concepts/adaptive-forgetting.md|adaptive forgetting]] (장기 추론의 망각 위험)
        ↓ 보완 메커니즘 재정의
[[entities/agentic-harness-engineering.md|agentic harness engineering]] (롤백 → 추론 예산 점진 투입)
        ↓ 거시적 역설
[[concepts/capability-cooperation-paradox.md|capability cooperation paradox]] (더 많은 도구 = 더 낮은 신뢰성)
```

이 연결 맵이 보여주듯, Wiki의 기존 엔티티들은 각자 파편화된 증거를 담고 있었지만, **"추론 노력이 도구 접근성보다 일차 시도 신뢰성의 결정적 요인"**이라는 발견이 이들을 하나의 일관된 서사로 연결합니다. 아키텍처 최적화의 방향은 "더 많은 도구"에서 "더 깊은 추론, 더 적은 도구"로의 전환이 맞지만, 이를 [[concepts/adaptive-inference.md|adaptive inference]]의 적응적 메커니즘과 [[concepts/adaptive-forgetting.md|adaptive forgetting]]의 보완 메커니즘 없이 단순히 적용하면 새로운 [[concepts/bottleneck-misattribution.md|bottleneck misattribution]]을 낳게 될 것입니다.

## 핵심 통찰

도구 인프라 확충에서 추론 예산 배분으로의 전환은 단순한 대체가 아니라, 인지세 최소화를 위한 도구 감축과 재시도의 점진적 예산 투입화를 포함하는 계층적 재구성이어야 한다.
