Wiki의 **LLM Agent 합성 분석**(97편 기반)은 코드 생성(TDAD, Code Review Agent Benchmark), GUI 자동화(UI-Voyager, MAESTRO, ClawBench) 등 다양한 벤치마크를 통해 에이전트 능력을 비교한다. 이 벤치마크들이 LLM 판단에 기반한 평가 메트릭을 사용한다면, **에이전트 간 성능 순위의 인스턴스별 비일관성**이 숨겨져 있을 가능성이 높다.

### 3. 새로운 개념적 연결

**[새로운 연결 1] LLM-as-Judge 신뢰성 ↔ Multi-Agent System 평가 귀인 문제**

Wiki의 **Multi-Agent System** 엔티티는 "다중 AI 에이전트 시스템에서 지연된 피드백은 인간 감독자의 오류 귀인을 체계적으로 왜곡"한다는 인사이트를 기록한다. LLM-as-judge의 추이성 위반과 결합하면, **다중 에이전트 시스템의 평가는 이중으로 왜곡**될 수 있다: (1) LLM judge 자체의 비일관성, (2) 지연 피드백에 의한 귀인 오류. 이는 "CoopEval: Benchmarking Cooperation-Sustaining Mechanisms and LLM Agents in Social Dilemmas"의 벤치마크 설계에도 직접적 함의를 가진다.

**[새로운 연결 2] Conformal Prediction ↔ OpenClaw 평가 체계화**

Wiki의 **OpenClaw 합성 분석**은 "평가 체계를 어떻게 구축할 것인가"를 핵심 질문으로 식별한다. Diagnosing LLM Judge Reliability 논문이 제안하는 **conformal prediction sets**(개별 판단에 불확실성 구간을 부여)는 OpenClaw 보안 평가의 신뢰구간을 명시적으로 제공하는 방법론으로 직접 적용 가능하다.

**[새로운 연결 3] 추이성 위반 ↔ Reinforcement Learning 보상 신호 문제**

Wiki의 **Reinforcement Learning 합성 분석**이 강조하는 "현실 세계에서 작동하는 RL"의 핵심 과제와 연결된다. RL 기반 에이전트 학습(예: Nemotron-Cascade 2의 Cascade RL)에서 LLM-as-judge가 보상 함수로 사용될 경우, 추이성 위반은 **보상 신호 자체의 비일관성**을 의미하며, 이는 학습된 정책의 불안정성으로 직결된다.

### 4. 재평가 프레임워크 제안

Wiki의 벤치마크 기반 분석을 재평가하려면:

1. **인스턴스 수준 일관성 감사**: 집계 점수만 기록된 비교에 대해, 추이성 위반률(transitivity violation rate)을 메타데이터로 추가해야 한다.
2. **Conformal prediction 기반 불확실성 명시**: 모델 간 성능 차이가 conformal prediction set 내에서 겹치는 경우, Wiki에서 "유의미한 차이 없음"으로 재표기해야 한다.
3. **평가 방법론 태깅**: Wiki의 각 성능 비교에 평가 방법(인간 평가 / LLM-judge / 자동 메트릭)을 명시하여, LLM-judge 의존도가 높은 분석을 식별할 수 있도록 해야 한다.
4. **도메인별 차등 신뢰도**: VLM 공간 추론이나 에이전트 GUI 자동화처럼 주관적 판단이 개입하는 영역의 벤치마크는 신뢰도를 하향 조정하고, 코드 실행 성공률 같은 객관적 메트릭 기반 평가는 상대적으로 신뢰도를 유지한다.
