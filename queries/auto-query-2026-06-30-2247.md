# 에이전트 능력과 안전성의 근본적 불가분성: Agent-Native Immune System을 중심으로

## 1. 문제의 근원: 위협 표면과 방어 표면의 동일성

Agent-Native Immune System 논문(최근 논문 목록)은 핵심 전제를 명확히 밝힌다:

> "정적 챗봇에서 지속적 메모리·도구 사용 프로토콜·다중 에이전트 협력을 갖춘 자율 에이전트로의 전환은 AI 위협 표면을 근본적으로 확장했다."

그러나 동시에 이 논문은 **동일한 능력들을 방어 메커니즘의 기반으로 내장**한다고 선언한다. 이는 단순한 설계 선택이 아니라, 구조적 필연성이다. 왜냐하면 Wiki의 여러 엔티티들이 보여주듯, 에이전트의 능력을 안전성과 분리하려는 시도 자체가 지속적으로 실패하기 때문이다.

## 2. 분리 불가능성에 대한 Wiki의 실증적 증거

### 2-1. 평가 층위에서의 분리 실패

[[concepts/ai-safety.md|ai safety]]와 관련된 SafetyALFRED 논문은 이 분리의 근본적 한계를 가장 명확히 보여준다. **11개 SOTA 모델**에서 disembodied QA 기반 안전 평가가 체화 배포 환경에서의 안전성을 보장하지 못함을 실증했다. 즉, 안전성을 '추가된 평가 계층'으로 분리해 검증하려는 접근이 구조적으로 무효화된다.

이는 [[entities/ai-risk-regulation.md|ai risk regulation]]이 다루는 문제와 직결된다. Bounding the Black Box 논문이 EU AI Act·NIST RMF·CoE Convention의 규제 요구를 기술적 인증으로 번역하려 했을 때, 분리 가능한 '안전 모듈'에 대한 증명이 아니라 **통계적 인증이라는 아키텍처 수준의 증명**이 필요했던 이유가 여기 있다.

### 2-2. 인프라 층위에서의 융합 현상

[[entities/agentic-harness-engineering.md|agentic harness engineering]]이 Crab 논문을 통해 보여주는 변화는 결정적이다:

> "C/R 런타임이 하네스의 핵심 구성요소(결함 내성·롤백·분기)를 직접 구현하는 인프라 계층으로, 하네스 엔지니어링의 범위를 애플리케이션 로직에서 실행 환경까지 확장함."

안전 메커니즘(결함 내성, 롤백)이 애플리케이션 로직 위에 얹히는 별도 계층이 아니라, **실행 환경 자체의 속성**이 된다. 이는 Agent-Native Immune System이 주장하는 '내장'의 정확한 인프라적 의미다.

### 2-3. 의미론적 간극의 해소가 곧 안전성

[[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]이 Crab에서 현현하는 방식을 보면:

> "에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율이 간극의 본질이다."

이 간극을 해소하는 것—에이전트가 자신의 의미론적 의존을 이해하고 그에 맞게 상태를 보존·복원하는 것—은 순수한 '성능 최적화'인 동시에 '안전성 보장'이다. [[concepts/agent-os-semantic-gap.md|agent os semantic gap]]과 [[concepts/agent-os-semantic-preservation-failure.md|agent os semantic preservation failure]]가 존재하는 한, 분리된 안전 계층은 에이전트의 실제 의미론적 상태를 인지하지 못해 방어에 실패한다.

## 3. 재구성이 필요한 아키텍처: 능력-안전 융합체

### 3-1. [[concepts/architectural-safety-convergence.md|architectural safety convergence]]의 필연성

위 슬러그 목록에 존재하는 이 개념은 정확히 질문이 묻는 것을 가리킨다. 에이전트 아키텍처와 안전성이 수렴(convergence)하는 지점에서, 다음의 재구성이 필요하다:

| 기존 파라다임 | 재구성된 파라다임 |
|---|---|
| 메모리 = 상태 유지 능력 | 메모리 = 공격 이력 추적·패턴 인식 능력 (면역 기억) |
| 도구 사용 = 환경 행동 능력 | 도구 사용 = 자가 진단·자가 치료 능력 (면역 반응) |
| 다중 에이전트 협력 = 작업 분할 능력 | 다중 에이전트 협력 = 상호 감사·견제 능력 (면역 체계의 다층 구조) |

### 3-2. [[concepts/capability-cooperation-paradox.md|capability cooperation paradox]]의 해소

이 슬러그가 가리키는 역설—능력이 협력을 통해 극대화되지만 동시에 제어 가능성을 저하시킨다는 것—은 분리 파라다임 아래에서만 역설이다. Agent-Native Immune System의 관점에서 이 역설은 해소된다: **협력 자체가 제어 메커니즘**이 되기 때문이다. [[concepts/agent-reliability-auditing.md|agent reliability auditing]]이 별도 프로세스가 아니라 협력 프로토콜 내에 내장될 때, 능력 증가는 안전성 증가와 동일한 방향이다.

### 3-3. [[concepts/agent-native-serving.md|agent native serving]]으로의 확장

[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]이 보여주듯, 다중 서버 MCP 배포에서 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정된다. 이는 서빙 인프라 자체가 에이전트의 의미론적 구조를 이해해야 한다는 뜻이다. [[concepts/agent-native-serving.md|agent native serving]]이 단순한 성능 최적화가 아니라, 서빙 계층에서의 면역 메커니즘 구현을 의미한다면, 능력-안전 융합은 인프라 전체로 확장된다.

이 분석에서 다음의 연결이 새롭게 드러난다:

**[[entities/agentic-harness-engineering.md|agentic harness engineering]] → [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]] → [[concepts/architectural-safety-convergence.md|architectural safety convergence]]**

세 엔티티는 독립적으로 존재하지만, 실제로는 하나의 연쇄를 형성한다. 하네스를 인프라 계층으로 확장하는 것(agentic-harness-engineering)이 곧 의미론적 간극의 해소(algorithm-system-translation-gap)이며, 이것의 완성이 아키텍처-안전 수렴(architectural-safety-convergence)이다. Agent-Native Immune System은 이 연쇄의 최종 형태를 기술한다.

## 5. 결론: 분리 불가능성은 취약성이 아니라 설계 원리다

질문에 대한 답은 **그렇다**이되, 단순한 긍정이 아닌 구조적 필연으로 이해되어야 한다.

Wiki의 증거 사슬—SafetyALFRED의 평가 분리 실패, Crab의 인프라 융합, Bounding the Black Box의 아키텍처 수준 인증 요구—은 일관되게 보여준다: 에이전트의 핵심 능력(메모리, 도구 사용, 협력)을 안전성과 분리하여 설계하는 것은 가능하지 않을 뿐 아니라, **분리 자체가 취약성의 원인**이 된다.

Agent-Native Immune System이 제시하는 방향은 '안전을 위해 능력을 제한하는 것'도, '능력에 안전을 덧붙이는 것'도 아니다. **능력의 구조 자체가 방어의 구조가 되도록 아키텍처를 재구성**하는 것이며, 이것이 [[concepts/architectural-safety-convergence.md|architectural safety convergence]]의 실질적 의미다.
