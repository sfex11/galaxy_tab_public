## 1. Wiki 기존 분석의 한계: 유한 토폴로지 전제

Wiki에서 다중 에이전트 오케스트레이션은 [[agora-opt]]를 중심으로 분석됩니다. Agora-Opt는 "런타임 메모리·토론 구조를 통해 다중 에이전트 정렬을 달성"하며, MASPO는 "프롬프트 설계 시점에서 사전 정렬을 보장하는 상보적 경로"를 제공합니다. 핵심은 **설계자가 결정한 유한한 에이전트 집합 내에서 조율을 최적화**한다는 점입니다 — 런타임 정렬의 부담을 설계 시점으로 분산시키는 것이 최적화의 방향성입니다.

이는 본질적으로 **제어 문제(control problem)**입니다. 토폴로지가 고정되어 있으므로, 최적화 대상은 '누가 언제 발언하는가', '프롬프트를 어떻게 사전 정렬하는가'와 같은 조율 파라미터에 국한됩니다.

## 2. WebSwarm이 파괴하는 전제

WebSwarm은 "LLM 기반 웹 검색 에이전트가 단순 팩토이드 질의응답에서 복잡한 deep-and-wide 검색 및 연구 지향적 태스크로 정보 탐색을 변환"시키며, **에이전트가 임의 깊이로 하위 에이전트를 재귀적으로 생성**하는 패턴을 도입합니다.

이때 오케스트레이션의 본질이 근본적으로 전이되어야 하는 이유는 세 가지 차원에서 확인됩니다.

### 2-1. 병목의 동적 이동: 고정 최적화의 무효화

[[aggregate-pipeline-serving]]의 분석이 시사하는 바가 여기서 결정적입니다. 다중 서버 MCP 배포에서 "파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 **스키마 축적**에 의해 결정"됩니다. WebSwarm의 재귀적 생성에서 스키마 축적은 **재귀 깊이에 따라 기하급수적으로 증가**합니다. 유한 토폴로지에서는 스키마 축적량을 사전에 예측 가능했지만, 재귀적 구조에서는 불가능합니다.

이는 [[adaptive-bottleneck-migration]]의 개념이 단순한 '병목 이동'을 넘어 **재귀 깊이에 따른 병목의 발산적 이동**으로 재정의되어야 함을 의미합니다.

### 2-2. 적응의 차원 확장: 외부→내부→구조적 상태

[[adaptive-inference]]에서 SpecKV가 "적응적 추론의 결정 차원을 모델/라우트 선택에서 추측 길이(γ) 선택으로 확장"하고, "압축이라는 내부 시스템 상태에 반응하는 적응의 새로운 유형"을 제시한 바 있습니다. WebSwarm의 재귀적 패턴은 이 적응의 차원을 한 단계 더 확장합니다 — **현재 에이전트 트리의 깊이와 폭이라는 구조적 상태(structural state)에 반응하는 적응**이 필요합니다.

기존의 적응이 "어떤 모델을 쓸까"→"얼마나 추측할까"로 진화했다면, 재귀적 오케스트레이션에서는 "더 깊이 파들어갈 것인가, 현재 수준에서 종료할 것인가"가 적응의 핵심 결정이 됩니다.

### 2-3. 실패의 누적 구조: 공동 실패 한계의 재귀적 강화

재귀적 생성은 [[co-failure-ceiling]]의 문제를 극단적으로 악화시킵니다. 유한 토폴로지에서는 에이전트 간 실패가 선형적으로 전파되지만, 재귀 구조에서는 **각 하위 에이전트의 오류가 상위 에이전트의 입력 왜곡으로 누적되어 트리 전체로 전파**됩니다. 더욱이 [[cascading-distribution-shift]]가 재귀 각 레벨마다 발생하여, 깊이가 증가할수록 루트 에이전트가 받는 정보의 분포가 원래 태스크 분포에서 극단적으로 이탈합니다.

## 3. 전이가 요구하는 근본적 재구조화

따라서 WebSwarm 패턴 하에서 오케스트레이션은 다음과 같이 재구조화되어야 합니다.

| 차원 | 기존 (고정 토폴로지) | 전이 후 (재귀적 확장) |
|------|---------------------|----------------------|
| 최적화 대상 | 조율 파라미터 (발언 순서, 프롬프트 정렬) | **재귀 깊이/폭의 자원 할당 한계** |
| 병목 특성 | 예측 가능한 고정 지점 | 깊이 종속적 발산 병목 |
| 적응 상태 | 외부 환경 또는 내부 압축 상태 | **에이전트 트리의 구조적 상태** |
| 실패 전파 | 선형적/국소적 | **계층적 누적/전역적** |
| 설계 시점 통제 | MASPO식 사전 정렬 가능 | **사전 정렬 범위를 벗어나는 구조적 불확실성** |

특히 [[agora-opt]]의 런타임 정렬 메커니즘은 재귀적으로 생성된 하위 에이전트들에게 어떻게 전파되는가? MASPO의 "설계 시점 사전 정렬"은 설계자가 예상하지 못한 깊이의 에이전트에게 적용될 수 없습니다. 즉, **유한 전제의 정렬 메커니즘이 재귀적 구조에서는 정렬 보장 범위를 상실**합니다.

## 4. 새로운 개념적 연결 발견

이 분석을 통해 다음 연결이 새롭게 드러납니다.

**① [[agent-loop-as-computation]] × 재귀적 생성 = 계산 복잡도로서의 오케스트레이션**
에이전트 루프를 연산 단위로 보는 관점을 재귀적 생성과 결합하면, 오케스트레이션은 더 이상 '조율'이 아니라 **재귀적 계산의 자원 한계 내 수렴성 보장 문제**가 됩니다. 튜링 완전성의 관점에서, 유한 깊이 제한 없는 재귀적 에이전트 생성은 종료성(termination) 보장 문제로 귀결됩니다.

**② [[adaptive-credit-granularity]]의 재귀적 적용 필요**
유한 토폴로지에서는 각 에이전트의 기여도를 정량화하는 것으로 충분했지만, 재귀 구조에서는 "어느 깊이의 에이전트가 최종 출력에 얼마나 기여했는가"를 **깊이 가중치를 고려한 세분화된 크레딧**으로 재설정해야 합니다.

**③ [[agent-execution-semantic-opacity]]의 심화**
[[agent-os-semantic-gap]]이 이미 "에이전트의 의미론적 의존 그래프를 OS의 체크포인트 메커니즘으로 번역할 때 발생하는 비효율"을 지적한 바 있습니다. 재귀적 생성은 이 의미론적 불투명성을 **동적으로 변화하는 의존 그래프 자체를 추적 불가능하게 만드는** 차원으로 심화시킵니다.

## 결론

WebSwarm의 재귀적 패턴은 Wiki가 [[agora-opt]]와 [[aggregate-pipeline-serving]] 등을 통해 구축한 '고정 토폴로지 내 제어 최적화' 프레임워크의 **적용 가능 영역 밖**에 위치합니다. 오케스트레이션의 본질을 '재귀적 확장의 자원 한계 관리'로 전이시키지 않으면, 기존의 조율 최적화는 재귀 깊이가 증가함에 따라 스키마 축적 병목과 계층적 실패 누적에 의해 체계적으로 붕괴합니다. 이는 단순한 스케일링 문제가 아니라, 오케스트레이션의 **문제 정의 자체**가 달라져야 하는 패러다임 전이입니다.

## 핵심 통찰

재귀적 다중 에이전트 생성은 오케스트레이션을 '조율 파라미터 최적화'에서 '재귀적 계산의 자원 한계 내 수렴성 보장 문제'로 근본적으로 전이시키며, 기존 유한 토폴로지 전제의 정렬 메커니즘들은 구조적 불확실성 하에서 보장 범위를 상실한다.
