### 1. 기존 최적화의 한계: 원자적 호출 전제 안에 갇힌 진단

[[aggregate-pipeline-serving]] 하위의 *"Tool Attention Is All You Need: Dynamic Tool Gating and Lazy Schema Lo"* 논문은 다중 서버 MCP 배포에서 **파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정됨**을 보여주며, 파이프라인 서빙 최적화의 실제 병목 지점을 재정의합니다. 그러나 이 재정의는 여전히 **"원자적 호출의 축적을 어떻게 줄일 것인가"**라는 프레임 안에 머물러 있습니다. 스키마 로딩의 지연을 줄이더라도, 각 호출이 독립적인 선택-실행-관찰 단위로 모델 추론 궤적에 노출된다는 전제는 그대로입니다.

즉, 기존 접근법은 **파이프라인의 관점에서 병목을 진단**하지만, **모델 추론의 관점에서 발생하는 구조적 오정렬**은 다루지 않습니다.

### 2. HyperTool이 포착한 문제: 실행 입자도 불일치

HyperTool의 핵심 지적은 이렇습니다:

> *"Tool-augmented LLM agents commonly rely on step-wise atomic tool calls, where each invocation, observation, and value transfer is exposed in the main reasoning trace. This creates an execution-g..."*

국소적으로 결정론적인 도구 워크플로우—예컨대 "파일 읽기 → 파싱 → 필드 추출"이 사실상 하나의 복합 연산인 경우—가 불필요하게 개별 모델 가시 단계로 펼쳐집니다. 이는 단순한 효율 문제가 아니라 **추상화 단위의 오정렬**입니다:

- **실행 층**: 결정론적으로 묶일 수 있는 워크플로우 (하위 단위의 선택이 모델 추론에 의존하지 않음)
- **가시 층**: 각 하위 단위가 독립적인 원자적 호출로 모델 추론 트레이스에 노출됨

이 간극이 만들어내는 비용은 단순한 레이턴시가 아닙니다. 매 단계마다 불필요한 관찰-해석 주기가 삽입되면서 [[agent-execution-semantic-opacity]]가 누적되고, [[blind-tool-invocation]]의 위험이 증폭됩니다. 모델이 볼 필요 없는 중간 상태를 관찰하게 함으로써 오히려 추론의 일관성을 해치는 구조적 문제입니다.

### 3. 시사점: 최적화 대상의 전환

두 논문을 교차하면 명확한 전환이 보입니다:

| 차원 | Tool Attention (기존) | HyperTool (재정의) |
|------|----------------------|-------------------|
| **진단 대상** | 스키마 축적에 의한 파이프라인 병목 | 원자적 호출 단위 자체의 오정렬 |
| **최적화 축** | 호출 횟수/지연 최소화 | 도구 실행의 의미론적 단위 재정의 |
| **해결 방향** | Lazy Schema Loading, Dynamic Gating | 결정론적 하위 워크플로우의 복합 연산화 |
| **남는 문제** | 원자적 호출 전제 유지 | — |

HyperTool이 제시하는 방향은 [[aggregate-pipeline-serving]]이 포착한 병목을 **상위에서 해소**하는 경로입니다. 원자적 호출이 아니라 의미론적으로 응집된 복합 도구 연산을 단위로 삼으면, 스키마 축적 문제 자체가 구조적으로 감소합니다—하나의 복합 연산에 대해 하나의 스키마만 로드되면 되기 때문입니다.

### 4. 새로운 개념 연결 발견

이 분석에서 다음 연결이 새롭게 드러납니다:

- **[[aggregate-pipeline-serving]] ↔ [[agent-execution-semantic-opacity]]**: 파이프라인 최적화 논의와 실행 의미론 불투명성이 원자적 호출 전제라는 공통 근원을 공유함. Tool Attention이 파이프라인 관점에서, HyperTool이 추론 관점에서 동일한 근원적 문제를 다름.
- **[[adaptive-inference]]와의 유추**: SpecKV가 적응적 추론의 결정 차원을 "모델/라우트 선택"에서 "추측 길이(γ) 선택"으로 확장한 것과 구조적으로 동일한 패턴—최적화의 자유도를 외부 선택에서 내부 상태에 기반한 단위 재정의로 이동시키는 패러다임 전환.

### 결론

HyperTool이 지적하는 실행 입자도 불일치는 도구 호출 최적화가 **"원자적 호출을 더 빠르게/더 적게"**에서 **"어떤 의미론적 단위를 하나의 호출로 간주할 것인가"**로 전환해야 함을 명확히 시사합니다. 이는 [[aggregate-pipeline-serving]]가 식별한 스키마 병목 문제까지 상위에서 구조적으로 해소할 수 있는 근본적 재정의입니다.
