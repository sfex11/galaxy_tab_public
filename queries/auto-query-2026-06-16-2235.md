# 하네스 최적화 연구의 "독립적 효과" 추적 문제

질문하신 간극은 실제로 Wiki 내의 여러 엔티티가 암묵적으로 공유하는 전제와 AgentSpec의 실증 사이의 구조적 모순을 정확히 짚어냅니다.

## 1. 현행 Wiki의 암묵적 전제: 구성요소 독립성 가정

현재 Wiki의 하네스 최적화 관련 엔티티들은 각각 **특정 구성요소를 독립적으로 최적화 가능한 대상**으로 전제하고 있습니다.

- **[[agentic-harness-engineering]]**: Crab 논문은 C/R 런타임을 하네스의 핵심 구성요소(결함 내성·롤백·분기)를 직접 구현하는 **인프라 계층**으로 규정하며, 이를 애플리케이션 로직과 분리된 독립 최적화 차원으로 배치합니다.
- **[[aggregate-pipeline-serving]]**: Tool Attention 논문은 "파이프라인 전체의 레이턴시 병목이 **개별 LLM 호출이 아닌 스키마 축적**에 의해 결정된다"고 주장하며, **하나의 특정 지점을 '진짜 병목'으로 식별**하여 거기에 집중하면 파이프라인 전체가 최적화된다는 논리를 전개합니다.
- **[[agent-os-semantic-gap]]**: 에이전트의 의미론적 의존 그래프와 OS의 체크포인트 메커니즘 사이의 간극을 **고립된 번역 문제**로 규정하고, 이 간극 자체를 해결하면 하네스가 정상화된다고 암묵적으로 가정합니다.

이 세 엔티티의 공통된 논리 구조는 **"병목을 식별 → 그 병목을 독립적으로 해결 → 전체 하네스 개선"**입니다. 이는 구성요소 간 결합이 느슨하거나, 적어도 각 구성요소의 기여를 분리하여 측정할 수 있다는 전제 위에 서 있습니다.

## 2. AgentSpec이 제시하는 반증: 스캐폴드 구성요소의 밀결합

AgentSpec 논문("AgentSpec: Understanding Embodied Agent Scaffolds Through Controlled Composition")은 이 전제를 정면으로 부정합니다:

> "LLM agents are increasingly built not as single model calls, but as scaffolded systems that combine reasoning, memory, reflection, action execution, and learning. **While such scaffolds often improve performance, understanding how individual components contribute is difficult.**"

핵심은 **"understanding how individual components contribute is difficult"**라는 실증적 결론입니다. reasoning, memory, reflection, action execution, learning이라는 스캐폴드 구성요소들이 밀결합되어 있어, 개별 기여를 분리·측정하는 것이 근본적으로 어렵다는 것입니다.

## 3. 모순의 구체적 현현: 존재하지 않는 독립적 효과 추적

이 두 가지가 결합되면, 현행 하네스 최적화 연구가 직면하는 문제가 명확해집니다.

### [[bottleneck-misattribution]]의 위험

[[aggregate-pipeline-serving]]이 "스키마 축적이 병목"이라고 진단한 것은, 구성요소들이 독립적이라는 전제하에서만 유효한 귀인입니다. AgentSpec의 관점에서 보면, 스키마 축적이 느려 보이는 것은 **스키마 축적 자체의 문제가 아니라, 메모리 구성요소·추론 구성요소·액션 실행 구성요소 간의 밀결합이 스키마 축적 지점에서 현현된 것**일 수 있습니다. 즉, **[[bottleneck-misattribution]]**이 발생하고 있을 가능성이 높습니다.

### [[bottleneck-transference]]의 순환

더 심각한 문제는, 잘못 귀인된 병목을 "해결"하더라도 효과가 사라지는 것이 아니라 **다른 구성요소로 전이**된다는 점입니다. 스키마 축적을 최적화하면, 밀결합 구조상 그 부하가 메모리 조회나 추론 단계로 옮겨갈 뿐입니다. 이는 [[bottleneck-transference]]의 전형적인 순환 구조를 만듭니다 — 병목을 "고치는" 것처럼 보이지만 실제로는 병목이 **이동**만 할 뿐, 전체 하네스의 성능은 구조적 한계에 부딪힙니다.

### [[compositional-incoherence]]로서의 하네스 최적화

결국, 구성요소를 독립적으로 최적화하려는 시도 자체가 스캐폴드의 실제 구조([[compositional-incoherence]])와 부합하지 않는 분석 단위를 강제하는 것입니다. Crab이 [[agent-os-semantic-gap]]을 해결하려 설계한 의미론적 체크포인트도, 이것이 reflection 구성요소나 memory 구성요소와 어떻게 얽혀 있는지를 통제하지 않은 채 최적화한다면, 측정된 "개선 효과"는 통제되지 않은 변수들의 혼합 결과입니다.

## 결론

질문하신 판단은 정확합니다. 현행 Wiki의 하네스 최적화 연구 — [[agentic-harness-engineering]], [[aggregate-pipeline-serving]], [[agent-os-semantic-gap]] — 는 **구성요소 독립성 가정** 하에 각자의 차원에서 "진짜 병목"을 식별하고 해결한다는 서사를 공유합니다. 그러나 AgentSpec의 실증이 이 서사의 기반을 무너뜨립니다. 구성요소가 밀결합되어 개별 기여를 분리할 수 없다면, 현재 추적되고 있는 "독립적 최적화 효과"는 **[[bottleneck-misattribution]]**과 **[[bottleneck-transference]]**의 교차점에서 발생하는 착시일 가능성이 큽니다. 하네스 최적화 연구는 구성요소 단위에서 **스캐폴드 전체의 구성(composition) 단위**로 분석 단위를 전환해야 하는 방법론적 전환기에 직면해 있습니다.

## 핵심 통찰

현행 하네스 최적화 연구는 구성요소 독립성 가정 위에 서 있지만, 실제 스캐폴드는 밀결합 구조이므로 '병목 식별→독립적 해결' 패러다임은 병목 전이만 낳을 뿐 진정한 최적화를 달성할 수 없다.
