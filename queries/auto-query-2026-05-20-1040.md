## Wiki 답변

제공된 Wiki 지식에 기반할 때, 에이전트 하네스 엔지니어링을 단순히 '코드 생성'의 문제로 환원하기보다는, **'코드의 의미론적 구조'와 '외부 실행 인프라의 물리적 한계' 사이의 변환 간극을 해결하는 이중적인 문제**로 보아야 합니다. 

이를 Wiki의 구체적인 논문들을 인용하여 세 가지 측면에서 설명하겠습니다.

### 1. 인프라의 물리적 한계: 코드 생성만으로는 해결되지 않는 간극
Wiki의 [[entities/agentic-harness-engineering.md|agentic harness engineering]] 문서에서 'Crab' 논문은 하네스의 핵심 구성요소(결함 내성, 롤백, 분기)가 애플리케이션 로직(코드) 수준을 넘어 C/R(Checkpoint/Restore) 런타임이라는 실행 환경까지 확장됨을 시사합니다. 

특히 [[concepts/agent-os-semantic-gap.md|agent os semantic gap]]에서 명확히 밝히듯, 에이전트가 가진 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS가 제공하는 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 근본적인 비효율이 발생합니다. 즉, 에이전트의 의도를 완벽하게 담은 하네스 코드를 생성했다 하더라도, 이를 OS 수준에서 의미론적으로 보존하고 복원하는 인프라의 한계([[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]])가 병목으로 작동하므로 하네스 문제가 인프라 설계로 환원될 수밖에 없습니다.

### 2. 런타임 병목의 재정의: 코드를 넘어선 파이프라인 차원의 문제
[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]에 연결된 'Tool Attention Is All You Need' 논문은 하네스 엔지니어링이 코드 자체의 최적화를 넘어서는 이유를 명확히 보여줍니다. 다중 서버 MCP 배포 환경에서 파이프라인 전체의 레이턴시 병목은 개별 LLM 호출이나 에이전트 코드의 효율성이 아니라 **'스키마 축적'**에 의해 결정됩니다. 이는 에이전트가 사용할 도구들의 스키마를 파이프라인이 어떻게 취합하고 전달하는가라는 서빙 인프라의 구조적 문제이므로, 코드 생성만으로는 하네스를 완성할 수 없음을 증명합니다.

### 3. 코드가 곧 하네스가 되는 지점: 새로운 패러다임의 등장
그러나 질문자의 통찰처럼 코드 자체가 하네스의 역할을 대체하는 경로도 Wiki 내에 명시적으로 존재합니다. 최근 논문 목록에 등장하는 **"Code as Agent Harness"**가 바로 그것입니다. 이 연구는 LLM의 강력한 코드 이해 및 생성 능력이 에이전트 하네스 구축의 핵심 매개체가 될 수 있음을 시사합니다.

다만, 이 코드 기반 하네스조차 정적인 산출물이 아니라 [[concepts/agentic-language.md|agentic language]](RunAgent 논문)가 다루는 것처럼 자연어 계획을 제약 기반 실행으로 해석하는 '동적 번역 계층'으로서 기능해야 합니다.
