## 1. 하네스를 '중립적 인프라'로 전제하는 현재 프레임의 균열

현재 Wiki에서 [[agentic-harness-engineering]]은 Crab 논문과의 연결을 통해 "하네스 엔지니어링의 범위를 애플리케이션 로직에서 실행 환경까지 확장"한다고 기술하지만, 그 확장의 축은 여전히 **결함 내성·롤백·분기**라는 최적화 차원에 머물러 있습니다.

그러나 [[aggregate-pipeline-serving]]의 Tool Attention 논문이 이미 이 전제에 균열을 내고 있습니다:

> "다중 서버 MCP 배포에서 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 **스키마 축적**에 의해 결정됨"

여기서 '스키마 축적'은 단순한 레이턴시 변수가 아닙니다. 스키마가 축적됨에 따라 에이전트가 동시에 인지할 수 있는 도구 공간이 구조적으로 변형됩니다. 이는 최적화 문제가 아니라 **에이전트가 도달 가능한 정보의 위상(topology)을 하네스가 결정**한다는 문제입니다.

## 2. [[algorithm-system-translation-gap]]이 포착한 구조적 변형의 본질

Crab 논문이 [[algorithm-system-translation-gap]]에서 밝힌 핵심은 다음과 같습니다:

> "에이전트의 **의미론적 의존 그래프**(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율이 간극의 본질이다."

이를 '비효율'으로만 읽으면 최적화 관점에 갇힙니다. 그러나 "의미론적 의존 그래프 → 전체 스냅샷"이라는 번역은 **의미론적 구조의 손실**을 수반합니다. 체크포인트가 어떤 상태를 포착하고 어떤 상태를 누락하는가는 복원 시점의 에이전트 추론에 직접적으로 영향을 미칩니다. 즉, [[crab]]의 C/R 런타임은 상태를 투명하게 보존하는 것이 아니라 **상태를 재구성**하며, 이 재구성 자체가 추론 경로를 변경합니다.

## 3. 가장 직접적인 증거: "Is Grep All You Need?"

최근 논문 목록에 수록된 **"Is Grep All You Need? How Agent Harnesses Reshape Agentic Search"**는 이 질문에 가장 직접적으로 답하는 논문입니다. 제목 자체가 "하네스가 에이전트 검색을 **재구성(reshape)**한다"고 명시하며, 하네스가 정보에 대한 중립적 파이프가 아니라 **검색 구조 자체를 변형하는 주체**임을 선언합니다.

grep 기반 하네스, 벡터 검색 기반 하네스, 계층적 검색 기반 하네스가 동일한 코퍼스에서도 에이전트에게 **다른 정보 지형**을 제공한다면, 하네스 선택은 모델 선택만큼이나 추론 결과를 결정하는 변수가 됩니다. 이때 하네스를 '최적화 대상'으로 분류하는 것은, 프롬프트를 '토큰 효율 최적화 도구'로만 분류하는 것과 동일한 오류를 범하는 것입니다.

## 4. 새로운 개념적 연결 발견

이 재분류를 통해 다음 연결들이 새롭게 드러납니다:

**a) [[context-as-pipeline-structural-constraint]]와의 통합**
컨텍스트가 파이프라인 구조에 의해 제약받는다는 이 개념은, 하네스가 컨텍스트의 **내용이 아니라 구조**를 결정한다는 관점에서 재분류의 직접적 근거가 됩니다. 파이프라인 구조가 곧 추론의 구조라면, 하네스 설계는 프롬프트 설계와 동일한 위상에 있어야 합니다.

**b) [[agent-execution-semantic-opacity]]의 근인 재해석**
하네스가 정보를 구조적으로 변형한다면, 에이전트 실행의 불투명성은 "실행 환경의 복잡성" 때문이 아니라 **"하네스가 수행한 암묵적 정보 필터링이 불투명하기 때문"**이라고 재해석되어야 합니다. 불투명성의 근원이 모델 내부에서 하네스-모델 경계로 이동합니다.

**c) [[blind-tool-invocation]]의 근인 확장**
맹목적 도구 호출이 에이전트의 판단 오류로만 설명된다면, 하네스가 제공하는 도구 스키마 공간의 구조적 편향을 간과하는 것입니다. 하네스의 검색 구조가 특정 도구를 검색 가능하게/불가능하게 만드는 결정이 [[blind-tool-invocation]]의 근인 중 하나일 수 있습니다.

**d) [[bottleneck-misattribution]]의 차원 확장**
[[aggregate-pipeline-serving]]이 "레이턴시 병목의 귀인 오류"를 지적했다면, 이를 **"추론 병목의 귀인 오류"**로 확장해야 합니다. 추론 실패의 원인을 모델 능력 부족으로 돌리는 것은, 실제로는 하네스의 검색 구조가 정보를 변형한 데서 비롯된 실패를 오진하는 것일 수 있습니다.
