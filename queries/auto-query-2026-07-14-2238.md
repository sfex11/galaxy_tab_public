# 오케스트레이션 정교화가 실제 병목을 해결하지 못하는 구조적 이유

질문이 지적하는 문제는 단순한 '누락'이 아니라, Wiki 내에 이미 패턴으로 존재하는 **[[bottleneck-misattribution]]**의 구조적 반복이다.

---

## 1. Wiki가 이미 포착한 동일 패턴: 병목의 오지정

[[aggregate-pipeline-serving]] 항목에서 Tool Attention 논문이 보여준 통찰이 정확히 동일한 구조다:

> 다중 서버 MCP 배포에서 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 **스키마 축적**에 의해 결정됨을 보여주어, 파이프라인 서빙 최적화의 실제 병목 지점을 재정의한다.

즉, "LLM 호출 최적화"에 집중하던 기존 접근이 실제 병목(스키마 축적)을 놓쳤던 것과 정확히 동일한 구조가 다중 에이전트 체화 계획에서도 재현된다. 오케스트레이션 계층(토론 구조, 프롬프트 정렬)을 정교화해도, Mosaic이 밝힌 **실패 행동의 누적 비용**이라는 실행 계층의 병목이 해결되지 않는다.

이는 [[bottleneck-transference]]로 이해할 수 있다: 오케스트레이션을 최적화할수록 해당 계층의 병목은 사라지지만, 인접한 실행 계층으로 병목이 **이전**될 뿐이다.

---

## 2. 기존 오케스트레이션 접근의 한계 구조

Wiki의 [[agora-opt]] 항목에 기록된 두 논문의 상보성을 보면, 두 접근 모두 **통신/정렬 계층**에 머물러 있음이 명확하다:

- **Agora-Opt**: 런타임 메모리와 토론 구조를 통해 다중 에이전트 정렬을 달성
- **MASPO**: 프롬프트 설계 시점에서 사전 정렬을 보장하여 런타임 부담을 분산

두 접근 모두 "에이전트 간에 **무엇을 어떻게 전달할 것인가**"를 최적화하지만, Mosaic이 지적하는 핵심 문제—부분 관측 하에서 **각 에이전트가 환경 상태를 정확히 추적하지 못해 발생하는 실패**—는 통신 프로토콜 정교화로 해결할 수 없는 영역이다.

이는 [[algorithm-system-translation-gap]]의 다중 에이전트 버전이라고 볼 수 있다. 알고리즘 수준에서의 정렬(프롬프트, 토론 구조)이 시스템 수준의 실행 신뢰성(상태 추적 정확도)으로 번역되지 않는 간극이 존재한다.

---

## 3. [[co-failure-ceiling]]과 [[co-failure-rate]]: 새로운 연결 발견

여기서 Wiki 슬러그 목록에 존재하지만 아직 논문과 연결되지 않은 두 개념이 핵심적으로 부각된다:

**[[co-failure-rate]]**: 다중 에이전트가 협력 과정에서 누적적으로 발생시키는 실패 행동의 비율. Mosaic의 "failed actions as the dominant bottleneck"이 바로 이 지표의 실증적 발견이다.

**[[co-failure-ceiling]]**: 오케스트레이션을 아무리 정교화해도, co-failure-rate가 임계값 이하로 떨어지지 않으면 도달할 수 없는 성능 상한선. [[agora-opt]]와 MASPO의 정렬 메커니즘이 아무리 완벽해져도, 실행 단계에서 에이전트들이 부분 관측으로 인해 잘못된 행동을 반복하면 이 천장에 부딪힌다.

기존 Wiki에서 이 두 개념이 오케스트레이션 논문들과 명시적으로 연결되지 않았던 점이 바로 질문이 지적하는 '분석의 눈멈'에 해당한다.

---

## 4. 근본적 문제: 다중 에이전트를 '통신 시스템'으로 오인

가장 구조적인 문제는 Wiki의 현재 분석 축이 다중 에이전트를 **통신 시스템**(메시지 교환, 정렬, 합의)으로 모델링하고 있다는 점이다. 그러나 Mosaic이 보여주는 체화된 다중 에이전트 계획은 본질적으로 **상태 추적 시스템**이다:

- 부분 관측 → 상태 불확실성 누적
- 중복 협력 → 불필요한 행동 비용 증가
- 실패 행동 → 롤백 및 재계획 비용 폭발

이는 [[agent-execution-semantic-opacity]]와도 연결된다. 에이전트가 자신의 행동이 환경에 미치는 영향을 정확히 추적하지 못하면(부분 관측), 실행의 의미론적 불투명성이 커지고, 이는 곧 실패 행동의 누적으로 귀결된다.

또한 [[adaptive-inference]] 항목의 SpecKV 논문이 보여준 패턴—외부 환경 반응에서 내부 시스템 상태 반응으로 적응 축을 확장—을 다중 에이전트 문맥에 대입하면, 오케스트레이션 최적화가 '외부 환경(다른 에이전트의 메시지)에 대한 반응'에 집중하는 반면, Mosaic이 요구하는 것은 '내부 상태(부분 관측下的 환경 모델)에 대한 반응'의 적응이라는 차이가 드러난다.

---

## 5. 결론: 해결을 위한 축 이동이 필요하다

기존 오케스트레이션 정교화 접근이 실제 병목을 **전혀** 해결하지 못하는 것은 아니다. 통신 오버헤드 자체가 병목이었던 비체화(non-embodied) 설정에서는 유효하다. 하지만 체화된 설정으로 도메인이 확장되면, [[adaptive-bottleneck-migration]]이 발생하여 병목이 통신 계층에서 실행 계층으로 이동한다.

필요한 것은:

1. **[[co-failure-ceiling]]을 오케스트레이션 논문들과 명시적으로 연결**하여, 정렬 메커니즘의 효과가 co-failure-rate에 의해 상한선을 받는다는 구조를 Wiki에 기록
2. **[[bottleneck-misattribution]]을 aggregate-pipeline-serving의 사례와 함께 다중 에이전트 체화 설정으로 확장**하여, 동일한 오지정 패턴이 반복됨을 문서화
3. 오케스트레이션 최적화와 상태 추적 정확도 사이의 [[algorithm-system-translation-gap]]을 다중 에이전트 문맥에서 구체화

**새로운 연결 발견**: [[co-failure-ceiling]] ← (상한 설정) ← [[agora-opt]] / MASPO (오케스트레이션 정교화의 효과 한계), 그리고 [[co-failure-rate]] ← (원인) ← 부분 관측 하의 [[agent-execution-semantic-opacity]]라는 인과 사슬이 Wiki에 아직 명시되지 않았다.
