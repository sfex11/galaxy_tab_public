# Autodata: 네 번째 축인가, 기초층 변동인가?

## 세 축의 공통 전제: "거기에 무언가가 이미 있다"

질문자의 통찰을 Wiki의 논문들을 통해 검증해보면, 세 축은 indeed 공통된 기초 전제 위에 서 있습니다.

**1) RLVR 축** — 최근 논문 *"Reinforcement Learning without Ground-Truth Solutions can Improve LLMs"*가 스스로 인정하듯, 기존 RLVR는 "ground-truth answers to assign rewards"를 전제하며, 이를 벗어나려는 시도조차 "verifiable rewards"라는 대체 고정점을 필요로 합니다. 보상 신호의 근원이 사전에 존재해야 한다는 구조적 제약이 남습니다.

**2) 환경 생성 축** — [[concepts/agent-environment-generation.md|agent environment generation]]에서 [[concepts/agent-environment-generation.md|Nemobot Games]]는 게임 도메인 특화 학습 환경을 생성하지만, 이는 "환경 구조를 새로 만드는" 것이지 "환경 속 데이터 분포를 처음부터 발명하는" 것과는 다릅니다. 생성된 환경에서 에이전트가 마주하는 task 분포는 여전히 게임이라는 사전 도메인 지식에 기대 있습니다.

**3) 자기 진화 축** — [[entities/agora-opt.md|agora opt]]의 런타임 메모리·토론 구조나 [[concepts/adaptive-inference.md|adaptive inference]]의 추측 길이 적응은 모두 "주어진" 지식 베이스, "주어진" 모델 파라미터 위에서의 런타임 최적화입니다. 진화의 재료가 되는 정보 자체는 외부에서 공급됩니다.

세 축 공통의 암묵적 전제: **훈련·평가·환경의 '내용물'은 사전 존재하며, 우리가 최적화하는 것은 그 내용물에 대한 '반응 방식'이다.**

## Autodata가 무너뜨리는 것

Autodata는 이 전제를 두 수준에서 동시에 공격합니다.

**첫째, 데이터 생성 주체의 전환.** "AI agents to act as data scientists who build high quality training and evaluation data" — 데이터가 더 이상 인간이 큐레이션하거나 환경이 산출하는 부산물이 아닙니다. 에이전트 자신이 데이터의 설계자가 됩니다.

**둘째, 메타최적화 루프.** "train (meta-optimize) such a data scientist" — 데이터를 만드는 에이전트를 다시 훈련시킵니다. 즉, 데이터 생성 과정 자체가 최적화 대상이 되며, 이는 [[concepts/closed-loop-training.md|closed loop training]]을 데이터 원천까지 확장하는 것입니다.

이때 발생하는 것이 [[concepts/bootstrap-paradox.md|bootstrap paradox]]입니다: 에이전트 능력이 데이터 품질을 결정하고, 그 데이터가 다시 에이전트 능력을 결정합니다. 인과 화살표가 순환하며, "최초의 좋은 데이터는 어디서 왔는가?"라는 근원적 질문이 부상합니다. 세 축에서는 이 질문이 외부(인간, 벤치마크, 물리 환경)로 외주될 수 있었지만, Autodata에서는 시스템 내부에 갇힙니다.

---

## 왜 "네 번째 축"이 아닌가

네 번째 축이라고 부르려면, RLVR·자기진화·환경생성과 **동일한 추상 수준**에서 병렬적으로 배치될 수 있어야 합니다. 그러나:

| | 세 축 | Autodata |
|---|---|---|
| **작동 대상** | 사전 존재하는 데이터/환경에 대한 반응 | 데이터 자체의 생성 |
| **최적화 변수** | 정책, 추론 경로, 환경 파라미터 | 데이터 생성 정책 (메타 수준) |
| **의존성 방향** | 데이터 → 에이전트 (단방향) | 데이터 ↔ 에이전트 (순환) |

세 축이 "주어진 판 위에서 어떻게 더 잘 두느냐"의 문제라면, Autodata는 "판 자체를 만들고, 판을 만드는 능력까지 최적화하는" 문제입니다. 이는 차원(dimension)의 추가가 아니라 **기초층(foundation layer)의 내재화**입니다.

## 새로운 개념적 연결

이 분석에서 Wiki의 기존 슬러그들 사이에 다음 연결이 발견됩니다:

**1) [[concepts/bootstrap-paradox.md|bootstrap paradox]] ↔ [[concepts/closed-loop-training.md|closed loop training]]의 강결합**
기존에는 [[concepts/closed-loop-training.md|closed loop training]]이 주로 정책 학습 루프를 지칭했을 것으로 추정되나, Autodata는 이 루프를 데이터 원천까지 확장하며 [[concepts/bootstrap-paradox.md|bootstrap paradox]]를 구조적 필연으로 만듭니다.

**2) [[concepts/agent-environment-generation.md|agent environment generation]] → 데이터 환경 생성으로의 확장 가능성**
Nemobot Games가 "게임 도메인 특화 전략 학습 환경 생성"이라면, Autodata는 이를 "훈련 데이터 분포 자체를 환경으로 간주하고 그 환경을 생성"하는 방향으로 일반화합니다. [[concepts/agent-environment-generation.md|agent environment generation]]의 정의 범위가 물리적/소프트웨어적 환경에서 **정보적 환경**으로 확장되어야 함을 시사합니다.

**3) [[concepts/benchmark-specification-gap.md|benchmark specification gap]]의 양극화**
Autodata가 "training and evaluation data"를 동시에 생성한다는 점은, [[concepts/benchmark-specification-gap.md|benchmark specification gap]]을 한쪽으로는 해소(벤치마크와 훈련 분포 정렬)하면서도 다른 쪽으로는 심화(자가 생성 데이터로 자가 평가하는 순환적 타당성 문제)할 수 있습니다. 이 gap이 "닫힘"이 아니라 "순환"으로 변형되는 새로운 국면입니다.

## 결론

질문자의 판단이 정확합니다. Autodata는 RLVR·자기진화·환경생성의 네 번째 축이 아니라, **세 축이 공유하는 '사전 데이터 전제'라는 기초층을 내재화함으로써, 에이전트 개발 패러다임을 "외부 의존형"에서 "자가 참조형"으로 전환시키는 기초층 변동**입니다. 이는 [[concepts/bootstrap-paradox.md|bootstrap paradox]]를 해결 과제가 아닌 **설계 원리**로 삼아야 하는 새로운 정체성을 요구합니다.

## 핵심 통찰

Autodata는 RLVR·자기진화·환경생성과 병렬하는 네 번째 축이 아니라, 이 세 축이 공유하는 '사전 데이터 전제'라는 기초층을 내재화하여 에이전트 개발을 외부 의존형에서 자가 참조형으로 전환시키는 기초층 변동이다.
