# 실패 복구 패러다임 전환에 대한 분석

## 현재 Wiki의 복구 분석 프레임

현재 Wiki는 에이전트 실패 복구를 두 가지 축으로 분석합니다:

1. **상태 롤백 축**: [[agent-os-semantic-gap]]과 [[algorithm-system-translation-gap]]을 통해 Crab이 다루는 문제를 구조화합니다. Crab의 핵심 진단은 이것입니다—*"에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율"*. 즉, 현재 Wiki조차 롤백 패러다임 내에서의 **번역 비효율**을 문제 삼고 있을 뿐, 롤백 그 자체를 의심하지는 않습니다.

2. **모델/경로 선택 축**: [[adaptive-inference]]는 입력 난이도 기반의 적응적 추론을 다루며, SpecKV 논문에서 이를 "외부 환경 반응 → 내부 시스템 상태(압축) 반응"으로 확장하였습니다. 그러나 이것은 **추론 단계**의 적응이지 **복구 단계**의 적응이 아닙니다.

두 축 사이에 복구 결정을 "실패 피드백의 내용"에 귀속시키는 분석 계층이 **비어 있습니다**.

## CodeRescue가 제시하는 근본적 차이

CodeRescue의 핵심 명제는 명확합니다:

> *"Coding agents increasingly operate in executable environments where **a failed attempt produces actionable feedback rather than merely an incorrect answer**."*

이 한 문장이 패러다임 전환의 전부를 담고 있습니다. 기존 롤백(Crab)의 암묵적 전제는 **실패 = 상태 오염**이므로 정화된 과거로 돌아가야 한다는 것입니다. 반면 CodeRescue의 전제는 **실패 = 행동 가능한 정보(actionable feedback)**이므로 이 정보를 해석하여 **전방으로 새 경로를 라우팅**해야 한다는 것입니다.

이는 단순한 구현 방식 차이가 아닙니다:
- **롤백의 존재론**: 실패는 부정적 상태 → 제거해야 할 것
- **전방 라우팅의 존재론**: 실패는 정보적 사건 → 해석해야 할 것

## 전환되어야 하는가? — 조건부 "예"

**단순 환경에서는 전환이 필요합니다.** 코딩, 수학 증명, 툴 호출 등에서 실패 피드백이 구조화된 정보(컴파일 에러, 테스트 실패 로그, API 응답 코드)로 제공되는 경우, 롤백은 정보를 파기하는 행위입니다. Crab이 [[algorithm-system-translation-gap]]에서 진단한 "의미론적 의존 그래프를 전체 스냅샷으로 번역하는 비효율"은 여기서 더 심각해집니다—**복구에 필요한 정보까지 함께 파기**되기 때문입니다.

**그러나 무조건적 전환은 아닙니다.** [[agent-os-semantic-gap]]이 다루는 문제 영역—체크포인트/복원이 의미론을 보존하지 못하는 상황(예: 외부 서비스 상태 변화, 시간 의존적 컨텍스트)—에서는 롤백 자체가 불가능하거나 무의미하며, 이런 경우 전방 복구가 **유일한 선택지**가 됩니다. 반대로, 상태가 불가역적으로 오염되고 피드백이 무의미한 경우(예: 메모리 손상, 보안 침해 후 상태)에는 롤백이 여전히 필요합니다.

따라서 정확한 진단은 이것입니다:

> 하네스의 복구 패러다임은 '상태를 되돌리는 것'에서 '피드백에 기반한 전방 복구 라우팅'으로 **기본 전제를 전환**해야 하며, 롤백은 전방 라우팅이 불가능한 예외 경로로 격하되어야 한다.

이는 [[checkpoint-restore-efficiency-spectrum]]이 슬러그 목록에 존재한다는 점과 맞닿아 있습니다—스펙트럼의 양 끝이 아니라, **기본 전제의 우선순위가 역전**되어야 한다는 것입니다.

## Wiki에서 새로 발견된 연결 및 결손

### 1. CodeRescue가 어떤 엔티티에도 연결되지 않음
CodeRescue는 "최근 논문"에만 나타나고 어떤 Wiki 엔티티와도 연결되어 있지 않습니다. 이것은 다음 연결이 필요함을 의미합니다:
- **[[agentic-harness-engineering]]**: Crab이 "하네스 엔지니어링의 범위를 애플리케이션 로직에서 실행 환경까지 확장"했다면, CodeRescue는 그 범위를 **"실패 의미론 해석"까지** 확장하는 후속 연구로 위치지어져야 합니다.
- **[[adaptive-inference]]**: SpecKV가 적응의 차원을 "외부 환경 → 내부 압축 상태"로 확장했다면, CodeRescue는 그것을 **"정상 입력 난이도 → 실패 피드백 유형"**으로 확장하는 또 다른 적응 차원을 제시합니다. 두 논문은 [[adaptive-inference]] 내에서 상보적 하위 유형으로 재구성될 수 있습니다.

### 2. 복구 라우팅 결정을 위한 독립 개념의 부재
현재 Wiki에는 **"실패 피드백을 복구 경로 결정의 1등급 신호로 사용한다"**는 개념을 담을 엔티티가 없습니다. [[adaptive-inference]]는 입력 기반, [[agentic-harness-engineering]]은 인프라 기반입니다. 실패 **사건(event)** 자체를 라우팅 신호로 삼는 개념—이를 임시로 *failure-feedback-routing*이라 부르겠지만 슬러그 목록에 없으므로 링크하지 않습니다—이 새롭게 식별되어야 합니다.

### 3. DeltaBox에 대한 Wiki 지식 결손
질문에서 언급된 DeltaBox는 제공된 Wiki 지식 어디에도 등장하지 않습니다. 이는 현재 Wiki가 체크포인트/복원 계열 연구를 Crab에만 국한시키고 있어, **복구 인프라의 설계 스펙트럼을 불완전하게 포착**하고 있음을 시사합니다.

### 4. [[algorithm-system-translation-gap]]의 재해석 가능성
Crab이 이 간극을 "의미론적 의존 그래프 → OS 스냅샷" 번역의 비효율로 진단했다면, CodeRescue는 이 간극을 **더 근본적으로 회피**합니다. 전방 복구 라우팅은 의미론적 의존 그래프를 OS 메커니즘으로 번역할 필요 자체가 없으므로, CodeRescue는 [[algorithm-system-translation-gap]]에 대한 **우회 경로**로도 해석될 수 있습니다.

## 결론

CodeRescue가 실행 환경의 실패 피드백을 1등급 신호로 삼는다는 것은, 복구 패러다임의 **기본 전제**를 전환해야 함을 보여줍니다. 그러나 이것은 롤백의 "완전한 폐기"가 아니라 [[checkpoint-restore-efficiency-spectrum]]에서 롤백을 **예외 경로로 격하**하고 전방 라우팅을 **기본 경로로 승격**하는 스펙트럼 내 우선순위 역전입니다. 현재 Wiki는 이 전환을 포착할 엔티티와 연결이 결손된 상태입니다.
