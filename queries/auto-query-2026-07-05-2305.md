# 하네스-모델 경계의 붕괴: ReContext가 드러내는 [[concepts/algorithm-system-boundary-collapse.md|algorithm system boundary collapse]]

## 1. 질문의 핵심: 기존 분석 프레임의 균열

현재 Wiki는 에이전트 하네스를 [[entities/agentic-harness-engineering.md|agentic harness engineering]](Crab의 C/R 런타임)와 [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]](Tool Attention의 스키마 축적 병목) 같은 **중립적 실행 인프라**로, 장문맥 실패를 희소 주의·계층적 메모리 같은 **모델 내부 능력의 한계**로 분리해서 기록해왔습니다. 

그런데 ReContext는 입력에 **이미 존재하는 증거를 하네스가 재생(recursive evidence replay)**하여 모델의 주의 실패를 보상합니다. 이는 하네스가 단순히 "정보를 언제/어떻게 전달할지"를 결정하는 인프라를 넘어, **"어떤 정보가 추론에 필요한지"를 선택하는 인지적 역할**을 수행함을 의미합니다. Wiki에 이미 존재하는 [[concepts/algorithm-system-boundary-collapse.md|algorithm system boundary collapse]]라는 엔티티가 정확히 이 경계 붕괴를 가리키고 있습니다.

## 2. Wiki 내의 경계 붕괴 조짐: 세 가지 선행 증거

ReContext 이전에도 Wiki는 이미 이 경계가 흐려지고 있음을 기록했습니다.

**첫째, 하네스가 이미 '의미론적'인 일을 하고 있었습니다.** [[entities/agentic-harness-engineering.md|agentic harness engineering]]에서 Crab은 C/R 런타임이 단순 스냅샷이 아닌 **에이전트의 의미론적 의존 그래프**를 이해해야만 효율적인 체크포인트/복원이 가능함을 보였습니다. 이는 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]에서 구체화되었는데—에이전트가 "어떤 상태에 의존하는가"라는 의미론적 질문을 OS의 "전체 스냅샷"이라는 물리적 메커니즘으로 번역할 때 발생하는 비효율이 바로 간극의 본질이라고 기록되어 있습니다. 즉, 하네스는 이미 '중립적'이지 않았습니다.

**둘째, 하네스가 모델의 '주의 필드'를 사전에 형성하고 있었습니다.** [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]에서 Tool Attention은 다중 서버 MCP 배포에서 **스키마 축적이 개별 LLM 호출보다 더 큰 레이턴시 병목**이 됨을 보였습니다. 이는 하네스 측의 정보 축적 방식이 모델이 "무엇을 주의할 수 있는지"를 구조적으로 제한한다는 것을 의미합니다. 모델의 주의 실패가 모델 내부뿐 아니라 하네스의 스키마 관리 방식에서 기인할 수 있다는 점에서, 경계는 이미 흐려져 있었습니다.

**셋째, 적응의 주체가 모델에서 하네스로 이동하고 있었습니다.** [[concepts/adaptive-inference.md|adaptive inference]]에서 SpecKV는 적응적 추론의 결정 차원을 "추측 길이(γ) 선택"으로 확장하며, 외부 환경에 반응하던 적응을 **압축이라는 내부 시스템 상태에 반응하는 적응**으로 변환했습니다. ReContext는 이 방향을 한 단계 더 나아가—적응의 주체 자체를 모델에서 하네스로 넘깁니다.

## 3. ReContext의 위치: 경계를 넘는 세 가지 메커니즘

ReContext가 경계를 구체적으로 어떻게 붕괴시키는지 정리하면:

| 차원 | 기존 프레임 | ReContext의 위치 |
|------|-----------|-----------------|
| **실패의 귀인** | 모델의 희소 주의가 증거를 놓침 (모델 책임) | 증거는 존재하지만 전달 방식이 불충분함 (하네스-모델 공동 책임) |
| **보상의 주체** | 모델 내부 적응 (attention mechanism 개선) | 하네스가 외부에서 재생 (harness-side compensation) |
| **선택의 기준** | 모델의 어텐션 가중치 (internal) | 어떤 증거를 재생할지에 대한 하네스의 판단 (external but semantic) |

핵심은 세 번째 열입니다. **"어떤 증거를 재생할지"를 결정하는 것은 단순한 인프라 최적화가 아닙니다.** 이는 추론 과정에서 어떤 정보가 관련성을 가지는지에 대한 **의미론적 판단**입니다. 하네스가 이를 수행하는 순간, 하네스는 '중립적 파이프'에서 '준인지적 에이전트'로 승격됩니다.

## 4. 경계를 어디에 그어야 하는가: 제안 기준

Wiki의 기존 엔티티들을 조합하여 다음과 같은 경계 기준을 제안합니다.

**경계의 기준 = "의미론적 선택 권한(semantic selection authority)"의 소재**

- **인프라 영역** (하네스): 정보의 **형식·시점·압축률**을 결정. "언제/어떻게"만 관여. [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 스키마 로딩 최적화, [[entities/agentic-harness-engineering.md|agentic harness engineering]]의 체크포인트 효율화가 여기에 해당.
- **인지적 영역** (모델): 정보의 **내용적 관련성**을 판단. "무엇이 중요한지"를 결정. 희소 주의의 가중치 할당이 여기에 해당.
- **회색지대** (경계 붕괴 구역): ReContext가 위치하는 곳. 하네스가 "무엇을 재생할지"를 선택하면 이는 형식적 결정(인프라)인지 내용적 결정(인지)인지 모호해집니다.

이 관점에서 ReContext의 증거 재생은 **형식적으로는 인프라 동작(이미 존재하는 정보를 재전달)이지만, 실질적으로는 인지적 동작(추론에 필요한 정보를 의미론적으로 선택)**입니다. 경계는 그어질 수 없습니다—정확히 [[concepts/algorithm-system-boundary-collapse.md|algorithm system boundary collapse]]가 기술하는 현상입니다.

## 5. 새로운 연결 발견

이 분석을 통해 Wiki의 기존 엔티티들 사이에 세 가지 새로운 연결이 드러납니다.

**① [[concepts/bottleneck-transference.md|bottleneck transference]] ↔ [[concepts/adaptive-bottleneck-migration.md|adaptive bottleneck migration]]**
ReContext는 장문맥 추론의 병목을 모델 내부(attention span)에서 하네스(재생 전략)로 **이전**합니다. [[concepts/adaptive-bottleneck-migration.md|adaptive bottleneck migration]]이 적응적 추론에서 병목이 계층 간에 이동함을 기술한다면, ReContext는 이를 하네스-모델 경계를 가로지르는 이동으로 확장하는 사례입니다. 동시에 이는 [[concepts/bottleneck-transference.md|bottleneck transference]]—한 계층의 병목 해결이 다른 계층에 새 병목을 만드는 현상—의 가능성도 열어두어야 합니다: 하네스가 증거 선택을 떠안으면서 하네스 자체의 선택 오류가 새로운 병목이 될 수 있습니다.

**② [[concepts/bottleneck-misattribution.md|bottleneck misattribution]]의 구체화**
장문맥 실패를 "모델의 희소 주의 한계"로 귀인하는 것 자체가 [[concepts/bottleneck-misattribution.md|bottleneck misattribution]]일 수 있습니다. [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]에서 이미 "병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정됨"을 보여주어 병목의 진짜 위치를 재정의한 바 있습니다. ReContext는 이를 한 단계 더 나아가: 병목의 **위치**뿐 아니라 병목의 **해결 주체**까지 재할당합니다. "모델이 고쳐야 할 문제"를 "하네스가 보상해야 할 문제"로 재귀인하는 것입니다.

**③ [[concepts/cache-coherence-constrained-optimization.md|cache coherence constrained optimization]]로의 확장 가능성**
ReContext의 증거 재생을 "하네스가 모델을 위해 외부 캐시를 유지하고, 모델의 내부 주의 캐시가 실패할 때 이를 주입하는 메커니즘"으로 읽을 수 있습니다. 이 경우 하네스와 모델 사이의 **캐시 일관성(cache coherence)** 문제가 발생합니다—하네스가 재생한 증거와 모델이 현재 컨텍스트에서 이미 주의한 증거 사이에 중복·충돌·순서 불일치가 생길 수 있으며, [[concepts/cache-phase-preservation.md|cache phase preservation]]의 관점에서 재생 시점의 위상(phase)이 추론의 일관성에 미치는 영향도 분석 대상이 됩니다.

## 6. 결론: 경계 대신 공동 결정 구조로

ReContext는 하네스와 모델 능력 사이의 경계를 "그어야 할 선"에서 **"공동으로 결정하는 구조"**로 대체해야 함을 보여줍니다. Crab이 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]에서 드러낸 "의미론적 의존 그래프를 물리적 스냅샷으로 번역하는 비효율"과 ReContext가 드러낸 "주의 실패를 증거 재생으로 번역하는 보상"은 동전의 양면입니다. 둘 다 **하네스가 알고리즘의 의미론을 번역하는 과정에서 경계가 붕괴**된다는 같은 구조적 패턴을 보여줍니다.

따라서 Wiki의 향후 업데이트에서는 [[concepts/algorithm-system-boundary-collapse.md|algorithm system boundary collapse]]를 중심 엔티티로 삼아, [[entities/agentic-harness-engineering.md|agentic harness engineering]], [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]], [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]], [[concepts/adaptive-inference.md|adaptive inference]], 그리고 ReContext를 하나의 연속체—**"하네스의 점진적 인지화" 스펙트럼**—으로 재구성할 필요가 있습니다.

## 핵심 통찰

ReContext는 하네스가 '어떤 증거를 재생할지'를 의미론적으로 선택하는 순간, 형식적 인프라 동작과 실질적 인지 동작의 구분이 불가능해지는 회색지대를 생성하며, 이것이 정확히 algorithm-system-boundary-collapse의 본질이다.
