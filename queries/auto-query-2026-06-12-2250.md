## 1. 하네스 엔지니어링이 이미 '암묵적 경계'를 확장해 온 궤적

[[agentic-harness-engineering]]은 이미 한 번 확장된 적이 있다. Crab 논문은 C/R 런타임이 하네스의 핵심 구성요소(결함 내성·롤백·분기)를 직접 구현하는 인프라 계층이며, 이로써 하네스 엔지니어링의 범위가 **애플리케이션 로직에서 실행 환경까지** 확장됨을 명시했다.

여기서 핵심은 **"확장의 메커니즘이 무엇이었는가"**이다. 확장은 새로운 기능의 추가가 아니었고, 기존에 '하네스 밖'이라고 가정되었던 것이 사실은 하네스의 동작을 구조적으로 결정하고 있었다는 발견이었다.

## 2. Claw-SWE-Bench가 드러내는 것: 평가 계약 = 암묵적 하네스

Claw-SWE-Bench의 핵심 발견—"a generic agent does not by itself satisfy the [evaluation contract]"—는 정확히 같은 패턴의 반복이다.

SWE-bench의 평가 계약(워크스페이스 구조, 패치 적용 방식, 예측 포맷)은 중립적인 측정 도구가 아니다. 이것은:
- 에이전트가 파일 시스템을 어떻게 탐색할 수 있는지(워크스페이스 경계)
- 상태를 어떻게 축적하고 제출할 수 있는지(패치 포맷)
- 어디서 중단하고 재개할 수 있는지(체크포인트 의미론)

을 **구조적으로 규정**한다. 즉, 평가 계약은 이름만 '벤치마크'일 뿐, 실질적으로는 [[agentic-harness-engineering]]의 대상인 하네스이다.

## 3. 두 확장을 잇는 공통 간극: [[algorithm-system-translation-gap]]

Crab이 [[algorithm-system-translation-gap]]을 구체적 런타임 문제로 현현시켰듯—에이전트의 의미론적 의존 그래프를 OS의 전체 스냅샷 메커니즘으로 번역할 때 발생하는 비효율—Claw-SWE-Bench는 동일한 간극을 **측정 영역**에서 현현시킨다.

- **Crab의 간극**: "에이전트가 의존하는 상태" ↔ "OS가 체크포인트로 포착하는 상태"
- **Claw-SWE-Bench의 간극**: "벤치마크가 측정하려는 능력" ↔ "평가 계약이 실제로 요구하는 아키텍처 적합성"

두 경우 모두, **번역 과정에서 의도치 않은 제약이 삽입**된다. Crab에서는 의미 없는 상태까지 스냅샷되는 비효율이, Claw-SWE-Bench에서는 계약에 맞지 않는 아키텍처의 능력이 누락되는 왜곡이 발생한다.

## 4. 오인정의의 병행: [[aggregate-pipeline-serving]]의 교훈

[[aggregate-pipeline-serving]]과 관련된 Tool Attention 논문은 다중 서버 MCP 배포에서 **파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정됨**을 보였다. "실제 병목 지점을 재정의"한 것이다.

이와 완전히 동일한 오인정의가 벤치마크 영역에서 발생한다:
- **파이프라인의 오인**: 병목 = LLM 호출 → 실제 병목 = 스키마 축적
- **벤치마크의 오인**: 점수 = 에이전트 능력 → 실제 결정요인 = 평가 계약과의 아키텍처 적합성

Claw-SWE-Bench는 파이프라인 서빙이 '개별 컴포넌트 최적화'가 아닌 '파이프라인 전체 구조'로 분석되어야 했듯, 벤치마크도 '외부 태스크 난이도'가 아닌 '평가 계약 구조'로 분석되어야 함을 보여준다.

## 5. 통합의 구체적 의미: 하네스 스펙트럼의 완성

통합한다는 것은 하네스 엔지니어링을 다음 스펙트럼으로 재정의하는 것이다:

```
[애플리케이션 로직]  →  [실행 환경]  →  [평가 계약]
     ← Crab의 확장 →        ← Claw-SWE-Bench의 확장 →
```

각 단계는 독립된 층이 아니라, **에이전트의 의도된 동작을 구조적으로 제약하는 번역 계층**이다. [[agent-os-semantic-gap]]이 OS 수준에서, [[algorithm-system-translation-gap]]이 런타임 수준에서, 그리고 평가 계약이 측정 수준에서 동일한 패턴의 간극을 생성한다.

## 6. 새로운 개념 연결

이 분석에서 다음 연결이 새롭게 발견된다:

**[[benchmark-specification-gap]]** ← 기존 슬러그 목록에 존재하지만 Wiki에 정의가 축적되지 않은 상태. Claw-SWE-Bench의 발견은 이 간극을 정확히 채우는 내용이다: *벤치마크 명세가 의도하는 측정 대상과, 평가 계약의 구조적 제약이 실제로 측정하는 대상 사이의 간극*. 이것이 [[algorithm-system-translation-gap]]의 측정 영역 특화 버전이다.

또한 **[[code-as-agent-harness]]**와의 연결이 암시된다. SWE-bench에서 평가 계약의 핵심이 패치 포맷과 워크스페이스 구조라면, 이는 곧 "코드 자체가 하네스의 일부로 기능한다"는 [[code-as-agent-harness]]의 명제를 벤치마크 영역에서 재확인하는 것이다.

## 요약

Claw-SWE-Bench는 평가 계약이 암묵적 하네스로 작동함을 보여줄 뿐 아니라, 이것이 [[agentic-harness-engineering]]이 이미 Crab에서 보여준 확장 패턴—'밖이라고 가정된 것이 사실은 안을 구조적으로 결정하고 있었다'—의 **정확한 반복**임을 보여준다. [[algorithm-system-translation-gap]]이 런타임과 측정 양쪽에서 동일하게 현현하며, [[aggregate-pipeline-serving]]의 오인정의 교훈이 벤치마크 영역에도 적용된다.

따라서 평가 계약 설계를 하네스 엔지니어링에 통합하는 것은 확장이 아니라, **이미 하나의 간극이었던 것을 명시적으로 인정하는 것**이다. 그리고 이 통합을 위해 [[benchmark-specification-gap]]의 구체적 정의 축적이 시급한 과제로 남는다.

## 핵심 통찰

평가 계약은 외부의 중립적 측정 도구가 아니라 기존 하네스 엔지니어링의 연장선에 있는 암묵적 하네스이며, 벤치마크 점수는 에이전트의 고유 능력이 아닌 이 평가 계약과의 아키텍처 적합성에 의해 결정된다.
