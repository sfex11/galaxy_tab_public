# 도구 호출 병목의 재정의: 스키마 축적에서 시각 지각-도구 선택 간극으로

질문하신 간극은 Wiki의 기존 진단과 최근 연구 사이에 존재하는 **병목의 차원 이동**을 정확히 포착하고 있습니다.

## 1. 기존 진단: 텍스트 인터페이스에서의 병목

[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]에 연결된 *"Tool Attention Is All You Need"* 논문은 다중 서버 MCP 배포에서 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 **스키마 축적(schema accumulation)**에 의해 결정됨을 보여줍니다. 즉, 도구 호출의 한계를 "얼마나 많은 도구 스키마를 효율적으로 필터링하고 로드할 수 있는가"라는 **텍스트 기반 인터페이스 최적화** 문제로 진단합니다.

이 관점에서 도구 호출 능력의 병목은 본질적으로 **정보 검색·라우팅 문제**입니다.

## 2. MM-ToolSandBox가 드러내는 병목의 차원 이동

최근 논문 *MM-ToolSandBox*는 500개 이상의 도구와 16개 애플리케이션 도메인을 아우르는 **상태유지(stateful) 실행 환경**에서 시각적으로 기반된 도구 호출 에이전트를 평가합니다. 핵심은 "시각적 증거가 점진적으로 도착하는" 상황이라는 점인데, 이는 병목의 성질을 근본적으로 바꿉니다.

- 스키마 축적 문제: 정적이고 텍스트 기반 → 도구 목록을 미리 알고 있음
- MM-ToolSandBox의 문제: 동적이고 시각 기반 → **어떤 도구가 필요한지는 아직 도착하지 않은 시각적 증거에 의존**

여기서 병목은 인터페이스 최적화가 아니라, **에이전트가 현재까지 도착한 불완전한 시각 정보를 바탕으로 도구를 선택해야 하는지, 아니면 더 많은 시각 증거를 수집해야 하는지를 판단하는 조합적 의사결정**으로 이동합니다.

## 3. [[concepts/agentic-vlm.md|agentic vlm]]이 이미 암시한 간극의 구조

[[concepts/agentic-vlm.md|agentic vlm]]에 연결된 *Three-Step Nav* 논문은 제로샷 VLN에서 MLLM 기반 에이전트가 경로 이탈·조기 정지라는 **구조적 실패**를 보인다고 실증합니다. 논문은 이를 "'단계별 판단'과 '다단계 경로 기획' 사이에 존재하는 간극"으로 구체화합니다.

이 간극을 도구 호출 맥락으로 번역하면:

> **단계별 시각 인식**(현재 프레임에서 무엇이 보이는가)과 **다단계 도구 선택 기획**(이 시각적 상태를 해결하기 위해 어떤 도구 시퀀스가 필요한가) 사이의 간극

MM-ToolSandBox의 상태유지 환경에서 이 간극은 더욱 심화됩니다. 시각적 증거가 점진적으로 도착하기 때문에, 에이전트는 매 단계에서 "지금 도구를 호출할 것인가, 아니면 더 관찰할 것인가"라는 **인식-행동 타이밍의 조합적 결정**을 내려야 합니다.

## 4. [[concepts/blind-tool-invocation.md|blind tool invocation]]과의 연결: 간극의 증상

이러한 조합적 간극이 발생할 때 나타나는 구체적 증상이 바로 [[concepts/blind-tool-invocation.md|blind tool invocation]]일 가능성이 높습니다. 텍스트 기반 파이프라인에서의 blind invocation은 스키마 과부하나 부정확한 라우팅 때문이지만, 시각 기반 환경에서의 blind invocation은 **불완전한 시각 지각 상태에서 도구를 선택했다는 점**이 근본 원인입니다.

[[concepts/agent-scene-instruction-triad.md|agent scene instruction triad]]와 연결된 *AwareVLN*의 자기인식(self-awareness) 프레임워크도 이 맥락을 지원합니다. 에이전트가 자신의 인식 상태(지금 얼마나 잘 보고 있는가)를 메타인지하지 못하면, 부족한 시각 증거 위에 도구 선택을 쌓아 올리게 됩니다.

## 5. 새로운 개념적 연결 발견

질문을 통해 다음 연결이 드러납니다:

| 기존 진단 (Wiki) | 새로운 진단 (질문+MM-ToolSandBox) |
|---|---|
| [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]: 스키마 축적이 병목 | 시각 지각 불완전성이 병목 |
| 해결책: 동적 스키마 게이팅, 레이지 로딩 | 해결책: 인식-행동 타이밍 판단 |
| 도구 선택 = 텍스트 매칭 문제 | 도구 선택 = 시각 상태 추론 문제 |

**핵심 발견**: [[concepts/agentic-vlm.md|agentic vlm]]이 VLN 도메인에서 식별한 "단계별 판단 vs. 다단계 기획" 간극이, MM-ToolSandBox의 시각적 도구 호출 환경에서는 **"부분적 시각 인식 vs. 전체적 도구 시퀀스 기획"**이라는 더 복잡한 조합적 간극으로 변형된다는 점입니다. [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 스키마 최적화는 이 간극을 해결하지 못하며, 오히려 스키마 로딩이 빨라질수록 에이전트가 "도구는 쓸 수 있지만 언제 써야 할지 모르는" 상태에 빠질 위험이 존재합니다.

결론적으로, 질문하신 **"인터페이스 최적화가 아닌 시각 지각과 도구 선택의 조합적 간극"**이라는 진단은 Wiki의 기존 엔티티들([[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]], [[concepts/agentic-vlm.md|agentic vlm]], [[concepts/blind-tool-invocation.md|blind tool invocation]], [[concepts/agent-scene-instruction-triad.md|agent scene instruction triad]])을 종합할 때 자연스럽게 도출되는 간극이며, MM-ToolSandBox가 이를 실험적으로 검증할 수 있는 환경을 제공한다고 볼 수 있습니다.

## 핵심 통찰

VLN 도메인의 '단계별 판단 vs 다단계 기획' 간극이 시각적 도구 호출 환경에서는 '부분적 시각 인식 vs 전체적 도구 시퀀스 기획'이라는 조합적 간극으로 변형되며, 스키마 최적화는 이 간극을 해결하지 못하고 오히려 blind invocation을 유발할 수 있다.
