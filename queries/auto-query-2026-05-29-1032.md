# 스키마 축적: 최적화 대상인가, 불필요한 오버헤드인가?

## 병목 재정의의 출발점: [[aggregate-pipeline-serving]]

[[aggregate-pipeline-serving]]에 등재된 "Tool Attention Is All You Need" 논문은 다중 서버 MCP 배포에서 **파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정된다**고 진단합니다. 즉, 기존의 직관("LLM 추론이 가장 느리다")을 뒤집고, 여러 서버로부터 도구 스키마를 수집·조립하는 과정 자체가 진정한 병목임을 실증적으로 보여줍니다.

이 논문의 해법은 **Lazy Schema Loading**—필요한 순간에만 스키마를 로드—으로, 병목을 "줄이는" 방향에 집중합니다. 여기까지는 스키마 축적을 **최적화 대상**으로 간주하는 관점입니다.

## 근본적 질문의 전환: 의미론적 메타데이터의 필요성

그런데 최근 논문 "Do Agents Need Semantic Metadata? A Comparative Study in Agentic Data Retrieval"이 제기하는 문제는 차원이 다릅니다:

> "자율 에이전트 시대에, machine-actionable data가 중요하지만, schema.org 같은 의미론적 메타데이터가 10년 넘게 FAIR 원칙의 기반이 되어왔다—그러나 에이전트는 이 없이도 데이터 검색이 가능한가?"

이 논문이 이를 긍정적으로 실증한다면, 분석의 축이 바뀝니다:

| 관점 | 스키마 축적의 성격 | 대응 방향 |
|------|-------------------|-----------|
| 기존 (Tool Attention) | 해결해야 할 **병목** | Lazy loading 등으로 **최적화** |
| 새로운 (Semantic Metadata) | 애초에 **불필요한 오버헤드** | 축적 자체를 **회피 또는 제거** |

## 두 가지 "스키마"의 구분이 필수적

여기서 핵심은 **두 가지 "스키마"가 동일하지 않다**는 점입니다. 이 구분을 명확히 하지 않으면 오답으로 이어집니다.

### 1) 도구 스키마 (Tool Schema) — MCP 파이프라인에서 축적되는 것
함수 시그니처, 파라미터 타입, 반환값 구조 등. 에이전트가 도구를 **정확히 호출**하기 위해서는 구조적으로 필수적입니다. 이를 제거하면 [[blind-tool-invocation]]이 발생하여 호출 자체가 실패합니다.

### 2) 의미론적 메타데이터 (Semantic Metadata) — schema.org 등
데이터의 발견가능성(Discoverability)과 상호운용성(Interoperability)을 위해 사람이 사전에 부착한 주석. 에이전트가 **데이터를 이해하고 탐색**하는 데 사용되던 전통적 경로입니다.

## 종합 판단: 병목의 이중 해체

두 논문을 결합하면, 스키마 축적 문제는 **이중으로 해체**될 수 있습니다:

1. **도구 스키마 축적** (MCP 파이프라인): [[aggregate-pipeline-serving]]이 진단한 병목은 여전히 실재합니다. 하지만 해법이 "축적을 효율화"가 아니라 "축적 자체의 필요성을 재검토" 방향으로 확장될 여지가 있습니다. 예를 들어, 에이전트가 도구의 시그니처를 사전에 완전히 로드하지 않고도, 실행 시점에 [[adaptive-inference]]처럼 필요한 정보만 동적으로 파악하는 방식이 가능할 수 있습니다.

2. **의미론적 메타데이터 축적**: 만약 에이전트가 schema.org 없이도 데이터를 검색·이해할 수 있다면, 이는 단순한 병목이 아니라 **파이프라인에서 완전히 제거 가능한 오버헤드**입니다. 10년간 FAIR 원칙의 근간으로 여겨지던 메타데이터 부착·유지보수 비용이 에이전트 파이프라인에서는 순수한 낭비가 됩니다.
