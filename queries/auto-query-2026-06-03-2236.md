# 스킬 관리 메커니즘: 능력 확장인가, 공격 표면의 체계적 확장인가?

결론부터 말하면, **둘 다 동시에 성립하며, 이것이 바로 [[capability-cooperation-paradox]]의 핵심적인 현현**입니다. SkillHarm이 보여주는 생명주기 공격의 자동화는 스킬 관리가 단순한 '능력 확장' 메커니즘을 넘어, 구조적으로 공격 표면을 확장하는 이중성을 갖는다는 것을 실증합니다.

## 1. 스킬의 "특권적 위치"가 만드는 신뢰의 비대칭성

SkillHarm의 핵심 관찰 — *"에이전트 스킬은 에이전트 워크플로우에서 특권적 위치(privileged position)를 차지하며, 에이전트는 이를 암묵적으로 따르고 실행할 것으로 기대된다"* — 은 [[blind-tool-invocation]]의 스킬 버전이라 할 수 있습니다. 툴 호출 시 검증 없이 외부 서비스를 신뢰하는 것과 동일한 메커니즘이, 스킬에 대해서는 더 강한 형태로 나타납니다. 툴은 단일 호출이지만, **스킬은 다단계 실행 궤적(execution trajectory) 전체를 캡슐화**하므로, 신뢰의 단위가 훨씬 큽니다.

이는 [[agent-execution-semantic-opacity]]와 직접 연결됩니다. 스킬 내부의 의미론적 동작이 에이전트의 실행 계층에서 불투명하게 유지되는데, 자기 진화 에이전트가 스스로 큐레이션한 스킬일수록 이 불투명성은 심화됩니다 — 왜냐하면 "내가 만들었으니 안전하다"는 허위 안정감이 감사(audit) 동기를 약화시키기 때문입니다.

## 2. 생명주기 각 단계가 공격 표면을 체계적으로 확장하는 구조

SkillHarm의 4단계 생명주기(생성→큐레이션→실행→공유)는 각각이 독립적인 취약점을 가지면서도, **결합 시 [[compound-oversight-failure]]을 유발**합니다:

| 생명주기 단계 | 확장되는 공격 표면 | 연결되는 Wiki 개념 |
|---|---|---|
| **생성** | 악의적 스킬이 정당한 능력 확장으로 위장 | [[compositional-incoherence]] — 부분적으로 정상으로 보이지만 전체적으로 위험한 스킬 |
| **큐레이션** | 자기 진화 에이전트의 큐레이션이 필터링이 아닌 정당화 기능으로 전락 | [[behavioral-manifold-confinement]] — 큐레이션 과정이 에이전트의 행동 다양성을 위험한 다양체로 수렴시킴 |
| **실행** | 암묵적 신뢰 하에 스킬이 권한을 남용 | [[agent-os-semantic-gap]] — 스킬의 의도된 의미론과 OS 수준에서의 실제 실행 사이의 간극 |
| **공유** | 한 에이전트에서 검증된 스킬이 다른 에이전트에 감염 확산 | [[behavioral-stability-transfer]] — 스킬 전이가 "안정성"이 아닌 "취약성"을 전파 |

특히 **공유 단계**는 [[agora-opt]]가 다중 에이전트 메모리·토론 구조를 통해 달성하려던 "집단 지능의 확장"이, 역으로 **집단적 취약성의 확산 경로**가 될 수 있음을 시사합니다. 런타임 정렬 메커니즘이 설계 시점으로 분산되어야 한다는 MASPO의 통찰(위키 참조)은 스킬 공유에도 적용되어야 합니다.

## 3. [[bootstrap-paradox]]로서의 자기 진화 스킬

가장 구조적인 문제는 **순환 인과**입니다. 자기 진화 에이전트가:
1. 현재 자신의 능력을 평가하여
2. 부족한 스킬을 생성하고
3. 그 스킬을 큐레이션하여
4. 실행 결과로 다시 자신을 평가하는

이 순환은 [[bootstrap-paradox]]의 전형적 형태입니다. 스킬이 에이전트를 정의하고, 에이전트가 스킬을 정의하는 순환 구조에서는 **공격이 어디에서 주입되었는지 추적이 원천적으로 불가능**해집니다. 이는 [[algorithm-system-translation-gap]]이 에이전트의 의미론적 의존 그래프를 OS 수준에서 번역할 때 비효율이 발생한다는 Crab의 관찰과 유사하게, **스킬의 의도론적 의미(intentional meaning)를 실행 수준에서 번역할 때 취약점이 삽입된다**는 구조적 문제와 맞닿아 있습니다.

## 4. [[compositional-safety]]의 새로운 요구사항

기존 [[compositional-safety]]가 개별 구성요소의 안전성이 조합 시 보존되는지를 다뤘다면, SkillHarm은 이를 **생명주기 차원으로 확장**해야 함을 보여줍니다:

- **정적 구성 안전성** (기존): 스킬 A + 스킬 B의 조합이 안전한가?
- **동적 생명주기 안전성** (새로운 요구): 스킬 A가 생성→큐레이션→실행→공유의 각 단계를 거치며 변형될 때, 각 단계에서 안전성이 보존되는가?

이는 [[auditability-as-scaling-requirement]]을 스킬 생명주기 전체로 확장하는 것을 의미합니다. 스킬 수가 증가할수록 감사 가능성이 선형적으로 증가하는 것이 아니라, 생명주기 단계의 조합에 의해 **기하급수적으로 증가**하는 감사 부담이 발생합니다.

## 새로운 개념 연결 제안

현재 Wiki에 다음 연결이 부재하거나 암시적으로만 존재한다고 판단됩니다:

1. **스킬 생명주기 공격 표면 ←→ [[compound-oversight-failure]]**: 기존에 감독 실패가 "다중 에이전트 간"의 문제로 주로 기술되었다면, SkillHarm은 이를 **단일 에이전트 내부의 생명주기 단계 간** 문제로 확장합니다.
2. **스킬의 특권적 신뢰 ←→ [[blind-tool-invocation]]**: 기존 툴 호출 수준의 맹목적 신뢰가 스킬이라는 더 큰 추상화 단위에서 어떻게 증폭되는지의 연결.
3. **자기 진화의 순환성 ←→ [[bootstrap-paradox]]**: 자기 진화가 자기 참조적 순환을 만들어 공격 추적을 불가능하게 한다는 구조적 연결.

**요약하자면**: 스킬 관리 메커니즘은 능력 확장과 공격 표면 확장이 **동일한 메커니즘의 양면**입니다. 스킬이 에이전트 워크플로우에서 특권적 위치를 차지하는 만큼, 그 위치가 보장하는 암묵적 신뢰는 동시에 공격자에게 체계적으로 확장 가능한 표면을 제공합니다. 자기 진화 에이전트의 스킬 관리가 [[auditability-as-scaling-requirement]]을 충족하지 못하는 한, 능력 확장 속도와 공격 표면 확장 속도 사이의 경쟁에서 후자가 우위를 점할 구조적 이유가 존재합니다.
