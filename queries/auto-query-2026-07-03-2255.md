# 성능 최적화 벤치마크가 "에이전트 발전"을 잘못 서사화하는 구조

## 1. 문제의 출발점: 벤치마크 아티팩트에 대한 과적합

최근 논문 **"Are Performance-Optimization Benchmarks Reliably Measuring Coding Agents?"**은 GSO, SWE-Perf, SWE-fficiency 등 리포지토리 수준의 성능 최적화 벤치마크가 코딩 에이전트를 평가하는 방식의 근본적 결함을 지적합니다. 이 벤치마크들은 실제 리포지토리에 패치를 적용하고 런타임을 비교하는 방식으로 작동하는데, 이것이 측정하는 것은 에이전트의 **일반적 코딩 능력**이 아니라 **해당 벤치마크 특정 아티팩트에 대한 최적화**일 가능성이 높습니다.

이는 Wiki의 기존 개념인 [[concepts/artifact-bound-optimization.md|artifact bound optimization]]과 정확히 직결됩니다. 에이전트가 벤치마크 리포지토리라는 특정 아티팩트의 구조(빌드 시스템, 벤치마크 실행 스크립트, 프로파일링 출력 형식 등)에 과적합하면, 점수는 상승하지만 실제 배포 환경에서의 코딩 능력은 향상되지 않는 [[concepts/ceiling-performance-problem.md|ceiling performance problem]]에 직면하게 됩니다.

## 2. 평가 패러다임의 잘못된 서사화 기제

### 2.1 점수 상승 → 능력 향상의 허위 동등화

리더보드 점수의 상승을 곧바로 "에이전트 발전"으로 서사화하는 것은 [[concepts/circular-validity-problem.md|circular validity problem]]의 전형적 사례입니다. 벤치마크가 측정하려는 것(일반적 코딩 역량)과 실제로 측정하는 것(벤치마크 특정 아티팩트에 대한 패치 생성 능력) 사이에 [[concepts/benchmark-specification-gap.md|benchmark specification gap]]이 존재하는데, 이 간극이 점수 상승의 서사에 의해 은폐됩니다.

특히 성능 최적화 벤치마크는 일반적인 기능 구현 코딩과 달리, **특정 런타임 환경에서의 성능 프로파일링 결과를 읽고 병목을 식별하는 능력**을 요구합니다. 이는 [[concepts/benchmark-format-blindspot.md|benchmark format blindspot]]의 특수한 형태로, 벤치마크의 입출력 형식(프로파일링 출력, 벤치마크 실행기 인터페이스)에 익숙한 에이전트가 실제 코딩 역량이 뛰어난 에이전트보다 높은 점수를 받을 수 있는 구조적 왜곡을 만듭니다.

### 2.2 [[concepts/codebase-dependency-contamination.md|codebase dependency contamination]]의 가속

성능 최적화 벤치마크는 특정 리포지토리의 코드베이스 구조에 깊이 의존합니다. 에이전트가 학습 데이터나 평가 과정에서 해당 리포지토리의 내부 구조(함수 호출 그래프, 모듈 의존성, 최적화 가능한 핫패스 위치 등)에 대한 사전 지식을 갖게 되면, 이는 [[concepts/codebase-dependency-contamination.md|codebase dependency contamination]]으로 작동하여 점수를 인플레이트시킵니다. 이 경우 리더보드는 "에이전트의 발전"이 아니라 "에이전트의 벤치마크 지식 축적"을 측정하게 됩니다.

## 3. 구조적 한계: 왜 기존 패러다임이 수정되기 어려운가

### 3.1 [[concepts/co-failure-ceiling.md|co failure ceiling]]의 위장

여러 에이전트가 동시에 동일한 벤치마크 아티팩트에 과적합할 경우, [[concepts/co-failure-ceiling.md|co failure ceiling]]이 발생합니다. 모든 에이전트가 벤치마크 특정 병목(예: 특정 프로파일링 툴의 출력 파싱, 특정 빌드 시스템의 플래그 세팅)에서 비슷한 패턴의 성공/실패를 보이면, 상대적 순위 변화는 미미해지고 점대점 비교가 무의미해집니다. 그러나 리더보드의 절대점수 상승은 "전체적인 발전"으로 서사화되어, 실제로는 아무도 벤치마크 외부에서 검증되지 않았다는 사실이 가려집니다.

### 3.2 [[concepts/attractor-state-evaluation-confound.md|attractor state evaluation confound]]의 고착화

성능 최적화라는 특정 평가 차원이 [[concepts/attractor-state-evaluation-confound.md|attractor state evaluation confound]]으로 기능할 수 있습니다. 에이전트 개발 커뮤니티가 "GSO 점수 = 코딩 에이전트 역량"이라는 암묵적 등식을 수용하면, 연구 방향이 점진적으로 이 흡인 상태(attractor)로 수렴합니다. 결과적으로 벤치마크 외부의 코딩 능력(보안 취약점 식별, 아키텍처 설계, 크로스 플랫폼 호환성 등)은 평가되지 않은 채 방치됩니다.

## 4. 새로운 개념적 연결 발견

이 분석을 통해 다음과 같은 연결이 드러납니다:

**[[concepts/artifact-bound-optimization.md|artifact bound optimization]] → [[concepts/codebase-dependency-contamination.md|codebase dependency contamination]] → [[concepts/circular-validity-problem.md|circular validity problem]]의 인과 사슬**: 벤치마크 아티팩트에 최적화된 에이전트가 코드베이스 의존성에 오염되고, 이것이 순환적 타당성 문제로 이어지는 3단계 붕괴 경로입니다. 기존에 이 세 엔티티가 개별적으로 존재했지만, 성능 최적화 벤치마크라는 구체적 맥락에서 이들이 **인과적으로 연쇄**됨을 보여줍니다.

**[[concepts/benchmark-format-blindspot.md|benchmark format blindspot]]과 [[concepts/benchmark-specification-gap.md|benchmark specification gap]]의 변별**: 전자는 벤치마크의 입출력 형식에 대한 편향(프로파일링 출력 읽기 능력 ≠ 최적화 코딩 능력), 후자는 벤치마크가 명세하려는 것과 실제로 측정하는 것의 괴리(일반적 코딩 역량 vs. 특정 리포지토리 패치 능력)로, 성능 최적화 벤치마크에서 두 가지가 **동시에** 작동하며 상승 작용을 일으킵니다.

**[[concepts/ceiling-performance-problem.md|ceiling performance problem]]의 정치적 차원**: 기술적 한계인 천장 효과가, 리더보드 기반의 "에이전트 발전" 서사를 유지하기 위한 **인센티브 구조**와 결합할 때 단순한 측정 문제를 넘어 연구 방향의 왜곡으로 확대됩니다. 점수가 천장에 가까워지면 미세한 점수 차이를 위해 점점 더 벤치마크 특화적인 최적화가 유도되는 악순환입니다.

## 5. 결론: 서사화의 오차 크기

기존 평가 패러다임이 "에이전트 발전"을 잘못 서사화하는 정도는 **구조적**입니다. 단순히 노이즈가 있는 측정이 아니라, 측정 대상 자체가 왜곡되어 있습니다. 리더보드 점수가 상승하는 것은 "코딩 에이전트가 더 나아지고 있다"가 아니라 **"벤치마크 아티팩트에 대한 최적화가 심화되고 있다"**는 서사로 재해독되어야 합니다. 이 재해독 없이는 SWE-bench, GSO 등의 리더보드가 에이전트 발전의 지표가 아니라 [[concepts/artifact-bound-optimization.md|artifact bound optimization]]의 지표로 전락할 위험이 있습니다.

## 핵심 통찰

성능 최적화 벤치마크의 근본적 결함은 측정 오차가 아니라, 점수 상승을 '발전'으로 서사화하는 구조적 메커니즘 자체가 평가 대상을 대체한다는 점이다.
