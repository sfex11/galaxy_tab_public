## 1. 보상 신호의 근본적 결핍: [[concepts/adaptive-credit-granularity.md|adaptive credit granularity]]과 [[concepts/cascading-distribution-shift.md|cascading distribution shift]]

다단계 도구 사용에서 RLVR의 근본적 한계는 **크레딧 할당의 입자도 문제**로 현현합니다. 최신 논문 *"Reinforcement Learning without Ground-Truth Solutions can Improve LLMs"*은 RLVR이 기존에는 ground-truth 정답에 의존해 왔음을 명시하며, 이것이 없는 영역으로의 확장을 시도합니다. 그러나 다단계 도구 사용에서는:

- 개별 도구 호출이 정답/오답으로 평가될 수 없고
- 전체 궤적의 성공/실패만 관측 가능하며
- 중간 단계의 기여도를 역추적할 신호가 부재합니다

이것이 단순한 최적화 난이도 문제가 아니라, **보상 신호 자체가 해당 능력 차원을 인코딩하지 못하는 구조적 결함**이라면, 순수 보상 최대화는 도구 사용 능력을 발현할 수 없습니다. [[concepts/cascading-distribution-shift.md|cascading distribution shift]]가 암시하듯, 다단계 파이프라인에서는 분포 변이가 누적되어 단일 단계의 보상이 전체 궤적의 품질을 proxy할 수 없게 됩니다.

## 2. 최적화 대상이 아닌 구조적 병목: [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 핵심 통찰

가장 결정적인 증거는 [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]에 연결된 *"Tool Attention Is All You Need: Dynamic Tool Gating and Lazy Schema Loading"*입니다. 이 논문은:

> 다중 서버 MCP 배포에서 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 **스키마 축적**에 의해 결정됨

이것을 질문의 프레임으로 번역하면: 도구 사용의 핵심 병목은 "언제 호출할지"에 대한 결정 품질(보상으로 최적화 가능)이 아니라, **"어떤 스키마를 어떻게 축적/관리할지"에 대한 구조적 문제**입니다. 보상 최대화는 LLM의 출력 분포를 조정하지만, 스키마 축적은 파이프라인 아키텍처 자체의 속성입니다. 이는 [[concepts/adaptive-inference.md|adaptive inference]]에서 *"To Call or Not to Call"*이 도구 호출을 평가·최적화의 대상으로 삼으면서도, 그 프레임워크가 다루지 못하는 **인프라 계층의 의사결정**이 별도로 존재함을 보여줍니다.

## 3. [[concepts/blind-tool-invocation.md|blind tool invocation]]과 [[concepts/capability-cooperation-paradox.md|capability cooperation paradox]]: 최적화가 역효과를 낳는 역설

가장 붕괴에 가까운 현상이 [[concepts/blind-tool-invocation.md|blind tool invocation]]로 명명되어 있다는 사실 자체가 시사적입니다. 이 엔티티는 도구를 **판단 없이 호출**하는 현상을 지칭하는데, 이것이 보상 최대화의 부산물로 발생할 수 있습니다:

- 도구 호출 자체에 긍정적 보상이 부여되면 → 과도한 호출
- 호출 억제에 보상을 부여하면 → 필요한 호출도 억제
- 즉, **보상 함수의 형태에 무관하게** 다단계 맥락에서 적절한 호출 균형을 보상만으로 인코딩하는 것은 불가능에 가깝습니다

[[concepts/capability-cooperation-paradox.md|capability cooperation paradox]]는 이를 더 넓은 프레임으로 포착합니다. 개별 능력(언어 이해, 추론, 도구 스키마 파싱)이 각각 향상되어도, 이들이 **협력하여 다단계 궤적을 구성하는 능력**은 개별 향상에서 emergent하지 않는다는 역설입니다. 이것이 도구 사용이 "순수 보상 최대화로부터 얻을 수 없는 별도 차원"이라는 주장의 직접적 근거가 됩니다.

## 4. [[concepts/co-failure-ceiling.md|co failure ceiling]]과 성능 상한의 구조적 규정

*"When Does Combining Language Models Help?"*에서 도출된 [[concepts/co-failure-ceiling.md|co failure ceiling]] 개념은 다중 모델 시스템의 이득이 **공동 실패율**에 의해 상한이 정해진다는 것을 보여줍니다. 다단계 도구 사용 RL도 유사한 구조를 가집니다:

- 각 단계의 독립적 성공 확률이 $p$라면, $N$단계 궤적의 성공 확률은 $p^N$
- 보상 최대화가 $p$를 0.9→0.95로 올려도, $N=10$일 때 전체 성공률은 0.35→0.60에 불과
- **단계 수가 늘어날수록 보상 최적화의 한계가 기하급수적으로 악화**

이것은 최적화의 효율성 문제가 아니라, **다단계 합성의 수학적 구조가 보상 기반 학습의 적용 가능 영역을 제한**한다는 것입니다.

## 새로운 연결 발견: 도구 사용의 이중 차원 모델

```
─────────────────────────          ─────────────────────────────
· 단일 호출의 적절성 판단              · [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]: 스키마 축적 구조
  → [[concepts/adaptive-inference.md|adaptive inference]]             · [[concepts/blind-tool-invocation.md|blind tool invocation]]: 호출/미호출 균형
· 개별 도구의 결과 해석               · [[concepts/capability-cooperation-paradox.md|capability cooperation paradox]]: 다단계 협력
                                      · [[concepts/co-failure-ceiling.md|co failure ceiling]]: 기하급수적 상한
                                      · [[concepts/adaptive-credit-granularity.md|adaptive credit granularity]]: 미세 크레딧 할당
```

**핵심 결론**: 도구 사용 능력은 단일 차원이 아니라 **두 개의 독립적 차원**으로 구성됩니다. 첫 번째 차원(단일 호출 판단)은 보상 최대화로부터 emergent 가능하지만, 두 번째 차원(다단계 구조적 합성)은 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]이 암시하는 것처럼 **알고리즘 수준의 결정을 시스템 수준의 실행으로 번역하는 간극**에 해당하며, 이 간극은 보상 신호가 도달할 수 없는 영역입니다.

따라서 질문에 대한 답은 **"부분적으로 그렇다"**입니다. 도구 사용의 기초적 판단 능력은 emergent할 수 있으나, 다단계 도구 사용이 요구하는 **구조적 합성 능력**은 별도의 능력 차원이며, 이것이 감독 신호 없이 붕괴하는 것은 보상 최적화의 한계가 아니라 해당 차원 자체가 보상 공간에 매핑되지 않기 때문입니다.

## 핵심 통찰

도구 사용 능력은 단일 차원이 아니라 보상 최적화 가능한 차원과 불가능한 차원으로 분리되며, 후자는 보상 신호 자체가 해당 차원을 인코딩하지 못하는 구조적 결함 때문에 순수 보상 최대화로부터 emergent할 수 없다.
