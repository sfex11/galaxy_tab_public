이 요약은 에이전트가 **예산의 절반을 결함 위치 파악에 소모한다**는 핵심 사실만 확인시켜 줍니다. SHERLOC의 세부 논증—위치 파악을 '파일 검색'이 아닌 '수리 가능한 진단'으로 평가해야 한다는 주장—은 Wiki에 구체적으로 기록되어 있지 않습니다.

다만, 질문하신 핵심 논리 구조—**기존 평가가 위치 파악의 질적 부족을 최종 수정 성공의 분산으로 은폐한다**—는 Wiki에 존재하는 다음 엔티티와 구조적으로 동일한 패턴을 보입니다:

### [[concepts/bottleneck-misattribution.md|bottleneck misattribution]]과의 구조적 동치성

Wiki에 등록된 `bottleneck-misattribution`은 "실제 병목을 잘못 지정하는 현상"을 의미하는 엔티티로, 이와 유사한 패턴이 Wiki의 다른 곳에서도 구체화되어 있습니다:

- **[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]**의 관련 논문("Tool Attention Is All You Need")에서는 다중 서버 MCP 배포에서 **파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정됨**을 보여주며, "파이프라인 서빙 최적화의 실제 병목 지점을 재정의한다"고 기술합니다. 이는 '최종 출력(응답 지연)'의 분산이 '중간 단계(스키마 축적)'의 병목을 은폐하는 패턴입니다.

- **[[concepts/adaptive-inference.md|adaptive inference]]**의 SpecKV 관련 기술에서는 기존 컨텍스트 기반 적응이 "외부 환경에 반응"하는 데 집중한 반면, 실제 적응이 필요한 차원이 "내부 시스템 상태(압축)"에 있음을 지적합니다. 즉, **평가/최적화의 초점이 진짜 결정 차원을 놓치는** 구조입니다.

### [[concepts/benchmark-specification-gap.md|benchmark specification gap]]의 적용 가능성

`benchmark-specification-gap`은 벤치마크가 측정하고자 하는 것과 실제로 측정하는 것 사이의 간극을 의미합니다. 질문하신 시나리오는 이 간극의 구체적 사례로 해석할 수 있습니다:

- **명세된 측정 대상**: "결함을 고쳤는가" (최종 패치 정확성)
- **실제 측정 대상**: "결함을 고친 것처럼 보이는가" (위치 파악의 질적 결함이 수정 단계에서 우연히 보정되었거나, 반대로 정확한 진단이 잘못된 수정으로 mask-off된 경우)

SHERLOC의 주장은 이 gap이 단순한 누락이 아니라 **구조적 은폐**라는 점을 강조하는 것으로 이해됩니다. 최종 수정 성공률이라는 분산이 높은 지표 안에 위치 파악 품질이라는 저차원 변수가 흡수되어, 개선 시도가 잘못된 곳(수정 로직)으로 향하게 된다는 것입니다.

### 새로운 연결 발견

Wiki의 기존 엔티티들을 조합하면, SHERLOC가 지적하는 문제를 다음과 같이 재구성할 수 있습니다:

1. **[[concepts/blind-tool-invocation.md|blind tool invocation]]** → 위치 파악 단계에서 에이전트가 진단 없이 도구를 남발하는 현상
2. **[[concepts/bottleneck-misattribution.md|bottleneck misattribution]]** → 이를 '탐색 비용'으로 분류하고 병목으로 인식하지 못함
3. **[[concepts/benchmark-specification-gap.md|benchmark specification gap]]** → 평가가 최종 패치만 측정하여 1→2의 문제를 은폐
4. SHERLOC의 제안 → 위치 파악을 '수리 가능한 진단'이라는 독자 평가 차원으로 끌어올려 3의 gap을 메우는 것
