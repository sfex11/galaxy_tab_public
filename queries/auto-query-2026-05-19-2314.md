### 1. 기존 연구가 이미 스스로의 문제 공식화를 정정한 사실

[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]와 연결된 논문은 사실 초기 문제 공식화가 잘못되었음을 **스스로 증명**했습니다:

> "다중 서버 MCP 배포에서 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 **스키마 축적**에 의해 결정됨을 보여주어, 파이프라인 서빙 최적화의 실제 병목 지점을 재정의한다."

즉, 이 연구는 "LLM 호출 속도를 최적화한다"는 초기 문제 공식화가 잘못되었음을 발견하고, "스키마 축적이 병목이다"로 문제를 재공식화했습니다. 이는 [[concepts/bottleneck-misattribution.md|bottleneck misattribution]]의 구체적 사례입니다 — 병목의 위치를 잘못 지목했다가 스스로 정정한 것입니다.

### 2. 그러나 재공식화 이후에도 남는 구조적 한계

질문의 핵심 통찰은 여기서 한 단계 더 들어갑니다. 스키마 축적이 병목이라는 재공식화도 여전히 **순차적 파이프라인이라는 전제를 유지**하고 있습니다. 스키마가 축적되는 이유가 파이프라인이 순차적으로 구성되어 있기 때문이라면, 병목 지점을 옮기는 것만으로는 근본 해결이 안 됩니다.

이때 [[concepts/context-as-pipeline-structural-constraint.md|context as pipeline structural constraint]]라는 Wiki 개념이 중요해집니다. 스키마 축적이란 결국 **컨텍스트가 파이프라인의 구조적 제약으로 기능**하는 현상입니다. 순차적 파이프라인에서는 이 제약을 회피할 구조적 경로가 원천적으로 존재하지 않습니다.

### 3. 병렬 재구성이 "다른 문제"를 해결하는 것이 아니라 "동일 문제의 상위 계층"을 해결하는 이유

여기서 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]이 핵심 연결고리가 됩니다:

> "Crab이 이 간극을 구체적 런타임 문제로 현현한다: 에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율이 간극의 본질이다."

이 간극의 본질 — **의미론적 의존 그래프를 시스템 메커니즘으로 번역할 때의 비효율** — 은 파이프라인 최적화와 병렬 재구성이 사실 같은 문제의 다른 계층에서 발생함을 보여줍니다:

| 계층 | 문제 | 기존 접근 | 재구성 접근 |
|------|------|----------|------------|
| 표현 계층 | 의존 그래프가 순차적으로 모델링됨 | — | 의존 그래프 자체를 병렬로 재설계 |
| 번역 계층 | 순차 그래프를 시스템에 매핑 | 스키마 축적 최적화 | 병렬 그래프를 병렬 시스템에 매핑 |
| 실행 계층 | 매핑된 시스템의 레이턴시 | 개별 호출 최적화 | 분산 실행 |

[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]가 해결한 것은 **번역 계층의 최적화**입니다. 질문이 지적하는 대로, 표현 계층에서 순차성이 내재되어 있다면 번역 계층 최적화에는 구조적 한계가 있습니다.

### 4. 그럼에도 기존 연구가 "잘못된 문제"를 푼 것이 아닌 이유

세 가지 이유로 기존 연구를 폐기할 수 없습니다:

**첫째**, [[concepts/agent-native-serving.md|agent native serving]](Pythia 논문과 연결)로 이어지는 연구 방향은 이미 파이프라인 최적화의 한계를 인식하고 에이전트 네이티브 서빙이라는 **상위 계층으로 이동**하고 있습니다. 즉, 기존 연구가 스스로의 한계를 진단하고 다음 문제로 이행하는 경로를 내장하고 있습니다.

**둘째**, [[entities/agentic-harness-engineering.md|agentic harness engineering]]이 보여주듯, 실제 배포 환경에서는 파이프라인 재구성이 항상 가능하지 않습니다:

> "C/R 런타임이 하네스의 핵심 구성요소(결함 내성·롤백·분기)를 직접 구현하는 인프라 계층으로, 하네스 엔지니어링의 범위를 애플리케이션 로직에서 실행 환경까지 확장함을 시사한다."

레거시 의존성, 규제 제약, 기존 인프라 호환성 등으로 인해 파이프라인 구조 자체를 바꾸지 못하고 병목만 최적화해야 하는 상황은 실존합니다.

**셋째**, [[concepts/adaptive-inference.md|adaptive inference]]의 SpecKV 사례가 보여주듯, 적응적 접근은 **내부 시스템 상태에 반응하여 국소적으로 순차성을 완화**할 수 있습니다:

> "적응적 추론의 결정 차원을 모델/라우트 선택에서 추측 길이(γ) 선택으로 확장한다. ... 압축이라는 내부 시스템 상태에 반응하는 적응의 새로운 유형을 제시한다."

이는 파이프라인 전체를 병렬로 재구성하지 않더라도, 국소적 적응을 통해 순차성의 비용을 줄일 수 있음을 보여줍니다.

## 새로운 연결 발견

이 분석 과정에서 다음 연결이 식별됩니다:

1. **[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]] → [[concepts/bottleneck-misattribution.md|bottleneck misattribution]]**: 기존 연구가 스스로 병목 오인을 정정한 사례로, [[concepts/bottleneck-misattribution.md|bottleneck misattribution]]의 구체적 실증 사례가 됨
2. **[[concepts/context-as-pipeline-structural-constraint.md|context as pipeline structural constraint]] → [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]**: 스키마 축적을 "컨텍스트가 파이프라인 구조적 제약으로 기능하는 현상"으로 재해석 가능
3. **[[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]] → [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]**: 파이프라인 최적화의 근본 한계가 의존 그래프-시스템 번역 간극의 하위 사례임을 보여줌
4. **[[concepts/adaptive-inference.md|adaptive inference]] → [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]**: 순차성 제거(거시 재구성)와 순차성 완화(미시 적응)가 상보적 관계에 있음

## 요약

기존 [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]] 연구는 **잘못된 문제를 푼 것이 아니라, 올바른 문제의 하위 계층만 푼 것**입니다. 스키마 축적을 병목으로 재정의한 것은 의미 있는 진전이지만, 그 재정의조차 순차적 파이프라인 전제 위에 서 있었습니다. 순차성 원천 제거는 동일한 문제 공간의 **표현 계층**을 공격하는 것이며, 두 접근은 배타적이 아니라 계층적으로 상보적입니다. 다만, Wiki의 [[concepts/agent-native-serving.md|agent native serving]] 방향성이 이미 이 계층 이동을 예고하고 있다는 점은 주목할 만합니다.

## 핵심 통찰

병목 지점의 오인을 넘어 문제가 발생하는 계층 자체(표현/번역/실행)를 오인하는 것이 파이프라인 최적화의 근본적 한계 원인이다.
