# 도구 호출의 상태성과 누적 아키텍처 효과 포착 문제

제시하신 질문은 현재 도구 호출 프레임워크가 가진 근본적 한계—즉, 모든 도구 호출을 동일한 상태less 연산으로 취급한다는 점—을 정확히 지적합니다. 라이브러리 임포트 같은 불가역적 아키텍처 결정을 "선택-실행-결과 수신"의 3단계로 환원하면, 결정의 상태성과 누적 효과가 프레임워크 층에서 완전히 증발합니다. Wiki의 논문들이 이 문제를 세 가지 차원에서 조명합니다.

## 1. 파이프라인 수준의 누적 효과: 개별 호출이 아닌 축적이 병목이다

[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 "Tool Attention" 논문은 다중 서버 MCP 배포에서 **파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적(schema accumulation)에 의해 결정됨**을 실증합니다. 이는 도구 호출이 상태less라는 전제가 거짓임을 보여주는 핵심 증거입니다—매 호출마다 스키마가 누적되어 파이프라인의 구조적 특성이 변화하며, 이 변화가 후속 모든 호출의 비용에 영향을 미칩니다.

라이브러리 임포트는 스키마 축적의 "강한 형태"입니다. 단일 호출로 보이지만, 실제로는 (1) 의존성 그래프에 새 노드 추가, (2) 네임스페이스 오염, (3) 후속 코드 생성의 가능성 공간 축소라는 세 가지 누적 효과를 동시에 수행합니다. 기존 프레임워크는 이 중 반환값(예: "임포트 성공")만 포착하고, 세 가지 부작용은 투명하게 통과시킵니다.

## 2. 의미론적 의존 그래프의 소실: OS 수준 메커니즘으로 번역 불가

[[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]에서 Crab 논문이 밝히는 간극이 바로 이 문제의 본질입니다:

> 에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율이 간극의 본질이다.

라이브러리 임포트 후 에이전트는 "이 모듈이 임포트되었다"는 **의미론적 사실**에 의존하여 후속 코드를 생성합니다. 그러나 기존 도구 호출 프레임워크는 이 의존 관계를 기록하지 않고, 최대한 파일 시스템 스냅샷([[concepts/agent-os-semantic-gap.md|agent os semantic gap]])으로만 복원을 시도합니다. 스냅샷은 "무엇이 바뀌었는가"는 알지만 "왜 바뀌었는가(어떤 결정의 결과인가)"를 모르기 때문에, 롤백 시 의미론적 일관성을 보장할 수 없습니다.

예를 들어, `import torch` 후 작성된 50줄의 코드를 롤백할 때, 파일 단위 스냅샷은 단순히 삭제하지만, 의미론적 의존 그래프는 "torch 임포트 → 이 50줄 생성"이라는 인과链을 유지해야 정상적 분기 탐색이 가능합니다.

## 3. 하네스가 상태 컨테이너가 되어야 한다

[[entities/agentic-harness-engineering.md|agentic harness engineering]]에서 Crab 논문이 시사하는 바는:

> C/R 런타임이 하네스의 핵심 구성요소(결함 내성·롤백·분기)를 직접 구현하는 인프라 계층으로, 하네스 엔지니어링의 범위를 애플리케이션 로직에서 실행 환경까지 확장함

이를 도구 호출에 적용하면, 하네스는 단순한 "도구 실행기"가 아니라 **"아키텍처 결정의 세마포어"**가 되어야 합니다. 각 도구 호출 전에 해당 호출이 가지는 부작용 프로필(side-effect profile)을 검사하고, 불가역적 결정(build-vs-buy)인 경우 의미론적 의존 그래프에 명시적으로 기록한 뒤에야 실행을 허가하는 구조입니다.

이는 [[concepts/code-as-agent-harness.md|code as agent harness]]—코드 자체가 에이전트의 하네스가 된다는 개념—과 결합할 때 특히 강력합니다. 코드가 하네스라면, 코드에 대한 불가역적 변경(임포트, 의존성 추가)은 하네스 자체의 구조 변경이며, 이를 상태less 연산으로 취급하는 것은 하네스의 무결성을 간과하는 것입니다.

## 발견되는 새로운 개념적 간극

위 분석에서 세 가지 기존 엔티티를 연결하면, 현재 Wiki에 **명시적으로 포착되지 않는 개념**이 드러납니다:

### 도구 호출의 부작용 스펙트럼 (Tool Call Side-Effect Spectrum)

기존 [[concepts/blind-tool-invocation.md|blind tool invocation]]이 "도구를 맹목적으로 호출하는 문제"를 지적한다면, 그 이면에는 **도구 호출 간의 부작용 차이를 구분하는 택소노미가 부재**한다는 근원적 결함이 있습니다. 적어도 다음 세 가지 등급이 필요합니다:

| 등급 | 예시 | 롤백 가능성 | 아키텍처 효과 |
|------|------|-------------|---------------|
| 순수 조회 | `search("...")` | 완전 | 없음 |
| 가역적 변이 | `create_file("temp.py")` | 파일 삭제로 복원 | 국소적 |
| 불가역적 아키텍처 결정 | `import torch`, `npm install` | 의존성 그래프 복원 필요 | 전역적·누적적 |

이 택소노미가 없으면, [[entities/agentic-harness-engineering.md|agentic harness engineering]]가 제안하는 롤백·분기 메커니즘도 등급 구분 없이 동작하게 되어, 순수 조회와 아키텍처 결정을 동일한 비용으로 롤백하려 시도하는 비효율이 발생합니다.

### 요약

기존 프레임워크가 도구 호출을 상태less로 처리하는 것은 [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 스키마 축적 실증에 의해 이미 반증되어 있으며, 그 누적 효과를 포착하기 위해서는 (1) [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]이 지적하는 의미론적 의존 그래프를 도구 호출 단위로 구축하고, (2) 이를 [[entities/agentic-harness-engineering.md|agentic harness engineering]]의 인프라 계층에서 일급 객체로 관리하며, (3) [[concepts/code-as-agent-harness.md|code as agent harness]] 관점에서 코드 변경을 하네스 구조 변경으로 재해석하는 세 단계 확장이 필요합니다. 현재 Wiki는 이 세 가지를 각기 독립적으로 보고 있어, **불가역적 도구 호출의 부작용을 하네스 무결성 문제로 통합하는 개념**이 누락되어 있습니다.
