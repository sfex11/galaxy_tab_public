# 영속적 도메인 상태와 하네스 상태 경계의 간극

## 1. 기존 하네스가 담당하는 상태 경계

[[entities/agentic-harness-engineering.md|agentic harness engineering]]의 핵심 논문인 **Crab**는 C/R 런타임이 하네스의 결함 내성·롤백·분기를 구현하는 인프라 계층이라고 규정합니다. 즉, 하네스의 상태 경계는 **샌드박스 내 파일시스템·프로세스·런타임 아티팩트**로 그려집니다.

그러나 [[concepts/agent-os-semantic-gap.md|agent os semantic gap]]이 Crab을 통해 현현하는 바와 같이, 에이전트의 **의미론적 의존 그래프**(어떤 상태에 의존하는가)를 OS의 **체크포인트 메커니즘**(전체 스냅샷)으로 번역할 때 이미 비효율이 발생합니다. [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]의 표현을 빌리면, 이것은 "에이전트가 의미적으로 의존하는 것"과 "OS가 스냅샷으로 포착하는 것" 사이의 번역 손실입니다.

**여기까지가 기존 Wiki가 포착한 간극입니다:** 샌드박스 내부에서도 의미론적 상태와 시스템적 상태 사이에 균열이 존재한다는 점.

---

## 2. 질문이 지적하는 간극: 상태 경계 자체의 부족

질문의 핵심은 이 균열을 넘어서는 것입니다:

> 하네스가 통제하는 상태는 **에피소딕(episodic)** — 세션 단위로 생성·소멸함
> 코딩 에이전트가 수정하는 코드베이스는 **영속적(persistent)** — 세션을 초월하여 누적됨

개별 PR이 정상이라 하더라도, 누적되면 공격 페이로드가 되는 현상은 [[concepts/codebase-dependency-contamination.md|codebase dependency contamination]]이 포착하려는 현상과 정확히 일치합니다. 그런데 기존의 [[entities/agentic-harness-engineering.md|agentic harness engineering]] 프레임워크는 이 오염이 발생하는 **영속 계층을 상태 경계 밖에 두고 있습니다**.

Crab의 C/R이 롤백할 수 있는 것은 "이번 세션의 샌드박스 스냅샷"이지, "이전 47번의 PR이 누적된 코드베이스"가 아닙니다. 다시 말해:

| | 하네스가 통제하는 상태 | 하네스가 통제하지 못하는 상태 |
|---|---|---|
| **수명** | 에피소딕 (세션 내) | 영속적 (세션 간) |
| **단위** | 파일시스템 스냅샷, 프로세스 상태 | 코드베이스 의존성 그래프 |
| **롤백 가능성** | C/R로 복원 가능 | 개별 PR별로는 정상 → 누적 롤백 불가 |
| **공격 표면** | 샌드박스 탈출 | 정상 PR의 누적 합성 |

## 3. 왜 이 간극이 구조적으로 발생하는가

[[concepts/agent-os-semantic-gap.md|agent os semantic gap]]이 지적하는 번역 문제를 영속적 도메인으로 확장하면 구조가 보입니다:

**Crab의 한계를 넘어선 2단계 번역 실패**

```
[에이전트 의도] 
  → 1차 번역: 샌드박스 내 아티팩트 상태 (Crab이 포착, 여기서 semantic-gap 발생)
    → 2차 번역: 영속적 코드베이스 상태 (하네스가 아예 포착 불가 ← 질문의 핵심)
```

1차 번역의 손실은 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]이 설명합니다. 그러나 2차 번역 — 에이전트의 개별 액션(정상 PR)이 영속적 도메인(코드베이스)에 미치는 **누적 효과** — 에 대해서는 하네스가 관측 모델 자체를 갖지 못합니다.

이는 [[concepts/blind-tool-invocation.md|blind tool invocation]]의 문제와도 구조적으로 동일합니다: 개별 도구 호출이 "정상"이므로 하네스가 개입하지 않지만, 호출의 **조합**이 비정상을 만들어내는 상황. 다만 blind-tool-invocation이 단일 세션 내의 문제라면, 여기서는 **세션 간 누적**으로 시축이 확장됩니다.

## 4. 새로운 개념적 연결 발견

이 분석에서 기존 Wiki 슬러그 간에 새로운 연결이 드러납니다:

### 4-1. [[concepts/cascading-distribution-shift.md|cascading distribution shift]] → 코드베이스 누적 편이
원래 데이터 분포의 연쇄적 변화를 의미하는 이 개념을, **코드베이스 의존성 그래프의 연쇄적 변화**로 재해석할 수 있습니다. 개별 PR은 각각 "정상 분포" 내에 있지만, 누적되면 원래 코드베이스의 구조적 불변식(structural invariant)을 벗어나는 분포 이동이 발생합니다.

### 4-2. [[concepts/co-failure-rate.md|co failure rate]] / [[concepts/co-failure-ceiling.md|co failure ceiling]] → 누적 정상성의 함정
개별 컴포넌트의 성공률이 높아도 **결합된 시스템의 실패율**이 급증하는 현상. N개의 PR이 각각 99% 안전하더라도, 누적되면 안전성이 보장되지 않는다는 점에서 co-failure의 본질과 일치합니다.

### 4-3. [[concepts/asymmetric-alignment-fragility.md|asymmetric alignment fragility]] → 개별-누적 정렬의 비대칭성
개별 PR 수준에서는 정렬(aligned)되어 있지만, 누적 수준에서는 정렬이 깨지는 비대칭성. 이것은 micro-alignment가 macro-alignment를 보장하지 못하는 구조적 취약성입니다.

## 5. 결론: 하네스 상태 모델의 근본적 확장이 필요함

질문의 통찰을 정리하면:

**기존 [[entities/agentic-harness-engineering.md|agentic harness engineering]]의 상태 모델은 에피소딕 샌드박스 상태만 포착하고, 에이전트가 실제로 조작하는 영속적 도메인 상태(코드베이스)는 상태 경계 밖에 방치한다.** 이것은 [[concepts/agent-os-semantic-gap.md|agent os semantic gap]]의 문제를 세션 간 영속 계층으로 확장한 것이며, [[concepts/codebase-dependency-contamination.md|codebase dependency contamination]]이 실제로 발생하는 메커니즘에 대한 구조적 설명을 제공합니다.

Crab식 C/R이 "이 세션의 상태를 되돌리는" 하네스라면, 코딩 에이전트에게 필요한 것은 **"누적된 도메인 상태의 구조적 건전성을 평가하는" 하네스**입니다. 이것은 롤백이 아니라 **누적 의미론적 감사(cumulative semantic audit)**의 문제이며, 기존의 에피소딕 C/R 패러다임만으로는 해결할 수 없는 하네스 설계의 새로운 차원을 요구합니다.

## 핵심 통찰

기존 하네스의 상태 경계가 에피소딕 샌드박스에 한정되어 있어, 개별적으로 정상적인 액션들이 세션 간 영속적 도메인에 누적될 때 발생하는 구조적 독성을 감지할 수 없다.
