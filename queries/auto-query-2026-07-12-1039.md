## 1단계: 요청 내부에서의 추상화 붕괴 — [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]

[[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]에 기록된 **"Tool Attention Is All You Need"**(2026-04-26)의 발견이 이미 이 논의의 시발점이다. 이 논문은 다중 서버 MCP 배포에서 **파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정된다**고 실증했다.

이것의 의미를 명확히 하자: 기존 서빙 최적화는 "하나의 요청 = 하나의 LLM 추론"이라는 암묵적 추상화에 기반한다. 그러나 에이전트 파이프라인에서는 하나의 요청 내부에 툴 스키마 로딩, 여러 LLM 호출, 툴 실행 등이 연쇄적으로 존재하며, **진짜 병목은 LLM 추론이 아니라 스키마 축적**이었다. 즉, 요청이라는 추상화 단위가 이미 요청 내부의 실제 병목 지점을 가려버린 것이다.

SMetric의 주장은 이것의 **논리적 연장선**이다. 요청 내부에서조차 추상화가 붕괴하는데, 여러 요청이 연쇄적으로 연결된 세션 전체를 요청 단위로 쪼개어 스케줄링하는 것은 더욱 근본적인 오류다.

---

## 2단계: 세션 수준에서의 추상화 부정합 — [[concepts/agent-native-serving.md|agent native serving]]

SMetric가 제기하는 핵심은 이렇다: 에이전트는 개별 LLM 응답 토큰이 아니라 **완전한 응답(행동 완수)에 대해서만 다음 결정을 내린다.** 그런데 기존 스케줄러는 각 LLM 호출을 독립적인 요청으로 취급하여 개별 레이턴시를 최소화하려 한다.

이는 [[concepts/agent-native-serving.md|agent native serving]]이라는 엔티티가 존재한다는 것 자체가 시사하듯, 에이전트 워크로드에는 에이전트 고유의 서빙 패러다임이 필요함을 의미한다. 요청 중심 스케줄링이 잘못된 것이 아니라, **에이전트 서빙이라는 문제 영역에 대해 충분히 낮은 추상화 계층으로 내려가지 못한 것**이다.

---

## 3단계: 의미론적 단위와 시스템 단위의 간극 — [[concepts/agent-os-semantic-gap.md|agent os semantic gap]] · [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]

이 문제를 더 깊이 파고들면, [[concepts/agent-os-semantic-gap.md|agent os semantic gap]]과 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]이 동일한 구조적 문제의 다른 현현임을 알 수 있다.

**"Crab"**(2026-05-03)은 [[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]을 구체적 런타임 문제로 현현한다고 Wiki에 기록되어 있다: **에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율**이 간극의 본질이라는 것.

이것과 SMetric의 문제는 정확히 동일한 구조다:
- **Crab의 경우**: 에이전트의 의미론적 상태 의존성 → OS의 비의미론적 스냅샷 단위로 번역 실패
- **SMetric의 경우**: 에이전트의 세션 수준 행동 완수 단위 → 스케줄러의 개별 요청 단위로 분할 실패

둘 다 **"에이전트가 의미론적으로 묶여 있는 단위"와 "시스템이 임의로 정한 단위" 사이의 불일치**다.

---

## 4단계: 미시적 적응도 요청 단위를 벗어난다 — [[concepts/adaptive-inference.md|adaptive inference]]

추가로, [[concepts/adaptive-inference.md|adaptive inference]]에서 **SpecKV**(2026-05-05)가 보여주는 적응은 모델/라우트 선택에서 **추측 길이(γ) 선택**으로 결정 차원을 확장하며, 이 적응은 "압축이라는 내부 시스템 상태"에 반응한다고 Wiki에 기록되어 있다.

즉, 적응의 단위 역시 요청이 아니라 **토큰 생성 과정 내부의 동적 상태**다. 요청 단위 최적화는 이 미시적 적응의 공간까지 포착하지 못한다.

## 종합: 왜 "근본적으로 잘못된 추상화 계층"인가

Wiki의 논문들을 종합하면, 요청 중심 최적화가 에이전트 서빙에서 잘못된 추상화 계층인 이유는 **세 가지 수준에서 동시에 추상화 오류가 발생하기 때문**이다:

| 수준 | 실제 단위 | 기존 스케줄링의 단위 | 관련 Wiki 증거 |
|------|-----------|---------------------|----------------|
| 미시 | 토큰 내부 압축 상태 (γ) | 요청 전체 일괄 처리 | [[concepts/adaptive-inference.md|adaptive inference]] (SpecKV) |
| 중간 | 파이프라인 내 스키마 축적 | 개별 LLM 호출 레이턴시 | [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]] (Tool Attention) |
| 거시 | 세션 수준 행동 완수 | 독립적 요청 | [[concepts/agent-native-serving.md|agent native serving]] (SMetric) |

요청 중심 최적화는 이 세 수준 모두에서 **실제 병목과 결정 단위를 보이지 않게 만드는 추상화**로 작동한다.

## 새로운 연결 발견

이 분석 과정에서 다음 연결이 명시적으로 드러났습니다:

> **[[concepts/algorithm-system-translation-gap.md|algorithm system translation gap]]과 [[concepts/aggregate-pipeline-serving.md|aggregate pipeline serving]]의 통합적 이해**: 두 엔티티는 서로 다른 논문에서 독립적으로 기술되어 있으나, 본질적으로 동일한 구조 — **에이전트의 의미론적 처리 단위와 시스템의 임의적 분할 단위 사이의 불일치** — 를 각각 런타임(Crab)과 서빙(Tool Attention, SMetric) 영역에서 현현한 것으로 보아야 한다. 이를 아우르는 상위 개념으로 **추상화 계층 부정합(abstraction-layer mismatch)**을 제안할 수 있을 것이다.

또한, [[entities/agentic-harness-engineering.md|agentic harness engineering]]이 Crab을 통해 "하네스의 범위를 애플리케이션 로직에서 실행 환경까지 확장"한다는 기록은, SMetric가 스케줄링의 범위를 요청에서 세션으로 확장해야 한다고 주장하는 것과 **동일한 방향성** — 즉, 에이전트 워크로드의 실제 단위에 맞게 시스템 경계를 재설정해야 한다는 방향성 — 을 공유함을 알 수 있다.

## 핵심 통찰

요청 중심 최적화가 에이전트 서빙에서 '근본적으로 잘못된' 이유는 단일 수준의 오류가 아니라 미시·중간·거시 세 수준에서 동시에 추상화 오류가 발생하여 실제 병목을 체계적으로 은폐하기 때문이며, 이는 algorithm-system-translation-gap과 aggregate-pipeline-serving이 본질적으로 동일한 구조—의미론적 단위와 임의적 분할 단위의 불일치—의 다른 영역별 현현임을 보여준다.
