## Parallax의 "사고-행동 분리" 논거가 도구 증강 에이전트 아키텍처의 보안·신뢰성 전제를 무효화하는 방식

---

### 1. 핵심 충돌: LLM Agent의 정의 vs. Parallax의 명제

Wiki에서 LLM Agent는 *"LLM을 핵심 추론 엔진으로 활용하여 환경을 인식하고, 도구를 사용하며, **자율적으로 의사결정과 행동을 수행**하는 AI 시스템"*으로 정의된다. 이 정의는 **사고(추론)와 행동(실행)의 통합**을 에이전트의 본질적 속성으로 전제한다.

Parallax("Why AI Agents That Think Must Never Act", 2026-04-16)는 이 전제를 정면으로 부정한다. 핵심 논거는: **추론하는 시스템이 동시에 실세계 행동(파일 읽기, 명령 실행, 네트워크 요청, DB 수정)을 수행할 권한을 가져서는 안 된다**는 것이다. 이는 단순한 가드레일 보강이 아니라, 에이전트 아키텍처의 **존재론적 재설계**를 요구한다.

---

### 2. 현재 보안 프레임워크의 전제와 그 취약점

Wiki의 논문들은 도구 증강 에이전트의 보안을 세 가지 층위에서 접근하고 있다:

#### (a) 프레임워크 아키텍처 수준 — OpenClaw 보안 평가
"A Systematic Security Evaluation of OpenClaw and Its Variants"(2026-04-07)는 6개 OpenClaw 변형(AutoClaw, QClaw, KimiClaw, MaxClaw, ArkClaw)을 205개 테스트 케이스로 평가하며, **보안이 모델이 아닌 프레임워크 아키텍처에 의해 결정**된다고 결론짓는다. 그러나 이 평가의 전제는 "프레임워크가 도구 호출 권한을 안전하게 관리할 수 있다"는 것이다. Parallax는 이 전제 자체를 공격한다: **추론 엔진이 도구 호출 권한을 보유하는 한, 어떤 아키텍처적 차이도 근본적 안전을 보장하지 못한다.**

#### (b) 런타임 가드레일 수준 — ClawGuard
ClawGuard(2026-04-15)는 간접 프롬프트 인젝션에 대응하여 2단계(사전 규칙 유도 + 도구 호출 수준 강제) 방어를 구현한다. 공격 채널로 (1) 웹/로컬 콘텐츠 인젝션, (2) MCP 서버 인젝션, (3) 스킬 파일 인젝션을 식별한다. 그러나 Parallax의 관점에서 ClawGuard는 **"사고하는 시스템이 행동할 수 있다"는 전제 위에 세워진 사후적 보완책**에 불과하다. 도구 호출 자체를 LLM의 추론 루프에서 분리하지 않는 한, 새로운 공격 벡터는 계속 출현한다.

#### (c) 궤적 수준 평가 — TraceSafe
TraceSafe(2026-04-10)는 **취약점 표면이 최종 출력에서 중간 실행 궤적으로 이동**한다는 핵심 통찰을 제시한다. 12개 위험 범주에 걸쳐 다단계 도구 호출 궤적을 평가하며, 기존 가드레일이 자연어 응답에는 잘 작동하지만 **다단계 도구 사용에서는 부적절하게 테스트되었음**을 밝힌다. 이것은 Parallax의 논거를 **경험적으로 뒷받침**하는 증거다: 사고와 행동이 통합된 시스템에서 궤적 전체의 안전성을 보장하는 것은 본질적으로 어렵다.

---

### 3. Box Maze와의 관계: 수렴과 분기

Box Maze(프로세스 제어 아키텍처)는 RLHF와 출력 필터링을 넘어 **추론 프로세스 자체의 무결성을 아키텍처 수준에서 강제**하는 3계층 분해(Memory Loop, Logic Loop, Heart Anchor)를 제시한다. 경계 위반율을 40%에서 <1%로 감소시킨다.

**수렴점**: Parallax와 Box Maze 모두 "프롬프트 수준 가드레일의 한계"를 인식하고 아키텍처적 해법을 추구한다.

**분기점**: Box Maze는 추론 프로세스의 **내부 무결성**을 강제하되 여전히 추론과 행동의 통합을 유지한다. Parallax는 이보다 급진적으로, 추론과 행동의 **물리적 분리**를 요구한다. 이는 Box Maze의 프로세스 제어가 충분조건이 아님을 시사한다.

---

### 4. ClawBench가 드러내는 실증적 격차

ClawBench(2026-04-13)는 153개 일상 과제를 144개 실제 웹 플랫폼에서 평가하며, 최고 성능 모델(Claude Sonnet 4.6)이 **샌드박스 벤치마크에서 65-75%인 성공률이 실제 라이브 웹사이트에서 33.3%로 급락**함을 보여준다. 이 일반화 격차는 Parallax의 논거에 구조적 근거를 제공한다: 통제된 환경에서의 안전성 검증이 실세계 배포에서의 안전성을 보장하지 못한다면, **행동 권한 자체를 제한하는 것이 유일한 안전 보장 방법**이라는 결론에 도달한다.

---

### 5. 새롭게 발견되는 개념적 연결

1. **"궤적 불투명성(trajectory opacity)" 문제의 재해석**: TraceSafe의 궤적 수준 평가와 Claw-Eval(2026-04-09)의 궤적 분석 확장은 "사고-행동 통합 시스템에서 중간 상태를 감시하는 것의 본질적 한계"를 드러낸다. Parallax는 이를 감시 문제가 아닌 **설계 문제**로 재프레이밍한다 — 감시해야 할 궤적 자체를 생성하지 않는 아키텍처가 해답이라는 것.

2. **OpenClaw 보안 평가의 "아키텍처 결정론"과의 긴장**: OpenClaw 보안 평가는 "보안은 모델이 아닌 아키텍처가 결정한다"고 결론짓는다. Parallax는 이 명제를 한 단계 더 밀어붙인다: **어떤 도구 증강 아키텍처도, 추론-행동 통합을 유지하는 한, 충분히 안전하지 않다.** 즉, 아키텍처 선택의 문제가 아니라 아키텍처 **패러다임**의 문제다.

3. **Multi-Agent System과의 접점**: Wiki의 다중 에이전트 시스템 연구에서 "지연된 피드백이 인간 감독자의 오류 귀인을 체계적으로 왜곡"한다는 발견은 Parallax의 논거를 보강한다. 사고와 행동이 분리되면, 행동 실행 단계에서 독립적인 검증 에이전트(또는 인간)가 개입할 수 있어 이 왜곡을 완화할 수 있다.

### 결론

Parallax는 현재 에이전트 보안 연구의 **암묵적 전제** — "추론과 행동의 통합은 유지하되, 그 실행을 안전하게 관리할 수 있다" — 를 정면으로 부정한다. ClawGuard의 런타임 방어, TraceSafe의 궤적 평가, Box Maze의 프로세스 제어, OpenClaw 변형 간 아키텍처 비교 모두 이 전제 위에서 작동한다. Parallax의 논거가 타당하다면, 이들은 **근본적으로 불충분한 전제 위에 세워진 점진적 개선**이며, 에이전트 아키텍처는 "어떻게 안전하게 행동하게 할 것인가"에서 **"추론과 행동을 어떻게 구조적으로 분리할 것인가"**로 질문 자체를 전환해야 한다.

## 핵심 통찰

에이전트 보안은 아키텍처 선택(architecture selection)의 문제가 아니라 추론-행동 통합이라는 아키텍처 패러다임(architecture paradigm) 자체의 문제이며, 궤적 불투명성과 샌드박스-실세계 격차가 이를 경험적으로 뒷받침한다.
