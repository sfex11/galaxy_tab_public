문제 정의: 툴 호출 에이전트의 상태 관리

고객 서비스 도메인의 툴 호출 에이전트는 여러 턴에 걸쳐 작업 상태를 유지하고 도메인 정책을 준수해야 합니다.

작업 상태의 구성 요소

  • 관련 사실 (Facts)
  • 식별자 (Identifiers)
  • 제약 조건 (Constraints)
  • 조건 (Conditions)

기존 에이전트의 한계

  • 상태를 별도로 표현하지 않음
  • 관측·툴 반환·정책이 프롬프트에 혼재
  • 매 차례마다 상태를 재구성해야 함
  • 상태 관리가 암묵적으로 처리됨

두 가지 핵심 실패 모드

실패 모드 1

상태 기반 오류 (State Grounding Failure)

올바른 사실을 검색했음에도, 나중에 오래되거나 누락되거나 잘못된 정보를 바탕으로 의사결정을 내림

"예약 기록을 정상적으로 조회했지만, 관련 상태가 프롬프트에 암묵적으로만 존재하여 정책 위반 취소를 실행"

실패 모드 2

정책 위반 실행 (Policy Violation)

구문적으로 유효한 툴 호출이라도, 현재 작업 상태에 따른 도메인 정책을 위반할 수 있음

"반품 가능 여부나 환불 대상 결제 수단 등 상태 의존적 규칙에 대한 별도 검증 없이 액션이 실행됨"

공통 원인: 작업 상태가 명시적으로 분리되어 표현되지 않음

LedgerAgent: 접근법 개요

모델 가중치는 변경하지 않는 추론 시점(inference-time) 메서드로, 에이전트 루프에 두 가지 결정적 컴포넌트를 추가합니다.

원장 (Ledger)

성공적인 툴 반환값을 스키마 기반의 형식화된 딕셔너리에 저장하여 프롬프트에 주입

정책 게이트 (Policy Gate)

환경 변경 툴 호출 전, 원장 상태를 기반으로 정책 위반 여부를 사전 검증

핵심 철학: 사후 수정(post-hoc correction)이 아닌 사전 예방(prevention)으로 실패 경계를 이동

원장 상태와 업데이트 메커니즘

형식적 정의

원장은 형식화된 딕셔너리 L : 𝒫 → 𝒱로, 𝒫는 정규 스키마 경로 집합, 𝒱는 툴 반환값 집합입니다.

경로 예시

  • user — 사용자 정보
  • orders.* — 주문 기록
  • products.* — 상품 정보
  • reservations.* — 예약 정보

업데이트 규칙

  • 성공적인 읽기 툴 반환값만 반영
  • 실패한 툴 / 쓰기 툴은 업데이트 제외
  • 쓰기 후 반드시 읽기 호출로 확인
  • 툴 경로 맵은 LLM이 아닌 도메인 정의

관찰-가정-불가 원칙 (Observe-not-assume): 쓰기 실행 후 에이전트는 반드시 읽기 호출을 통해 새 상태를 관찰해야 하며, 원장은 항상 외부 시스템에 기반합니다.

원장 기반 생성 (Ledger-Grounded Generation)

모델 호출 전, 원장 전체 블록을 프롬프트에 결정적으로 렌더링합니다.

1
툴 반환 수신
읽기 툴의 성공적 JSON 반환값 획득
2
원장 흡수
경로 맵에 따라 원장에 기록 저장
3
프롬프트 주입
원장 블록을 프롬프트에 추가
4
모델 생성
에이전트가 원장을 참조하여 응답

기존 방식

대화 기록에서 관련 정보를 검색하여 상태 재구성

LedgerAgent

안정적인 구조화된 표현을 통해 직접 조회(lookup)

정책 게이트 (Policy Gate)

환경 변경 툴 호출이 실행되기 전에, 원장 상태를 기반으로 도메인 정책을 검증합니다.

통과 시

제안된 호출이 정책을 만족하면 그대로 환경에 실행

차단 시

위반 규칙과 충돌하는 상태를 식별하는 피드백 반환 → 에이전트가 계획 수정 또는 차단

정책 게이트의 핵심 특성

  • 정책 규칙은 원장 필드 위의 술어(predicates)로 표현
  • 읽기 전용 호출은 게이트 대상이 아님 (환경 변경 호출만 검증)
  • LLM 호출 없이 결정적(deterministic)으로 동작
  • 환경이 변경되기 전에 검증 → 사전 예방 달성

에이전트 루프 전체 구조

1

사용자 입력 수신

고객의 요청 또는 질문을 대화에 추가

2

원장 렌더링 + 프롬프트 구성

현재 원장 상태 블록을 프롬프트에 결정적으로 주입

3

모델 생성

에이전트가 원장을 참조하여 응답 또는 툴 호출 생성

4

환경 변경 호출 → 정책 게이트 검증

위반 시 차단 + 피드백 반환, 통과 시 실행

5

읽기 툴 반환 → 원장 흡수

성공적인 읽기 반환값을 스키마 경로에 따라 원장에 저장

다음 턴으로 반복

업데이트된 원장과 함께 1번으로 돌아감

실험 설정

벤치마크 도메인

τ²-bench 및 τ-Trait의 4개 고객 서비스 도메인

  • 예약 관리 (Reservation)
  • 주문 처리 (Order)
  • 계정 관리 (Account)
  • 서비스 플랜 변경 (Service Plan)

모델 패널

오픈 가중치 + 폐쇄 가중치 모델 혼합 패널

  • 다양한 규모의 언어 모델 평가
  • 도메인–모델 조합에서 교차 평가
  • 사용자 시뮬레이션 모델 별도 구성

평가 지표

passk — k회 독립 시행 중 최소 1회 성공 확률. k가 클수록 일관성(consistency)을 더 엄격하게 측정합니다.

환경 변경 작업에서의 성능

LedgerAgent의 성능 향상은 환경 변경 액션이 필요한 작업에서 가장 두드러집니다.

작업 유형 표준 에이전트 LedgerAgent 개선 요인
환경 변경 작업
(환불, 취소, 변경 등)		 정책 위반 빈발 최대 개선폭 정책 게이트의 사전 차단
읽기 전용 작업
(조회, 검색 등)		 상태 재구성 오류 안정적 개선 원장 기반 상태 참조

인사이트: 정책 게이트는 읽기 전용 작업에는 관여하지 않으므로, 환경 변경 작업에서의 추가 개선은 게이트의 고유한 기여로 해석됩니다.

비교 분석: 기타 에이전트 방법론

방법론 상태 표현 정책 검증 LLM 호출 추가
표준 프롬프트 기반 암묵적 (프롬프트 내 혼재) 없음 (모델에 의존) —
IRMA 입력 재구성 규칙 프롬프트 삽입 없음
FAMA 다중 에이전트 협력 전문 헬퍼 에이전트 다수
파인튜닝 / RL 모델 가중치에 내재 학습된 행동에 의존 —
LedgerAgent 명시적 형식화 원장 결정적 사전 게이트 없음

LedgerAgent는 모델 가중치를 변경하지 않고, 추가 LLM 호출 없이 상태 추적과 정책 준수를 시스템 수준에서 해결합니다.

오류 분석 및 제약 조건

ABLATION 결과

  • 원장만 제거 시 성능 저하 → 원장의 필수성 확인
  • 정책 게이트만 제거 시 환경 변경 작업에서 저하 → 게이트의 독립적 기여
  • 두 컴포넌트 모두 필요한 상보적 구조

한계점

  • 도메인별 툴 경로 맵 수동 정의 필요
  • 정책을 술어로 명시적으로 작성해야 함
  • 읽기 툴로 관측 가능한 상태에만 한정
  • 장기 기억이나 비관측 세계 상태 복원 불가

결론 및 기여 요약

기여 1: 실패 모드 식별

정책 준수형 툴 호출 에이전트에서 상태 기반 오류(state grounding failure)를 핵심 실패 모드로 명시적으로 식별

기여 2: LedgerAgent 제안

스키마 기반 형식화 원장으로 관측 상태 유지 + 정책 게이트로 환경 변경 호출 사전 검증하는 추론 시점 메서드

기여 3: 실험적 검증

4개 고객 서비스 도메인 × 다양한 백본 모델에서 passk 일관성 개선 확인, 특히 환경 변경 작업에서 최대 효과

모델을 더 똑똑하게 만드는 대신, 상태를 명시적으로 표현하고 정책을 액션 경계에서 강제하는 것이 핵심 접근법입니다.
