Workflow as Knowledge

LLM 중재 워크플로우를 위한 의미론적 지속성

Emanuele Quinto (UNHCR) · Carlo Andrea Rozzi (CNR) · Francesco Zanitti (ZeLe & F)

동기: 워크플로우의 표현적 한계

LLM 시스템은 단일 턴 프롬프팅에서 구조화된 워크플로우로 이동했지만, 실행을 명시화하는 것만으로는 더 깊은 표현적 문제를 해결하지 못합니다.

정의의 산재

워크플로우 정의가 소스 코드, 설정 파일, DB 등에 분산

실행 흔적의 단절

중간 상태가 로그, 트레이스, 채팅 기록으로만 잔존

상관관계 ≠ 의미

인프라 상관관계가 공유 의미 객체 모델을 대체할 수 없음

"실행을 명시적으로 만드는 것만으로는 워크플로우 정의, 실행 인스턴스, 추론 기록을 하나의 의미적 객체 모델로 통합하지 못한다."

핵심 제안: 워크플로우를 지식으로

워크플로우 정의, 인스턴스, 추론 기록을 공유 지식 기판(Knowledge Substrate) 내에서 조회 가능한 지식 객체로 유지하자.

1
워크플로우 정의
데이터 객체로서의
형식화된 프로세스
2
워크플로우 인스턴스
체크포인트로서의
지속적 실행 상태
3
추론 · 승인 기록
검사 가능한
의미적 객체들

의미론적 지속성(Semantic Persistence)이란 단일 실행 주기를 넘어, 타입화된 워크플로우 객체와 추론 기록, 그들의 정체성과 관계, 후속 검사에 필요한 맥락을 보존하는 것을 의미합니다.

세 가지 개념 계층

상위 계층
의미론적 계층
  • 워크플로우 정의
  • 워크플로우 인스턴스
  • 추론 · 승인 · 심의 기록
중간 계층
제어 계층 (DSL 머신)
  • 객체 해석 및 맥락 조립
  • 모델/도구 호출 중재
  • 허용된 전이 유효성 검사
하위 계층
런타임 서비스 계층
  • 모델 어댑터
  • 도구 · 외부 프로세스
  • 지속성 / 인덱싱

제어 계층은 의미론적 객체를 읽고, 실행 결과를 지속적 의미론적 객체나 관계로 다시 쓰는 양방향 관계를 가집니다.

핵심 구분: derive vs infer

본 모델의 중심적인 의미론적 구분입니다. 결정론적 계산과 LLM 중재 판단을 문법적으로도, 의미론적으로도 분리합니다.

비교 항목 derive infer
성격 결정론적 계산 LLM 중재 판단
입력 가용 상태 전체 선언된 맥락 하에서
통제 실행기 완전 통제 실행기 정책 하에 능력 제한
재현성 동일 입력 → 동일 출력 비결정론적
기록 요건 간단한 상태 전이 맥락, 입력, 출력, 정책 모두 기록

LLM은 자율적 프로세스 감독자가 아니라 중재된 주변기기(mediated peripheral)다.

핵심 의미론적 스키마

워크플로우 정의, 인스턴스, 맥락 스냅샷, 추론 기록, 의존 관계가 타입화된 영속 객체로 표현됩니다.

워크플로우 정의

단계, 상태 전이, 인간/모델 상호작용을 선언한 프로세스 명세. 실행 가능한 절차가 아닌 데이터 객체.

워크플로우 인스턴스

실행기가 정의를 해석할 때 생성 또는 재개됨. 상태가 체크포인트로 기능.

맥락 스냅샷

infer 호출 시점의 선언된 맥락을 기록. 후속 검사에 필수.

추론 기록

infer 호출의 맥락, 입력, 출력, 능력 정책을 포함하는 별개의 의미론적 객체.

승인 · 심의 기록

승인은 권한 부여 이벤트, 패널은 동의, 논증, 결정을 포함하는 구조화된 심의 발생.

의존 관계

객체 간의 명시적 링크. 숨겨진 인과 단계가 아닌 선언적 관계.

실행 의미론과 의미론적 체크포인트

워크플로우 인스턴스의 상태가 곧 체크포인트입니다. 인스턴스는 언제든 검사 가능하고 재개 가능한 라이브 객체입니다.

정의 해석

실행기가 워크플로우 정의 객체를 읽어 인스턴스 생성 또는 재개

derive 단계 실행

결정론적 상태 계산 → 인스턴스 상태 갱신

infer 슬롯 중재

선언된 맥락 + 능력 정책 하에 LLM 호출 → 추론 기록 객체 생성

의미론적 체크포인트 기록

인스턴스 상태 + 추론 기록 + 의존 관계를 지식 기판에 영속화

재개(Resumption)

중단 후 체크포인트에서 인스턴스 상태를 복원하여 실행 속행

실행 중심 접근법과의 대조

기존 에이전트 워크플로우 시스템은 실행 관심사를 잘 해결하지만, 의미론적 객체 모델은 별도의 문제입니다.

관점 실행 중심 시스템 본 모델 (의미론적 지속성)
워크플로우 정의 소스 코드 / 설정 파일 타입화된 지식 객체
실행 인스턴스 런타임 상태 (휘발적) 영속적 라이브 객체
LLM 호출 함수 호출과 동일 취급 derive/infer 구분
인간 상호작용 UI 이벤트 / 비구조화 채팅 타입화된 승인/심의 기록
프로베니언스 인프라 상관관계 명시적 의미론적 관계

네 가지 설계 주장

1
정의는 데이터 객체다

워크플로우 정의는 실행 가능한 절차나 외부 설정뿐 아니라 데이터 객체로 표현되어야 한다.

2
인스턴스는 라이브 객체다

워크플로우 인스턴스는 상태가 체크포인트로 기능하는 영속적 라이브 객체로 존재해야 한다.

3
derive와 infer를 분리하라

LLM 호출은 문법적으로도 의미론적으로도 결정론적 상태 계산과 구별되어야 한다.

4
인간 상호작용을 기록하라

인간 대면 추론 상호작용은 UI 이벤트나 비구조화 채팅으로 사라지지 않고 타입화된 기록으로 유지되어야 한다.

Lisp에서 빌려온 설명적 렌즈

Lisp는 구현 약속이 아닌 설명적 렌즈로 사용됩니다. 언어 독립적인 개념 모델입니다.

기호 형식 (Symbolic Forms)

객체 구조에 가까이 머무는 표현 형식. 코드가 데이터를 직접 반영.

객체 정체성 (Object Identity)

각 워크플로우 객체가 고유한 정체성을 가지며, 관계가 명시적으로 추적됨.

반성적 검사 (Reflective Inspection)

실행 중에도 워크플로우 구조와 상태를 조회하고 검사 가능.

라이브 이미지 (Live Image)

중단-재개 가능한 실행 상태 이미지. 선험이 아닌 유추적 비유.

확장 가능한 객체 프로토콜

새로운 기록 유형과 관계를 선언적으로 추가 가능한 유연한 스키마.

탐색적 어휘 스캔

제안된 모델을 기술하기 위한 기본 어휘(primitive vocabulary)의 개요입니다.

제어
  • resource
  • context
  • guard
  • derive
  • infer
인간 상호작용
  • approval
  • panel
  • record
  • state
효과
  • capability / action
  • handoff / promotion

이 어휘는 최종 명세가 아닌 탐색적 스캔의 결과입니다. 각 원시어가 모델 내에서 어떤 역할을 수행하는지, 어떤 경계를 그리는지를 실험적으로 검토한 것입니다. 향후 형식적 전이 의미론 작업에서 정교화될 예정입니다.

논의 및 한계

명시적 한계
  • 개념 모델 제안 — 구현 및 경험적 평가 없음
  • 형식적 전이 의미론은 향후 작업
  • 의미론적 지속성이 정확한 귀인, 재현성, 신뢰, 감사 품질을 자동으로 보장하지 않음
  • 특정 디바이스 아키텍처나 구현에 의존하지 않으므로 구체적 성능 논의 불가
기여의 위치
  • 객체 모델과 어휘 자체가 핵심 기여
  • 의미론적 객체 스키마와 원시 어휘 제공
  • derive/infer 경계의 명시적 구분
  • PROV-DM에 대한 예비 매핑 (부록 C)

"네 가지 주장은 이 프레이밍에서 따른 설계 약속이며, 검증된 주장으로 취급되기 전에 향후 경험적 또는 형식적 평가가 필요한 항목들이 명시되어 있다."

결론

워크플로우는 단순히 지식을 생산하고 흔적을 남기는 것이 아니라, 그 자체로 검사 가능하고, 재개 가능하고, 검토 가능한 지식 객체로 표현될 수 있습니다.

derive와 infer의 구분, 세 계층 아키텍처, 타입화된 기록 객체들은 이 변화를 뒷받침하는 개념적 도구입니다. 형식적 의미론과 경험적 검증은 아직 남은 과제입니다.
