왜 AI 에이전트로 칩 설계인가?

전통적 칩 설계의 문제

  • 비용: $4억 이상, N2 마스크만 $3천만+
  • 기간: 18~36개월, 수백 명 인력
  • 리스핀: 버그 발견 시 막대한 추가 비용
  • 소프트웨어와 달리 패치 배포 불가

칩 설계의 고유한 요구

  • 논리 설계부터 물리 설계까지 전 단계 통합
  • 소프트웨어보다 훨씬 엄격한 검증 필수
  • 코딩 에이전트와는 다른 에이전트 설계 필요
  • 타이밍, 파워, 면적 등 물리적 제약 고려

VerTQ: TurboQuant 추론 가속기

Design Conductor 2.0이 자율적으로 설계한 LLM 추론 전용 가속 칩

TurboQuant란?

Google이 2026년 3월 발표한 KV-캐시 압축 알고리즘

  • KV 메모리 사용량 4.3배 압축
  • 어텐션 내부 곱셈 16배 감소
  • 성능/품질 완전 보존
  • 발표 당시 하드웨어 구현 존재하지 않음

Google 발표 직후 메모리 칩 제조사 주가에 직접적 영향

VerTQ의 역할

프로세서와 DRAM 사이에 배치되어 KV-캐시를 관리

  • 압축된 상태 그대로 FlashAttention 수행
  • Online Softmax 온칩 구현
  • KV 압축/복압축 처리
  • Python vLLM 프레임워크 직접 통합

타겟: 자율주행, 드론, 로봇 등 엣지 AI

Conductor 2.0은 아키텍처 단계에서 기술 제약을 사전 반영

XCVU29P-3 FPGA 구조

  • 총 4개 SLR (Super Logic Region) 다이
  • VerTQ는 3개 SLR에 맵핑
  • SLR 간 신호 교차는 지연이 큼
  • 다이 간 교차를 최소화하도록 아키텍처 설계

설계 인사이트

Conductor 2.0은 아키텍처/명세 단계에서 기술 제약을 고려하여, 사용 가능한 기술 자원에 최적으로 맵핑되는 칩을 설계했다.

이는 단순한 RTL 생성을 넘어, 물리적 현실을 이해하는 설계 능력을 보여줌

Conductor 2.0에 제공된 입력

약간의 요구사항 텍스트 + TurboQuant 논문 참조 + 소프트웨어 구현만 제공

사용자 입력 요약

  • 타겟: Xilinx XCVU29P-3 FPGA, Vivado 사용
  • 역할: 메모리 앞단 블록, KV 캐시 압축/복압축 + 어텐션 계산
  • vLLM 프레임워크 통합 (시뮬레이션 기반)
  • 타겟 클럭: 125 MHz, 가장 공격적인 합성/P&R 옵션
  • FP16 KV 캐시 정밀도, K: 3 PQ 비트 + 1비트 QJL
  • 데모 모델: Qwen3-4B
  • 에러 핸들링 불필요, DAZ/FTZ 적용 간소화

이 입력만으로 Conductor 2.0은 80시간 동안 아키텍처부터 레이아웃까지 전 과정을 자율 수행

Conductor 2.0의 핵심 새로운 능력

컨셉 → 레이아웃

논문/아이디어에서 시작하여 물리 레이아웃까지 전 과정 자율 수행

DC 1.0은 RTL 수준에 머물렀으나, DC 2.0은 합성, P&R, 타이밍 클로저까지 처리

아키텍처 이해와 제1원리 추론

논문을 읽고 하드웨어 아키텍처를 스스로 설계하는 능력

SLR 간 교차 최소화 등 기술 제약을 아키텍처 단계에서 사전 반영

루프 종결 (Closing the Loop)

합성/타이밍 결과를 바탕으로 RTL을 자동 수정

백엔드 결과 → 마이크로아키텍처 수정 → 재합성의 피드백 루프 자율 운영

수학적 추론 능력

TurboQuant 수학적 알고리즘을 이해하고 하드웨어로 변환

양자화 수학, 소프트맥스 누적 등 복잡한 수치 연산의 하드웨어 최적화

검증 방법과 도전 과제

검증 접근법

  • 모듈별 광범위한 테스트벤치 자동 작성
  • Python 참조 구현으로 기능 및 사이클별 동작 디버그
  • Qwen3-4B 실제 데이터로 시스템 시뮬레이션
  • 36레이어 전체 통과 (컨텍스트 64)

에뮬레이션 환경 시 컨텍스트 32K 테스트 가능성 언급

주요 도전 과제

  • 부동소수점 수치 디버그에 막대한 토큰 소모
  • FP16/FP32 혼합 사용으로 Python 참조와 미세 차이 발생
  • 모든 불일치가 잠재적 하드웨어 버그일 수 있어 개별 원인 분석 필요
  • 고노름 입력 벡터에서 더 큰 차이 발생 가능

한계점과 개선 방향

과도하게 방법론적

때로 불필요하게 철저한 접근으로 토큰 낭비 발생

백엔드 vs 마이크로아키텍처 수정

타이밍 문제를 RTL 수정이 아닌 백엔드 옵션으로 해결하려는 경향

목표 설정의 한계

적절한 설계 목표를 독립적으로 설정하는 능력 여전히 부족

인간 검토 병목

자율성 향상에도 불구하고 인간 검토 단계가 병목으로 작용

핵심 관찰

DC 1.0 시절 "숙련된 구현자"에 가까웠다면, DC 2.0은 진정한 "설계자"에 가까워졌지만, 여전히 완전한 자율 설계에는 이르지 못했다.

모델과 하네스의 빠른 공진화가 핵심 동력이며, 이 속도는 점점 가속화되고 있음
