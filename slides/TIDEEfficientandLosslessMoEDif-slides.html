배경

Diffusion LLM이 주목받는 이유

자기회귀(AR) 모델의 한계

  • 토큰을 순차적으로 하나씩 생성
  • 단방향 컨텍스트만 활용
  • 하드웨어 활용도 낮음 (memory-bound)

Diffusion LLM(dLLM)의 장점

  • 블록 단위 병렬 디코딩
  • 양방향 컨텍스트 활용
  • 높은 하드웨어 활용도

LLaDA 시리즈를 비롯한 dLLM은 AR 모델과 동등한 성능을 달성하면서 훨씬 높은 디코딩 처리량을 제공합니다.

문제 정의

MoE-dLLM 배포의 핵심 과제

LLaDA-2는 MoE 백본을 채택하여 8B → 100B 파라미터로 확장했지만, 자원 제한 환경에서의 배포는 미해결 과제입니다.

1

광범위한 전문가 활성화

한 디노이징 스텝에서 블록 내 모든 토큰의 전문가를 동시에 활성화 → OOM 위험

2

과도한 I/O 오버헤드

매 스텝마다 전문가를 GPU-CPU 간 스왑하면 막대한 I/O 트래픽 발생

3

CPU 컴퓨트 병목

토큰 연산을 CPU로 라우팅하면 GEMM에서 GPU 대비 수십 배 느림

필요: 최소 I/O 오버헤드 + 최대 컴퓨트 효율의 오케스트레이션

핵심 통찰

전문가 라우팅의 시간적 안정성

블록 내 인접한 디노이징 스텝 간에 전문가 활성화 패턴이 매우 유사하다는 발견

유사도 히트맵 관찰

  • 인접 스텝 간 코사인 유사도 ≈ 0.95
  • 5스텝 차이에서도 높은 유사도 유지
  • 대각선 밴드 패턴으로 안정성 확인

고유 전문가 수 추이

  • 초기 스텝에서 적은 수 활성화
  • 디노이징 진행에 따라 점진적 증가
  • 하지만 인접 스텝 간 변화는 작음

→ 전문가 재사용 기회: 매 스텝마다 전문가를 교체할 필요 없이, 일정 간격 동안 동일한 전문가 배치를 유지해도 높은 적중률을 보장할 수 있습니다.

방법론

간격 기반 전문가 갱신 전략

Refresh Step

  • 각 전문가의 토큰 적중 수(hit count) 집계
  • 가장 많은 토큰을 처리한 CPU 전문가를 GPU로 승격
  • GPU 메모리 예산에 맞춰 최적 집합 구성
  • I/O-aware: 실제 데이터 전송량 최소화

Skip Step

  • 현재 전문가 배치를 그대로 유지
  • GPU 전문가 → GPU에서 연산
  • CPU 전문가 → 비동기 라우팅으로 CPU 연산
  • 불필요한 메모리 이동 제로화

핵심 아이디어: 라우팅 안정성을 활용해 간격이 짧을수록 GPU 적중률이 높지만 I/O가 잦고, 간격이 길수록 I/O는 줄지만 CPU 연산이 증가합니다. 최적 간격을 찾는 것이 핵심입니다.

최적화

수학적 프로그래밍을 통한 최적 간격 탐색

추론 스케줄링을 제약 조건부 수학적 프로그래밍(MP) 문제로 공식화

1단계: 하드웨어 프로파일링

GPU-CPU 전송 대역폭, GPU GEMM 속도, CPU GEMM 속도 등 시스템 파라미터 측정

2단계: 분석적 지연 모델 구축

간격 τ에 대한 총 지연 = I/O 지연(τ 감소에 따라 증가) + CPU 컴퓨트 지연(τ 증가에 따라 증가)

3단계: 탐욕적 탐색으로 최적 τ 도출

프로파일링 결과와 결합하여 제약 조건 하에서 총 처리량을 최대화하는 τ* 탐색

런타임 오버헤드 최소화: 프로파일링은 한 번만 수행되며, 탐색 공간은 매우 작음

CPU Rerouting
[Kamahori '24]		 누락 전문가를 CPU에서 연산 없음 매우 높음 부적합
TIDE (본 연구) 간격 기반 I/O-aware 갱신 최소화 최소화 최적

dLLM은 AR과 달리 매 스텝에서 전체 블록의 토큰을 동시에 처리하므로, 활성화되는 전문가 풀이 훨씬 크고 다양합니다. 기존 AR용 방법은 이를 처리하지 못합니다.
