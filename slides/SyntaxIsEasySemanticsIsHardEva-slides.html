02 / 12 — 연구 배경

왜 LTL 번역이 중요한가?

LTL이란?

선형 시간 논리(Linear Temporal Logic)는 소프트웨어, 네트워크, 시스템의 요구사항 및 보안/개인정보 보호 정책을 형식적으로 명세하는 데 사용되는 논리 체계입니다.

핵심 문제

LTL의 복잡한 의미론 때문에 도메인 전문가들이 요구사항을 LTL로 표현하기 어렵습니다. LLM이 이 간극을 메울 수 있을까요?

핵심 가설: LLM이 자연어를 LTL로 자동 번역할 수 있다면, 형식 검증 도구의 접근성을 크게 높일 수 있습니다.

03 / 12 — 연구 질문

연구 질문 및 과제 설계

구문적 과제 (Syntax)

  • RQ1syn — 원자 명제 추출
    자연어에서 원자 명제를 정확히 식별
  • RQ2syn — 올바른 형태 판별
    LTL 수식의 문법적 유효성 판단

의미적 과제 (Semantics)

  • RQ1sem — 자연어 → Future LTL 생성
  • RQ2sem — 트레이스 분류 (만족/위반)
  • RQ3sem — 트레이스 생성
  • RQ4sem — 자연어 → Past LTL 생성

평가 기준: 동치성(Equivalence), 건전성(Soundness), 완전성(Completeness), 구문 유효성(Syntax Validity)

검증 도구: NuSMV 모델 체커를 사용한 형식 검증

08 / 12 — 오류 분석

LLM은 어디서 실패하는가?

시간 연산자 범위 오류

G, F, X 연산자의 범위(scope)를 잘못 지정하는 지배적인 실패 모드

부정(Negation) 오류

복합 수식에서 NOT을 잘못된 부분수식에 적용

연산자 혼동

"항상(G)"과 "결국(F)"을 혼동하거나, 과거 연산자(Y, H, S)의 상호작용을 오해

중첩 복잡도

연산자 중첩 깊이가 증가할수록 정확도가 급격히 하락

구체적 오류 예시

입력: "Mary는 팀에 합류하지 않고 플루트를 연주하지 않을 것이다"
정답
~p ∧ ~q
LLM 오류
단일 원자 명제로 처리 → 논리 구조 소실
중첩 연산자 트레이스 분류 오류
X(X(X(X(F(x1)))))
단순한 만족 트레이스임에도 빈번하게 오분류 — 깊이가 깊어지면 실패

보안 vs 일반 도메인

보안 도메인에서 동치성이 3~16%p 하락합니다. 원자 명제 매핑의 영향이 일반 도메인(1~2%p)보다 보안 도메인(6~19%p)에서 훨씬 큽니다.

Self-Refinement

자기 수정(Self-refine) 전략이 보안 도메인에서 약간의 개선을 보이지만, 근본적인 의미적 오류를 해결하지 못합니다.

11 / 12 — 핵심 발견

6가지 핵심 발견

1

구문-의미 격차

~50% 구문 유효 vs 24.4% 의미 동치 — 패턴 암기는 하되 시간 관계 이해 부족

2

Python 인터페이스 우월

코드 완성 형태로 재구성하면 동치성 +23.8%p 향상, 통계적으로 유의미

3

상세 프롬프트 효과

BNF 문법과 의미 정의를 포함하면 전반적 성능 향상, 특히 올바른 형태 판별에서 큰 효과

4

복잡도에 취약

연산자 중첩 깊이 증가 시 의미적 정확도 급격히 하락

5

위반 트레이스 어려움

만족 트레이스보다 위반 트레이스 생성이 일관되게 더 어려움

6

보안 도메인 추가 난이도

보안 정책 변환에서 동치성 3~16%p 추가 하락, 원자 명제 매핑 영향이 극대화

12 / 12 — 결론

결론 및 향후 연구

핵심 결론

LLM은 LTL 형식 명세에 잠재력을 보이나, 구문적 정확성만으로는 충분하지 않습니다. 의미적 검증이 필수적이며, 현재 LLM 출력에 대한 인간 검증 없이는 신뢰할 수 없습니다.

향후 연구 방향

하이브리드 접근법

LLM 생성 + 형식 검증 도구 결합으로 의미적 오류 자동 포착

학습 데이터 확장

더 다양한 LTL 예제로 훈련 데이터 보강, 도메인 특화 미세조정

다단계 수정 프로세스

반복적 자기 수정 및 검증 루프를 통한 점진적 품질 향상

의미적 정확성 메트릭

구문 검증을 넘어선 더 정교한 의미적 평가 지표 개발
