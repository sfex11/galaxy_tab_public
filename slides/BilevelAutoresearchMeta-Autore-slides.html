자동연구 루프를 자동연구 루프로 최적화하는 자기참조적 프레임워크

자동연구가 그 자체로 연구의 한 형태라면, 자동연구를 연구 자체에 적용할 수 있다.

기존 자동연구의 한계

기존 시스템들은 왜 한계에 부딪혔는가?

기존 자동연구 시스템들

  • Karpathy (2026) — 단일 트랙 루프: LLM이 코드를 읽고, 변경을 제안하고, 실행하고, 결과를 측정
  • AutoResearchClaw — 다중 배치 병렬 탐색 확장
  • EvoScientist — 지속적 경험 메모리 추가

공통적 문제점

모든 개선은 인간 연구자가 직접 코드를 읽고, 병목을 식별하고, 새 코드를 작성해야 했음. 자동연구 루프 자체를 개선하는 것은 자동화되지 않았다.

외부 루프가 동일한 설계 단계를 자율적으로 수행할 수 있는가?

수학적 프레임워크

이중 수준 최적화의 형식화

이중 수준 최적화

상위 수준: minφ F(φ, θ*(φ))

하위 수준: θ*(φ) ∈ arg minθ f(θ, φ)

  • φ = 탐색 메커니즘 (runner 코드) — 이산적 프로그램
  • θ = 학습 설정 (하이퍼파라미터)

기존 방식

상위 수준이 실수값 파라미터를 경사 기반으로 최적화

본 논문

상위 수준이 코드(프로그램)를 LLM으로 생성하여 메커니즘을 교체

동일한 LLM(DeepSeek deepseek-chat)이 모든 수준에서 사용됨
메타 수준에 더 강한 모델 불필요

Level 2: 4라운드 대화

메커니즘 연구 및 코드 주입 프로세스

1

탐색 (Explore)

LLM이 runner.py 소스와 탐색 추적을 읽고, 인접 분야(조합 최적화, 온라인 학습, 실험 설계, 베이지안 최적화)의 메커니즘을 조사

2

비평 (Critique)

관찰된 실패 모드에 대해 후보를 평가하고 가장 유망한 것을 선택

3

명세 (Specify)

클래스명, 생성자 인수, 메서드 시그니처, runner.py 통합 지점을 포함한 인터페이스 명세 작성

4

생성 (Generate)

완전하고 실행 가능한 Python 코드 작성 → importlib로 동적 로드 → 검증 성공 시 활성화, 실패 시 자동 복원

실험 설계: 4그룹 제거 실험

벤치마크: Karpathy의 GPT 사전학습 과제 (50M 파라미터)

그룹 활성 수준 설명
A Level 1만 순수 자동연구, 외부 개입 없음
B Level 1 + 1.5 내부 루프 + 파라미터 조정
C Level 1 + 1.5 + 2 전체 이중 수준 + 메커니즘 연구
D Level 1 + 2 메커니즘 연구만 (전략 조정 없이)

  • GPU: RTX 5090 (32GB) x 3대
  • 학습 예산: 300초
  • 탐색 예산: 30회 반복

코드 예시: 타부 서치 매니저

Group C R1에서 LLM이 자율 생성한 코드 (발췌)

class TabuSearchManager: def __init__(self, tenure=5, thresholds=None): self.tabu_list = [] self.tenure = tenure self.distance_thresholds = thresholds or {} def is_tabu(self, config, iteration): # 만료된 항목 제거 self.tabu_list = [e for e in self.tabu_list if iteration <= e["expires_at"]] for entry in self.tabu_list: for param, tabu_val in entry["config"].items(): if param in config: try: diff = abs(float(config[param]) - float(tabu_val)) thresh = self.distance_thresholds.get( param, 0.0) if diff <= thresh: return True # 너무 가까움, 차단 except (ValueError, TypeError): if config[param] == tabu_val: return True return False # 허용됨

역할: 실패한 변경과 유사한 방향을 다시 시도하지 못하게 차단 → LLM이 새로운 탐색 방향으로 전환하도록 강제

결정적 발견: TOTAL_BATCH_SIZE

5~8배 성능 향상의 원인

발견된 최적화

TOTAL_BATCH_SIZE를 219에서 217~218로 감소
→ 검증 손실 -0.039 ~ -0.065 향상

왜 효과적인가?

더 작은 배치 → 고정 시간(300초) 내 더 많은 경사 스텝 → 50M 파라미터 모델에서 더 나은 수렴. 원래 219는 H100에 최적화되었으나, RTX 5090 + SDPA 환경에서는 최적이 아님.

Group A/B가 놓친 이유

DeepSeek의 기본 탐색: 배치 크기 증가를 먼저 시도 ("큰 배치가 더 좋다"는 암묵적 편향). 증가가 거부된 후 A는 반복하고, B는 이 파라미터를 동결하여 감소 방향을 완전히 차단.

Group C/D가 발견한 이유

Level 2 메커니즘이 결정적 역할: 타부 서치가 실패한 방향 재방문을 차단하고, 직교 탐색이 차원 다양성을 강제하여 감소 방향을 시도하게 함.

가설 검증 결과

H1: Group B >> Group A  미지지

Group B 평균(-0.006)이 Group A(-0.009)보다 오히려 수치적으로 나쁨. B의 R1은 개선 없음(-0.000)으로 전체 최악 결과.

H2: Group C >> Group B  지지됨

Group C 절대 개선(-0.045)이 Group B(-0.006)의 7.5배. 두 반복에서 극적 향상(-0.065, -0.058).

H3: Level 2가 자율적으로 메커니즘 발견  지지됨

조합 최적화, 온라인 학습, 실험 설계 분야에서 메커니즘 생성. 도메인 명세 없이 코드 첫 시도 성공, 6건 중 5건 검증 통과.

H4: Group D ≈ Group C  조건부 지지

D(-0.034)와 C(-0.045) 분산 범위 내. Level 2만으로 충분하나, D의 R1은 개선 없음 → Level 1.5가 약간의 안정성 제공.

탐색 행동 비교 분석

Group A: 거의 결정론적 반복

3회 반복 모두 거의 동일한 제안 순서. WEIGHT_DECAY 감소(Δ≈-0.008) → WINDOW_PATTERN 변경(Δ≈-0.002) → 22회 연속 거부. LLM이 동일 상태에서 거의 동일한 제안을 생성.

Group B: 방향 전환되지만 한계 존재

Level 1.5가 정체된 파라미터를 올바르게 식별하고 LR, MATRIX_LR 등으로 리디렉션. 더 많은 파라미터를 탐색하지만 동일한 수락/거부 프레임워크 내에서 유사한 개선.

Group C: Level 2가 새로운 방향 개방

메커니즘 주입 후 TOTAL_BATCH_SIZE 감소 발견 (219 → 217~218). 단일 변경으로 -0.039 ~ -0.065 — Group A/B 어떤 변경보다 5~8배 큰 효과.

Group D: Level 2만으로도 가능하지만 불안정

R2, R3에서 독립적으로 배치 크기 감소 발견. 그러나 R1은 실패(두 메커니즘 모두 import 검증 실패) → 더 높은 분산(±0.031).

한계점

작은 표본 크기

그룹당 3회 반복은 엄밀한 비교에 불충분. Group C 표준편차(±0.030)가 절대 평균의 67%. 신뢰할 수 있는 추정에는 n≥10 필요.

단일 벤치마크

모든 결과가 하나의 과제(GPT 50M, 300초, RTX 5090)에서만 검증. 다른 모델 크기, 예산, 과제에 대한 일반화 미입증.

기준선 변동

기준선 val bpb가 반복마다 변동(1.094~1.114). Δ 사용으로 완화하지만 완전히 제거되지 않음.

동적 로드 취약성

예비 실행에서 sys.modules 등록 버그로 모든 메커니즘 주입이 원본으로 조용히 폴백. 무음 실패는 위험한 실패 모드.

프롬프트 유도 편향

Level 2 프롬프트가 명시적으로 특정 도메인(조합 최적화, RL, 진화 알고리즘 등)을 제안 → 발견 가능 도메인이 제한.

결론

검증된 핵심 원리

자동연구는 자기 자신을 연구할 수 있다

외부 루프가 인간이 설계할 필요 없음 — 내부 루프를 실행하는 동일 모델이 구조적 개선을 자율 생성할 수 있음.

주요 성과

  • 내부 루프 대비 5배 검증 bpb 개선
  • 3개 분야에서 자율 메커니즘 발견
  • LLM 기본 탐색이 피하는 방향으로 탐색 강제

향후 연구

  • n≥10 반복 + 고정 시드
  • 다양한 벤치마크로 일반화 검증
  • 코드 생성 품질 향상을 위한 테스트 하네스
