동기: 열린 세계의 가정 서비스 로봇
"데운 치킨 버거를 내어주세요"
— 단순해 보이지만, 로봇에게는 복잡한 문제
  • 버거의 위치 추론
  • 치킨 재질인지 판별
  • 전자레인지 '+' 버튼의 가열 시간 효과 추정
가정 환경의 본질적 불확실성
  • 객체가 숨겨져 있을 수 있음
  • 객체 속성을 알 수 없음
  • 행동 효과가 불명확함
  • 예상치 못한 상황 발생
기존 접근법의 한계
닫힌 세계 기반 계획
  • 사전 정의된 상태 표현에 의존
  • 수동 설계된 지식 베이스 필요
  • 예상치 못한 상황에서 실패
  • 지식 확장 능력 부재
LLM 직접 계획
  • 환각(hallucination) 위험
  • 장기horizon 일관성 부족
  • 제약 조건 강제 어려움
  • 불확실성 하 능동적 정보 획득 불가
최근 하이브리드 방법들도 수동적 추론에 머물러 있음 — 모델 생성 후 불확실성을 무시하면 침묵적 실패(silent failure) 발생 가능

가설 생성: FM 기반 불확실성 표현
가설의 유형
  • 상태 가설: 객체 위치, 속성 (예: "이 버거는 치킨이다")
  • 전이 가설: 행동 효과 (예: "'+' 버튼은 가열시간을 30초 증가")
핵심 원칙
  • FM 출력을 사실이 아닌 불확실한 잠재 변수로 취급
  • 작업 지향적 가설 생성 (task-oriented)
PDDL 확장 예시
(:predicates
  (is-chicken ?obj)
  (h-is-chicken ?obj) ; 가설
  (h-not-chicken ?obj) ; 대립가설
)

(:action check-chicken
  :precondition (and ...)
  :effect (and
    (not (h-is-chicken ?obj))
    (not (h-not-chicken ?obj))
  ))
불확실성 하 계획 수립
비용 기반 행동 분할 (Cost-Based Action Splitting)
각 원본 행동을 가설별 분기 행동으로 변환하여, 고비용 분기가 검증 행동을 유도
확장 문제 구성
  • 기존 모델 + 가설 서술자 결합
  • 동적 검증 행동 자동 생성
  • 적절한 결정화(determinization) 적용
  • 고전적 계획기(FF 등) 사용 가능
계획의 특징
  • 부분 정지(plan partial grounding): 검증 전 행동은 조건부
  • 작업 실행과 검증이 명시적으로 교차(interleave)
  • 불필요한 검증은 비용으로 자연스럽게 회피
 고전적 계획기의 비용 최소화가 자동으로 효율적인 검증-실행 순서를 발견
가설 검증 및 모델 갱신
1
행동 실행

2

센서, 이미지 등 새로운 정보 수집
3

가설의 이중 역할
상상(Imagination): 아직 관측되지 않은 속성에 대한 추론
기억(Memory): 이전 검증 결과의 유지·활용
오류 처리
가설이 틀린 경우 → 모델 갱신 후 재계획
실행 실패 → 새로운 가설 생성 후 반복

주요 실패 원인 (H Only): 가설 검증 없이 잘못된 속성 추정 → 잘못된 행동 실행 → 되돌릴 수 없는 상태
핵심 분석: 왜 HUME이 잘 동작하는가
침묵적 실패 방지
검증 없이 잘못된 모델로 계획하면, 실행은 되지만 목표 달성 불가 → HUME는 검증을 계획에 편입하여 사전 차단
효율적 검증 순서
비용 기반 분할이 불필요한 검증을 자연스럽게 회피하고, 작업 경로 상의 검증만 선택
구조적 추론 + 개방적 지식
PDDL의 구성적 계획 구조와 FM의 상식적 사전지식을 불확실성을 통해 통합
지속적 모델 성장
가설이 검증 → 확정 사실로 모델에 통합 → 이후 작업에서 재사용 가능
제한사항 및 향후 과제
제한사항
  • 불확실성 표현이 이진 가설로 제한됨
  • 가설 검증 조건이 FM 의존적
  • 추상화 수준을 수동으로 설정 필요
  • 가설 생성의 완전성 보장 어려움
향후 과제
  • 더 풍부한 불확실성 표현 (확률 분포 등)
  • 검증 조건 예측의 일반화
  • 다중 출처 가설 생성 (비전, 물리 엔진 등)
  • 더 복잡한 실제 가정 환경으로 확장
결론
열린 세계에서 로봇의 자율적 지식 확장은 모델 확장 자체를 불확실하고 능동적으로 검증 가능한 과정으로 다루어야 한다.
