Sovereign Execution Brokers: Enforcing Certificate-Bound Authority

Jun He, Deying Yu · OpenKedge.io

자율 에이전트의 인프라 제어 위험

LLM 기반 자율 에이전트가 클라우드, 배포, 데이터 제어 워크플로우에 직접 연결되는 추세다. 하지만 비결정적 추론 과정에 프로덕션 변경 권한을 부여하는 것은 심각한 보안 위험을 초래한다.

환각(Hallucination) 위험

단일 환각 또는 적대적 프롬프트 주입이 데이터베이스 삭제, 보안 그룹 개방 등 파괴적 인프라 변경을 유발할 수 있다.

정적 자격증명의 문제

에이전트에 정적 IAM 자격증명을 부여하면, 프롬프트가 탈취되거나 추론 루프가 오작동할 때 모든 권한이 남용될 수 있다.

기존 접근법의 한계

메커니즘 강제하는 것 부족한 것
IAM 신원 권한 제안 의미론(Semantics)
OPA 정책 정책 술어(Predicates) 인증서 기반 권한
감사 로그 사후 기록 사전 실행 방지
SAB 승인 인증서 발급 실행 시점 강제
SEB (본 논문) 인증서 기반 실행 강제 강제 변경 경로 배포 필요

IAM은 "이 신원이 이 작업을 할 수 있는가?"를 묻지만, SEB는 "이 특정 인증된 변경 요청이 현재 시스템 상태에서 여전히 유효한가?"를 묻는다.

SAB만으로는 부족한 이유

SAB(Sovereign Assurance Boundary)는 제안의 의미적 안전성을 검증하여 서명된 인증서 Ω를 발급하지만, Ω는 암호적 주장일 뿐 강제 수단이 아니다.

1
바이패스 가능

에이전트가 정적 자격증명으로 API를 직접 호출하여 승인 과정을 우회

2
시간 간격 취약점

승인과 실행 사이에 상태 드리프트나 보안 환경 변화 발생 가능

3
재생 공격

만료되거나 취소된 인증서로 반복 실행 시도 가능

SEB: 주권 실행 브로커 소개

SEB는 인증서 기반 에이전트 인프라를 위한 런타임 강제 경계다. 에이전트 런타임은 프로덕션 변경 자격증명을 보유하지 않으며, 모든 자율 변경은 SEB를 통과해야 한다.

1
인증서 검증
SAB 서명 Ω 검증
2
계약 일치 검사
요청 vs 인증된 계약 C
3
상태 검사
유효성, 에포크, 드리프트
4
범위 지정 실행
단명 자격증명 발급 후 실행
5
기록
서명된 결정/결과 기록

실행 모델: 인증서 검증 술어

브로커 실행 인터페이스는 다음과 같이 정의된다:

Execute(Ω, req, St, Platform) → D | O

  • Ω 검증: SAB 인증서의 암호적 서명 및 체인 유효성 확인
  • 계약 일치: 요청된 변경이 인증된 계약 C와 정확히 일치하는지 검증
  • 유효성 창: 인증서의 시간 유효성 창 내인지 확인
  • 정책/취소 에포크: 현재 활성 에포크와 일치하는지 확인
  • 라이브 상태 드리프트: 현재 인프라 상태 St가 계약과 여전히 호환되는지 확인
  • 논스 예약: 재생 공격 방지를 위한 일회성 논스 검증

범위 지정 실행 신원 모델

SEB는 최소 권한 원칙에 따라 실행 시점에만 단명 자격증명을 발급한다. 에이전트는 재사용 가능한 변경 권한을 절대 받지 않는다.

범위 차원 (Scope Dimensions)
  • 인증된 계약 C에 바인딩
  • 대상 리소스에 제한
  • 유효성 창에 제한
  • 취소 에포크에 바인딩
바이패스 방지 배포
  • AWS: IAM 정책으로 비브로커 신원 거부
  • K8s: Admission Webhook으로 비브로커 요청 차단
  • 모든 자율 경로가 SEB를 통과하도록 강제

TOCTOU 보호와 드리프트 탐지

에이전트 시스템에서 검사 시점과 사용 시점 사이(TOCTOU)의 간격은 본질적인 문제다. 승인과 실행 사이에 인프라 상태가 변할 수 있다.

에포크 기반 취소

정책 에포크와 취소 에포크를 분리하여, 대기 중인 인증서를 실행 전에 무효화 가능

드리프트 술어

실행 시점에 라이브 상태 St를 조회하여 인증서 계약과의 호환성 실시간 검증

실패-폐쇄(Fail-Closed)

상태 확인 불가 시 기본적으로 실행을 거부하는 보수적 의미론 채택

동작 예시: 방화벽 포트 개방

1
제안 (Proposal)

에이전트 Aops가 포트 443을 CIDR 10.0.0.0/16에 개방하는 계약 Cfw 제안

2
승인 (Admission)

SAB가 기업 방화벽 정책에 대해 Cfw 검증 후 인증서 Ωfw 발급

!
개입 사건 (Intervening Event)

실행 전 서브넷이 폐기되거나 긴급 취소 에포크가 전진함

런타임 검사 — 거부

SEB가 라이브 상태를 확인하고 대상 서브넷 무효 또는 에포크 만료로 인해 실행 거부

실행 (Execution) — 모든 검사 통과 시

해당 보안 그룹에만 바인딩된 단명 IAM 토큰 발급 후 실행, 결과 로깅

시스템 구현

코어 컴포넌트
  • Go 기반 브로커 서비스
  • 인증서 파서 및 검증기
  • 논스 예약 및 재생 방지
  • 결정/결과 원장(Ledger)
  • 정책 및 취소 캐시
플랫폼 어댑터
  • AWS STS 어댑터 (단명 자격증명)
  • Kubernetes TokenRequest 어댑터
  • 클라우드 프록시 서비스
  • K8s Admission Webhook
  • 장애 처리 및 크래시 복구

성능 평가 결과

AWS 및 Kubernetes 클러스터에서 프로토타입 평가 (p50 / p95 / p99 지연)

Exp1
검증 마이크로벤치마크

암호적 검증 및 드리프트 검증의 개별 지연 측정

Exp2
엔드투엔드 지연

제안부터 실행 완료까지 전체 파이프라인 지연 측정 및 베이스라인 비교

Exp3-5
보안 및 장애 모드

취소 전파 지연, 드리프트 감지 민감도, 파티션/크래시 복구 시나리오

평가는 통계적 방법론을 따르며, 시뮬레이션 경계를 명확히 정의하여 브로커 자체의 오버헤드와 외부 API 지연을 분리 측정함.

보안 분석

정리 1 (인증서 기반 변경): 배포가 비브로커 변경 신원을 거부한다면, SEB를 통과하지 않은 임의의 인프라 변경은 발생할 수 없다.

증명 스케치 — 5가지 거부 사례:
  • 사례 1: 브로커 바이패스 — 플랫폼이 비브로커 신원 거부
  • 사례 2: 유효 인증서 없음 또는 재생 — 논스 불일치
  • 사례 3: 인증서 만료 또는 취소됨
  • 사례 4: 요청-계약 불일치 또는 안전하지 않은 드리프트
  • 사례 5: 필요한 범위를 강제할 수 없음

논의 및 한계

인정된 한계
  • 클라우드 제공자에 대한 신뢰 의존
  • 콜드 스타트 지연
  • 브로커 가용성이 치명적 의존성
  • 운영 복잡도 증가
  • 취소 일관성 vs 부하 트레이드오프
  • 의미적 안전성은 외부에 존재
향후 논의 사항
  • 인간-루프(HITL) 에스컬레이션 통합
  • 성능 확장 및 합의(Consensus) 방안
  • 상태 관측의 신선도(Staleness) 문제
  • 크로스 클라우드 및 멀티 계정 배포
  • 대상 API 제한 사항

결론

SEB는 제안, 승인, 실행의 분리를 통해 인증된 권한을 단명하고 취소 가능하며 감사 가능한 런타임 기능으로 전환한다.

  • 에이전트 런타임에서 프로덕션 변경 자격증명을 완전히 제거
  • 인증서-계약 일치, 유효성, 에포크, 드리프트를 실행 시점에 종합 검증
  • 범위 지정 단명 신원으로 최소 권한 원칙 실현
  • 서명된 결정/결과 기록으로 완전한 감사 추적 제공
  • AWS 및 Kubernetes에서 프로토타입 구현 및 경험적 평가 완료

"인증서는 강제가 아니다 — 타겟 인프라 변경 경로가 브로커 검증을 적극적으로 요구할 때만 강제된다."
