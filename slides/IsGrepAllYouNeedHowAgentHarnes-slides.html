Grep만으로 충분한가?
에이전트 하니스가 어떻게
에이전틱 검색을 재구성하는가

Is Grep All You Need? How Agent Harnesses Reshape Agentic Search

Sahil Sen, Akhil Kasturi, Elias Lumer, Anmol Gulati, Vamse Kumar Subbiah

PricewaterhouseCoopers, U.S.

도입 및 동기

최근 LLM 에이전트는 자율적으로 정보를 검색하고 도구를 호출하며 대규모 말뭉치를 추론하는 복잡한 에이전틱 워크플로우를 가능하게 함. 그러나 검색 전략이 에이전트 아키텍처 및 도구 호출 패러다임과 어떻게 상호작용하는지에 대한 체계적 비교는 부재함.

검색 보강 생성(RAG)의 확산

에이전트 시스템에서 RAG 도입이 급증하나, 검색 전략 선택이 아키텍처와 결합될 때의 효과는 거의 연구되지 않음

실무적 차원의 미탐구 영역

도구 출력의 표현 방식, 무관한 텍스트가 증가할 때 성능 변화, 하니스 간 차이 등 핵심 실무 질문이 남아있음

연구 기여

1

검색 · 하니스 · 표현 방식의 상호작용

어휘적 검색(grep)과 밀집 검색(vector)의 선택이 에이전트 오케스트레이션 계층 및 도구 출력 방식(인라인 vs 파일 기반)과 결합될 때의 증거 제시

2

노이즈와 규모에 대한 특성 규명

무관한 주변 콘텐츠가 과제 관련 자료에 비해 증가함에 따라 엔드투엔드 동작이 어떻게 변화하는지, 검색기 동작과 에이전트 루프 간 상호작용 포함

3

에이전트 스택 간 이질성 비교

동일한 텍스트 말뭉치를 사용해도 아키텍처적으로 다른 하니스(커스텀 vs 제공자 네이티브 CLI) 간에 검색 효과가 안정적이지 않음을 직접 입증

검색 전략 개요

어휘적 검색

Lexical Search

  • BM25, grep, 정규식 기반 패턴 매칭
  • 임베딩 모델 불필요, 계산 비용 무시할 수준
  • BEIR 벤치마크에서 여전히 강력한 베이스라인
  • SPLADE 등 학습된 희소 표현도 존재

의미론적 검색

Semantic Search

  • 쿼리·문서를 밀집 벡터로 인코딩 후 ANN 탐색
  • 의미적 유사성 기반 → 의역(paraphrase) 처리에 강점
  • 임베딩 품질·인프라·인덱싱 지연에 의존
  • 선택적 리랭킹으로 후보 정제 가능

하이브리드 접근

Hybrid Approaches

  • RRF로 순위 리스트 병합 (점수 보정 불필요)
  • ColBERT: 토큰 수준 정교한 유사도 계산
  • 두 방식이 서로 다른 관련 문서 검색 → 상보적
  • 에이전트가 두 도구 중 선택하는 형태로도 자연 발생

에이전트 하니스 비교

커스텀 하니스 (Chronos)

  • 에이전트 루프의 모든 단계에 세밀한 제어
  • 시스템 프롬프트, 도구 정의, 컨텍스트 구성, 결과 포맷팅, 종료 조건 모두 개발자 관리
  • ReAct 패러다임 채택 (추론·행동 교차)
  • 도메인 특화 최적화 가능 (동적 프롬프팅, 결과 절단, 리랭킹)

단점: 상당한 개발 오버헤드

제공자 네이티브 CLI 하니스

Claude Code, Codex, Gemini CLI

  • 도구 호출을 셸 기반 인터페이스에 내장
  • 모델이 grep 등 CLI 도구에 직접 접근
  • 커스텀 하니스와 근본적으로 다른 아키텍처 클래스
  • 개발자가 루프 제어·컨텍스트 관리에 대한 통제가 제한됨

단점: 내부 동작 불투명, 제어력 부족

도구 호출 아키텍처

도구 결과를 모델에 어떻게 전달하는지가 추가적인 아키텍처 차원을 형성함.

표준 (인라인)

Standard / Inline

도구 호출 결과가 컨텍스트 윈도우 내에 직접 삽입됨. 모델이 추가 읽기 작업 없이 즉시 결과를 확인 가능. 전통적인 RAG 파이프라인과 일치하는 방식.

프로그래매틱 (파일 기반)

Programmatic / File-Based

도구 결과가 파일로 기록되고, 모델이 명시적으로 읽기 작업을 수행해야 함. 추가적인 도구 호출 단계가 발생하며, 에이전트 루프의 복잡성이 증가.

이 두 방식의 차이는 사소해 보이지만, 에이전트 루프 내에서 검색 결과를 소비하는 방식에 근본적 변화를 초래할 수 있음.

방법론: 데이터셋 및 설정

  • 구조화된 이벤트 형태로 제공 (Chronos 활용)

실험 1: 검색 모드·하니스·도구 호출 방식

목표: Grep과 Vector 검색을 Chronos 및 제공자 CLI 하니스에서, 인라인·파일 기반 도구 결과 방식 모두에 대해 비교

핵심 결과

  • Grep이 Chronos 및 제공자 CLI 전반에서 Vector보다 일반적으로 더 높은 정확도 달성
  • 그러나 전체 점수는 동일 대화 데이터에서도 하니스와 도구 호출 스타일에 강하게 의존
  • 검색 방식보다 하니스 선택이 최종 정확도에 더 큰 영향

시사점

"최고의 검색기"를 찾는 것만으로는 불충분. 동일한 검색 결과라도 어떤 하니스를 통해 어떻게 모델에 전달되는지가 성능을 좌우함. 에이전틱 검색 평가에서 아키텍처를 통제하지 않은 비교는 오도될 위험이 큼.

Grep이 더 나은 성능을 보였지만, 그 우위는 하니스와 도구 호출 방식의 조합에 의해 크게 조절됨.

실험 2: 노이즈 증가에 따른 컨텍스트 스케일링

목표: Grep 전용과 Vector 전용 검색을 비교하면서, 각 쿼리에 점진적으로 무관한 대화 이력을 혼합하여 관련 없는 자료의 비율을 증가시킴

관찰 결과

  • 노이즈 비율 증가에 따라 두 검색 방식 모두 성능 저하
  • 그러나 저하 속도와 패턴이 검색 방식별로 상이
  • 에이전트 루프 전체의 행동 변화(쿼리 수, 반복 횟수 등)도 노이즈에 따라 다르게 나타남

실무적 함의

대규모 노이즈가 있는 말뭉치에 RAG를 배포할 때, 단순한 정확도 비교를 넘어 노이즈 스케일링 곡선을 확인해야 함. 특정 노이즈 수준에서만 우위인 검색기는 실제 환경에서 취약할 수 있음.

핵심 발견 요약

1

Grep의 예상 밖의 강점

의미론적 벡터 검색이 RAG의 기본으로 자리잡았으나, 에이전틱 검색 환경에서는 단순한 grep이 더 높은 정확도를 보여줌. 임베딩 없이도 충분할 수 있음.

2

하니스가 검색보다 중요

동일한 검색 결과라도 Chronos, Claude Code, Codex, Gemini CLI 간에 정확도 차이가 큼. 검색기 선택보다 오케스트레이션 계층이 성능을 더 크게 결정.

3

표현 방식의 영향

인라인 vs 파일 기반 결과 전달 방식이 정확도에 유의미한 차이를 만듦. 사소해 보이는 인터페이스 결정이 에이전트의 정보 소비 패턴을 바꿈.

4

노이즈에 대한 차별적 내성

무관 자료 비율이 증가할 때 Grep과 Vector가 다른 저하 패턴을 보임. 노이즈 환경에서의 스케일링 거동이 검색기 선택의 핵심 기준이 되어야 함.

한계점

데이터셋 범위

LongMemEval의 116개 질문 샘플에 한정. 더 다양한 도메인, 언어, 과제 유형에서의 일반화는 추가 검증 필요.

하이브리드 검색 미포함

Grep과 Vector의 이진 비교에 집중. RRF 기반 하이브리드나 ColBERT 등 통합 방식은 실험에 포함되지 않음.

제공자 CLI의 불투명성

Claude Code, Codex, Gemini CLI의 내부 프롬프트·루프 로직이 불투명하여, 성능 차이의 근본 원인을 완전히 분석하기 어려움.

시간적 제약

제공자 CLI의 버전 업데이트 주기가 빠르므로, 특정 시점의 결과로 고정되는 한계가 있음. 지연 시간·비용 분석도 미포함.

결론

에이전틱 검색에서 "어떤 검색기를 쓸까?"라는 질문만으로는 충분하지 않음. 본 연구는 검색 전략의 효과가 에이전트 하니스와 도구 호출 아키텍처에 의해 크게 조절된다는 것을 실증적으로 보여줌.

  • Grep이 벡터 검색보다 일반적으로 더 높은 정확도를 기록했으나, 이 우위는 하니스에 의존적
  • 커스텀 하니스와 제공자 CLI 간에 동일 말뭉치에서도 검색 효과가 안정적이지 않음
  • 노이즈 스케일링 실험에서 두 검색 방식이 상이한 저하 패턴을 보임
  • 인라인 vs 파일 기반 결과 전달이 성능에 유의미한 영향을 미침

에이전틱 검색 시스템을 설계할 때, 검색기 선택과 동등한 무게로 하니스 아키텍처와 도구 인터페이스 설계를 고려해야 함.
