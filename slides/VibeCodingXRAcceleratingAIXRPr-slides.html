01 — 문제 정의

XR 프로토타이핑은 왜 어려운가?

"LLM이 소프트웨어 개발을 민주화했지만, XR 경험 프로토타이핑은 복잡한 게임 엔진과 저수준 센서 통합의 마찰로 여전히 접근이 어렵다."
🎮

복잡한 엔진

Unity, Unreal 등 무거운 게임 엔진에 대한 전문 지식이 필요

🧩

센서 통합 마찰

핸드 트래킹, 깊이 센서 등 저수준 하드웨어 API 직접 관리

🔒

폐쇄적 생태계

기존 AI+XR 도구(LLMR, DreamCodeVR)는 비동기적이고 폐쇄적

핵심 통찰

LLM의 능력 부족이 아니라, 의미론적이고 인간 중심적인 공간 개념이 부족한 것이 근본 원인

02 — 핵심 기여

두 가지 핵심 기여

🧱

XR Blocks

공간 컴퓨팅의 복잡성을 고수준, 인간 중심 프리미티브로 추상화하는 오픈소스 모듈형 WebXR 프레임워크

WebXR three.js TensorFlow.js
✨

Vibe Coding XR

자연어 프롬프트를 기능적 XR 애플리케이션으로 변환하는 LLM 기반 워크플로우

자연어 입력 1분 이내 생성
"민들레를 만들어주세요. 손으로 잡으면 씨앗이 날아가는"
→ 1분 이내에 인터랙티브 WebXR 애플리케이션으로 변환
03 — XR Blocks 아키텍처

Reality Model: 3가지 핵심 프리미티브

👤 사용자
🌍 세계
🤖 에이전트

사용자 (User)

위치, 핸드 트래킹, 인터랙션 상태를 일급 객체로 관리

세계 (World)

환경 기하학, 깊이 감지, 조명 추정, 물리 엔진

에이전트 (Agent)

행동 로직과 메모리를 갖춘 지능형 엔티티

모듈형 코어 엔진

환경 인식 깊이 센싱 핸드 인터랙션 핀치 제스처 강체 물리 LLM 쿼리 에이전트 메모리

08 — 기술 스택

기술 사양 및 코딩 표준

기반 기술

  • WebXR — 웹 기반 XR 표준
  • three.js — 3D 렌더링
  • TensorFlow.js — AI 추론
  • troika-three-text — 텍스트 렌더링

코딩 규칙

  • Y축 = 위, Z축 = 앞/뒤
  • 초기화: z = -user.objectDistance
  • 인터랙션: onSelectStart/End/Selecting
  • 2D 오버레이 대신 공간 UI 우선

설계 철학

오픈 웹 접근 방식 — 투명한 웹 표준 코드베이스 위에 구축, 빠른 반복을 위해 설계, 즉시 배포 가능. 기여가 공개적이고 누적적으로 발전할 수 있는 구조.
