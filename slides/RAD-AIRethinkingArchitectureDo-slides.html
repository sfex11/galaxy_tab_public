배경

arc42 & C4 모델의 구조적 공백

arc42
12개 실용적 섹션으로 구성된 아키텍처 문서화 템플릿. 소개, 제약사항, 컨텍스트, 빌딩블록뷰, 런타임뷰, 배포뷰 등을 포함하나 AI 컴포넌트 전용 섹션 없음
C4 모델
계층적 시각화: System Context, Container, Component, Code 레벨. 모든 컨테이너가 안정적 인터페이스를 가진 결정적 코드 모듈로 취급됨

5가지 문서화 공백 (G1–G5)

✘G1: 모델 생명주기, 데이터 파이프라인, 드리프트 전용 arc42 섹션 부재
✘G2: ML 컴포넌트 및 비결정성 경계를 위한 C4 다이어그램 타입 부재
✘G3: EU AI Act Annex IV와 arc42/C4 간 매핑 부재
✘G4: AI 전용 아키텍처 결정 기록(ADR) 템플릿 부재
✘G5: Model/Data Cards의 아키텍처 문서 통합 부재

8개 AI 전용 섹션 — E1–E4

🔍
E1: AI 경계 구분
Context & Scope 확장
결정적 vs 비결정적 시스템 경계를 명시. 출력 유형, 신뢰도 사양, 업데이트 빈도, 폴백 동작의 4-부분 계약 정의
📋
E2: 모델 레지스트리 뷰
Building Block 확장
AI 모델을 1급 빌딩 블록으로 격상. 모델 ID, ML 프레임워크, 학습 데이터 해시, 배포 상태 등 기록
📊
E3: 데이터 파이프라인 뷰
Runtime View 확장
수집 → 전처리 → 피처 엔지니어링 → 학습 → 추론 → 피드백 루프. 각 단계별 품질 게이트 주석
⚖️
E4: 책임 있는 AI 개념
Cross-Cutting 확장
공정성, 설명가능성, 인간 감독, 프라이버시, 안전성의 5개 카테고리 × AI 컴포넌트 매트릭스
arc42 확장 (2/2)

8개 AI 전용 섹션 — E5–E8

📝
E5: AI 결정 기록 (AI-ADR)
Architecture Decisions 확장
모델 대안, 데이터셋 사양, 공정성 모니터링, 모델 수명, 재학습 트리거, 설명가능성 방법, 규제 분류 등 7개 필드 추가
🎯
E6: AI 품질 시나리오
Quality Requirements 확장
데이터 드리프트, 모델 노후화, 적대적 입력 등 AI 특화 소스-자극-응답 시나리오. 정량적 트리거와 측정 가능한 응답
💳
E7: AI 부채 레지스터
Risks & Tech Debt 확장
경계 침식, 얽힘, 숨겨진 피드백 루프, 데이터 의존성, 파이프라인 부채 등 ML 특화 부채 체계적 추적
🛠️
E8: 운영 AI 뷰 (신규)
완전히 새로운 섹션
모니터링, 재학습 정책, 배포 전략(카나리/블루-그린/섀도), 롤백 정책의 4개 필수 하위 섹션
E8 운영 AI 뷰는 기존 arc42에 동등한 섹션이 전혀 없는 완전히 새로운 확장입니다.
C4 모델 확장

3가지 C4 다이어그램 확장

C4-E1: AI 컴포넌트 스테레오타입
5개 새로운 스테레오타입으로 AI 컴포넌트를 시각적으로 구별:
«ML Model» «Data Pipeline» «Feature Store» «Monitor» «Human-in-the-Loop»
C4-E2: 데이터 계보 오버레이
데이터 출처에서 변환, 모델 소비까지의 계보 추적 보조 다이어그램 레이어. 스키마 기대값, 데이터 신선도 제약, 프라이버시 분류를 주석으로 표시
C4-E3: 비결정성 경계 다이어그램
아키텍처를 결정적/비결정적 영역으로 분할. 각 경계에 신뢰도 사양, 폴백 전략, 성능 저하 프로파일의 3-속성 계약 주석

RAD-AI가 발견한 생태계 수준 문제

🌀 연쇄 드리프트 (Cascading Drift)
이상 탐지 드리프트 → 거짓 안전 경보 → 불필요한 경로 재계산 → UX 저하. 데이터 계보 오버레이로 의존성 체인 시각화
⚖️ 차별화된 규제 준수
AI 경계 구분으로 고위험/제한위험 컴포넌트를 구별, 법적으로 필요한 곳에만 Annex IV 문서 집중 적용
🤝 연합 거버넌스
공유 피처 스토어 소유권, 교차 운영자 데이터 협약, 책임 경계를 책임 있는 AI 개념 섹션에서 문서화
