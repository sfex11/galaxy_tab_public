문제 정의: 왜 AI 에이전트는 회귀를 만드는가?

🔴

벤치마크의 맹점

SWE-bench는 "문제 해결 여부"만 측정하고 기존 테스트의 회귀(regression)는 무시한다. 패치가 통과해도 다른 곳이 깨질 수 있다.

💥

높은 회귀율

기존 에이전트의 6.08%가 회귀를 유발한다. 실패한 시도의 33%에서 기존 테스트가 깨진다 — 실제 배포 시 큰 위험.

🧪

전체 테스트 실행 불가

대규모 프로젝트에서 모든 테스트를 실행하면 시간/비용이 폭증. 영향받는 테스트만 선택적으로 실행해야 한다.

📋

TDD 프롬프팅의 역설

"TDD를 따르라"는 절차적 지시만으로는 오히려 회귀가 증가. 에이전트는 절차보다 구체적 컨텍스트가 필요하다.

1AST 의존성 그래프

def calculate_tax()
class InvoiceService
def validate_input()

코드의 AST를 파싱하여 함수, 클래스, 모듈 간 호출/임포트 관계를 그래프로 구축. 변경된 노드에서 출발하여 연결된 테스트를 탐색.

2가중 영향 점수

직접 호출: w = 1.0
간접 의존: w = 0.5
동일 모듈: w = 0.3

의존 관계의 유형과 거리에 따라 가중치를 부여. 직접 호출하는 테스트가 더 높은 위험도를 가짐. 임계값 이상만 선택.

3테스트 선택 및 전달

test_tax_calculation ⚠️
test_invoice_total ⚠️
test_validation ○

"이 테스트들이 깨질 위험이 있다"는 컨텍스트를 에이전트에 직접 전달. 절차 지시 없이 무엇을 확인할지 알려줌.
