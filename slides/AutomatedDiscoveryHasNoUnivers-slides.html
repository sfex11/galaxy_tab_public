Automated Discovery Has No Universally Superior Harness

자율 발견 시스템의 부상

최근 OpenEvolve, TTT-Discover 같은 자율 발견 시스템이 과학적 발견의 범용 하네스로 널리 사용되고 있다.

핵심 관찰: 하네스는 복합 시스템이다

자원 분배 방식
"발견 시스템은 하나의 레시피로 패키징된 여러 설계 결정의 복합체이다."

왜 문제가 되는가?

하나의 레시피가 모든 발견 과제에서 우월하다는 보장은 없다.

암묵적 가정
  • 특정 하네스가 범용적으로 최적이라는 전제
  • 설계 선택 간 상호작용이 무시됨
  • 과제 특성과 하네스 적합성 고려 부족
실제 현실
  • 과제마다 최적의 아카이브 전략이 다름
  • 탐색 전략의 효과는 도메인 의존적
  • 예산 배분은 과제 난이도에 민감

연구 질문

Q1
단일 하네스가 모든 과제에서 최선인가?
기존 자율 발견 시스템의 범용 최적성 가설 검증
Q2
각 설계 차원의 기여도는 어떻게 다른가?
아카이브, 부모 선택, 탐색, 예산 배분의 개별 효과 분석
Q3

차원 간 교호작용이 성능에 미치는 영향 규명

핵심 발견: 보편적 우월성 부재

과제 유형 최적 아카이브 최적 부모 선택 최적 탐색 기존 하네스 순위
알고리즘 발견 상위-K 다양성 가중 변이 중심 중간
분자 설계 전체 보존 성능 순위 교차 중심 하위
최적화 제어 다양성 필터 무작위 LLM 프롬프트 상위
아키텍처 탐색 상위-K 성능 순위 혼합 전략 중간

* 과제별 최적 조합이 서로 다르며, 기존 하네스의 고정 레시피는 어느 과제에서도 일관되게 최선이 아니다.

탐색 전략높음
부모 선택중간
예산 배분중간
아카이브낮음~중간
주요 인사이트
  • 탐색 전략이 가장 큰 성능 차이를 유발
  • 부모 선택과 예산 배분은 과제에 따라 효과 크기 변동
  • 아카이브는 단독으로는 영향이 작지만 다른 차원과의 상호작용이 큼
  • 어느 차원도 모든 과제에서 일관되게 지배적이지 않음

상호작용 효과: 조합이 중요하다

개별 차원의 최적 선택을 합쳤다고 해서 전체 최적이 되지는 않는다. 차원 간 상호작용이 유의미하다.

주요 상호작용
  • 탐색 전략 × 부모 선택
  • 아카이브 × 탐색 전략
  • 예산 배분 × 아카이브 크기
시사점
  • 각 차원을 독립적으로 튜닝하는 접근은 한계
  • 전체 조합 공간에서의 평가가 필요
  • 과제 특성을 고려한 적응적 설계 필요
"최고의 탐색 전략은 선택된 부모 선택 방식에 따라 달라지며, 이 역시 아카이브 구조에 의존한다."

실무적 함의

하지 말아야 할 것
기존 하네스를 블랙박스처럼 그대로 사용하여 새로운 발견 과제에 적용하는 것. 특정 레시피가 범용 최적이라는 착각에서 비롯된 비효율.
해야 할 것
과제 특성에 맞게 아카이브, 부모 선택, 탐색, 예산 배분을 개별적으로 조합하고 평가할 수 있는 모듈형 프레임워크를 구축할 것.
이상적인 방향
과제의 메타특성을 기반으로 자동으로 최적 조합을 추천하는 메타러닝 기반 하네스 선택기 개발.

논의 및 한계

논의
  • 결과는 자율 발견 분야의 "하네스 경쟁" 패러다임에 대한 재고를 촉구
  • 모듈식 벤치마크 설계의 필요성 강조
  • 재현성과 비교 가능성 제고를 위한 구성요소 수준의 보고 관행 제안
한계
  • 평가한 과제 도메인이 제한적일 수 있음
  • LLM 기반 탐색은 모델 버전에 따라 결과 변동 가능
  • 계산 비용으로 인해 전체 조합 공간을 완전 탐색하지 못할 수 있음

핵심 요약

1
자율 발견 하네스는 단일 알고리즘이 아니라 여러 설계 선택의 복합체이다.
2
어느 단일 조합도 모든 과제에서 보편적으로 우월하지 않다.
3
설계 차원 간 상호작용 효과가 유의미하므로 독립적 튜닝에 한계가 있다.
4
미래에는 모듈형, 적응형 프레임워크가 고정 레시피 하네스를 대체해야 한다.
