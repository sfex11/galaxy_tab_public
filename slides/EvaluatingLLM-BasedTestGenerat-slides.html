LLM 기반 테스트 생성의
소프트웨어 진화 환경 평가

LLM이 생성한 테스트는 진정한 프로그램 이해에 기반하는가,
아니면 표면적 패턴의 재현에 불과한가?

8개 LLM 모델 22,374개 프로그램 변형 돌연변이 기반 분석

Sabaat Haroon • Mohammad Taha Khan • Muhammad Ali Gulzar

연구 동기

왜 이 연구가 필요한가?

✅
현재 상태
LLM은 원본 코드에서 79% 라인 커버리지, 76% 브랜치 커버리지로 우수한 기본 성능 달성
❓
핵심 의문
이 성능이 진정한 코드 이해에서 비롯되는가, 아니면 훈련 데이터 패턴 암기에 불과한가?
코드가 지속적으로 변화하고 리팩토링되는 CI/CD 환경에서, LLM 기반 테스트 생성은 여전히 매우 신뢰하기 어려운 도구로 남아있다.

RQ2 결과

의미 변경(SAC) 시 성능 하락

기능적으로 코드가 변경되었을 때, LLM 테스트는 급격히 무너진다.

테스트 통과율 100% → 66.5% ▼ 33.5%p
라인 커버리지 79.3% → 67.4% ▼ 11.9%p
브랜치 커버리지 76.1% → 60.6% ▼ 15.5%p

💡 핵심 발견: 브랜치 커버리지가 라인 커버리지보다 더 크게 하락 — 변경된 제어 흐름 추론에 어려움을 겪고 있음. 테스트 수는 오히려 13.2 → 13.9로 증가하여 "산탄식 테스트(scattershot testing)" 양상을 보임.

🔎 핵심 역설: 구문적으로 더 큰 변화(2.4줄)가 의미적으로 더 작은 영향(0)을 가짐에도, LLM은 기존 테스트를 1.2배 더 많이 폐기. "미사용 매개변수" 변경만으로 테스트가 8.1개로 급감 — 구문 민감성의 명확한 증거.

LLM은 의미적 코드 변경을 추론할 정밀성이 부족하며, 대신 훈련 중 관찰된 패턴에 크게 의존한다. 모든 SAC 카테고리에서 98.6~100%의 잔류 정렬률을 보인다.
RQ5 결과

회귀 인식의 부재

LLM은 기존 테스트를 유지하지 않고 매번 새로 생성한다.

지표 SAC 환경 SPC 환경
평가 프로그램 수 8,585 10,563
새로 생성된 테스트 9.7개 11.8개
폐기된 기존 테스트 8.8개 10.9개
총 테스트 변동 18.5개 22.7개
테스트 일치율 29.3% 18.1%

⚠ 심각한 문제: 폐기된 테스트가 새로 생성된 테스트보다 더 높은 커버리지 효과를 가지고 있었음. LLM은 기존 고효율 테스트를 적응시키지 못하고, 처음부터 스위트를 재생성.

핵심 발견

5가지 핵심 결론

1
잔류 정렬 — 실패한 SAC 테스트의 99%+가 원본 코드에서 통과. 훈련 데이터 패턴 암기의 증거.
2
구문 민감성 — SPC가 SAC보다 더 큰 테스트 스위트 혼란을 야기. 기능적 동등성에도 불구하고 표면 패턴에 의존.
3
커버리지 저하 — 브랜치 커버리지가 라인 커버리지보다 일관적으로 더 크게 하락. 제어 흐름 추론의 한계.
4
테스트 불안정 — 구조적 노이즈 발생 시 1.2~1.4배 더 많은 테스트 생성, 동시에 기존 효과적 테스트 높은 비율 폐기.
5
언어 차이 — Python의 동적 타이핑이 Java의 정적 타이핑 대비 훨씬 큰 기본 과제를 부과.
