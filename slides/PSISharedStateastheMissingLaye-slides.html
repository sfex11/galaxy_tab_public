PSI

개인 AI 에이전트의 일관된 도구를 위한 공유 상태 아키텍처

독립적으로 생성된 AI 모듈들을 일관된 도구(Instrument)로 통합하는 공유 상태 계층

Zhiyuan Wang, Erzhen Hu, Mark Rucker, Laura E. Barnes
arXiv:2604.08529
01 / 문제 정의

AI가 도구를 만들 수 있지만,
통합은 여전히 불가능하다

자연어 요청으로 개인화된 앱을 생성할 수 있지만, 생성된 도구들은 서로 고립된 사일로로 남습니다.

실제 상황

"왜 요즘 이렇게 피곤할까?"라는 질문에 답하려면
운동 데이터 + 수면 품질 + 일정 부하 + 심박수를 통합해야 합니다.

하지만 이 데이터는 각각 다른 앱에 분산되어 있습니다.

📱

앱 A

건강 데이터

📱

앱 B

운동 기록

📱

앱 C

일정 관리

서로 연결되지 않는 독립된 사일로

02 / 핵심 아이디어

공유 상태: 누락된 시스템 계층

PSI는 독립적으로 생성된 모듈을 도구(Instrument)로 전환합니다.

💾

영속적 (Persistent)

재생성 없이 지속적으로 사용 가능한 아티팩트

🔗

연결됨 (Connected)

공유 개인 맥락 버스에 상태를 게시하고 쓰기 기능 노출

💬

채팅 보완적

GUI 모니터링과 채팅 추론이 동일한 상태를 공유

핵심 원리: 영속적 GUI 도구와 범용 채팅 에이전트(Facai)가 동일한 공유 맥락 버스 위에서 동작합니다.

03 / 아키텍처

PSI의 3계층 아키텍처

1. 생성 계층 (Generation Layer)
에이전틱 코딩 워크플로우를 통한 모듈 작성
명세 코드 생성 자동 수정 컴파일 검증
▼
2. 공유 개인 맥락 계층 (Personal-Context Bus)
모든 등록 모듈의 현재 상태 스냅샷을 수집하는 중앙 레지스트리
개인 범위 기기 내 영속 태그된 텍스트 형식
▼
3. 상호작용 계층 (Interaction Layer)
두 개의 동기화된 인터페이스가 동일한 변경 가능한 상태에 접근
영속 GUI 대시보드 범용 채팅 에이전트 (Facai)
04 / 프로바이더 계약

통합을 지역적 의무로 전환

단일 Swift 프로토콜이 플러그 앤 플레이 모듈 통합을 가능하게 합니다.

protocol ToolkitDataProvider {
  var toolkitIdentifier: String
  var relevanceKeywords: [String]
  func buildContextSummary() -> String?
}

출력 형식

[Health Data]
오늘: 1030 cal, 62g 단백질
운동: 12분, 65 cal 소모
[End Health Data]

설계 원칙

  • 통합 복잡도: O(n) (쌍별 O(n²) 아님)
  • 빈 모듈은 자동 생략
  • 모듈 간 직접 통신 없음

전체 핵심 메커니즘: 4개 파일, 173줄 (프로토콜 38줄 + 조립 60줄 + 레지스트리 59줄 + 부트스트랩 16줄)

10 / 설계 트레이드오프

공유 상태의 장점과 도전

✔

장점

  • 교차 모듈 합성으로 단일 모듈의 8배 작업 성공률
  • 새 모듈은 프로토콜만 구현하면 자동 통합
  • 읽기(추론)와 쓰기(액션) 양방향 지원
  • LLM 매개 통신으로 결합도 최소화
⚠

도전 과제

  • 맥락 오염: 오래되거나 중복된 요약이 시스템 전체 응답 품질 저하
  • 확장성: 모듈 수 증가 시 프롬프트 길이 증가
  • 시스템 수준 실패: 사일로 앱의 국소 실패 vs 공유 상태의 전역 실패
  • 라우팅 필요: 대규모 모듈에서 선택적 맥락 주입 필요
