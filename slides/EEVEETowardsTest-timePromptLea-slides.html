문제 정의: 테스트타임 프롬프트 학습

기존 접근법

  • 단일 데이터셋/벤치마크에 최적화
  • 하나의 통합 프롬프트로 적응
  • GEPA, ACE, Combee 등

실세계 요구사항

  • 이질적인 도메인의 혼합 스트림
  • 다양한 태스크 포맷과 평가 기준
  • 크로스 데이터셋 간섭 문제

멀티 데이터셋 테스트타임 프롬프트 학습: 모델이 단일 정상 소스가 아닌, 여러 데이터셋과 도메인에서 추출된 스트림을 받는 환경

핵심 문제: 크로스 데이터셋 간섭

벤치마크가 추가될수록 단일 프롬프트 방식은 이전 태스크 성능을 저하시킴

EEVEE 프레임워크 개요

전문화된 프롬프트 집합과 라우터의 조합으로 이질적 스트림 처리

1
입력 수신
혼합 태스크 스트림 x
2
라우팅
R(x; 𝒫) → 슬롯 z 선택
3
프롬프트 적용
전문화된 p_z로 추론
4
출력 생성
ŷ = M(x; p_z)

프롬프트 집합 𝒫

K개의 전문화된 프롬프트 슬롯 {p₁, ..., pK} 유지

라우터 R

입력을 태스크 클러스터로 분할하여 적절한 슬롯에 할당

라우터-프롬프트 공진화 전략

라우터와 프롬프트의 상호 의존성을 해결하기 위한 교차 최적화

R

라우터 진화 단계

프롬프트 집합 𝒫_T를 고정하고 더 나은 라우터 R_{T+1} 탐색

P

프롬프트 진화 단계

라우터 R_{T+1}를 고정하고 각 슬롯 프롬프트 𝒫_{T+2} 업데이트

핵심 아이디어: 라우팅 결정과 프롬프트 전문화가 동시에 개선되도록 번갈아가며 최적화

라우터 진화 상세

진화 프로세스

  • 임시 라우터 풀 ℬ_R 초기화
  • 최소 하나의 슬롯이 풀 수 있는 예제만 샘플링
  • 참조 라우터에서 돌연변이 R_mut 생성
  • 실패 사례 분석 → 반성 R_ref 생성
  • 검증셋에서 베이스라인 개선 시만 채택

라우터 평가 지표

정확도 A(R) — 라우팅된 정답률

일관성 C(R) — Compact + Separate

균형 B(R) — 슬롯 사용률 + 분포 균형

가중치는 일관성/균형 → 정확도로 어닐링

프롬프트 진화 상세

라우터에 의해 분할된 데이터로 각 슬롯을 독립적·병렬로 진화

𝒟tr,k

슬롯 k에 라우팅된
학습 데이터 그룹

ℬP,k

슬롯별 임시
프롬프트 풀

병렬

비어있지 않은 슬롯끼리
독립적으로 진화

라우터 진화와의 차이점

  • 돌연변이(mutation)와 반성(reflection) 사용 — 동일
  • 라우터 특화 분석 단계 없음 — 프롬프트는 직접 정답 여부로 평가
  • 슬롯별 검증셋 𝒟val,k에서 개선 시 채택

3단계 학습 프로세스

1

초기화 (Initialization)

유용한 프롬프트 슬롯을 사전에 구성하여 공진화의 안정적인 시작점 확보

2

탐색 (Exploration)

라우터-프롬프트 결합 업데이트를 효율적으로 탐색, 다양한 라우팅 파티션 실험

3

수렴 (Convergence)

안정화된 라우터 하에서 프롬프트 정교화, 평가 가중치를 정확도로 어닐링

3단계 설계 없이 바로 공진화를 시작하면 라우터 불안정성으로 프롬프트 최적화가 방해받음

절제 연구 및 태스크 스케일링

절제 연구 핵심 발견

  • 수동 라우터는 행동적으로 유용한 파티션을 복원하지 못함
  • 공진화 없이 고정 라우터 + 프롬프트 학습 시 성능 저하
  • 3단계 학습 프로세스가 안정성에 필수적
  • 어닐링 스케줄과 가중치에 대한 강건성 확인

태스크 스케일링

  • 데이터셋 수가 증가해도 EEVEE는 안정적 성능 유지
  • 기존 방법들은 태스크가 추가될수록 보유 점수 하락
  • 일반화: 학습에 없던 도메인에서도 긍정적 전이

토큰 비용 효율성
