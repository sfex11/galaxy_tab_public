활발한 연구
실제 소프트웨어 개발은 여러 메서드를 조정하고, 공유 상태를 관리하며, 도메인 간 로직을 통합하는 구성적 코드 생성을 요구한다.

구성적 코드 생성이란?

명세서(specification)로부터 완전하고 내부적으로 구조화된 클래스를 처음부터 빌드하는 능력

요구되는 능력

  • 다중 메서드 간 조정
  • 공유 상태 관리
  • 메서드 간 의존성 처리
  • 인터페이스 제약 준수
  • 도메인 특화 로직 통합

기존 벤치마크의 한계

  • 함수 수준: 단일 함수만 평가
  • 저장소 수준: 기존 코드 수정만
  • ClassEval: 수동 큐레이션 (500인시)
  • 2023년 데이터 → 오염 위험
  • 7개 도메인으로 제한

Stage 1 상세

최신 GitHub 리포지토리에서 실제 클래스 수집 → Evol-Instruct 기반 복잡도 증강(메서드 추가, 의존성 심화) → 서로 다른 도메인의 클래스를 결합하여 새로운 복합 클래스 스켈레톤 생성

LLM Judge Ensemble

단일 모델이 아닌 다중 LLM 패널이 명세-테스트-해법의 정합성을 교차 검증. 다수결 투표로 품질 보장하여 수동 큐레이션의 인간 편향 제거

실험 설계

평가 대상 모델 (5개)

  • GPT-5.1
  • Gemini-2.5-Pro
  • Qwen3-480B
  • Qwen3-30B
  • Kimi-K2

생성 전략 (5가지)

  • Holistic: 전체 클래스를 한 번에 생성
  • Incremental: 메서드를 순차적으로 추가
  • Compositional: 독립 단위를 조합
  • Top-down: 인터페이스 → 구현 순서
  • Bottom-up: 기본 메서드 → 통합 순서

평가 지표

Class-level Pass@1: 생성된 클래스가 전체 테스트 스위트를 1회 시도에서 통과하는 비율. 함수 수준 Pass@1과 달리 모든 메서드가 동시에 정확해야 함.

모델 간 격차

함수 수준보다 훨씬 넓음

클래스 수준 Pass@1은 함수 수준 벤치마크에서 보이는 모델 간 격차보다 훨씬 크다. 이는 구성적 과제가 프론티어 모델 간의 진정한 차별력을 보여준다.

함수 수준 벤치마크에서 상위 모델들이 90%+에 도달하는 것과 대조적으로, 클래스 수준에서는 최고 모델조차 절반 미만의 성능을 기록

RQ3: 생성 전략의 영향

전략 선택은 모델 능력과 강하게 상호작용한다

효과적 전략

Bottom-up +9.4p

기본 메서드부터 점진적 구축. 약한 모델에 가장 효과적 — 구조적 분해가 인지 부하를 줄여줌. Holistic 대비 최대 9.4%p 향상

Holistic

전체를 한 번에 생성. 강한 모델에서 최고 성능 달성. 충분한 컨텍스트 윈도우를 가진 모델에 적합

실패 전략

Compositional 1.3%

독립 단위를 분리 생성 후 조합. 심각한 성능 붕괴 — 메서드 간 의존성과 공유 상태를 조정하지 못해 최저 1.3% 기록

Top-down

인터페이스 먼저, 구현 나중. 개념적으로 우수하나, 실제로는 후반 구현에서 일관성 유지 실패

RQ4: 오류 분석 (500건 수동 주석)

56.2%

논리 오류

개별 메서드 내부의 알고리즘적 오류, 경계 조건 미처리, 잘못된 계산 로직

38.0%

의존성 오류

메서드 간 호출 순서 오류, 공유 상태 불일치, 인터페이스 위반

5.8%

기타

구문 오류, 임포트 누락 등

핵심 병목: 단순한 코드 형성이 아니라 메서드 간 조정(cross-method coordination)이 클래스 수준 코드 생성의 근본적 한계다. 논리 오류와 의존성 오류를 합치면 전체 실패의 94.2%를 차지한다.

Compositional 전략에서는 의존성 오류 비율이 더욱 높게 나타남 — 분리 생성 시 메서드 간 조정이 더 어려워짐을 확인

시사점: 클래스 수준 생성에서 모델 개선은 단순한 코드 완성 능력이 아닌, 다중 메서드 간 상태 일관성을 유지하는 능력을 목표로 해야 한다

주요 발견 요약

1

구성적 과제는 진정한 차별력을 가진다

함수 수준에서 모델 간 격차가 좁아진 반면, 클래스 수준에서는 17.7%p의 넓은 격차가 유지된다.

2

전략-모델 상호작용이 존재한다

강한 모델은 Holistic이 최적이지만, 약한 모델은 Bottom-up이 최적이다. "범용 최고 전략"은 없다.

3

분해가 항상 정답은 아니다

Compositional(독립 분리 후 조합) 전략은 1.3%까지 붕괴 — 의존성을 무시한 분해가 오히려 독이 된다.

4

핵심 병목은 메서드 간 조정이다

논리 오류(56.2%) + 의존성 오류(38.0%) = 94.2%. 표면적 코드 형성이 아닌 구조적 조정이 한계다.

시사점 및 향후 방향

모델 설계

  • 다중 메서드 상태 일관성 추론 능력 강화
  • 클래스 수준 사전 학습 데이터 확보
  • 메서드 간 의존성 그래프 이해 학습

에이전트 설계

  • 모델 능력에 적응적 전략 선택
  • 의존성 인식 분해 (무작위 분해 지양)
  • 상태 일관성 검증 단계 삽입

평가 방법론

  • Pass/Fail 너머 오류 유형별 세분화
  • 오염 저항적 벤치마크 지속 갱신
  • 교차 도메인 구성 능력 평가 확대

한계 (Threats to Validity)

  • Python에만 한정됨
  • LLM Judge 자체의 편향 가능성
  • 5개 모델로만 평가
