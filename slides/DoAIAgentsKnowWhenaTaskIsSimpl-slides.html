문제 정의

최대 컨텍스트 우선 (Maximum-Context-First) 전략

대부분의 LLM 에이전트는 불확실성에 직면하면 가능한 한 많은 컨텍스트를 수집한 뒤 모든 위험을 제거하려 한다. 복잡한 작업에서는 이러한 신중함이 정당화되지만, 단순한 작업에서는 큰 인지적·실행적 오버헤드를 생성한다.

결핍된 판단 능력

에이전트가 갖추지 못한 것은 지식이나 검색 능력이 아니라, 다음의 네 가지 판단이다:

  • 작업이 얼마나 어려운지
  • 어떤 정보가 실제로 필요한지
  • 이미 본 정보 중 현재 변경과 무관한 것
  • 가장 짧은 신뢰할 수 있는 경로

연구 질문

RQ1

작업이 요구해야 하는 노력을 형식화하고, 에이전트의 실제 노력이 그것에서 얼마나 벗어나는지 측정할 수 있는가?

RQ2

에이전트가 행동하기 전에 작업의 실행 범위를 충분히 저렴하고 신뢰성 있게 추정하여 최소 실행 경로를 선택할 수 있는가?

RQ3

최소에서 시작하고 실패 시에만 확장하는 방식이, 미리 최대 컨텍스트를 수집하는 것에 비해 성공률을 유지하면서 비용을 줄이는가?

형식화: 최소 충분 실행과 ACRR

최소 충분 실행 (MSE)

성공 제약 조건을 만족하는 최소 비용 에이전트 궤적. 특정 작업에 대해 오라클이 보장하는 가장 저렴한 실행 경로를 의미한다.

모든 작업마다 오라클 최소 충분 궤적이 존재하므로, 중복을 정확하게 측정 가능

에이전트 인지 중복 비율 (ACRR)

오라클 대비 낭비된 노력의 정규화된 측도. 에이전트의 실제 실행 비용이 최소 충분 실행 비용에서 얼마나 벗어나는지를 비율로 나타낸다.

ACRR이 1.0이면 오라클과 동일, 값이 클수록 중복 심화

초기 운영 점 (Initial Operating Point)

전력 시스템의 전력 흐름 해석에서 평탄 시작(Flat Start)이나 DC 추정으로 초기 해를 설정한 뒤 Newton–Raphson 반복으로 정제하는 것과 유사하게, 에이전트도 초기 추정점에서 시작하여 점진적으로 정제해야 한다.

E3 프레임워크 개요

E3는 Estimate, Execute, Expand의 세 단계로 구성된 복잡도 인식 실행 프레임워크다.

E3 각 단계 상세

Estimate

  • 작업 지시문 분석
  • 복잡도 등급 추정 (단일 파일 / 교차 파일 / 리포지토리 수준)
  • 필요한 파일 범위와 편집 유형 예측
  • 초기 운영 점 설정

Execute

  • 추정된 최소 파일 집합만 읽기
  • 최소 실행 가능 경로로 편집 수행
  • 검증 단계 실행 (테스트 / 구문 검사 등)
  • 성공하면 즉시 종료

Expand

  • 검증 실패 시에만 트리거
  • 다음 복잡도 등급으로 확장
  • 추가 파일과 의존성 탐색
  • 재시도 후 다시 검증

MSE-Bench: 벤치마크 설계

설계 원칙: 능력 불변 평가 (Capability-Invariant Evaluation)

실행 중복을 원시 모델 능력과 분리하기 위해, 결정론적 오프라인 시뮬레이터에서 정책을 평가. "에이전트가 편집을 할 수 있는가?"를 일정하게 유지하고, 정책이 컨텍스트를 얼마나 수집하는지만 변동시킨다.

중복 분석: 단순할수록 더 심하다

가장 단순한 작업에서 중복이 가장 크게 발생한다는 핵심 발견. Maximum-Context 정책은 단일 파일 편집에서 리포지토리 수준 편집보다 ACRR이 훨씬 높게 나타난다.

절제 연구 및 강건성

절제 연구 (Ablation)

  • Estimate 제거 → 비용 증가, 추정의 역할 입증
  • Expand 제거 → 성공률 하락, 확장의 필요성 입증
  • 두 구성 요소가 상호 보완적으로 작동

강건성 검증

  • Held-out 지시문: 추정기의 어휘 단서를 의도적으로 파괴해도 100% 성공률 유지
  • 비용 가중치 변화: 거의 모든 가중치 조합에서 E3가 가장 저렴한 완전 성공 정책

핵심 시사점

효율성은 추정기가 벤치마크 템플릿에 맞았거나 유리한 비용 모델 덕분이 아니라, Estimate–Expand 아키텍처 자체의 속성이다.

실제 LLM 검증: LLM-Case 하네스

실제 환경에서의 교차 검증

시뮬레이터 결과를 실제 GPT-4o 에이전트가 오픈소스 라이브러리를 편집하는 환경에서 검증.

검증 설계

  • 수동 작성 정책 없음 (실제 코드)
  • 실제 결합 (real coupling)
  • 측정된 오라클 (주장이 아닌 실측)
  • 허수무단이 아닌 최선의 헤드라인

결과

  • 과독서(over-reading)는 시뮬레이터보다 완화되지만 실제로 존재
  • E3가 가장 날씬하고 빠른 정책
  • 유일한 실패 원인: 프로바이더 속도 제한 (잘못된 편집이 아님)

모든 후보 패치는 프로젝트의 실제 pytest 스위트를 실행하여 측정된 오라클로 평가

전력 시스템 공학 사례 연구

초기 운영 점 비유를 실제 전력 시스템 해석으로 검증. Newton–Raphson 전력 흐름 해석에서 좋은 초기점이 불필요한 탐색을 어떻게 줄이고 수렴을 빠르고 안정적으로 만드는지를 실측 데이터로 확인.

전력 흐름 해석 (유추 원천)

  • 상태 공간을 열거하지 않음
  • 구조화된 초기 운영 점 계산
  • Flat Start 또는 DC 추정 → Newton–Raphson 정제

E3 (유추 적용)

  • 모든 파일을 미리 읽지 않음
  • 추정을 통해 초기 실행 범위 설정
  • 실패 시에만 점진적 확장 → 정제

좋은 초기점이 정답이 아니더라도, 불필요한 탐색을 극적으로 줄이고 수렴을 빠르게 한다.
