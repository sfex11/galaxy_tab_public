동기: 코딩 에이전트의 실패 후 결정

코딩 에이전트는 실행 가능한 환경에서 동작하며, 실패 시 행동 가능한 피드백을 생성합니다.

컴파일 에러, 테스트 실패, 잘못된 출력, 타임아웃 등은 단순한 오답이 아니라 진단 가능한 복구 문제로 변환됩니다.

기존 접근의 한계

  • 실패 = 곧 더 강한 모델 필요이라는 암묵적 가정
  • 실행 피드백을 활용한 저렴한 복구 기회 간과
  • 예산 변화 시 재훈련 필요

핵심 통찰: 복구 행동의 비단조적 성공 패턴

"실행 피드백은 저렴한 모델의 추가 호출이 강한 모델로의 전환만큼 효과적일 수 있게 만든다. 관련 결정은 어떤 모델을 호출할 것인가가 아니라 어떤 복구 행동을 취할 것인가이다."

비단조

더 비싼 행동이 항상 더 높은 성공률을 보장하지 않음

상보적

저렴한 복구와 모델 승격이 각각 다른 사례를 해결

예산 의존

최적 행동은 사용자 예산과 작업 부하에 따라 변화

세 가지 복구 행동

저렴한 모델의 실패 후, 라우터는 문제陈述, 실행 결과, stderr를 관찰하여 세 행동 중 선택

Reflect (반영)

실행 피드백을 활용하여 기존 코드를 수정

가설: 현재 궤적이 국소적으로 수정 가능

Replan (재계획)

저렴한 모델로 새로운 계획부터 재시도

가설: 궤적이 틀렸으나 저렴한 모델로 성공 가능

Escalate (승격)

더 강하고 비싼 모델에 위임

가설: 능력 격차로 인해 강한 모델 필요

핵심 설계 원칙: 라우터는 한 번만 훈련되며, 예산 변화 시 CRC 레이어가 비용 페널티 테이블을 재스캔하는 것만으로 대응 가능 — 재훈련 불필요

지도 학습 복구 라우터

롤아웃 구성

각 실패 사례에 대해 세 가지 복구 행동을 모두 실행하여 성공/실패를 기록

라벨링 규칙

최저비용 성공 행동을 정답 라벨로 지정

reflect ≤ replan ≪ escalate (비용 순서)

라우터 입력

  • 문제 설명 (problem statement)
  • 실행 판정 (verdict)
  • 표준 에러 출력 (stderr trace)

평가 지표

단순 분류 정확도가 아닌 해결률(solve rate)로 평가 — 여러 행동이 동일 task를 해결할 수 있음

비용 정규화 정책과 CRC 보정

비용 정규화 결정 규칙

비용 페널티 λ를 도입하여 라우터의 점수에 비용을 반영:

a* = argmax_a [ s(a|x) − λ · c(a) ]

λ=0: 무제한 argmax (품질 최우선)
λ→∞: 항상 최저비용 행동 선택

Conformal Risk Control

사용자 예산 B에 대해 λ̂(B)를 보정하여:

  • 한계 기대 비용 제어 (exchangeability 하)
  • 라우터 재훈련 불필요
  • 비단조 성공 패턴에도 적용 가능

보장 대상은 비용이지 해결률이 아님 — 이것이 비단조성을 다룰 수 있는 핵심

예산 제어 가능 프론티어의 직관

예산이 타이트할 때

큰 λ → reflect/replan 우선 → 낮은 비용, 낮은 해결률

예산이 여유로울 때

작은 λ → escalate 허용 → 높은 비용, 높은 해결률

이산적 프론티어 형성

λ 변화에 따라 reflect → replan → escalate로 점진적 전환, 비용-품명 트레이드오프 곡선 생성

기존 이진 캐스케이드(accept/defer)와 달리, 세 가지 이질적 행동에 대해 연속적인 예산 조절이 가능한 구조

실험 설정

데이터 및 롤아웃

  • 5개 코딩 벤치마크 (TACO 등)
  • 저렴한 모델 시도 후 실패 사례 수집
  • 실패 사례당 3가지 복구 행동 모두 실행

모델 설정

메인 설정: GPT-5.4-nano (저렴) / GPT-5.4 (강함)

핵심 발견

  • CRC 보정 프론티어가 고정 행동 baseline 전체를 상회
  • 프롬프트 전용 라우터보다 학습 라우터가 일관되게 우수
  • 이진 캐스케이드보다 더 세밀한 비용-품질 트레이드오프 제공
  • 5개 벤치마크 모두에서 저렴한 복구와 승격의 상보성 확인

라우터 품질 및 복구 이질성 분석

복구 이질성

실패 사례마다 최적 복구 행동이 다름:

Reflect만 해결하는 사례 존재
Replan만 해결하는 사례 존재
Escalate만 해결하는 사례 존재
어떤 행동도 해결 못하는 사례도 존재

라우터 절단 분석

  • 전체 입력(문제+verdict+stderr) 사용 시 최고 성능
  • stderr 제거 시 성능 저하 — 실행 피드백의 중요성 확인
  • Cross-model 일반화: 한 모델 쌍으로 학습한 라우터를 다른 모델 쌍에 적용해도 의미 있는 프론티어 형성

Gemini 모델 쌍에 대한 cross-model 실험에서도 유사한 상보성 패턴 관찰

결론 및 핵심 기여

1

실패 후 복구 라우팅 공식화

코딩 에이전트의 복구를 reflect, replan, escalate 세 행동에 대한 라우팅 문제로 정의

2

예산 제어 가능 배포 메커니즘

CRC 보정 레이어로 단일 학습 라우터에서 다중 예산 운영점 생성, 재훈련 없이 한계 기대비용 제어

3

실증적 복구 트레이드오프 증명

5개 벤치마크에서 저렴한 복구와 모델 승격의 상보성을 확인하고, 이산적 비용-품질 프론티어 실현
