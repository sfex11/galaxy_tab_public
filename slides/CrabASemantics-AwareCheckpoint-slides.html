Crab: 에이전트 샌드박스를 위한
의미론적 인식 체크포인트/복원 런타임

Checkpoint-and-Restore for Agent Sandboxes

Tianyuan Wu, Chaokun Chang, Lunxi Cao, Wei Gao, Wei Wang

HKUST (홍콩과학기술대학교)

왜 에이전트 샌드박스에서 C/R이 필요한가

수백 턴

장시간 실행되는 에이전트 워크로드는 수백 번의 상호작용 턴을 포함하며, 크래시 시 전체 재실행 비용이 매우 큼

스팟 인스턴스

클라우드 스팟 인스턴스는 비용 절감이 크지만 단기 선점이 발생하며, C/R로 진행 상황을 보존해야 함

Tree RL

Tree GRPO 등 트리 기반 RL 알고리즘은 중간 상태에서 분기가 필요하며, 체크포인트 포크로 공유 접두사 재실행 방지

안전 롤백

치명적이거나 적대적으로 유도된 액션 후 알려진 양호 상태로 샌드박스 복원이 필수적

핵심 문제: 에이전트–OS 의미론적 간극

에이전트 프레임워크는 툴 호출은 보지만 OS 효과를 모르고,
OS는 상태 변화를 보지만 턴 수준의 맥락으로 복원 관련성을 판단하지 못한다.

에이전트 레이어

  • 툴 호출 내용 파악
  • OS 부수효과 불가지
  • 설치된 패키지, 프로세스 상태 누락

OS 레이어

  • 상태 변화 관측 가능
  • 턴 경계 인식 불가
  • 모든 스텝을 동등하게 처리 → 비효율

→ 두 레이어 모두 효율적인 C/R에 필요한 정보를 단독으로 가질 수 없다

기존 접근의 두 극단

시스템 레이어 캡처 상태 정확성 비용
Claude (App) 채팅 + Git/FS 불완전 낮음
LangGraph (Framework) 대화 기록 불완전 낮음
Docker (OS) 전체 컨테이너 완전 보통
E2B (VM) 전체 VM 완전 높음
Crab (Cross-layer) 적응적 완전 낮음

앱 레벨은 효율적이나 불완전하고, OS/VM 레벨은 정확하나 비용이 높다

숨겨진 희소성: 75% 이상의 턴은 무상태

에이전트–OS 의미론적 간극은 대규모 희소성을 숨기고 있다:

핵심 통찰: 런타임이 에이전트를 블랙박스로 취급하고, 턴 경계에서 OS 가시적 효과만 관측하여 체크포인트 입자도를 결정하면 된다

설계 요구사항

R1
복원 정확성
각 복원점은 채팅 기록뿐 아니라 파일시스템, 프로세스 등 이후 실행에 의존하는 전체 샌드박스 상태를 재구성해야 함
R2
낮은 노출 오버헤드
체크포인트가 엔드투엔드 에이전트 지연을 지배해서는 안 되며, 샌드박스 일시정지로 C/R 이점이 상쇄되어서는 안 됨
R3
밀집 배치 확장성
다수 샌드박스의 체크포인트 트래픽이 스토리지 대역폭 등 공유 호스트 자원을 붕괴시키지 않아야 함

세 가지를 동시에 만족하려면 어떤 턴이 복원 관련 상태를 생성하는지 추론하고, 밀집 배치된 샌드박스 간에 체크포인트 작업을 조율해야 한다

Crab 아키텍처 개요

Coordinator

제어 평면 (컨트롤 패스)

  • 에이전트–LLM 경로 프록시
  • 턴 경계 식별
  • LLM 대기 시간과 C/R 중첩

Inspector

상태 추적기

  • eBPF 기반 경량 관측
  • FS/프로세스/메모리 변화 감지
  • 턴별 체크포인트 입자도 분류

C/R Engine

데이터 평면

  • 호스트 범위 스케줄링
  • runc + ZFS + CRIU 활용
  • 버스트 평활화 및 우선순위 조절

에이전트나 C/R 백엔드 수정 없이 호스트 사이드에서 투명하게 동작

워크플로우: 턴 경계에서의 의미론적 분류

1
에이전트가 툴 액션 실행

샌드박스 내에서 셸 명령, 파일 편집, 패키지 설치 등

2
Coordinator가 턴 경계 감지 → Inspector에 질의

LLM API 호출 직전, 해당 턴의 OS 가시적 효과 분류 요청

3
Inspector가 eBPF로 상태 변화 분석 후 분류

FS 변경만? 프로세스 변경만? 둘 다? 또는 무상태?

4
C/R Engine이 LLM 대기 시간 동안 비동기 체크포인트 수행

필요 없는 턴은 스킵; 필요한 턴은 최소 입자도로 스냅샷

Inspector: eBPF 기반 턴 분류

각 턴 종료 시 Inspector는 OS 가시적 효과를 4단계로 분류:

SKIP 체크포인트 불필요

읽기 전용 조회, 대화 내용만 변경 등 복원 관련 상태 없음

FS-ONLY 파일시스템만

소스 코드 편집, 파일 생성/삭제 등 (ZFS 스냅샷만 사용)

PROC-ONLY 프로세스만

백그라운드 서비스 시작, 포트 바인딩 등 (CRIU만 사용)

FULL 전체 상태

패키지 설치 + 서비스 시작 등 FS와 프로세스 모두 변경

eBPF 훅: fanotify(FS), tracepoint(프로세스), kprobe(메모리 매핑)

C/R Engine: 호스트 범위 스케줄링

백엔드 조합

  • runc — 컨테이너 라이프사이클 관리
  • ZFS 스냅샷 — 파일시스템 상태 (초고속, Copy-on-Write)
  • CRIU — 프로세스 상태 (메모리, FD, 네트워크)

핵심 전략

  • 입자도별 최소 백엔드만 호출
  • 체크포인트 버스트를 시간적으로 평활화
  • 에이전트 진행을 차단할 위험이 높은 작업 우선 처리

LLM 대기 시간 숨기기

에이전트는 자연적으로 툴 실행과 LLM 응답 대기를 교대로 반복함

Crab은 이 LLM 대기 창을 체크포인트 작업에 활용하여 대부분의 비용을 숨김

p95 노출 지연: 작업 시간의 0.44% (64 샌드박스 밀집)

평가: 복원 정확성과 엔드투엔드 오버헤드

워크로드: SWE-Bench (코드 수정) + Terminal-Bench (셸 집약적) · 3종 에이전트/LLM 조합

밀집 배치에서의 확장성

96개 샌드박스 동시 배치 시:

  • 매 턴 전체 C/R: 3.78× 느려짐 (호스트 I/O 병목)
  • Crab: 정상 수준 유지

C/R Engine의 버스트 평활화가 공유 스토리지 대역폭 붕괴를 방지

사례 연구: 롤백을 에이전트 툴로 노출

Crab의 C/R 능력을 에이전트가 직접 호출할 수 있는 툴 API로 노출:

기존: 셸 레벨 자가 복구

에이전트가 git reset, 파일 수동 복원, 프로세스 재시작 등 복잡한 셸 명령으로 복구 시도

→ 취약하고, 불완전하며, 많은 토큰 소모

Crab: 단일 샌드박스 복원

에이전트가 rollback(checkpoint_id) 툴을 한 번 호출하여 알려진 양호 상태로 즉시 복원

→ 신뢰성 높고, 빠르고, 토큰 절약

29%↓

벽시계 시간 감소

36%↓

롤백 토큰 감소

결론

의미론적 간극의 식별과 정량화

채팅만 복원 시 8–13% 성공, 매 턴 전체 C/R 시 3.78× 느려짐

Crab: 호스트 사이드 의미론적 인식 C/R 런타임

에이전트/C/R 백엔드 수정 없이, eBPF 관측 + LLM 대기 중첩 + 호스트 범위 스케줄링

검증된 성과

100% 복원 정확성, 체크포인트 트래픽 87% 감소, 무장애 시간의 1.9% 이내 오버헤드 (96 샌드박스)

에이전트 샌드박스에서 C/R은 선택이 아닌 필수 인프라이며,
Crab은 정확성과 효율성의 극단을 동시에 달성한다.
