모델 변경 없는 동시성:
Future 기반 비동기 함수 호출 프레임워크

Concurrency without Model Changes: Future-based Asynchronous Function Calling for LLMs

Guangyu Feng, Huanzhi Mao, Prabal Dutta, Joseph E. Gonzalez

UC Berkeley

문제 정의: 동기식 함수 호출의 병목

현대 LLM 에이전트의 핵심 능력인 함수 호출(Function Calling)은 표준적으로 동기식 실행 의미론을 따릅니다.

  • 모델이 함수 호출을 발행하면 디코딩이 블로킹됨
  • 함수 실행 완료 후에야 다음 디코딩 재개
  • 함수 호출이 많아질수록 종단간 지연(end-to-end latency) 선형 증가
  • 독립적 함수 호출 간에도 병렬 실행 불가

동기식 실행 타임라인

디코딩
F1 실행
디코딩
F2 실행
디코딩

실행과 디코딩이 완전히 직렬화됨

핵심 관찰: LLM은 미해결 결과를 추론할 수 있다

LLM은 미해결 실행 결과를 나타내는 심볼릭 Future 플레이스홀더에 대해 원래부터 추론할 수 있는 능력을 갖추고 있다. 이를 활용하면 모델 수정 없이 비동기 패러다임을 도입할 수 있다.

1

Future 인식

모델은 future 객체를 일반 반환값처럼 처리 가능

2

Future 전달

미해결 future를 후속 함수 호출의 인자로 전달 가능

3

해결 시 적응

이후 턴에서 실제 결과로 대체되면 자연스럽게 수용

AsyncFC: 개요

AsyncFC는 순수 실행 계층 프레임워크로, 모델 디코딩과 함수 실행을 분리하여 두 가지 형태의 동시성을 달성합니다.

디코딩-실행 중첩

함수가 백그라운드에서 실행되는 동안 모델 디코딩이 계속 진행됩니다. Future 플레이스홀더가 즉시 반환되어 블로킹을 제거합니다.

함수 간 병렬 실행

의존성이 없는 함수 호출들은 병렬로 실행됩니다. 개발자가 제공한 리소스 상호작용 주석을 기반으로 안전한 병렬성을 보장합니다.

파인튜닝 불필요 모델 수정 없음 기존 함수 구현 변경 없음 표준 FC 프로토콜 호환

방법론 1: 논블로킹 디코딩

1

2

런타임이 백엔드 실행기로 전달

블로킹 없이 후속 호출 생성

스키마 자동 변환 — 출력

함수 스키마를 자동으로 변환하여 단일 Future 객체를 반환값으로 가지도록 변경합니다. 런타임이 반환하는 심볼릭 future가 함수의 즉시 반환값이 됩니다.

스키마 자동 변환 — 입력

함수 입력을 구체적 값 또는 Future 식별자 모두 수용하도록 듀얼 타입으로 변환합니다. 이를 통해 미해결 future를 인자로 사용한 후속 호출이 가능해집니다.

방법론 2: 결과 검색 및 의존성 스케줄링

결과 통합 매커니즘

  • 함수 실행 완료 시 턴 경계에서 결과를 컨텍스트에 통합
  • Future 플레이스홀더가 실제 해결된 결과로 대체
  • await_future 함수로 모델이 명시적으로 future 해결 가능
  • 에러 메시지도 동일한 인터페이스로 통합

의존성 인식 스케줄링

  • 기본적으로 보수적 직렬 실행
  • 개발자 제공 리소스 상호작용 주석으로 병렬성 해제
  • 같은 리소스에 쓰는 호출은 직렬화
  • 독립적 리소스 접근은 안전하게 병렬 실행

모델이 발행한 호출 순서를 기준으로 상태 업데이트, 함수 반환, 모델 컨텍스트가 결정론적이 됩니다 — 기존 병렬 FC API의 레이스 컨디션 문제를 해결합니다.

확장: 비동기 사고(Asynchronous Thinking)

AsyncFC의 Future 기반 실행 인터페이스는 외부 함수 호출뿐만 아니라 비동기 사고로도 자연스럽게 확장됩니다.

기존 비동기 사고 접근법

  • 모델을 명시적으로 fork/join 서브쿼리하도록 훈련 필요
  • 모델 아키텍처 또는 학습 과정 변경 요구
  • 추가 파인튜닝 비용 발생

AsyncFC의 비동기 사고

  • 실행 계층 변경만으로 병렬 사고 노출
  • 모든 긴 지연 연산을 동일한 Future 인터페이스로 표현
  • 별도 훈련 없이 모델의 추론 병렬성 활용

외부 함수 호출과 내부 사고 과정을 통일된 Future 기반 인터페이스로 처리함으로써, AsyncFC는 도구 사용과 추론의 경계를 허물는 범용 프레임워크가 됩니다.

설계 직관: 가속 패턴 분석

AsyncFC의 가속 효과는 작동 체제(Operation Regime)에 따라 다르게 나타나며, 모델 능력에 대한 민감도도 존재합니다.

저장 분해

전체 지연 시간에서 디코딩 시간과 실행 시간의 비율에 따라 가속 효과가 결정됩니다. 실행 지연이 클수록 가속 비율이 높아집니다.

모델 능력 민감도

모델이 Future 플레이스홀더를 더 잘 이해할수록 더 많은 병렬성을 추출하고 가속 효과를 극대화합니다.

의존성 구조

함수 호출 간 의존성이 적을수록 병렬 실행 기회가 많아지며, 의존성이 많으면 직렬 실행으로 인해 가속이 제한됩니다.

논의 및 한계점

논의

  • 실행 동시성 관리를 모델에서 시스템 런타임으로 이동하는 것이 올바른 설계 방향
  • LLM의 숨겨진 능력(Future 추론)을 발견한 것 자체가 중요한 기여
  • 순차 FC 모델도 병렬 FC 이상의 성능을 낼 수 있음을 실증

한계점

  • 의존성 주석이 개발자가 수동으로 제공해야 함
  • 보수적 기본 정책으로 인해 주석 없이는 직렬 실행
  • 함수 실행 지연이 매우 짧으면 가속 효과 미미
  • Future reasoning 능력의 모델 간 편차 존재

결론

AsyncFC는 순수 실행 계층 프레임워크로서, 모델 수정 없이 LLM 함수 호출에 비동기 프로그래밍 패러다임을 도입합니다.

일관된 지연 감소

도구 사용, 소프트웨어 엔지니어링, 다중 홉 추론 벤치마크에서 정확도 유지

프로토콜 호환성

표준 동기 FC 프로토콜을 보존하며 기존 에코시스템과 완전 호환

새로운 상호작용 패턴

LLM이 심볼릭 Future를 추론하는 원래 능력을 발견, 비동기 사고로 확장

동시성 관리는 모델이 아니라 시스템의 책임이다.
