배경: 복합 에이전트 시스템의 부상

단일 턴 → 복합 시스템

단순 챗봇에서 도구 사용, 동적 메모리, 다단계 제어 흐름을 오케스트레이션하는 정교한 운영 단위로 진화

복잡한 실행 궤적

추론 체인, 코드 실행 환경, 상태 전이를 포함하는 복잡한 실행 트레이스가 생성됨

이러한 궤적 정보를 효과적으로 활용하여 에이전트 시스템을 더욱 향상시키는 것이 핵심 과제가 되었다.

지배적인 최적화 패러다임: 반사적 최적화(Reflexive Optimization) — LLM이 실패를 진단하고 에이전트 정책을 개선

문제 정의: Context-Noise Trade-off

문제 A 전체 궤적

무관한 성공 단계와 저신호잡음비(SNR) 컨텍스트가 과부하를 유발 → 허구적 상관관계, 환각적 프롬프트 패치

문제 B 짧은 절단

국소 근접성을 인과적 관련성의 신호로 처리 → 멀리 있으나 인과적으로 필수적인 기록을 삭제할 위험

해결 STRACE

인과적 컨텍스트 증류를 통해 컴팩트한 인과 슬라이스를 추출하여 고SNR 최적화 컨텍스트 제공

핵심 현실: 에러가 나타나는 노드(하류 증상)는 종종 에러가 기원하는 노드(상류 근본 원인)가 아니다. 예: Step 50의 Code Interpreter 충돌은 Step 5의 Planner 오류의 지연된 결과일 수 있다.

실행 로그를 단일 선형 텍스트가 아닌 인과 그래프로 취급하는 최적화 프레임워크

배치 수준

실패 패턴 마이닝으로 중복 트레이스 필터링, 대표적 실패만 보존

개별 트레이스 수준

텍스트 의존성 그래프 상에서 인과 지역화로 비인과 단계 제거, 진정한 근본 원인 모듈 식별

Phase 1: 구조적 모델링

텍스트 의존성 그래프 구성

  • 에이전트 컴포넌트 간 데이터 및 제어 관계를 의존성 사전으로 기록
  • 컴팩트한 텍스트 의존성 그래프(Textual Dependency Graph) 추론
  • 노드: 에이전트 모듈 (Planner, Code Interpreter, Retriever 등)
  • 엣지: 모듈 간의 정보 흐름 및 제어 의존성

역할

이 토폴로지는 후속 단계들의 구조적 가이드로 활용됨

단순한 선형 텍스트가 아닌, 구조적 관계를 명시적으로 모델링하여 인과 추적의 기반 제공

Phase 2: 실패 패턴 마이닝 & 트레이스 필터링

1

배치 진단 요약

각 배치의 실패 트레이스를 통계적 및 구조적 진단으로 요약

2

재발 및 심각도 신호 분석

실패 패턴의 빈도와 심각도를 기준으로 중복/저가치 실패 식별

3

대표적 실패 트레이스 보존

컴팩트한 트레이스 집합만 남겨 최적화 효율성 극대화

모든 실패 트레이스에서 최적화를 시도하는 것은 계산적으로 비실용적이며, 취약하고 과적합된 업데이트를 초래함

Phase 3: 인과적 지역화 (Causal Localization)

의존성 역추적

  • 의존성 그래프를 따라 역방향 추적 수행
  • 무관한 단계를 제거하여 컴팩트한 인과 슬라이스 추출
  • 에러의 논리적 기원을 발현 노드가 아닌 위치에서 식별

근본 원인 모듈 식별

최적화 대상으로서 진정한 루트-카우스 모듈을 정확히 지정

인과 슬라이스 vs 기존 방법

절단(Truncation): 시간적 근접성 기준 → 인과적 연결 단절 위험

슬라이딩 윈도우: 고정 범위 → 장거리 인과 의존성 누락

인과 슬라이스: 의존성 그래프 기반 → 증상과 기원 간 시간적 간극을_bridge하면서 노이즈 동시 필터링

Phase 4: 귀납적 정책 최적화

최적화 과정

  • 지역화된 실패 에피소드를 활용하여 일반화된 자연어 휴리스틱 합성
  • 예방적 가이드라인을 식별된 근본 원인 모듈의 명령어에만 주입
  • 기본 실행 코드 변경 없이 지속적, 안전, 비용 효율적 개선

안전성 보장

코드 수정이 아닌 프롬프트 수준의 개입만으로 정책 업데이트

기존 시스템의 동작을 파괴하지 않고 점진적 향상 가능 — 롤백도 용이

핵심 차별점: 증상 수리가 아닌 근본 원인 모듈에만 타겟팅된 최적화를 수행함으로써, 잘못된 컴포넌트 업데이트를 방지

최적화 확장성 및 비용 분석

확장성

  • 트레이스 필터링으로 불필요한 최적화 비용 대폭 절감
  • 인과 슬라이스로 컨텍스트 길이 효율적 관리
  • 배치 크기 증가에도 안정적 성능 유지

비용 효율성

  • Full Trajectory 대비 토큰 사용량大幅 감소
  • 단계별 비용 분석에서 Phase 2의 필터링이 가장 큰 비용 절감 기여
  • 코드 수정 없이 프롬프트만 업데이트하므로 배포 비용 최소화

STRACE는 더 적은 컨텍스트로 더 정확한 최적화를 달성하여, 비용-성능 트레이드오프에서 기존 방법들을 일관되게 상회

Step 5: Planner 오류

기존 방법

Code Interpreter(Step 50)의 명령어를 수정 시도 → 증상만 수리, 근본 원인 미해결 → 동일 패턴 재발

STRACE

Planner(Step 5)를 근본 원인으로 식별 → Planner 명령어에 예방적 가이드라인 주입 → 패턴 근절

절제 연구 (Ablation Study)

구성 제거된 단계 영향
STRACE (Full) — 최고 성능
w/o Structural Modeling Phase 1 제거 성능 하락 의존성 정보 부재
w/o Trace Filtering Phase 2 제거 성능 하락 중복 트레이스 과부하
w/o Slicing (Current Node) Phase 3 — 현재 노드만 부분 하락 근본 원인 탐지 실패

네 단계 모두가 상호 보완적으로 작동하며, 특히 구조적 모델링(Phase 1)과 인과 슬라이싱(Phase 3)의 결합이 핵심 성능 driverr
