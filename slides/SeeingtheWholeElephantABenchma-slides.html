"전체 코끼리를 보라" — 완전 실행 추적의 필요성

문제 정의

실패 귀인의 형식적 정의

LLM 기반 MAS는 유한한 기능 컴포넌트 집합 𝒜 = {a₁, ..., aₙ}으로 구성되며, 턴 기반 프로토콜 하에 이산 단계로 실행됩니다. 각 단계 t에서 활성 컴포넌트 a(t)는 입력 xₜ을 받아 출력 yₜ을 생성합니다.

단계 수준 귀인

실패가 불가피해지는 최초 시점을 식별합니다. 단순한 첫 번째 실수가 아닌, 복구 가능성이 상실되는 지점입니다.

에이전트 수준 귀인

해당 단계에서 실패를 유발한 책임 컴포넌트를 식별합니다. 반드시 별도 에이전트가 아닐 수 있으며, 기능 모듈 단위입니다.

추적 스키마: oₜ = (xₜ, yₜ, a(t), step_idₜ, agent_idₜ) — 각 단계의 입력, 출력, 활성 컴포넌트, 단계 식별자, 에이전트 식별자를 기록

분석

실패 및 귀인 패턴 분석

단계 수준의 높은 민감도

단계 수준 귀인은 에이전트 수준보다 정보 누락에 훨씬 민감합니다. 정확한 단계를 찾으려면 더 세밀한 컨텍스트 이해가 필요하며, 입력 정보의 부재가 치명적입니다.

아키텍처별 편차

다중 에이전트 오케스트레이션 vs 단일 에이전트 스캐폴드 간에 귀인 난이도가 다르며, 아키텍처 인식적 귀인 접근이 필요합니다.

에이전트 유형별 차이

기획자, 실행자, 검증자 등 컴포넌트 역할에 따라 실패 패턴과 귀인 난이도가 상이하게 나타납니다.

단계 위치 의존성

실패 단계의 위치(초기/중기/후기)에 따라 귀인 성능이 달라지며, 후기 단계일수록 복잡한 인과 사슬을 추적해야 합니다.

시사점

실무적 시사점과 함의

개발자를 위한 조언

  • 디버깅 시 가능한 모든 실행 상세 정보를 활용할 것
  • 출력만 보고 귀인하면 상당수 실패 원인을 놓침
  • 재현 가능한 환경 구축으로 가설 검증을 지원할 것

연구자를 위한 방향

  • 아키텍처 인식적 귀인 기법 개발 필요
  • 단계 수준 귀인에 특화된 접근법 연구
  • 다양한 관측성 수준에서의 견고한 귀인 모델

"필드는 다양한 관점에서 귀인 기법을 평가할 수 있는 더 다채로운 벤치마크를 통해
MAS에서의 실패 귀인에 대한 더 견고하고 누적적인 이해를 구축할 수 있을 것입니다."
