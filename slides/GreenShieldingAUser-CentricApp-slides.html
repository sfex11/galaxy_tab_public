Green Shielding: 사용자 중심의
신뢰할 수 있는 AI를 위한 접근법

LLM 기반 의료 진단을 사례 연구로
Aaron J. Li, Nicolas Sanchez, Hao Huang 외 · UC Berkeley, UCSF, Univ. of Melbourne 등

문제 제기: 일상적 입력 변화의 위험성

AI Safety I 적대적 위협

레드팀(Red-teaming)이 중심. 최악의 시나리오, 적대적 공격, 재난적 리스크에 집중. 기존 연구의 주류.

AI Safety II 일상적 위험 ← 본 논문

비적대적이고 일상적인 사용자 입력 변화가 모델 출력에 미치는 영향. 실제 배포 환경에서의 안정성 문제.

"LLM 출력은 사용자가 질문을 표현하는 방식의 일상적이고 비적대적인 변화에 매우 민감할 수 있으며, 이는 기존 레드팀 노력으로는 충분히 다뤄지지 않는 간극이다."

Green Shielding이란?

상용 제품의 사용 설명서처럼, LLM을 고정된 배포 제품으로 취급하고 실제 사용 환경에서의 증거 기반 지침을 구축하는 연구 아젠다.

01
모델 수정 불필요

모델 내부를 변경하는 대신, 배포된 모델의 행동을 분석

02
비적대적 관점

적대적 공격이 아닌, 일상적인 사용자 입력 변화에 초점

03
실증적 근거

PCS 프레임워크 기반의 체계적인 경험적 설계

Frontier AI · Responsible AI · Pragmatic AI 세 가지 사회기술적 상상력을 연결하는 다리 역할

CUE 기준: 경험적 기반의 세 가지 축

PCS(예측가능성, 계산가능성, 안정성) 프레임워크에 기반한 사용자 중심 지침 구축 기준

C
Context — 맥락

진정한 실제 사용 사례를 대변하는 태스크. 시험형 벤치마크가 아닌, 실제 환자가 작성한 질문 등

U
Utility — 유용성

태스크 관련 유용성을 측정하는 참조 표준과 평가 지표. 단일 정답이 아닌, 임상적으로 의미 있는 속성 측정

E
Elicitation — 유도

사용자 입력 및 상호작용의 변화가 어떻게 다른 모델 행동을 유도하는지를 반영하는 섭동 체계

파일럿 연구: 기존 벤치마크에서의 프롬프트 민감도

MedQA, MedXpertQA 벤치마크에서 세 가지 차원의 프롬프트 섭동 실험

내용 (Content)

임상 정보의 순서 변경, 추가 힌트 제공/제거, 질문 재구성

형식 (Format)

개방형 생성 vs. 선택형, 출력 구조 지시, 답변 길이 요구

어조 (Tone)

전문적/비전문적, 긴급/비긴급, 공감적/객관적 표현

핵심 발견

모든 차원의 섭동이 일관되게 응답 정확도를 변화시킴. 특히 정보 완전하고 단일 정답인 시험형 벤치마크는 실제 진단의 불확실성과 다중 가능성을 포착하지 못하는 근본적 한계 노출.

HCM-Dx: 환자 작성 쿼리 벤치마크

HealthCareMagic-100K에서 진단 태스크에 맞게 필터링 및 적응한 데이터셋

기존 벤치마크의 한계
  • 정보 완전, 단일 정답 구조
  • 실제 환자 표현 방식과 차이
  • 진단 불확실성 미반영
  • 감별진단 목록 평가 불가
HCM-Dx의 특징
  • 실제 환자가 작성한 자연어 쿼리
  • 노이즈, 모호성, 정보 부족 포함
  • 다중 진단 가능성 반영
  • 감별진단 목록 평가에 적합

참조 구조 및 평가 지표

동반 의사 응답 대신, 복수 프론티어 LLM으로 생성한 구조화된 참조 진단 세트

Plausible

임상적으로 타당한 진단

Highly Likely

가장 가능성이 높은 진단

Safety-Critical

놓치면 안 되는 위중증

구조적 지표
  • Plausible Precision — 타당 진단의 정밀도
  • Likely Coverage — 고가능성 진단 포함률
  • Safety-Critical Coverage — 위중증 포함률
  • Differential Breadth — 제안 진단 수
의미적 지표
  • Evidence Grounding — 근거 기반성
  • Epistemic Uncertainty — 인식적 불확실성 표현

프롬프트 중립화 (Prompt Neutralization)

사용자 수준 요인을 제거하면서 임상 내용은 보존하는 자동화된 재작성 모듈

1
원본 쿼리
환자가 작성한 자연어 질문 (감정, 비전문적 표현 포함)
2
요인 식별
감정적 표현, 비전문 용어, 서술 방식 등 사용자 요인 탐지
3
중립화 재작성
객관적이고 표준화된 의학적 설명으로 변환
4
중립화 쿼리
임상 내용 보존, 사용자 요인 제거된 입력
제거 대상 요인

감정적 언어 · 불안/공포 표현 · 비전문적 증상 묘사 · 시간/강조 수식어 · 개인적 맥락 (직업, 나이 등 비임상 정보) · 반복 표현

실험 설정

평가 모델
  • GPT-4o, GPT-4o-mini
  • Claude 3.5 Sonnet, Claude 3 Haiku
  • Gemini 1.5 Pro, Gemini 1.5 Flash
  • Llama 3.1 405B, 70B
  • Mixtral 8x22B
실험 조건
  • Raw — 원본 환자 쿼리 직접 입력
  • Neutralized — 프롬프트 중립화 적용
  • 의사 응답과의 비교 분석
  • 개별 요인 통제 섭동 실험
  • 전문가 검증 평가 파이프라인

참조 진단 세트 생성: 복수 프론티어 LLM 활용 → 전문가 검증 → 자동화 평가 파이프라인 구성

주요 결과: 벤치마크 성능

원본 환자 입력(Raw) 조건에서의 다중 프론티어 LLM 성능

높음
Plausible Precision

대부분 제안 진단이 임상적으로 타당한 범위 내

낮음
Likely Coverage

가장 가능성 높은 진단을 놓치는 비율 상당

낮음
Safety Coverage

위중증 진단 포함률 현저히 부족

프론티어 모델이라 할지라도 가장 확률이 높거나 놓치면 안 되는 조건을 안정적으로 제시하지 못함. 단일 점수 평가에서는 이 문제가 가려짐.

핵심 결과: 프롬프트 중립화의 트레이드오프

중립화가 유도하는 일관되고 해석 가능한 파레토형 트레이드오프

지표 Raw → Neutralized 변화 방향
Plausible Precision 증가 개선
Differential Breadth 감소 (더 간결) 의사 유사
Likely Coverage 감소 악화
Safety-Critical Coverage 감소 악화

중립화는 더 간결하고 타당한 감별진단을 생성하지만, 고가능성 및 위중증 진단의 포함률을 희생함. 어느 한 조건이 일률적으로 우월하지 않음.

의사 응답과의 비교

실제 임상의 감별진단 패턴 vs. LLM 감별진단 패턴

임상의 패턴
  • 매우 좁은 감별진단 목록
  • 소수의 선도 가설에 우선 집중
  • 포괄적 열거보다는 우선순위 부여
  • 임상 맥락에 기반한 좁히기
LLM 패턴
  • 상대적으로 긴 감별진단 목록
  • 다수 진단을 나열하는 경향
  • 중립화 시 의사와 더 유사해짐
  • 그러나 위중증 포함률은 여전히 낮음

중립화가 형태적으로는 의사와 유사해지지만, 위중증 진단 포함이라는 안전적 차원에서는 의사의 판단과 다른 방향으로 변화함.

논의 및 시사점

PCS as Guiding Framework

예측가능성·계산가능성·안정성 렌즈를 통해 실증적 절차가 태스크에 정렬되고, 실제 제약 하에서 실행 가능하며, 현실적 이질성에 강건하도록 설계.

트레이드오프의 해석

단일 점수가 아닌 다차원 평가가 필수. 상황, 위험 수용도, 하류 의사결정에 따라 바람직한 운영점이 달라짐. "정답" 프롬프트는 존재하지 않음.

확장 가능성

의료 진단에 한정되지 않음. 법률, 교육, 재무 등 다른 의사결정 지원 설정과 에이전트 AI 시스템으로 자연스럽게 확장 가능.

한계

태스크 공식화의 한계 (감별진단 ≠ 전체 진단 과정), 평가 프레임워크의 한계 (LLM 기반 참조), 전문가 불일치와 규범적 불확실성, 대화형 평가 필요성.

결론

1
상호작용 선택이 체계적으로 출력 속성을 변화시킨다

비적대적이고 일상적인 프롬프트 차이가 임상적으로 의미 있는 차원에서 모델 행동을 이동시킴

2
단일 점수 평가는 이러한 트레이드오프를 은폐한다

정밀도-포함률 트레이드오프를 명시적으로 측정 가능하게 만들어야 함

3
Green Shielding은 증거 기반 배포 지침의 토대를 제공한다

CUE 기준을 통해 실제 사용 환경에 정렬된 평가 체계를 구축 가능

"최적의 프롬프트를 찾는 것이 아니라,
상황에 맞는 운영점을 선택하기 위한 증거를 구축하는 것"
