Self-Evolution through Source-Level Rewriting in Autonomous Agent Systems

문제 제기: 배포 후 정적 상태의 에이전트

현재 상황

  • 배포 후 사용자 상호작용으로부터 학습하지 않음
  • 동일한 실패 모드가 사용자 간 반복 발생
  • 수정은 인간 주도 업데이트에 전적으로 의존

근본 원인

기존 자가 진화 에이전트는 텍스트 가변 산출물만 수정

  • 스킬 파일, 프롬프트 설정, 메모리 스키마, 워크플로우 그래프
  • 에이전트 하니스(라우팅, 훅, 상태 관리, 디스패치)는 수정 불가

텍스트 계층에서 물리적으로 도달할 수 없는 구조적 실패 클래스가 존재한다 — 버그가 프롬프트 텍스트에 없다면, 프롬프트 재작성으로는 해결할 수 없다.

소스 수준 적응의 4가지 우위

1

튜링 완전성

프로그래밍 언어는 튜링 완전 → 소스 코드 설계 공간은 모든 텍스트 가변 설계 공간을 엄격한 부분집합으로 포함

2

엄격한 상위 집합

프롬프트 편집이 달성할 수 있는 것은 동등한 코드 편집도 달성 가능하지만, 역은 성립하지 않음

3

결정적 효과

라우팅 로직, 훅 순서, 상태 머신 불변성은 코드로 실행 → 베이스 모델의 준수 여부에 의존하지 않음

4

롱컨텍스트 드리프트 내성

텍스트 수정은 매 턴마다 재독해 필요 → 누적 시 준수도 희석. 소스 수정은 행동으로 인코딩되어 노후화에 강함

MOSS: 개요

프로덕션 에이전트 기질(substrate)에 대해 소스 수준 자가 재작성을 수행하는 시스템

  • 각 진화는 자동 큐레이션된 프로덕션 실패 증거 배치에 고정
  • 결정적 다단계 파이프라인으로 진행
  • 코드 수정은 플러그인 가능한 외부 코딩 에이전트 CLI에 위임
  • MOSS는 스테이지 순서와 판정(verdict)만 보유
  • 후보 검증: 일시적 trial worker에서 배치 재생
  • 승격: 사용자 동의 게이트 + 인플레이스 컨테이너 스왑 + 헬스 프로브 롤백

OpenClaw에서 4태스크 평균 그레이더 점수 0.25 → 0.61 (단일 사이클, 인간 개입 없음)

시스템 아키텍처: 5가지 구성 요소

Substrate (기질)

사용자 대면 에이전트를 실행하는 메인 컨테이너 (OpenClaw 등)

Control Surface (제어 표면)

에이전트가 진화를 주도하는 moss evo CLI (9개 서브커맨드)

External Coding Agent (외부 코딩 에이전트)

호스트 사이드 서브프로세스로 호출되는 플러그인 가능 CLI (Claude Code, Codex, DeepSeek 등)

Host-Daemon (호스트 데몬)

전체 프로세스를 감독하는 호스트 상주 데몬

Ephemeral Trial Workers (일시적 시험 워커)

후보 이미지에 대해 배치를 재생하여 검증

제어 표면: moss evo CLI

status

batches

3계층 인터페이스 계약

  • CLI 입력: 에이전트의 셸 툴에서 호출
  • 웹훅 출력: evolution-converged, evolution-failed, apply-complete
  • 능력 문서: 온디스크 참조로 에이전트가 온디맨드 독해

기질이 5가지 프리미티브(셸 실행, 파일시스템 읽기, 주기적 스케줄링, 웹훅-에이전트 전달, 시스템 프롬프트 주입)만 제공하면 MOSS 코드 변경 없이 호스팅 가능

외부 코딩 에이전트

왜 외부 위임인가?

대규모 기질에서 단일 추론 루프로 코드베이스를 읽고 재작성하는 것은 불가능 — 교차 파일 불변성과 동시 상태 상호작용이 한 루프의 용량을 초과

MOSS의 역할

  • 스테이지 순서 관리
  • 판정(verdict) 내림
  • 루프 종료 결정
  • 스왑 시점 결정

코딩 에이전트의 역할

  • 주어진 범위 내 코드 편집
  • 호스트 사이드에서 실행 (쉘 환경, 네트워크, 마운트 필요)
  • 4-메서드 runner 인터페이스로 추상화

내장 러너: Claude Code OpenAI Codex DeepSeek-T — 플러그인 가능 구조로 확장 용이

진화 프로세스: 방향성 진화

합성 벤치마크가 아닌 실제 프로덕션 실패 증거에 고정된 진화

증거 수집: 2가지 상보적 채널

백그라운드 자동 스캔

주기적 스캔이 최근 사용자 세션에서 저성능 대화 세그먼트를 자동 발굴

사용자 플래깅

사용자가 대화 중 임의의 턴을 직접 플래그 — 동일한 배치로 유입

핵심 원칙

모든 다운스트림 스테이지는 합성 벤치마크가 아닌 해당 배치를 수정하는 것에 고정됨

스테이지 파이프라인

1
분석
배치 내 실패 패턴 분석, 수정 대상 코드 영역 식별
2
계획
수정 계획 수립 — 변경 파일, 예상 효과, 리스크 평가
3
편집
외부 코딩 에이전트가 소스 코드 수정 수행
4
빌드
수정된 코드베이스 빌드 → 후보 이미지 생성
5
검증
일시적 trial worker에서 배치 재생 후 채점

모든 산출물(계획, diff, 빌드 로그, 반복별 채점 행렬)은 에이전트가 온디맨드로 읽을 수 있어 사용자가 사전 감사 가능

런타임 검증 및 인플레이스 컨테이너 스왑

런타임 검증

  • 단위 테스트나 합성 하니스가 아닌 실제 배치 재생
  • 일시적(ephemeral) trial worker 사용
  • 프로덕션 등가 컨테이너에서 실행
  • 수렴 시 에이전트에게 대화 채널로 프로액티브 통지

인플레이스 컨테이너 스왑

  • 명시적 사용자 승인 후에만 호스트 데몬이 수행
  • 사용자 상태 보존
  • 스왑 후 헬스 프로브 실패 시 자동 롤백
  • 라이브 배포 중단 없음

사용자 동의 게이트 + 헬스 프로브 게이트 롤백 = 안전한 프로덕션 배포 보장

사례 연구: OpenClaw 설정

기질: OpenClaw

  • 프로덕션 등급 에이전트 시스템
  • 멀티채널 게이트웨이 (Slack, Discord, Web)
  • 플러그인/훅 배관, 세션/스킬 기계장치
  • 영속적 사용자 상태

4개 태스크

  • 다단계 작업 수행
  • 그레이더 점수로 평가
  • MOSS 개입 전 베이스라인: 평균 0.25
  • 단일 진화 사이클만 실행

벤치마크 점수가 아닌 실제 프로덕션 실패 배치에 고정 → 합성 평가와의 근본적 차이

사례 연구: 결과

0.25 → 0.61

4태스크 평균 그레이더 점수 · 단일 사이클 · 인간 개입 없음

MOSS의 수행 내용

  • 프로덕션 실패 배치 자동 큐레이션
  • 소스 수준 코드 수정 (텍스트가 아닌 하니스 포함)
  • trial worker에서 런타임 검증
  • 인플레이스 컨테이너 스왑으로 프로덕션 반영

핵심 시사점

  • 텍스트 계층 수정만으로는 달성 불가능한 개선
  • 하니스 수준의 구조적 버그 해결 가능
  • 벤치마크가 아닌 실제 실패에 기반한 진화
  • 단일 사이클로 유의미한 점수 향상

관련 연구와의 차별성

연구 스케폴드 피드백 패러다임 프로덕션
SICA 최소 벤치마크 점수 탐색적
Darwin Gödel Machine 최소 벤치마크 점수 탐색적
HyperAgents 최소 벤치마크 점수 탐색적
Meta-Harness 최소 실행 트레이스 탐색적
MOSS 프로덕션 등급 프로덕션 실패 배치 방향성 진화

기존 연구는 최소 스케폴드 + 벤치마크 점수 기반 탐색적 패러다임. MOSS는 프로덕션 기질 + 실제 실패 기반 방향성 진화로 전환

결론

  • 기존 자가 진화 에이전트는 텍스트 가변 산출물에만 국한 → 하니스 수준의 구조적 실패에 도달 불가
  • 소스 수준 적응은 튜링 완전, 엄격한 상위 집합, 결정적 효과, 드리프트 내성의 4가지 우위 보유
  • MOSS는 프로덕션 기질에서 소스 수준 자가 재작성을 안전하게 수행하는 최초의 시스템
  • 실제 프로덕션 실패에 고정된 방향성 진화로 벤치마크 의존성 극복
  • OpenClaw에서 0.25 → 0.61 달성 (단일 사이클, 무인간 개입)
