LLM과 멀티모달 LLM의 발전으로 브라우저, 터미널 등 일상 도구를 직접 조작하는 능동적 에이전트(Proactive Agent)가 등장하고 있습니다.

핵심 질문: 실제 환경에서 이들의 진정한 능력을 어떻게 체계적으로 측정할 것인가?

기존 벤치마크의 3가지 구조적 한계

1

실제 환경 복잡도 포착 불가

WebArena의 자체 호스팅 웹사이트 미러, OSWorld의 캐시된 페이지 등 샌드박스 환경에 의존하여 실제 성능과의 극심한 격차 발생

2

단일 턴 평가 패러다임

사용자가 반복적으로 피드백을 제공하는 폐루프 인터랙션을 완전히 무시

3

시나리오 기반 분류의 한계

"사무실", "연구" 등 시나리오로 분류하면 실패 원인을 식별 불가 — 시각 인식 문제인지, 장문 추론 문제인지, 도구 사용 문제인지 구분 불가

해결해야 할 2가지 근본적 과제

안정적인 정답(Ground Truth) 부재

실제 환경에서는 정답이 고정되지 않음. 예: 아마존 상품이 오늘 $199, 내일 $179. 사전 녹음된 답안은 즉시 유효성을 상실함.

해결책: 단계별 완료 체크포인트 기반 평가

정보 격리의 역설

자연스러운 사용자 시뮬레이션이 정답이나 평가 기준에 접근하면, 정보 오염(Data Contamination) 위험. 하지만 접근 없이는 자연스러운 피드백 불가.

해결책: 3역할 폐루프 평가 전략 (정보 방화벽)

UniClawBench 개요

실제 환경에서 능동적 에이전트를 평가하기 위한 최초의 능력 주도형(Capability-Driven) 벤치마크

역할 기반
폐루프 평가

  • Docker 컨테이너 내 실제 소프트웨어, 라이브 브라우저, 로컬 파일시스템에서 실행
  • 사전 녹음된 정답 대신 단계별 완료 체크포인트 활용
  • 정보 방화벽을 통한 다중 턴 피드백 시뮬레이션

5가지 핵심 능력 차원

Skill Usage

사전 정의된 스킬을 올바르게 호출하고 인자를 구성하는 능력

Exploration

미지의 인터페이스와 환경을 자율적으로 탐색하는 능력

Long-Context Reasoning

긴 문맥을 유지하며 다단계 추론을 수행하는 능력

Multimodal Understanding

이미지, 스크린샷 등 시각 정보를 이해하고 활용하는 능력

Cross-Platform

브라우저, 터미널, 파일시스템 등 여러 플랫폼을 조율하는 능력

각 태스크는 단일 능력에 집중하여
실패 원인을 정확히 진단

YAML 기반 태스크 정의

기본 정의, 수명주기, 태스크 프롬프트, 주입(Injection), 평가 규칙을 구조화된 형식으로 관리

실제 환경 실행

Docker 컨테이너 내 실제 브라우저, 터미널, 로컬 파일시스템에서 라이브 실행

3역할 폐루프 평가 전략

정보 오염 없이 자연스러운 다중 턴 인터랙션을 시뮬레이션하는 정보 방화벽(Information Firewall) 설계

1
Executor Agent
실제 환경에서 태스크 수행. 트레이스와 산출물 생성
2
Supervisor Agent
숨겨진 평가 기준으로 트레이스·산출물 검증. 정답 접근 가능
3
User Simulator
트레이직 + coarse 진행 신호만 수신. 자연스러운 피드백 제공. 정답 접근 불가

핵심: User Simulator는 원시 진행 신호(예: "진행 중", "완료")만 받고, 상세 평가 기준이나 정답은 절대 볼 수 없음 → 정보 오염 방지

평가 시스템 상세 동작

1

태스크 초기화

YAML 파일에서 태스크 정의 로드 → Docker 환경 준비 → 조건부 주입(Conditional Injection) 적용

2

실행 사이클 (Executor)

Executor가 환경에서 액션 수행 → 트레이스 기록 → 산출물(파일, 스크린샷 등) 생성

3

감독 평가 (Supervisor)

숨겨진 eval_rule.md 기준으로 단계별 체크포인트 검증 → 상세 판정 + coarse 진행 신호 출력

4

사용자 피드백 (User Simulator)

coarse 신호 + 표면적 트레이직만 참조 → 자연스러운 사용자 피드백 생성 → Executor에 전달

사이클 반복 또는 종료

최대 사이클 수에 도달하거나 완료 판정 시 종료 → 최종 점수 산출

실험 1: 교차 모델 평가

OpenClaw 프레임워크 고정 상태에서 다양한 SOTA 모델의 기본 능력 비교

모델 Skill Explor. Long-CTX Multi. Cross-P. 전체
GPT-4o 높음 중간 높음 높음 중간 최상위권
Claude 높음 높음 높음 중간 높음 최상위권
Gemini 중간 중간 중간 높음 중간 중상위권
오픈소스 모델군 중간 낮음 중간 낮음 낮음 중위권

관찰: 모델 간 능력 프로필이 뚜렷하게 상이함 — Multimodal에서 강한 모델이 Long-Context에서 약할 수 있음

일부 차원에서 강점

프레임워크 간 성능 격차가 모델 간 격차보다 더 크게 관찰됨 — 프레임워크 설계가 실제 환경 성능에 결정적 영향

핵심 발견: 프레임워크 > 모델

토큰 사용량 vs 성능 진행

더 많은 토큰을 소비한다고 항상 더 나은 성능을 보장하지 않음. 프레임워크의 액션 설계와 메모리 관리가 효율성을 좌우함.

평가 신뢰성 연구

체크포인트 기반 평가의 반복 측정 결과 높은 일관성 확인 — 동적 환경에서도 안정적인 평가 가능

프레임워크 설계가 실제 환경 성능의 더 큰 변동 요인

기본 모델 능력은 프레임워크 내에서 상한을 설정

두 요인의 상호작용이 최종 성능을 결정

기여 요약 및 한계점

주요 기여

  • 최초의 능력 주도형 실제 환경 에이전트 벤치마크
  • 5가지 능력 차원 기반 400개 이중어 태스크 설계
  • 정보 방화벽 기반 3역할 폐루프 평가 전략
  • 모델-프레임워크 교차 분석으로 설계 인사이트 제공

한계점

  • 5가지 능력으로 전체 에이전트 능력을 완전히 커버하지 못할 수 있음
  • User Simulator가 완벽한 인간 피드백을 재현하지 못할 가능성
  • 실제 웹사이트 변경으로 인한 태스크 유지보수 비용
  • 평가 속도 — 라이브 환경 실행으로 인한 시간 비용

"에이전트 성능을 개선하려면 기본 모델 능력 향상과 프레임워크 설계 최적화를 동시에 추구해야 한다."
