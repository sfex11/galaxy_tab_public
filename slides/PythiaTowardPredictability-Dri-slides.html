Exploiting Workflow Predictability for Efficient Agent-Native LLM Serving

Shan Yu, Junyi Shu et al. — UCLA, Alibaba Cloud, Intel, SJTU, UC Berkeley, Rice, Tsinghua, PKU

배경: LLM 서빙의 근본적 가정

기존 LLM 서빙의 통념: "모든 것이 예측 불가능하다"

불확실성의 근원

  • 언제 어떤 모델이 활성화될지 불명확
  • 각 모델에 대한 요청 유형·량 예측 불가
  • API 엔드포인트만 노출, 실행 제어 불가

반응적 설계의 결과

  • KV 캐시 크기·공유 모드 동적 조정
  • 오토스케일링 런타임 적응
  • FCFS 대기열 기반 요청 스케줄링
완전히 반응적인 서빙 시스템은 사전 최적화 기회를 영구적으로 소외시킨다.

멀티 에이전트 워크플로우의 부상

LLM 상호작용 패러다임이 고립된 채팅에서 자율적 목표 주도 에이전트로 전환

구조화된 워크플로우

고정된 에이전트 유형과 역할이 행동을 제약

전문화된 워커

엔지니어, 리서처 등 특정 모델+도구 보유

오케스트레이터

강력한 LLM이 복잡한 작업을 하위 작업으로 분해

산업 배포에서는 GPU 클러스터가 소수의 특정 에이전트 애플리케이션에 전용으로 할당되며,
동일 에이전트의 요청이 유사한 사용 패턴을 따르므로 리소스 사용에 상당한 예측 가능성이 도입된다.

프로덕션 트레이스 분석: 3가지 핵심 병목

에이전트 서빙 플랫폼 및 내부 코딩 어시스턴트의 대규모 트레이스 심층 프로파일링 결과

1

저조한 캐시 적중률

전문 에이전트마다 고유 프롬프트 사용, 긴 시간 간격으로 호출 → 캐시된 접두사 퇴거 → 잦은 제로 히트와 긴 프리필 지연

2

심각한 리소스 비효율

워크플로우 무관 스케줄링 → 이종 요청 임의 혼합 → 레플리카 간 불균형, 메모리 경합, 빈번한 선점 및 재계산

3

구조적 버스트 폭주

1분 내 50% 이상 스파이크 → 반응적 오토스케일링으로 대응 불가 → 워크플로우 의존성 무시로 연쇄적 대기열 병목

기존 블랙박스 접근법은 에이전트 워크로드를 범용 트래픽으로 취급하여 심각한 비효율을 초래한다.

핵심 통찰: 워크플로우는 예측 가능하다

범용 LLM 서빙과 달리, 멀티 에이전트 워크플로우는 고정된 에이전트 유형과 특정 역할이 모델 호출 방식을 엄격하게 제약하여 런타임 불확실성을 감소시킨다.

리소스 예측 가능성의 두 가지 근원

예측 가능한 워크플로우

  • 정적 워크플로우: 그래프 구조 완전 결정
  • 동적 워크플로우: 반복적 패턴 따름
  • 에이전트 간 데이터 흐름으로 요청 전파 경로 예측
  • 특정 워커의 캐시 혜택 최소화 여부 판별 가능

예측 가능한 워크로드

  • 각 워커 에이전트는 복잡도가 제한된 특정 작업 담당
  • 출력 길이가 비교적 단순한 분포 따름
  • 드라이 런 또는 온라인 프로파일링으로 정확 추정 가능
  • 사전 캐시 웜업 및 선제적 오토스케일링 가능

Pythia: 에이전트 네이티브 서빙 시스템

워크플로우 예측 가능성을 체계적으로 활용하여, 기존에는 불가능했던 최적화를 실현

Pythia는 요청의 잔여 실행 시간을 근사화 — 워크플로우 그래프 내 현재 위치와 에이전트 역할의 예상 출력 크기로부터 추론. 이 구조적 통찰을 우선순위 메커니즘으로 변환하여 예상 완료 시간 기반 스케줄링.

선견지명

Clairvoyance: 워크플로우 정보로 미래 리소스 수요 예측

글로벌 조정

단일 요청 최적화가 아닌 워크플로우 전체 관점에서 조율

선제적 실행

반응적이 아닌 사전 예방적 리소스 할당

도전 1: 예측 정보의 획득

문제: 에이전트 개발자가 전문 프로그래머가 아닐 수 있으며, 동적 그래프를 정적으로 기술하기 어렵다.

해결책: 예제 기반 합성 (Example-Driven Synthesis)

1
관찰 1: 대부분의 에이전트 앱은 단순한 구조를 가지며, 대표 입력 몇 번 실행으로 충분한 프로파일링 데이터 생성
2
관찰 2: 동적 그래프는 반복 패턴(예: Engineer-Executor 연속 상호작용)으로 구성 → 정규 표현식으로 완전 표현 가능
3
최소 변경: 에이전트 프레임워크가 OpenAI 호환 엔드포인트의 extra_body 페이로드로 앱·에이전트 유형 전달 (사용자 코드 변경 불필요)

게이트웨이 프로파일러가 워크플로우 그래프를 캡처하는 정규 표현식과 에이전트별 출력 길이를 도출하여 페이로드에 주석으로 주입

도전 2: 예측 정보의 활용

문제: 나이브하게 선제적 모델로 전환 시 리소스 쓰래싱 및 조기 캐시 퇴거 위험

Pythia는 트레이스 분석에서 식별된 3가지 기회에 정렬된 조화된 세 가지 메커니즘을 도입:

추측적 캐시 관리

노드 수준 | Belady 영감 정책으로 공유 컨텍스트 프리페치, 일시적 토큰 퇴거, 비동기 캐시 웜업

병목 1 해결

전망 요청 스케줄링

클러스터 수준 | 예측 출력 길이와 워크플로우 구조로 이종 워크로드 균형화, HoL 블로킹 감소

병목 2 해결

위상 적응형 오토스케일링

클러스터 수준 | 팬아웃·위상 전환 같은 구조적 워크로드 변화를 예측하여 선제적 스케일업/다운

병목 3 해결

추측적 캐시 관리 (Speculative Cache)

노드 수준 최적화 — 반응적 LRU를 Belady 영감 정책으로 강화

기존 LRU의 한계

  • 시간적 간격이 긴 에이전트 호출 시 캐시 퇴거
  • 워크플로우 컨텍스트 무시
  • 재방문 패턴 예측 불가

Pythia의 접근

  • 워크플로우 인사이트로 공유 컨텍스트 프리페치
  • 일시적 토큰 사전 퇴거로 메모리 확보
  • 비동기 캐시 웜업으로 프리필 지연 마스킹

핵심 아이디어: Belady의 최적 알고리즘에서 영감을 받아, 워크플로우 그래프 정보를 활용해 어떤 캐시 항목이 가장 먼 미래에 재사용될지를 예측하여 퇴거 결정

전망 요청 스케줄링 (Lookahead Scheduler)

클러스터 수준 최적화 — 예측 출력 길이와 워크플로우 구조 기반 스케줄링

1
출력 길이 예측
에이전트 역할별 프로파일링된 분포 활용
2
임계 경로 추정
워크플로우 그래프 상 잔여 실행 시간 근사
3
우선순위 스케줄링
예상 완료 시간 기반 요청 순서 결정

해결하는 문제

  • 이종 요청의 임의 혼합으로 인한 불균형
  • 메모리 경합 및 빈번한 선점
  • FCFS의 Head-of-Line 블로킹

달성 효과

  • 워크로드 균형화로 파이프라인 포화
  • 메모리 경합 완화
  • 종단간 작업 완료 시간 단축

위상 적응형 오토스케일링 (Phase-Adaptive)

클러스터 수준 최적화 — 구조적 워크로드 변화를 예측하여 선제적 스케일링

기존 반응적 오토스케일링의 문제

평활화된 메트릭에 의존 → 워크플로우 팬아웃 시 50%+ 스파이크 감지 지연 → 스케일업 레이턴시 동안 대기열 폭주 → 연쇄적 병목 발생

Pythia의 위상 적응형 접근

  • 워크플로우 정규 표현식에서 팬아웃 패턴 사전 식별
  • 위상 전환(예: 계획 → 실행) 시점을 워크플로우 진행 상태로부터 예측
  • 부하 급증 전에 모델 사전 스케일업
  • 위상 완료 후 불필요한 리소스 신속 스케일다운
반응적 "부하 올라가면 늘리기"가 아닌, "부하 올라갈 것이므로 미리 준비"하는 패러다임 전환

논의 및 의의

패러다임 전환

LLM 서빙의 기본 전제를 "불확실성"에서 "구조화된 예측 가능성"으로 전환. 에이전트 네이티브 서빙이라는 새로운 시스템 범주 제안.

최소 인터페이스

사용자 애플리케이션 코드 변경 없이, 기존 OpenAI 호환 엔드포인트의 페이로드만으로 워크플로우 시맨틱을 서빙 계층에 전달.

조화된 설계

세 메커니즘이 개별적으로도 효과적이지만, 결합 시 상승 효과를 달성. 캐시·스케줄링·스케일링이 동일한 예측 정보를 공유하여 일관된 의사결정.

산업적 검증

프로덕션 트레이스 기반 문제 정의부터 대규모 클러스터 파일럿 배포까지, 실제 환경에서의 적용 가능성 입증.

"에이전트 서빙을 범용 트래픽이 아닌 의미 있는 구조로 바라보는 것이 효율성의 핵심이다."
