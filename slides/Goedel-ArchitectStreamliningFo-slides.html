왜 새로운 접근이 필요한가?

기존 파이프라인의 한계

  • 재귀적 보조정리 분해는 막다른 전략에서 비효율적으로 루프
  • 상향식 분해 트리 → 전역적 맥락 공유 불가
  • 폐쇄형 모델 의존 → 재현 불가
  • 문제당 수백 달러의 API 비용

Goedel-Architect의 차별점

  • 전역 의존성 그래프 블루프린트로 전략을 한번에 수정
  • 병렬 증명 시도 간 맥락 공유
  • 백본과 파이프라인 모두 오픈소스
  • 문제당 ~$0.44 (최대 500× 저렴)

핵심 개념: 블루프린트

블루프린트는 메인 정리로 향하는 정의와 보조정리의 의존성 그래프입니다.

구조

  • 각 노드 = 형식적으로 명시된 정의 또는 보조정리
  • 각 간선 = 선언된 의존성 (증명 시 참조 가능)
  • 메인 정리가 유일한 싱크
  • 비순환, 모든 노드가 타겟에서 도달 가능

핵심 특성

  • 단일 Lean 파일로 출력
  • LeanArchitect 패키지가 그래프 구조 검증
  • 보조정리 본문은 비워둔 채 서명만 선언
  • 재귀 분해 트리와 달리 전역적 뷰 제공

"재귀적 분해 트리 대신 전역 의존성 그래프를 사용하면, 병렬 증명이 맥락을 공유하고 정제가 전체 전략에 한 번에 작용할 수 있다."

파이프라인 전체 구조

블루프린트 생성
형식 명제 + (선택) 자연어 증명 → 초기 의존성 그래프 G₀
2
정리 증명
각 보조정리를 병렬로 Lean 증명기에 디스패치, 의존성 제한
3
블루프린트 정제
실패 노드 진단 → 수정된 그래프 Gₖ₊₁ 생성 → 반복

블루프린트 → Lean 증명

루프

미해결 노드 존재 시

블루프린트 생성 단계

기본 과정

1
형식 명제를 입력으로 수신
2
정의와 보조정리 노드 생성, 의존성 선언
3

비순환 그래프 G₀ 출력 (모든 노드 미해결 상태)

자연어 증명 가이던스 선택적

더 어려운 문제에 대해 강한 모델이나 공식 해답의 자연어 증명을 구조적 가이드로 활용

  • 자연어 증명은 Lean 인식 불필요 — 어떤 비형식적 증명기도 사용 가능
  • 명명된 보조정리 그래프로 변환
  • Draft-Sketch-Prove의 flat 스케치가 아닌 그래프 구조로 정제 가능

정리 증명 단계

각 보조정리는 독립적으로 증명기에 디스패치되며, 병렬로 실행됩니다.

증명기의 접근 방식

  • 대상 보조정리와 선언된 의존성만 조회
  • 그래프의 나머지 부분은 보이지 않음
  • Lean 컴파일러 + Mathlib 검색 도구 사용
  • 목표 달성 또는 예산 소진까지 반복

노드 상태 분류

  • 해결됨 (Green) — 증명 성공
  • 미해결 (Blue) — 증명 실패
  • 부정됨 (Red) — 컴파일러 확인 반례 존재

증명기가 포기할 때는 구조화된 진단을 반환: 시도 내용과 간극 위치 기록. 반례를 발견하면 원문 대신 부정 명제의 증명을 등록.

블루프린트 정제 루프

명제가 가정 하에서 거짓

  • 부정된 노드(Red): 컴파일러가 반례 확인
  • 미해결 노드(Blue): 증명기가 거짓으로 판단했지만 반증 불가
  • 대응: 형식화 수정 (가정 추가·서명 변경)

proof_too_hard

증명 가능하지만 부모 노드만으로 체인 불가

  • 미해결 노드(Blue)에서 발생
  • 증명기가 증명 가능성은 확신하지만 간극 존재
  • 대응: 보조정리를 더 작은 헬퍼로 분해

이미 해결된 노드는 녹색으로 유지되어 증명이 재사용됩니다. 상위 편집은 의존 노드를 파란색으로 되돌릴 수 있습니다.
