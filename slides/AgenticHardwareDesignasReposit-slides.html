Cunxi Yu, Chenhui Deng, Nathaniel Pinckney, Brucek Khailany

왜 RTL 생성은 단일 턴으로 부족한가

RTL의 본질적 난이도

  • 사이클 수준 동작 정확성 요구
  • 리셋 및 인터페이스 규약 준수
  • 비트 폭 추론 필수
  • FSM 전이, ready-valid 프로토콜
  • 자연어 명세에 종종 누락된 코너 케이스

단일 턴 생성의 한계

  • 구문적으로 유효한 Verilog ≠ 기능적 정확성
  • 컴파일·시뮬레이션 피드백 없이는 수리 불가
  • 파형/트레이스 검사 필요
  • 검증 아티팩트(테스트벤치, 체커) 동시 생성 요구
  • 반복적 수정-평가 루프가 필수

"유용한 에이전트는 후보 아티팩트를 실행 가능한 워크스페이스에 배치하고, 도메인 도구를 호출하며, 실패를 해석하고, 명시적 수용 조건이 충족될 때까지 수정해야 한다."

HORIZON 프레임워크 개요

핵심 아이디어

하드웨어 설계 문제를 소프트웨어 진화 문제와 동일하게 관리한다. 설계, 컨텍스트, 하니스, 평가기가 모두 격리된 git worktree 안에 존재하며, 진행 상황은 연결되지 않은 채팅 턴이 아닌 저장소 상태 변화로 표현된다.

사용자가 구조화된 Markdown 하니스 제공
부트스트랩 에이전트가 프로젝트 팩으로 컴파일
핸즈프리 에이전트 루프가 자율 진화

프로젝트 팩 구성

  • 도메인 지식 — 태스크 관련 전문 지식
  • 실행 가능한 평가기 — 컴파일/시뮬레이션 파이프라인
  • 수용 조건 — 명시적 정확성 게이트
  • Git/런타임 정책 — 커밋, 분기, 실행 제약

하니스에서 실행 가능한 태스크로

구조화된 Markdown 하니스가 어떻게 자체 포함된 git worktree로 변환되는지 보여주는 파이프라인이다.

실행 가능 태스크
자율 에이전트 루프가 즉시 시작 가능한 상태

"같은 기계로 RTL 생성, 완성, 수리를 모두 커버하며, RTL 벤치마크 문제를 소프트웨어 엔지니어링 대리물로 재구성할 필요 없이 있는 그대로 처리할 수 있다."

주의

"에이전트 AI로 하드웨어 설계가 해결되었다고 주장하지 않는다. 이 벤치마크들은 칩 설계라는 훨씬 더 광범위한 공학 문제에 대한 통제된 프록시일 뿐이다."

토큰 소비 분석

토큰 소비 트렌드

패스률이 진화함에 따라 토큰 소비 패턴이 어떻게 변화하는지 분석했다.

  • 초기 시도: 높은 토큰 소비, 낮은 패스률
  • 중기 반복: 피드백 기반 수리로 효율 향상
  • 후기 수렴: 감소하는 시도 횟수로 패스 달성

연구 병목 이동

실행 피드백이 정확성을 수렴시킨 후, 주요 연구 병목이 이동한다.

이전: 첫 시도 정확성
이후: 수렴 효율성
이후: 검증 품질

테스트 생성 태스크 심층 논의

검증 지향 태스크의 특수성

CVDP 연구에 따르면, 최신 단발성 모델은 특히 다음 태스크에서 어려움을 겪는다:

  • 테스트벤치 스튜뮬러스 생성
  • 체커 생성
  • 어서션 생성
  • 버그 수정

HORIZON의 접근

에이전트 루프가 생성된 테스트벤치/체커/어서션을 실제 RTL과 함께 시뮬레이션하여 커버리지와 정확성을 검증한다. 단발성 생성으로는 달성 불가능한 실행 기반 피드백 루프가 검증 품질을 견인한다.

테스트 생성 커버리지 분석 결과는 트레이스 버퍼를 통해 상세히 기록되어 재생 및 분석 가능

논의와 한계

명시적 한계

  • 벤치마크는 통제된 프록시일 뿐, 실제 칩 설계를 대변하지 않음
  • 아키텍처 탐색, 마이크로아키텍처, 물리 설계는 미포함
  • 수렴 효율성이 여전히 연구 병목
  • 검증 품질의 자동 평가 난제
  • 명세-하니스 불일치로 인한 잔류 실패 가능

오픈 연구 과제

  • 에이전트 AI를 위한 아키텍처 탐색 통합
  • 마이크로아키텍처 설계 자동화
  • 검증 계획 자동 생성
  • 물리 설계와의 상호작용
  • EDA 소프트웨어 + 방법론 개발 통합

"저장소 네이티브 에이전트는 단일 도메인 트릭이 아니라 등장하는 패러다임이다. Git 의미론 — 분기, 계통, 체크포인팅 — 은 에이전트 탐색을 조직화하는 자연스러운 적합체다."
