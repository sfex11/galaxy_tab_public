잘못된 상태 전이

오래된 프롬프트에 반응한 이벤트 핸들러가 워크플로우를 잘못된 상태로 이동

정책 게이트 부재

제안과 쓰기 사이에 정책 검증이 없어 90% 할인이 고객에게 전달

진실의 원본 혼란

이벤트 로그와 버전된 행 중 어느 것이 진실의 원본인지 결정하지 않아 장기 프로세스 상태 상실

이 중 어느 것도 모델 결함이 아니다. 모두 LLM이 추론을 실행하기 이전에 팀이 내린 아키텍처 선택이다.

핵심 제안

확률적-결정론적 경계(SDB)

프로덕션 LLM 에이전트는 확률적 모델 출력을 결정론적 소프트웨어 시스템과 결합한다. 이 두 영역의 경계는 모든 프로덕션 에이전트의 구조적 하중을 받는 엔지니어링 표면이지만, 아직 이름이 없다.

19/21

5개 오픈소스 프레임워크에서 LLM→액션 호출 사이트 중 명시적 검증-커밋 로직을 가진 비율

71%

21개 에이전트 실패 포스트모템 중 경계 취약점에 국한되는 비율 (15건)

81%의 문서화된 수정 사항이 SDB의 네 가지 부분 중 하나를 강화함

SDB의 4파트 계약

LLM 출력이 시스템 액션이 되는 순간

1
Proposer
제안자: LLM이 생성한 확률적 출력
2
Verifier
검증자: 제안에 대한 결정론적 검사
3
Commit
커밋: 수락 후 내구성 있는 쓰기
4
Reject
거부: 검증 실패 시 제안자에게 반환되는 타입화된 응답

이 원시적(primitive)은 구현에서 새롭지 않다. 이름과 계약으로서 새롭다. 이름을 붙이면 실무자가 실패를 통해 재발견하지 않고 명시적으로 설계할 수 있다.

런타임 분류

세 가지 런타임 클래스

대화형

Conversational

  • 작업 단위: 세션
  • 지속 시간: 초 단위
  • 컨텍스트 윈도우가 세계를 담음
  • 지연(latency)이 설계를 지배
자율형

Autonomous

  • 작업 단위: 태스크
  • 지속 시간: 분 단위
  • 웹훅/스케줄/큐로 트리거
  • 큐가 실행 간 상태를 보관
장기형

Long-Horizon

  • 작업 단위: 프로세스
  • 지속 시간: 시간~일 단위
  • 다중 에이전트 참여, 일시정지/재개
  • 실행 중 세계가 변함

패턴 카탈로그

6개 런타임 패턴

P1 계층적 위임

Hierarchical Delegation · 계보: 액터 모델

P2 스캐터-개더 + 사가

Scatter-Gather plus Saga · 계보: 사가 패턴

P3 이벤트 기반 순서화

Event-Driven Sequencing · 계보: 로그(The Log)

P4 슈퍼바이저 + 게이트

Supervisor plus Gate · 계보: Erlang 감독

P5 공유 상태 머신

Shared State Machine · 계보: 워크플로우 넷 + CAS

P6 휴먼 인 더 루프

Human in the Loop · 계보: 분산 트랜잭션

각 패턴은 특정 분산 시스템 결과에서 계보를 추적하며, 워커가 확률적일 때 무엇이 전이되고 무엇이 전이되지 않는지 식별

방법론

5단계 패턴 선택 절차

1

런타임 분류

워크로드가 대화형, 자율형, 장기형 중 어디에 속하는가?

2

spine(척추) 선택

상태 관리 전략을 기반으로 핵심 패턴을 선택

3

조율 래핑

작업 분할/결합 방식을 spine 주위에 배치

4

제어 경계 설정

SDB의 4파트 계약 강도를 결정

5

구축 순서 지정

구현 순서와 검증 체크리스트 산출

출력물: 6줄 아키텍처 의사결정 기록(ADR) — 이벤트 시간 vs 처리 시간 의미론, 진실의 원본 질문, 가시성-제어 트레이드오프 등 기존 시스템 트레이드오프의 에이전트 영역 투영

신뢰성 모델

왜 모델 개선만으로는 부족한가

y(t) = μt + σξ(t)

σ — 호출별 분산

  • 확률적 제안자(Proposer)에서 발생
  • 모델 세대가 거듭될수록 압축
  • 모델 선택으로 제어 가능

μ — 아키텍처 모멘텀

  • 패턴 선택과 SDB 강도로 결정
  • 호출별 모델 품질과 구조적으로 독립
  • 모델 교체로는 변경 불가

σ가 축소될수록 μ가 장기 신뢰성의 지배적인 레버가 된다. SDB를 명시적으로 설계하는 것이 팀이 μ를 형성하는 방법이다.

프로덕션 진단

실패 서명 카탈로그와 관측 렌즈

프로덕션에서 관찰된 실패를 특정 패턴에 매핑하는 절차

패턴 대표적 실패 서명
P1 계층적 위임 하위 에이전트가 상위의 의도를 왜곡하여 상위가 잘못된 결정 기반으로 구축
P2 스캐터-개더 보상 사가가 불완전하여 부분 커밋이 정리되지 않음
P3 이벤트 순서화 이벤트 시간과 처리 시간의 불일치로 순서 위반
P4 슈퍼바이저 게이트 검증이 환각을 통과하여 잘못된 액션이 커밋됨
P5 공유 상태 머신 CAS 경합으로 인한 상태 전이 누락
P6 휴먼 인 루프 승인 지연으로 프로세스 타임아웃 또는 승인 피로

세 가지 관측 렌즈: 추적 렌즈(단일 요청 경로), 상태 렌즈(시간 경과에 따른 상태 스냅샷), 이벤트 렌즈(로그 재생 일관성)

새로운 실패 모드

리플레이 발산(Replay Divergence)

SDB가 명확하게 만들어 보이는 하나의 실패 모드: 결정론적 이벤트 로그의 LLM 기반 소비자가 모델 버전 변경 시 다른 다운스트림 출력을 생성하는 현상

전통적 시스템에서

동일한 로그를 재생하면 항상 동일한 결과. 이것이 이벤트 소싱의 핵심 전제다.

LLM 에이전트에서

동일한 로그 + 다른 모델 버전 = 다른 하위 상태 전이. 결정론적 로그 위에 확률적 소비자가 있기 때문.

리플레이 발산은 분산 시스템의 "정확히 한 번" 의미론이 확률적 워커와 만날 때 어떻게 붕괴하는지를 보여준다. SDB의 Verifier와 Commit 강도가 이 발산의 폭을 결정한다.

검증

5개 워크로드에 대한 종단간 적용

워크로드 런타임 선택된 spine
90일 계약 갱신 장기형 P5 공유 상태 머신
빌링/결제 어시스트 대화형 P1 계층적 위임
주문 관리 폴아웃 스캐너 자율형 P4 슈퍼바이저+게이트
번호 이동 포팅 조율 장기형 P3 이벤트 순서화
리드 워밍 장기형 P2 스캐터-개더+사가

동일한 장기형 런타임이라도 상태 술어(state predicate)가 다르게 발동하여 서로 다른 spine을 선택 — 포팅은 이벤트 시간 의미론이, 리드 워밍은 보상 의미론이 핵심
