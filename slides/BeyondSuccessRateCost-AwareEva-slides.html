성공률 그 너머: 공격 및 방어 보안 에이전트의 비용 인식 평가

Beyond Success Rate: Cost-Aware Evaluation of Offensive and Defensive Security Agents

Paul Kassianik, Blaine Nelson, Yaron Singer

arXiv:2607.15263v1 · 2026

기존 평가의 한계

현재 방식

  • 넉넉한 추론 예산 하에서 최대 공격 능력 측정
  • CTF 해결, 취약점 발견, 익스플로잇 개발에 집중
  • 백만 토큰 예산 + 공격자 지향 실행 권한 부여

무엇이 누락되었는가

  • 실제 운영 환경에서 모든 단계는 비용을 소비
  • 추론 단계, 도구 호출, 원격 분석, 보강 요청 등
  • SOC는 예산 내에서 분석가 수준의 답변이 필요

"운영 보안은 최대 능력 시연이 아니다. 달러당 얼마나 많은 능력을 살 수 있는가가 핵심이다."

핵심 아이디어: 비용-성공 트레이드오프

단일 리더보드 대신, 운영 포인트를 특성화한다

1

고정 예산을 공통 축으로 모델 비교

2

추론 비용과 도구 비용 분해

3

경제적 효율성과 운영 적합성 측정

성공률을 운영적 질문으로 변환: "추론 및 도구 비용 1달러당 얼마나 많은 능력을 구매하는가?"

평가 설계 개요

1
프레임워크
Inspect 평가 프레임워크, ReAct 스타일 에이전트, 자동 압축(90% 컨텍스트 임계값)
2
비용 제어
에이전트는 비용 인지 불가, 특정 고비용 도구 호출만 제한 인지
3
후향적 예산 캡
실행 후 토큰/비용 소비를 기준으로 성공 여부 재분류

제공자 구성

  • OpenAI / Anthropic: 직접 API
  • 기타 모델: OpenRouter (1주 + 2폴백)
  • 3제공자 평균 가격으로 비용 산정

도구 세트

  • 공격/방어 간 도구 세트만 변경
  • 에이전트 아키텍처는 동일 유지
  • 비용, 토큰, 도구 통계 자동 추적

공격 벤치마크: Cybench

구성

  • 39개 샌드박스 CTF 과제
  • 4개 사이버 보안 대회 출처 (HackTheBox, Glacier, SEKAI, HKCert)
  • 셸 및 파이썬 접근 권한 제공

도메인 범위

암호화 웹 익스플로잇 리버스 엔지니어링 포렌식 바이너리 공격

비용 구조

한계 비용의 대부분이 테스트 타임 추론과 명령어 실행에 집중

평가 방식

숨겨진 플래그 토큰(예: HTB{...}) 제출로 성공 판정. 후향적 토큰 예산 캡으로 비용-성능 곡선 생성

방어 벤치마크: BOTS v1

Splunk Boss of the SOC v1

  • 공개 Splunk 원격 분석 데이터
  • 공식 대회 문제 + 점수 가치 + 힌트 페널티
  • 이진 정답 정확도 기반 평가

SOC 작업 내용

  • 로그 스키마 발견 및 SPL 작성/수정
  • 대표 이벤트 검사 및 아티팩트 상관관계
  • 외부 보강(웹 검색, WHOIS 등) 필요 여부 판단

비용 구조 — 공격과 근본적으로 다름

모델 토큰 비용 (추론 소비)
Splunk 쿼리 비용 (분석가 자원)
이벤트 검사 비용 (원격 분석 탐색)
보강 호출 비용 (웹 검색, WHOIS 등)

→ BOTS 1,000점당 비용($/1K pts) 지표 도입

핵심 결과: 공격 스케일링

Cybench — 추가 테스트 타임 컴퓨팅이 성능을 지속적으로 향상

예산 증가 효과

  • Claude Opus 4.8, DeepSeek v4 Flash: 명확한 예산 여유(headroom) 확인
  • 더 높은 예산 캡에서 추가 해결 과제 발견
  • 스케일된 오픈 가중치 모델이 프론티어 독점 시스템에 근접하면서도 비용 경쟁력 유지

GPT-5.5의 특이한 패턴

  • 대부분의 성공 과제를 하위 예산 캡 이하에서 이미 해결
  • 예산 증가에 따른 한계 효과가 작음
  • 효율적 추론 경로를 먼저 찾는 경향

공격 성공률은 지출과 함께 안정적으로 상승 — 테스트 타임 컴퓨팅 스케일링이 유효

핵심 결과: 방어 스케일링

BOTS v1 — 추가 예산이 성능을 보장하지 않음

예산 증가의 부진

  • DeepSeek v4 Flash: 직접 예산 캡 증가에도 거의 향상 없음
  • 높은 도구 호출량이 오히려 낮은 점수와 동반
  • 강한 모델들은 ~20회 도구 호출 내에서 거의 최대 성능 도달

진정한 결정 요인

  • 규율 있는 도구 사용 (disciplined tool use)
  • 원격 분석 탐색 전략 (telemetry navigation)
  • 선택적 보강 (selective enrichment)
  • 원시 추론 예산 단독으로는 부족

방어 신호는 도구 볼륨이 아닌 규율을 추적한다

레드팀 vs 블루팀: 상이한 스케일링 체제

측면 공격 (Cybench) 방어 (BOTS v1)
예산-성능 관계 지속적 상승 빠르게 포화 또는 무관
한계 비용 주체 테스트 타임 추론 + 명령 실행 모델 토큰 + Splunk + 보강 호출
핵심 역량 추론 깊이, 시도 횟수 도구 규율, 탐색 전략, 선택적 보강
오픈 모델 경쟁력 프론티어 근접 가능 아직 큰 격차
추가 예산의 가치 높음 낮음/부정적

동일한 에이전트 아키텍처, 동일한 평가 프레임워크에서도 레드/블루 팀은 근본적으로 다른 스케일링 패턴을 보인다

BOTS v1 오염 통제

공개 오래된 벤치마크 → 절대 점수 해석 시 오염 검사 필수

오염 위험

  • BOTS v1은 2017년 공개 — 훈련 데이터 중복 가능성
  • 모델 행동에서 부분적 복원 가능
  • 겉보기 능력 과대평가 위험

통제 방법

  • 도구 없음(no-tools) 컨트롤 — 순수 지식 기반 답변 측정
  • 컨텍스트 없음(no-context) 프로브
  • 이들을 1차 능력 결과가 아닌 오염 통제로 취급

도구 없이도 높은 점수를 낸다면 → 실제 조사 기술이 아닌 사전 기억일 가능성

기여 요약 및 시사점

기여

  • Cybench + BOTS v1 공통 하네스 기반 비용-성공 비교
  • 샘플별 추가 지출이 점수를 변화시키는 예산 여유 분석
  • SOC 조사를 위한 모델 토큰 + 유료 도구 이중 비용 회계
  • BOTS v1 no-tools 컨트롤을 통한 오염 통제 방법론

시사점

  • 보안 에이전트 벤치마크는 경제적 효율성을 함께 측정해야 함
  • 방어 에이전트 개선 방향: 규율, 탐색, 보강 전략 — 예산이 아닌 기술
  • 공격 평가에서는 스케일드 오픈 모델이 실용적 대안이 될 수 있음

인터랙티브 결과 웹사이트: https://evals.frontier.security

결론

보안 에이전트 벤치마크는 과제 성공과 함께 경제적 효율성과 운영 적합성을 측정해야 한다.

공격: 컴퓨팅 스케일링이 유효, 오픈 모델의 경쟁력 입증

방어: 도구 규율이 핵심, 예산 단독으로는 불충분

평가: 비용 인식, SOC 네이티브 접근 필요

비용 인식, SOC 네이티브 평가는 어떤 모델이 오늘 실용적으로 유용한지, 그리고 방어 에이전트가 아직 어디를 개선해야 하는지 더 명확한 그림을 제공한다.
