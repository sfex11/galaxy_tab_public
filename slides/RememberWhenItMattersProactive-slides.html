Remember When It Matters

장기 수행 에이전트를 위한 사전적 메모리 에이전트

Yifan Wu, Lizhu Zhang, Yuhang Zhou, Mingyi Wang, Bo Peng, Serena Li, Xiangjun Fan, Zhuokai Zhao

Meta AI · July 2026

문제 정의: 행동적 상태 붕괴

"장기 실행 중, 미래 행동을 규제해야 할 정보가 더 이상 에이전트의 다음 결정에 영향을 미치지 못하는 현상"

전형적 실패 사례

  • 초기 요구사항 식별 후 나중에 위반
  • 실패한 명령어를 거의 동일하게 재시도
  • 이미 진단한 에러 패턴을 새로운 것으로 취급

핵심 관찰

정보가 컨텍스트 윈도우 내에 존재하더라도 행동에 영향을 주지 못함. 단순한 긴 컨텍스트 제공으로는 해결 불가.

기존 접근법의 한계

접근법 방식 한계
수동 메모리 (MemGPT, Mem0) 저장·검색 중심 언제 개입할지 결정 불가
컨텍스트 압축 (Context-Folding) 요약·분해 완료된 하위작업만 대상
학습된 메모리 (Mem-α, MemAct) 에이전트 자체 컨텍스트 관리 액션 에이전트 수정 필요
어드바이저 모델 광범위한 전략적 조언 메모리 기반이 아닌 일반적 지침

공통 결함: 메모리가 개입 시점을 결정하는 문제를 다루지 않음

핵심 아이디어: 메모리를 개입으로

요약기는 "무엇을 유지할지"를 묻지만, 우리의 메모리는 "유지된 상태가 다음 행동에 활성화되어야 하는지"를 결정

Phase 1: 메모리 관리

최근 트레이젝토리 세그먼트를 기반으로 메모리 뱅크를 갱신

트레이젝토리 분석
새로운 사실, 실패, 진단 추출
C
편집 연산 수행
ADD / UPDATE / DELETE 결정

설계 원칙

  • 메모리 뱅크는 컴팩트하게 유지 — 중복·만료 항목 삭제
  • 편집은 구조화된 출력으로 표현 — 파싱 가능한 형식
  • 트레이젝토리 세그먼트만 입력으로 사용 — 전체 이력 불필요

Phase 2: 개입 선택과 일시적 주입

갱신된 메모리 뱅크를 바탕으로, 다음 액션 에이전트 호출에 리마인더를 주입할지 결정

침묵 (Silent)

현재 메모리가 에이전트의 다음 행동에 영향을 줄 필요가 없다고 판단 시

→ 액션 에이전트는 평소대로 실행

리마인더 주입 (Intervene)

행동적으로 비활성화될 위험이 있는 메모리 항목을 간결한 문장으로 주입

→ 일시적(transient)으로만 추가, 영구 저장 아님

"불필요한 개입은 단순히 중복을 넘어 해로울 수 있다" — 선택적 개입이 핵심

트리거 메커니즘

메모리 에이전트는 고정 간격으로 실행되며, 모든 액션 단계마다 호출되지 않음

설계 근거

  • 비용 절감: 매 단계 호출 시 메모리 에이전트 비용이 액션 에이전트와 동등해짐
  • 지연 최소화: 주입 없이도 정상 진행 가능한 구간에서 불필요한 대기 방지
  • 충분한 정보: 세그먼트 단위로도 새로운 사실과 실패 패턴 포착 가능
  • 실용성: 실제 배포 시 비용-효과 균형을 위한 현실적 선택

개입 정책 학습 (Learning Intervention Policies)

SFT(지도 미세조정) + GRPO(그룹 상대적 정책 최적화)를 사용하여 메모리 에이전트가 언제 침묵하고 언제 개입할지 학습. 검증 보상 기준으로 개선 확인.

실험 설정

Terminal-Bench 2.0

  • 자율 명령줄 실행 벤치마크
  • 다단계 파일 조작, 디버깅, 시스템 관리
  • 긴 트레이젝토리에서 상태 유지 필수

τ²-Bench

  • 대화형 툴 사용 + 도메인 규칙
  • 항공·여행 도메인의 복잡한 정책 준수
  • 다중 턴 상호작용에서 규칙 기억 필수

에이전트 구성

메모리 에이전트: Claude Opus 4.6 (기본), Qwen3.5-27B (오픈 웨이트 실험)  |  액션 에이전트: Claude Sonnet 4.5 (약함), Claude Opus 4.6 (강함)

핵심 발견: 약한 에이전트에서 큰 폭의 개선, 강한 에이전트에서도 이득이 사라지지 않음 — 메모리 개입이 모델 능력과 상보적

제거 실험 및 비교

변형 설명 결과
제안 방법 메모리 관리 + 선택적 개입 최고 성능
수동 은행 노출 메모리 뱅크를 매 호출마다 프롬프트에 추가 열등
항상 주입 매 트리거마다 무조건 리마인더 주입 열등
어드바이저 전용 메모리 구조 없이 일반적 조언만 제공 열등
일반 검색 구조화되지 않은 R 기반 검색 열등

"선택적 개입"이 핵심 — 항상 주입하는 것이 침묵보다 못할 수 있으며, 구조화된 메모리 유지가 필수적

정성적 분석

Terminal-Bench 디버깅 연속성

에이전트가 에러를 진단한 후, 후속 단계에서 동일한 잘못된 경로를 재시도하는 패턴을 메모리 개입이 차단

  • 실패한 명령어와 원인을 메모리에 기록
  • 유사한 시도 감지 시 리마인더 주입
  • 디버깅 컨텍스트가 긴 트레이젝토리에서 유지

τ²-Bench 정책·상태 재활성화

대화가 길어질수록 도메인 규칙과 고객 요구사항이 행동적으로 비활성화되는 문제 해결

  • 초기 정책 제약을 메모리에 보존
  • 위반 위험 시 타이밍에 맞게 주입
  • 고객 상태(예약 정보 등) 재활성화

오픈 웨이트 메모리 에이전트

폐쇄형 모델(Clause Opus)에만 의존하지 않고, 오픈 웨이트 모델로 개입 정책 학습 가능성 탐색

학습 설정

  • 기반 모델: Qwen3.5-27B
  • 학습 데이터: SETA 벤치마크
  • SFT → GRPO 2단계 학습
  • 검증 보상 기준 평가

결과

  • SETA 검증 보상 지속적 개선
  • Terminal-Bench로 부분 전이(partial transfer) 달성
  • 초기 단계이나 오픈 웨이트 학습의 실행 가능성 입증

향후 방향: 더 다양한 도메인의 학습 데이터와 강화 학습 알고리즘을 통해 전이 성능 향상 기대

결론 및 기여

i

문제 식별

장기 에이전트의 핵심 실패 모드인 행동적 상태 붕괴를 명시적으로 정의

ii

아키텍처 제안

메모리 유지와 액션 선택을 분리한 2단계 개입 아키텍처

iii

학습 가능성

오픈 웨이트 모델로 개입 정책 학습 가능성을 초기 단계에서 입증

메모리는 단순한 저장·검색이 아닌 개입 정책으로 설계되어야 하며, 선택적 개입이 항상 주입보다 우월함. 플러그앤플레이 구조로 기존 에이전트 생태계와 호환.
