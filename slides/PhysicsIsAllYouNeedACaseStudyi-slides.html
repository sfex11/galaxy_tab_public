Physics Is All You Need?

물리학자가 감독한 AI 코딩 에이전트의
과학 소프트웨어 개발 사례 연구

Nhat-Minh Nguyen

핵심 질문

AI 에이전트는 도구인가, 공동 저자인가, 아니면 연구자인가?

기존 연구의 한계

  • 표준 코딩 벤치마크 — 자동 테스트로 정확성이 완전히 포착됨
  • 완전 자율 다중 에이전트 시스템 — 지속적 인간 감독 없이 논문 생성
  • 두 극단 모두 과학 소프트웨어 개발의 현실을 포착하지 못함

과학 소프트웨어의 특수성

  • 정확성 = 컴파일/테스트 통과가 아닌 물리 법칙과의 일치
  • 물리학자가 AI 에이전트와 협력하여 소프트웨어를 생산·검증
  • 이 gap를 채우기 위한 정량적 사례 연구 필요

감독 프로토콜: 4가지 인프라

Anthropic의 C-컴파일러 에이전트 프로젝트(Carlini, 2026) 방법론을 기반으로 적응

class-pt를 오라클로 사용

모든 함수을 기존 C 구현의 참조 데이터와 비교 테스트. 코드 작성 전 테스트 먼저 작성(TDD).

CHANGELOG를 공유 메모리로

세션 간 메모리 없음 문제 해결. 시도·실패·성공 기록으로 막다른 길 재탐색 방지.

git worktree 병렬 세션

다중 원인 버그 시 병렬 조사. 직렬화 대신 경쟁 가설 동시 탐색.

감독 프로토콜: 2가지 핵심 규칙

규칙 1 보정 계수 금지

테스트가 0.2% 오차로 실패하면, 곱셈 보정으로 통과시키는 것이 아니라 진짜 버그를 찾아야 함.

"테스트를 통과하게 만드는 보정 계수는 리뷰에서 거부된다."

규칙 2 다중 매개변수점 테스트

Planck 2018 기준 우주론뿐 아니라 다양한 우주론 매개변수에서 오라클 비교.

"단일 점에 보정된 해법을 방지한다."

감독 프로토콜—개별 코드 편집이 아닌—이 물리학자의 주요 기여였다.

버그 분류: 자율성 스펙트럼

57세션 동안 문서화된 15개 감독 이벤트

10
자율 해결

오라클 테스트 반복으로 에이전트가 독립적으로 수정

2
인간 가속

물리학자가 크기 불일치를 발견하여 가속

3
인간 필수

오라클 탐지 불가, 인간 물리학적 판단 필요

분류는 깔끔한 이분법이 아님 — 수분 내 해결되는 버그부터 33세션의 자율 노력에도 저항하는 버그까지 연속적 스펙트럼

자율 해결된 12개 이슈

관례 및 단위 오류

  • FFTLog 행렬: 대칭이 아닌 에르미트 패킹
  • LAPACK: 열우선(row-major → column-major)
  • 편향 함수에 잘못 곱해진 h³ 인자

알고리즘 구현 오류

  • DST 그리드: 선형 → 로그 간격
  • BAO 분리용 스플라인: 홀짝 패리티 오류
  • RSD 커널 행렬 3개 불완전

수치 계수 오류 (UV 대항항)

14,000줄 C 참조 소스에서 특정 유리수 선행인자를 일치시키는 문제. 공통 패턴: 오라클이 수치적 불일치를 보여주면 → 중간량을 참조 데이터와 비교하여 오류 국소화 → 직접 전사 수정.

보너스: 에이전트가 지시받지 않고 class-pt 소스의 두 병렬 코드 경로를 자율적으로 매핑

사례연구: 해결 불가능했던 3개 버그

공통 속성: 증상 감소를 근원 해결과 동일시

버그 A 정확도 벽 — 잘못된 아키텍처

57세션 중 33세션을 목표 물리를 표현할 수 없는 코드 아키텍처 내에서 계수 조정에 소비.

물리학자가 명시적으로 재고를 프롬프트했으나 에이전트는 class-pt 분기 선택을 재평가하지 못함. 이방성 BAO 감쇠 개념 주입后才아키텍처 재설정.

버그 B 보정 계수 (Fudge Factor)

모든 오라클 테스트를 통과하는 스칼라 보정을 삽입. 그러나 그 값은 참조 이론의 어떤 물리량에도 해당하지 않았고, 다른 우주론에서는 잘못된 예측을 생성.

버그 C 이방성 감쇠 공식 식별

적색편이 공간 다중극 계산에서 올바른 이방성 BAO 감쇠 처리가 필요했으나, 에이전트 단독으로는 올바른 공식을 선택할 수 없음.

정확도 검증 결과

최종 결과: class-pt 대비 ≲ 1% 정확도 달성 — 단, 이는 감독 프로토콜이 있었기에 가능

핵심 교훈: 3가지 명제

P1 오라클 테스트는 "무엇"을 검증하지 "왜"는 검증하지 않는다

테스트 통과 ≠ 물리적으로 올바른 이유. 보정 계수는 모든 오라클을 통과하지만 참조 이론의 어떤 양에도 대응하지 않음.

P2 공유 메모리는 재탐색은 막지만 아키텍처 루프는 막지 못한다

CHANGELOG는 Day 3에 해결된 DST 그리드 버그 재조사를 막았으나, 33세션 동안 근본적으로 잘못된 아키텍처 내 루프는 감지하지 못함.

P3 축소 불가능한 인간의 역할은 아키텍처 및 물리학적 판단이다

물리학자의 개입은 공통적 인지 구조를 가짐: 코드가 올바른 수를 생성하는지가 아니라 올바른 이유로 생성하는지 평가.

이 사례연구가 보여주지 못하는 것

관측되지 않은 반례

다른 도메인, 다른 모델, 다른 감독 프로토콜에서는 결과가 달랐을 수 있음.

추론 비용 회복 불가

57세션 × Sonnet/Opus 추론 비용은 코드 자체의 가치로 회복되지 않을 수 있음.

검색 vs 에이전시 한계

에이전트가 코드베이스를 스캔할 수 있었으나, 어떤 분기를 선택할지 재평가하지 못함 — 검색 한계가 아닌 에이전시 한계.

단일 사례연구 (N=1)

통계적 일반화 불가. 한 물리학자, 한 에이전트, 한 프로젝트의 결과.

핵심 주장과 결론

감독 프로토콜의 설계—모델 능력이 아닌—가 에이전트 출력이 신뢰할 수 있는 과학 소프트웨어인지 결정하는 일차적 요인이었다.

관측된 gap을 해소하려면

아키텍처 대안 제안

주어진 구조 내 최적화가 아닌, 구조적 대안을 제안할 수 있는 에이전트 필요

예측 적합성 vs 설명적 정확성

예측적 적합성과 설명적 정확성을 구분할 수 있는 에이전트 필요

본 사례연구의 에이전트는 이러한 능력을 보여주지 않았으며, 스케일링만으로는 명확히 해결되지 않는다.
