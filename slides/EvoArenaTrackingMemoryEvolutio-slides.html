기존 메모리 시스템의 한계

대부분의 메모리 기반 에이전트는 메모리를 단일 최신 상태로 유지한다.

  • 새 정보가 이전 정보를 단순히 덮어씀
  • 이전 행동과 그 유효성 맥락이 소실
  • 롤백이나 다른 버전 필요 시 대응 불가
  • 왜 변경되었는지에 대한 추적 불가

예시: 워크플로우 권한 변경

v1: deploy.sh 실행 가능

v2: 권한 변경으로 deploy.sh 차단 → 메모리 덮어씀

v3: v1 권한 복원 필요 → 하지만 v1 정보 이미 소실!

에이전트는 이전 행동과 그것이 유효했던 맥락을 모두 잃게 됨

Terminal-Bench-Evo

체인 스코프 내 메모리 관리로 오버헤드 최소화

  • 체인 범위로 패치 히스토리 제한
  • 저장 시점에 업데이트 수행
  • 검색 시점에 조건부 패치 복원

PersonaMem-Evo

패치 인덱싱으로 선택적 검색 효율화

  • 메모리 수집 시 패치 자동 생성
  • 패치별 인덱싱 및 저장
  • 쿼리 시 관련 패치만 선택적 검색
EvoMem은 기존 메모리 시스템에 경량 플러그인으로 통합되므로, 추가적인 대규모 리소스 없이도 진화 추적 기능을 제공한다.

핵심 시사점과 기여

평가의 패러다임 전환

정적 스냅샷에서 지속적 환경 진화 체인으로 평가 방식을 확장

메모리 = 진화 히스토리

메모리를 단일 최신 상태가 아닌, 근거 기반 업데이트 이력으로 관리해야 함

범용적 적용 가능성

EvoMem은 Terminus2, OpenHands, Memento-Skill, A-Mem 등 다양한 에이전트에 통합 가능

최신 상태와 이전 상태의 연결

신뢰 가능한 에이전트는 최신 상태를 이전 상태, 사유, 증거와 연결하여 보존해야 함

한계점과 향후 방향

현재 한계

  • 세 도메인(터미널, 소프트웨어, 사회)으로 제한
  • 패치 생성 품질이 백본 LLM에 의존
  • 장기 진화 체인에서의 패치 누적 비용
  • 다중 에이전트 협업 환경 미포함

향후 연구 방향

  • 더 많은 도메인으로 진화 체인 확장
  • 자동 패치 압축 및 요약 기법
  • 멀티모달 환경 진화 벤치마크
  • 진화 인식 메모리 아키텍처 설계
