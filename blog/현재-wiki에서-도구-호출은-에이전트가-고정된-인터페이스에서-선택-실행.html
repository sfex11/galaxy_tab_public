코딩 에이전트가 라이브러리를 임포트하는 순간, build-vs-buy 결정이 내려진다. 그런데 기존 프레임워크는 이를 단순한 "선택-실행-결과 수신"으로 환원한다. 무엇이 누락되고 있는가.

레스토랑 주문의 함정: 상태less 전제가 얼마나 위험한가

우리가 레스토랑에 가서 메뉴를 고르고 주문하면, 웨이터는 주문을 전달하고 음식이 나온다. 이 과정은 이전 손님의 주문과 무관하게 독립적으로 동작한다. 다음 손님이 같은 메뉴를 시켜도, 레스토랑의 구조 자체는 바뀌지 않는다. 현재 도구 호출 프레임워크가 에이전트의 동작을 모델링하는 방식이 바로 이것이다 — 모든 호출은 상태less한 독립 연산이라는 전제.

하지만 코딩 에이전트의 도구 호출은 레스토랑 주문이 아니라 집의 구조를 바꾸는 시공에 가깝다. 벽을 하나 뜯으면 그 공간의 용도가 영원히 바뀌고, 이후의 모든 설계 결정은 "벽이 없는 공간"이라는 사실 위에서 내려져야 한다. import torch 한 줄이 정확히 이런 일을 한다. 의존성 그래프에 노드가 추가되고, 네임스페이스가 오염되며, 후속 코드 생성의 가능성 공간이 축소된다.

라이브러리 임포트는 단일 호출로 보이지만, 실제로는 (1) 의존성 그래프에 새 노드 추가, (2) 네임스페이스 오염, (3) 후속 코드 생성의 가능성 공간 축소라는 세 가지 누적 효과를 동시에 수행한다. 기존 프레임워크는 이 중 반환값("임포트 성공")만 포착하고, 세 가지 부작용은 투명하게 통과시킨다.

문제의 핵심은 이 부작용들이 프레임워크 층에서 완전히 증발한다는 점이다. 에이전트는 임포트의 결과로 "성공"이라는 문자열만 받고, 아키텍처가 어떻게 변했는지에 대한 정보는 얻지 못한다. 이는 마치 시공 후 "작업 완료"라는 스티커만 받고, 설계도면은 업데이트되지 않는 것과 같다.

스키마 축적: 파이프라인이 조용히 무거워지는 메커니즘

도구 호출이 상태less라는 전제는 이미 실증적으로 반증되었다. "Tool Attention" 논문이 다중 서버 MCP 배포 환경에서 확인한 사실은 명확하다 — 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아니라 스키마 축적에 의해 결정된다.

이를 이해하려면 도서관 비유가 도움이 된다. 도서관에 책이 한 권 들어올 때마다 목록에 한 줄이 추가된다. 한두 권은 문제없지만, 수천 권이 누적되면 목록을 뒤지는 비용 자체가 병목이 된다. 책의 내용을 읽는 시간(개별 LLM 호출)이 아니라, 어떤 책이 있는지 확인하는 시간(스키마 검색)이 전체 파이프라인을 지연시키는 것이다.

라이브러리 임포트는 이 스키마 축적의 "강한 형태"다. 단순히 검색 대상이 늘어나는 수준이 아니라, 임포트된 모듈의 전체 API가 파이프라인의 스키마에 편입된다. PyTorch 하나 임포트하면 수천 개의 함수 시그니처가 스키마에 추가되고, 이것이 후속 모든 도구 호출의 비용에 영향을 미친다. 기존 프레임워크는 이 비용을 호출 단위로 분산시키지 않고, 누적이라는 개념 자체를 인식하지 못한다.

핵심 통찰

도구 호출의 비용은 고정되어 있지 않다. 이전 호출들이 만들어놓은 상태에 따라 동적으로 변한다. 상태less 전제 하에서는 이 변동 비용을 예측하거나 최적화할 수 없다. 이것이 단순한 효율성 문제가 아니라, 아키텍처 결정의 비용을 측정 불가능하게 만드는 근원적 결함이다.

스냅샷은 "무엇"을 알지만 "왜"를 모른다

상태를 복원해야 할 때 우리는 보통 스냅샷에 의존한다. 파일 시스템 스냅샷, 체크포인트, 커밋 해시 — 모두 "특정 시점의 상태를 기록했다가 복원"하는 메커니즘이다. 그런데 Crab 논문이 지적하는 간극은 정확히 이 지점에서 발생한다.

에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율이 간극의 본질이다.

비유하자면, 요리 과정을 복원하는 상황을 상상해 보자. 스냅샷은 "소금 3g이 들어갔다"는 사실을 기록하지만, "이 소금은 간장을 넣은 후에 맛을 보고 결정적으로 추가한 것"이라는 인과 맥락은 기록하지 않는다. 소금을 빼고 싶을 때, 스냅샷은 단순히 "소금 없는 상태"로 돌려놓을 뿐, 간장-소금 사이의 의존 관계가 끊어진 것에 대해서는 아무것도 알지 못한다.

코드에서도 마찬가지다. import torch 후 작성된 50줄의 코드를 롤백할 때, 파일 단위 스냅샷은 그 50줄을 단순히 삭제한다. 하지만 의미론적 의존 그래프는 "torch 임포트 → 이 50줄 생성"이라는 인과 사슬을 유지해야 정상적인 분기 탐색이 가능하다. 스냅샷 기반 롤백은 인과 사슬을 단절시키기 때문에, 롤백 후 에이전트가 동일한 결정을 반복하거나, 더 나아가 의미론적으로 불일치하는 코드를 생성하게 된다.

스냅샷의 근본적 한계: 상태의 내용은 복원할 수 있어도, 상태에 도달한 경로는 복원할 수 없다. 에이전트가 아키텍처 결정을 되돌리려면 "무엇을 지워야 하는가"가 아니라 "어떤 결정의 결과를 취소하는가"를 알아야 한다.

이 간극이 치명적인 이유는 코딩 에이전트가 분기 탐색(branch-and-bound)을 수행할 때마다 발생한다. 한 경로에서 import torch를 선택하고, 다른 경로에서 import jax를 선택한다고 해보자. 파일 스냅샷은 두 경로를 독립적으로 저장하지만, 두 경로가 동일한 의존성 그래프의 서로 다른 분기점이라는 사실은 포착하지 못한다.

부작용 스펙트럼: 모든 도구 호출이 같은 무게를 가질 리 없다

기존 연구들이 "도구를 맹목적으로 호출하는 문제(blind-tool-invocation)"를 지적해 왔지만, 그 이면에는 더 근원적인 결함이 숨어 있다 — 도구 호출 간의 부작용 차이를 구분하는 택소노미 자체가 존재하지 않는다는 사실.

이것을 이해하려면 은행 업무 비유가 직관적이다. 창구에서 잔액 조회하는 것, 당일 이체하는 것, 그리고 계좌 자체를 해지하는 것은 모두 "창구 업무"라는 공통 분모가 있지만, 부작용의 무게는 완전히 다르다. 이를 구분하지 않고 모든 창구 업무를 동일한 절차로 처리하는 은행은 존재하지 않는다. 그런데 도구 호출 프레임워크는 정확히 이런 방식으로 동작하고 있다.

등급 예시 롤백 가능성 아키텍처 효과
순수 조회 search("..."), read_file() 완전 (롤백 불필요) 없음
가역적 변이 create_file("temp.py"), 변수 할당 파일 삭제로 복원 가능 국소적
불가역적 아키텍처 결정 import torch, npm install 의존성 그래프 복원 필요 전역적 · 누적적

이 택소노미가 없으면, 롤백·분기 메커니즘이 등급 구분 없이 동작하게 된다. 순수 조회 결과와 불가역적 아키텍처 결정을 동일한 비용으로 롤백하려 시도하는 비효율이 발생한다. 조회 호출에 대해서는 스냅샷조차 불필요하고, 반면 아키텍처 결정에 대해서는 스냅샷만으로는 부족한데 — 동일한 롤백 파이프라인을 거치게 되면 양쪽 모두 비효율적이다.

build-vs-buy 결정으로서의 임포트

라이브러리 임포트는 단순한 "코드 한 줄 추가"가 아니다. 에이전트에게 import torch는 "PyTorch를 직접 구현하지 않고 가져다 쓰겠다"는 build-vs-buy 결정이다. 이 결정은 이후의 모든 코드 생성 경로에 제약을 걸고, 대안 경로(예: JAX, TensorFlow, 순수 NumPy)를 탐색할 수 없게 만든다. 기존 프레임워크는 이 결정의 무게를 전혀 측정하지 않는다.

패러다임 전환: 하네스를 아키텍처 결정의 세마포어로

지금까지의 분석에서 하나의 명확한 결론이 도출된다. 도구 호출을 상태less 연산으로 처리하는 패러다임은 더 이상 유지될 수 없다. 그렇다면 어떻게 바꾸어야 하는가. 세 가지 확장이 동시에 필요하다.

  1. 의미론적 의존 그래프를 도구 호출 단위로 구축하라. 각 도구 호출이 생성하는 의존 관계를 명시적으로 기록한다. import torch가 발생하면, "이후 코드가 torch 네임스페이스에 의존한다"는 사실을 그래프에 편입시킨다. 이 그래프는 파일 스냅샷이 아니라 결정의 인과 구조를 담는다.
  2. 이 그래프를 하네스 인프라 계층에서 일급 객체로 관리하라. 의존 그래프는 에이전트의 애플리케이션 로직이 아니라, 실행 환경(하네스)이 관리해야 할 인프라 데이터다. C/R(Checkpoint/Restart) 런타임이 이 그래프를 기준으로 롤백 범위와 복원 전략을 결정한다.
  3. 코드 변경을 하네스 구조 변경으로 재해석하라. "코드 자체가 에이전트의 하네스가 된다(code-as-agent-harness)"는 관점에서, 코드에 대한 불가역적 변경(임포트, 의존성 추가)은 단순한 파일 수정이 아니라 하네스 자체의 구조 변경이다. 구조 변경은 상태less 연산으로 처리될 수 없다.

세마포어 모델의 핵심: 하네스는 각 도구 호출 전에 해당 호출의 부작용 프로필을 검사하고, 불가역적 결정인 경우 의미론적 의존 그래프에 명시적으로 기록한 뒤에야 실행을 허가한다. 이것은 도구 호출을 "무조건 실행"에서 "조건부 허가"로 전환하는 것을 의미한다.

비유하자면, 기존 프레임워크는 출입통제 없는 건물이었다. 누가 들어오든, 무엇을 하든, 문이 열려 있었다. 새로운 모델은 건물의 각 문에 출입구 검표소를 설치하는 것이다. 단순한 방문객(순수 조회)은 빠르게 통과시키고, 가구를 반입하는 사람(가역적 변이)은 기록을 남기며, 벽을 뜯는 시공업자(불가역적 아키텍처 결정)은 관리자 승인과 구조 변경 로그를 요구한다.

현재 Wiki의 연구들이 — 스키마 축적, 알고리즘-시스템 번역 간극, 하네스 엔지니어링, 코드-하네스 동일시 — 를 각기 독립적으로 다루고 있다면, 이 네 가지를 하나의 렌즈로 통합할 때 비로소 "불가역적 도구 호출의 부작용을 하네스 무결성 문제로 포착하는" 새로운 개념이 완성된다. 이것이 코딩 에이전트의 도구 호출 프레임워크가 나아가야 할 다음 패러다임이다.
