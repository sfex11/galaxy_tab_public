모델이 아닌 하네스를 스케일링하라

에이전트 AI의 진정한 병목은 모델 크기가 아니라,
모델을 감싸는 시스템의 설계 깊이에 있다

대상 논문: From Model Scaling to System Scaling 2025년 분석

자동차와 도로: 우리가 놓치고 있던 비유

자동차의 성능을 높이고 싶을 때, 우리는 보통 엔진을 바꿉니다. 배기량을 키우고, 터보를 달고, 연료 분사를 정밀하게 만듭니다. 그런데 어느 순간 엔진을 아무리 개조해도 차가 빨라지지 않는 순간이 옵니다. 도로가 좁아서, 신호등이 많아서, 차선이 불명확해서 속도를 낼 수 없는 상황 — 그게 바로 지금 에이전트 AI가 직면한 현실입니다.

지금까지 에이전트 AI 연구는 "엔진(모델)"을 얼마나 크게 만들까에 집중해 왔습니다. 파라미터를 늘리고, 학습 데이터를 늘리고, 컴퓨트를 쏟아부었습니다. 하지만 최근 등장한 논문 From Model Scaling to System Scaling: Scaling the Harness in Agentic AI는 이 흐름에 정면으로 반기를 듭니다. 엔진이 아니라 도로(하네스)를 스케일링해야 한다는 주장입니다.

여기서 하네스(Harness)란, 기초 모델을 실제 에이전트로 동작하게 만드는 모든 시스템적 구조물을 말합니다. 체크포인트·롤백 메커니즘, 도구 호출 파이프라인, 상태 관리, 감사 로그 — 모델 바깥에서 모델을 구동하는 전체 뼈대가 하네스입니다. 이 논문은 이 하네스를 단순한 '부가 인프라'가 아닌, 모델 스케일링과 동등한 독립적 스케일링 차원으로 규정합니다.

핵심 질문을 다시 써 보겠습니다: "모델이 더 똑똑해지면 에이전트가 더 강해진다"는 믿음에서, "하네스가 더 정교해질수록 에이전트가 더 강해진다"로 어떻게 넘어가는가? 그 전환의 구조를 파헤쳐 보겠습니다.

기존 프레임: 하네스를 '최적화할 인프라'로만 본 관습

문제의 출발점은 언어적 습관에 있습니다. 기존 문헌과 Wiki 분석에서 하네스는 항상 "최적화의 대상"으로 등장합니다. 마치 도로를 '보수해야 할 인프라'로만 보는 시선과 같습니다.

C/R 런타임은 "인프라 계층"일 뿐?

기존 agentic-harness-engineering 엔티티는 Crab 논문을 인용하며 C/R(Checkpoint/Recovery) 런타임을 "하네스의 핵심 구성요소(결함 내성·롤백·분기)를 직접 구현하는 인프라 계층"이라고 규정합니다. 여기서 '인프라 계층'이라는 표현이 핵심입니다. 인프라는 애플리케이션 로직의 바깥에 있는 것입니다. 모델이 주연이고, C/R은 무대 조명 담당 — 이런 위치 부여입니다.

파이프라인 병목은 "해소해야 할 문제"?

aggregate-pipeline-serving 엔티티는 Tool Attention 논문을 통해 "파이프라인 전체의 레이턴시 병목이 스키마 축적에 의해 결정된다"는 사실을 보여줍니다. 하지만 분석의 방향은 늘 "이 병목을 어떻게 해소할까"였지, "이 병목의 깊이가 하네스 스케일의 지표다"라는 해석은 없었습니다.

번역 간극은 "설계 결함"?

agent-os-semantic-gap과 algorithm-system-translation-gap은 에이전트의 의미론적 상태를 OS 수준의 체크포인트로 번역할 때 발생하는 비효율을 '간극(gap) 문제'로 프레이밍합니다. 마치 번역기가 오역하는 걸 고치면 된다는 식의 접근입니다.

공통적 전제의 문제점

이 세 분석의 밑바닥에는 하나의 암묵적 전제가 깔려 있습니다: 모델은 고정된 능력 원천이고, 하네스는 그 모델을 효율적으로 구동하기 위한 최적화 대상이라는 전제입니다. 자동차 비유로 돌아가면, "엔진이 정해져 있으니 도로는 그저 엔진 성능을 발휘하게 해 주는 보조 수단"이라는 셈입니다.

그런데 정말 그럴까요? F1 레이싱에서 서킷의 설계가 레이스 결과를 결정하지 않나요?

네 가지 축: 하네스 스케일링의 설계 원칙

본 논문이 제시하는 핵심 문장을 다시 읽어보겠습니다.

"the design of auditable, persistent, modular, and verifiable architectures around foundation model" — From Model Scaling to System Scaling

이 네 가지 속성은 단순한 '좋은 설계 원칙'이 아닙니다. 이 논문의 관점에서 이들은 스케일링 축입니다. 모델 스케일링이 파라미터·데이터·컴퓨트를 축으로 삼듯, 하네스 스케일링은 감사 가능성·지속성·모듈성·검증 가능성을 축으로 삼습니다.

모델 스케일링 축 하네스 스케일링 축 (본 논문) 무엇을 깊게 만드는가
파라미터 수 Auditable — 감사 가능성 의사결정 경로의 추적 가능 깊이
학습 데이터 양 Persistent — 지속성 상태 보존의 시공간적 범위
컴퓨트 예산 Modular — 모듈성 구성 요소의 분해·재조립 정도
훈련 스텝 Verifiable — 검증 가능성 행위 결과의 엄밀한 확인 수준

이 표의 왼쪽 열은 익숙합니다. LLM 시대를 관통해 온 스케일링의 정석이죠. 그런데 오른쪽 열을 보면, 하네스 스케일링이 단순히 "코드를 깔끔하게 짜는 것"과는 전혀 다른 차원의 작업임을 알 수 있습니다.

감사 가능성의 깊이를 생각해 보세요. "이 에이전트가 왜 그런 결정을 내렸는가?"라는 질문에 답할 수 있는 깊이가 1단계(최종 출력만 확인)인지, 10단계(중간 체크포인트·분기 이력·롤백 사유까지 추적)인지에 따라 하네스의 '스케일'이 달라집니다. 이건 코드 리팩토링으로 해결되는 게 아닙니다. 아키텍처 차원에서 의사결정 그래프를 어떻게 기록하고, 어떻게 질의 가능하게 만드는가의 문제입니다.

지속성의 범위도 마찬가지입니다. 에이전트가 5분 동안의 대화 상태만 유지하는지, 아니면 며칠에 걸친 작업의 맥락을 파편화 없이 보존하는지 — 이 범위 자체가 스케일링 축이 됩니다. 더 넓은 지속성을 구현하려면 체크포인트 설계, 메모리 관리, 복구 로직 전체가 달라져야 합니다.

요약하면, 모델 스케일링이 "모델 안에 얼마나 많은 지식을 압축하는가"라면, 하네스 스케일링은 "모델 밖에서 모델의 행위를 얼마나 정교하게 관리·추적·복구하는가"입니다. 둘은 독립적이고, 상호 보완적입니다.

기존 지식을 뒤집어 읽기: 인프라에서 스케일링 계층으로

이 재정의가 강력한 이유는, 기존에 알려진 사실들을 완전히 다른 해석의 렌즈로 다시 볼 수 있게 해 주기 때문입니다. 마치 같은 X-ray 사진을 다른 의사가 보면 다른 진단이 나오는 것과 같습니다.

C/R 런타임: "구성요소"에서 "스케일링 구현체"로

기존에는 C/R 런타임이 하네스의 부품이었습니다. 본 논문의 관점에서는, C/R이 제공하는 결함 내성·롤백·분기 기능 자체가 Persistent와 Verifiable의 구현체가 됩니다. 롤백 가능성이 높을수록, 분기 검증이 엄밀할수록 하네스 스케일이 커지는 것입니다.

비유하자면, 롤백 기능을 "있으면 좋은 옵션"에서 "스케일링의 직접 지표"로 격상시키는 겁니다. 자동차의 브레이크를 "안전 장치"에서 "코너링 속도를 결정하는 성능 요소"로 다시 보는 것과 같습니다.

스키마 축적: "병목"에서 "모듈성의 척도"로

파이프라인에서 스키마가 누적되는 현상을 기존에는 레이턴시 병목으로 봤습니다. 본 논문의 관점에서는, 이것을 파이프라인이 얼마나 모듈적으로 분해 가능한가(Modular)의 척도로 재해석합니다.

스키마가 파이프라인 전체에 누적된다는 것은, 모듈 경계가 불투명하다는 뜻입니다. 각 단계가 독립적으로 교체·업그레이드될 수 없다는 거죠. 이건 병목이 아니라 하네스 스케일링의 미달 상태를 나타내는 지표입니다. 문제를 "고치는 것"이 아니라 "측정하는 것"으로 바뀝니다.

의미론적 간극: "버그"에서 "스케일링 불균형"으로

에이전트의 의미론적 상태를 OS 체크포인트로 번역할 때 발생하는 비효율을, 기존에는 설계 결함이나 버그로 프레이밍했습니다. 본 논문의 렌즈로 보면, 이건 모델 스케일링과 하네스 스케일링 사이의 불균형이 만들어내는 구조적 간극입니다.

직관적 이해: 고속도로와 차량의 불균형

시속 300km로 달릴 수 있는 슈퍼카(모델)가 있는데, 도로(하네스)는 시속 60km 구간에 감속 카메라와 좁은 차선만 있는 상황을 상상해 보세요. 이때 문제는 차의 속도가 아닙니다. 도로와 차 사이의 불균형이 문제입니다. 모델이 복잡해질수록 의미론적 의존 그래프가 복잡해지는데, 하네스 스케일이 이를 따라가지 못하면 간극이 벌어지는 것입니다.

새롭게 드러나는 연결망

본 논문의 네 가지 속성을 기존 Wiki 슬러그와 매핑하면서 의미심장한 연결이 드러납니다.

  • Auditable → auditability-as-scaling-requirement: 이 슬러그가 이미 존재한다는 것 자체가 논문의 논지와 직접 부합합니다. 감사 가능성이 스케일링 요건으로 규정되어 있습니다.
  • Persistent → agent-sandbox-state-management: 샌드박스 상태 관리가 지속성의 구현체로 재해석됩니다.
  • Verifiable → agent-reliability-auditing: 신뢰성 감사가 검증 가능성의 실행 형태가 됩니다.
  • Modular → agent-loop-as-computation: 에이전트 루프를 하나의 계산 단위로 모듈화하는 개념이 모듈성 스케일링의 기본 단위가 됩니다.

특히 auditability-as-scaling-requirement라는 슬러그의 존재는 주목할 만합니다. Wiki가 이미 부분적으로 "감사 가능성이 부가 기능이 아니라 스케일링의 필수 조건"이라는 직관을 포착하고 있었음을 시사합니다. 하지만 이를 모델 스케일링과 동등한 독립 축으로 체계화하지는 못했고, 본 논문이 바로 그 마지막 한 걸음을 완성합니다.

패러다임 전환: 어디서부터 바꿔야 하는가

이 논문이 가져오는 전환을 한 문장으로 압축하면 이렇습니다.

"모델이 더 똑똑해지면 에이전트가 더 강해진다"에서
"하네스가 더 감사 가능하고, 지속 가능하고, 모듈러하고, 검증 가능해질수록 에이전트가 더 강해진다"로.

그런데 이 전환은 단순히 '관점 바꾸기'로 끝나지 않습니다. 실제로 연구 방향과 엔지니어링 우선순위를 근본적으로 바꿔야 합니다.

구체적으로 무엇을 바꿔야 하는가

첫째, 벤치마크의 축을 추가해야 합니다. 현재 에이전트 벤치마크는 모델의 정답률·추론 능력을 측정합니다. 여기에 하네스 스케일링의 네 축을 측정하는 메트릭이 추가되어야 합니다. "이 에이전트의 의사결정 경로를 몇 단계까지 추적할 수 있는가?", "상태 복구 시 누락되는 컨텍스트 비율은 얼마인가?" 같은 질문이 벤치마크의 정식 항목이 되어야 합니다.

둘째, 연구 투자의 분배를 재조정해야 합니다. 지금은 모델 학습에 90%, 하네스 설계에 10%의 자원이 배분되는 구조에 가깝습니다. 본 논문의 논지를 수용한다면, 하네스 스케일링 연구 — 감사 가능 아키텍처, 상태 지속성 프로토콜, 모듈 경계 명세, 검증 프레임워크 — 에 훨씬 더 많은 연구 자원이 투입되어야 합니다.

셋째, 시스템 엔지니어의 역할을 재정의해야 합니다. 기존에는 ML 엔지니어가 주연이고 시스템 엔지니어가 조연이었습니다. 하네스 스케일링의 관점에서는, 시스템 설계 자체가 성능을 결정하는 독립 변수가 되므로 시스템 엔지니어의 역할이 주연급으로 격상됩니다.

넷째, 제품 평가 기준을 바꿔야 합니다. "어떤 모델을 쓰는가?"에서 "하네스의 스케일이 어느 수준인가?"로 평가의 중심이 이동해야 합니다. 같은 모델을 쓰더라도, 하네스 스케일이 높은 에이전트가 낮은 에이전트보다 실제 환경에서 압도적으로 강한 결과를 낼 것이기 때문입니다.

자동차 비유로 돌아가 마무리하겠습니다. 1960년대의 레이싱은 엔진 출력으로 승부가 결정되었습니다. 하지만 현대 F1에서는 엔진 출력보다 서킷 적응력, 타이어 관리, 피트 스톱 전략, 데이터 분석 시스템이 레이스를 지배합니다. 엔진의 차이는 미미해졌고, 엔진을 둘러싼 시스템의 차이가 결과를 좌우합니다.

에이전트 AI도 같은 지점에 도달하고 있습니다. 기초 모델 간의 능력 차이가 좁혀지는 시점에서, 진정한 경쟁력은 모델 안이 아니라 모델을 감싸는 하네스의 스케일에서 결정될 것입니다. 이 논문은 그 전환을 알리는 사이렌입니다. 들어야 할 때입니다.
