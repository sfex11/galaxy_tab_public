에이전트 아키텍처 분석

빠르게 가기 위해 남긴 발자국

추측적 도구 호출이 만든 프라이버시의 구조적 역설 — 지연을 숨기려다 통제 불가능한 공격 표면을 열어버린 모순에 대하여

섹션 1

병목을 숨기려는 유혹: 스키마 축적 문제와 추측적 실행

LLM 에이전트가 실무에서 의미 있는 작업을 수행하려면 반드시 외부 도구에 접근해야 합니다. 검색, 계산, 파일 조작, 데이터베이스 조회 — 이 도구들을 에이전트 파이프라인에 연결하는 표준이 MCP(Model Context Protocol)로 자리 잡으면서, 시스템 설계자들은 하나의 예상치 못한 병목과 마주했습니다.

"Tool Attention Is All You Need" 논문이 밝혀낸 핵심 발견은 이것입니다. 다중 서버 MCP 배포 환경에서 파이프라인 전체의 응답 지연을 지배하는 것은 개별 LLM의 추론 시간이 아닙니다. 바로 스키마 축적(schema accumulation) — 각 도구 서버로부터 사용 가능한 도구 목록과 인자 규격을 모아오는 과정 자체가 병목이라는 사실입니다.

直观的 비유

식당 주방을 상상해 보십시오. 요리사는 3초면 어떤 요리를 만들지 결정합니다. 하지만 조리대 위에 놓인 수십 개의 레시피 책을 한 권씩 펴서 확인하는 데는 30초가 걸립니다. 요리사의 사고 시간이 아니라 도구 목록을 파악하는 시간이 병목인 셈입니다.

이 병목을 해소하기 위해 등장한 전략이 추측적 도구 호출(speculative tool call)입니다. 에이전트가 아직 최종 결정을 내리지 않았더라도, "아마 이 도구를 쓸 것 같다"는 추측을 바탕으로 미리 해당 도구의 스키마를 로드하고, 심지어 실행까지 해두는 것입니다. 마치 요리사가 손님이 무엇을 시킬지 모르는 상태에서 예상되는 메뉴 5개의 재료를 미리 꺼내두는 것과 같습니다.

내부적으로 이 접근은 효과적입니다. 스키마 로딩 시간이 사용자가 느끼는 대기 시간 속에 숨겨집니다. 문제는 이 "숨김"이 에이전트의 경계를 넘어 외부 세계로 뻗어나갈 때 시작됩니다.

섹션 2

되돌릴 수 없는 편지: Ghost Tool Calls가 포착한 시점 비대칭성

최근 발표된 논문 "Ghost Tool Calls: Issue-Time Privacy for Speculative Agent Tools"는 이 모순을 정확히 꿰뚫습니다.

"Tool-augmented language agents speculatively issue likely future tool calls to hide latency, but those calls leak inferred user intent to external services before the agent commits to the branch." — Ghost Tool Calls, 2025

여기서 핵심은 시점의 비대칭성(temporal asymmetry)입니다. 에이전트는 도구를 추측적으로 호출하는 시점에 이미 외부 서비스에 사용자 의도가 담긴 요청을 전송합니다. 이후 에이전트가 "아, 이 분기는 아니었네" 하고 롤백(rollback)을 수행하더라도, 이미 외부 서버에 도착한 정보는 사라지지 않습니다.

直观的 비유

눈이 쌓인 산길을 간다고 상상해 보십시오. 당신은 어느 길로 갈지 아직 결정하지 못했지만, 시간을 아끼기 위해 세 갈래 길 모두에 정찰병을 미리 보냅니다. 정찰병들은 눈밭에 발자국을 남깁니다. 결국 당신은 왼쪽 길을 선택하고, 가운데와 오른쪽 길의 정찰병을 "철수"시킵니다. 하지만 눈밭에 찍힌 발자국은 지워지지 않습니다. 당신이 어느 길들을 고려했는지, 결국 어디로 갔는지가 그대로 노출됩니다.

이 비유가 담고 있는 구조적 진실은 명확합니다. 에이전트의 내부 상태는 체크포인트와 복원 메커니즘으로 되돌릴 수 있지만, 외부로 나간 정보는 되돌릴 수 없다는 것입니다. "Ghost Tool Call"이라는 이름 자체가 이 현상을 완벽하게 포착합니다 — 에이전트는 해당 분기를 포기했지만, 외부 서비스에는 이미 유령처럼 잔존하는 호출 기록이 남습니다.

왜 "Ghost"인가

에이전트의 실행 트리에서 해당 분기는 잘려 나가(pruned) 사라집니다. 에이전트 자신의 기억에는 존재하지 않는 호출입니다. 그러나 외부 서비스의 로그에는 실재하는 호출 기록으로 남아있습니다. 살아있지도 죽지도 않은 — 유령 같은 상태. 이 단어 선택이 암시하는 바는 정확히 존재론적 불일치입니다.

섹션 3

두 세계의 경계: 내부 가역성과 외부 비가역성의 간극

이 모순을 이해하려면 에이전트 시스템 내부와 외부 사이의 정보 흐름의 비대칭성을 구조적으로 정리할 필요가 있습니다.

차원 내부 상태 (에이전트/OS) 외부 상태 (서비스)
가역성 Checkpoint/Restore로 복원 가능 전송 후 되돌릴 수 없음
통제권 C/R 런타임이 관리 서비스 측 정책에 종속
의미론 보존 의미론적 C/R로 해결 시도 아무런 보장 없음

이 표가 보여주는 것은 단순한 "버그"가 아닙니다. 두 세계 사이의 근본적인 속성 불일치입니다. 내부 세계에서는 상태를 자유롭게 되감고 되돌릴 수 있지만, 외부 세계로 정보가 한 번 넘어가면 그 이후의 통제권은 완전히 상대방에게 넘어갑니다.

"Crab" 논문이 다룬 의미론적 간극 — 에이전트의 의미론적 의존 그래프를 OS의 체크포인트 메커니즘으로 번역할 때 발생하는 비효율 — 는 여전히 시스템 내부 경계 안에 있었습니다. 해결하려는 문제도, 해결 수단도 모두 에이전트의 통제 범위 내에 있었습니다.

그러나 추측적 도구 호출의 프라이버시 문제는 이 간극을 외부 서비스 경계로까지 확장시킵니다. 문제의 영역 자체가 에이전트의 통제 권한 밖으로 튀어나온 것입니다. 이것이 단순한 최적화 문제가 아닌 아키텍처 수준의 구조적 결함으로 보아야 하는 이유입니다.

새로운 간극의 식별

외부 상태 비가역성 간극(External-State Non-Revocability Gap) — 에이전트의 내부 롤백 가능성과 외부 서비스의 정보 잔존성 사이에 존재하는 되돌릴 수 없는 정보 비대칭성. 기존 Wiki의 어떤 엔티티로도 완전히 포섭되지 않는 간극으로, 샌드박스 내부 상태를 다루는 기존 프레임워크와 달리 샌드박스 외부에서 발생하는 정보의 비가역성을 다룹니다.

섹션 4

효율 최적화의 맹점: 적응 메커니즘이 보지 못하는 것

이 모순을 더욱 날카롭게 만드는 것은, 기존의 적응형 최적화 메커니즘이 이 문제를 전혀 인식하지 못한다는 점입니다.

SpecKV 같은 적응형 추론 시스템을 보십시오. 이 시스템은 압축이라는 내부 시스템 상태에 반응하여 추측 길이(γ)를 동적으로 조절합니다. KV 캐시가 효율적으로 압축되면 더 긴 추측을 실행하고, 그렇지 않으면 추측을 줄입니다. 순수하게 내부 효율 지표에만 반응하는 것입니다.

적응의 구조적 어긋남

SpecKV의 적응 루프가 관찰하는 것: KV 캐시 압축률, 내부 메모리 사용량, 추론 처리량

SpecKV의 적응 루프가 관찰하지 못하는 것: 외부 서비스로 유출되는 정보의 양, 해당 정보의 민감도, 유출 이후의 정보 잔존 기간

이것은 단순한 누락이 아닙니다. 적응 메커니즘의 관찰 가능성(observability) 경계가 정보 흐름의 실제 위험 영역과 구조적으로 어긋나 있는 것입니다. 마치 자동차의 연비 최적화 시스템이 엔진 효율만 보고, 배기가스가 어디로 흘러가는지는 전혀 보지 않는 것과 같습니다.

더 근본적으로, 추측적 실행은 도구 호출의 의도적 성격(intentionality)과 실행의 확정성(commitment) 사이에 본래 존재하던 간극을 프라이버시 공격 표면으로 전환시킵니다. "아마 이걸 쓸 것 같다"는 미확정 의도가, 외부 서비스 입장에서는 "이 사용자가 이 정보를 원한다"는 확정적 신호로 수신되기 때문입니다. 의도의 확정도(度)가 경계를 넘는 순간 왜곡되는 것입니다.

  • 내부 관점: "이것은 추측이다, 확정이 아니다" — 확률적 의미론 유지
  • 외부 관점: "요청이 도착했다" — 이분적 사실로 수렴
  • 간극: 경계를 넘나드는 순간, 확률적 표현이 이분적 사실로 강제 변환됨

이 강제 변환을 막을 수 있는 메커니즘이 현재의 추측적 실행 아키텍처에는 존재하지 않습니다. 정보가 외부로 나가는 순간, 그 정보의 "추측적이라는 메타데이터"는 소멸합니다.

섹션 5

패러다임 전환: 지연 숨김에서 정보 흐름 제어로

"Ghost Tool Calls"가 제안하는 issue-time privacy 접근법은 이 모순에 대한 최초의 정면 대응입니다. 도구 호출 발행 시점에 프라이버시 보호를 적용하겠다는 것입니다. 하지만 이것만으로는 근본적인 trade-off가 해결되지 않습니다.

필요한 패러다임 전환

첫째, 최적화의 목표를 재정의해야 합니다. 현재 추측적 실행의 목표는 단일 차원 — 레이턴시 최소화 — 입니다. 이를 다차원 목표 함수로 전환해야 합니다. 레이턴시 감소량 ΔL과 정보 유출 위험 ΔP를 동시에 최적화하되, ΔP에 대한 임계값을 하드 제약(hard constraint)으로 두어야 합니다. 속도가 아무리 빨라도 특정 프라이버시 임계값을 넘으면 추측적 실행이 아예 차단되는 구조여야 합니다.

둘째, 정보의 "추측성 메타데이터"를 경계를 넘어 보존해야 합니다. 외부 서비스가 이 요청이 "추측적 호출"임을 알 수 있는 표준화된 프로토콜 확장이 필요합니다. 이를 통해 서비스 측이 일정 기간 후 해당 데이터를 자동 폐기하거나, 적어도 일반 요청과 구분하여 취급할 수 있게 해야 합니다.

셋째, 샌드박스의 경계를 재설계해야 합니다. 현재의 샌드박스는 코드 실행 수준에서의 격리를 다룹니다. 그러나 추측적 도구 호출이 보여주듯, 정보 흐름 수준에서의 격리이 필요합니다. "이 정보는 롤백 가능한 상태에서 나간 것이다"라는 속성을 정보 자체에 부착하고, 이 속성이 외부 서비스에서도 의미를 가지는 아키텍처가 필요합니다.

넷째, 적응 메커니즘의 관찰 경계를 외부로 확장해야 합니다. SpecKV가 내부 상태만 보는 것에서 벗어나, 추측적 호출이 유발할 외부 정보 유출의 예상량을 적응 루프의 입력으로 포함해야 합니다. 이를 위해서는 도구별 정보 민감도 프로파일링이 전제되어야 합니다.

결론적으로, 지연 숨김을 위한 추측적 실행은 내부적으로는 효율적이지만, 외부적으로는 에이전트의 통제 범위를 벗어난 프라이버시 공격 표면을 구조적으로 생성합니다. 이것은 파라미터 튜닝으로 해결될 수 있는 문제가 아니라, 정보가 시스템 경계를 넘는 방식 자체를 재설계해야 하는 아키텍처 패러다임의 전환을 요구하는 문제입니다.

빠르게 가기 위해 남긴 발자국은 지워지지 않습니다. 우리가 만들어야 할 것은 발자국을 남기지 않는 길이 아니라 — 그것은 불가능합니다 — 발자국이 의미를 갖지 않도록 설계된 눈밭입니다.
