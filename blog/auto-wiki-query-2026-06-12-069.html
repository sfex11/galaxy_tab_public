평가 계약은 암묵적 하네스다

Claw-SWE-Bench가 드러낸 진실 — 우리가 '중립적 측정'이라 믿었던 것이 사실은 에이전트의 행동을 구조적으로 규정하는 틀이었다면?

목차
  1. 운전시험의 유리벽 — 하네스가 이미 한 번 확장된 방식
  2. 줄자가 길이를 재는 게 아니다 — 평가 계약의 실체
  3. 번역의 누락 — 두 영역에서 동일하게 현현하는 간극
  4. 병목의 오인 — 파이프라인에서 벤치마크까지
  5. 하네스 스펙트럼 — 패러다임 전환의 구체적 지도

1. 운전시험의 유리벽 — 하네스가 이미 한 번 확장된 방식

운전면허 시험을 생각해 보자. 시험장에는 코스가 있고, 규칙이 있고, 합격 기준이 있다. 우리는 이것을 '운전 능력을 측정하는 중립적인 도구'라고 믿는다. 하지만 만약 시험 코스가 오직 자동변속 차량만 주행 가능한 구조로 설계되어 있다면? 수동변속 능력이 뛰어난 운전자도 이 시험에서는 불이익을 받는다. 시험은 '운전 능력'이 아니라 '자동변속 차량에 대한 적합성'을 측정하고 있는 셈이다.

에이전트 하네스(Agent Harness)의 역사에서도 정확히 같은 일이 한 번 벌어졌다. Crab 논문은 C/R(체크포인트/롤백) 런타임이 단순한 인프라가 아니라, 결함 내성·롤백·분기 같은 하네스의 핵심 동작을 직접 구현하는 계층이라는 것을 보였다. 이 발견 이전에는 런타임이 '하네스 밖'의 독립적 인프라라고 가정되었다. 하지만 실제로는 런타임이 하네스의 동작 방식을 구조적으로 결정하고 있었다.

핵심 패턴

하네스 엔지니어링의 확장은 새로운 기능의 추가가 아니었다. 기존에 '밖'이라고 가정되었던 것이 사실은 '안'의 동작을 구조적으로 결정하고 있었다는 발견이었다. 확장의 메커니즘은 '발견'이지 '추가'가 아니다.

이때의 확장은 애플리케이션 로직에서 실행 환경으로 이루어졌다. 하네스의 경계가 코드 수준에서 OS 수준으로 밀려난 것이다. 그리고 이 패턴은, Claw-SWE-Bench라는 완전히 다른 영역에서 똑같은 방식으로 반복된다.

2. 줄자가 길이를 재는 게 아니다 — 평가 계약의 실체

줄자로 물체의 길이를 잰다고 가정하자. 그런데 이 줄자가 5cm 단위로만 눈금이 새겨져 있고, 측정 결과는 반드시 가장 가까운 5cm 배수로 제출해야 한다면? 이 줄자는 '길이'를 재는 게 아니다. '5cm 단위로 표현 가능한 길이'만을 재고 있다.

Claw-SWE-Bench가 보여준 것도 정확히 이 지점이다. 논문의 핵심 문장을 보자:

"A generic agent does not by itself satisfy the evaluation contract."

SWE-bench의 평가 계약 — 워크스페이스 구조, 패치 적용 방식, 예측 제출 포맷 — 은 중립적인 측정 도구가 아니다. 이것은 이름만 '벤치마크'일 뿐, 실질적으로는 다음 세 가지를 구조적으로 규정하는 하네스다:

  • 파일 시스템 탐색의 경계 — 에이전트가 워크스페이스를 어떻게 navigate할 수 있는지
  • 상태 축적과 제출의 방식 — 수정 내용을 어떤 포맷으로 패치로 변환해야 하는지
  • 중단과 재개의 의미론 — 어디서 체크포인트를 찍고 어디서 이어서 작업할 수 있는지

이 규정들에 맞지 않는 아키텍처는 — 코드 수정 능력이 뛰어나더라도 — 평가 계약을 충족하지 못해 낮은 점수를 받는다. 마치 수동변속 운전자가 자동변속 코스에서 불합격하는 것과 같다.

💡 직관적 비유

피아노 경연을 상상해 보라. 그런데 이 경연의 '제출 규칙'이 반드시 악보를 PDF로 스캔해서 제출해야 하는 것이라면? 즉흥 연주자, 악보 없이 연주하는 음악가, 혹은 전자음악 작곡가는 아무리 훌륭한 연주를 해도 이 '평가 계약'을 충족할 수 없다. 경연이 측정하는 것은 '피아노 연주 능력'이 아니라 'PDF 악보 제출 가능한 피아니즘'이다.

평가 계약은 벤치마크의 일부가 아니라 벤치마크와 에이전트 사이에 위치한 번역 계층이다. 그리고 이 번역 계층은 하네스 엔지니어링의 대상이다.

3. 번역의 누락 — 두 영역에서 동일하게 현현하는 간극

번역기를 생각해 보자. 한국어 원문을 영어로 번역할 때, 번역기의 구조적 한계 때문에 특정 표현이 누락되거나 왜곡되는 현상이 있다. 원문의 뉘앙스가 번역 과정에서 사라지는 것이다. 중요한 것은 — 이 누락은 원문의 문제가 아니라 번역 계층의 문제라는 점이다.

Algorithm-System Translation Gap(알고리즘-시스템 번역 간극)은 정확히 이런 현상을 설명하는 개념이다. 그리고 이 간극이 두 곳에서 — 런타임 영역과 측정 영역에서 — 동일한 패턴으로 현현한다는 것이 Claw-SWE-Bench의 핵심 통찰이다.

Crab의 간극 (런타임)
에이전트가 의존하는 상태
↔ OS가 체크포인트로 포착하는 상태

결과: 의미 없는 상태까지 스냅샷되는 비효율
Claw-SWE-Bench의 간극 (측정)
벤치마크가 측정하려는 능력
↔ 평가 계약이 실제로 요구하는 아키텍처 적합성

결과: 계약에 맞지 않는 아키텍처의 능력이 누락되는 왜곡

두 경우 모두 번역 과정에서 의도치 않은 제약이 삽입된다. Crab에서는 에이전트의 의미론적 의존 그래프를 OS의 전체 스냅샷 메커니즘으로 번역할 때 불필요한 상태가 포함된다. Claw-SWE-Bench에서는 벤치마크가 의도하는 코딩 능력을 평가 계약의 구조적 요구사항으로 번역할 때 특정 아키텍처에 대한 편향이 삽입된다.

여기서 새로운 개념 연결이 자연스럽게 발생한다. 기존에 존재하던 Benchmark Specification Gap(벤치마크 명세 간극)이라는 슬러그에 이제 구체적인 내용이 채워진다: 벤치마크 명세가 의도하는 측정 대상과, 평가 계약의 구조적 제약이 실제로 측정하는 대상 사이의 간극. 이것은 Algorithm-System Translation Gap의 측정 영역 특화 버전이다.

나아가 Code-as-Agent-Harness라는 명제와도 연결된다. SWE-bench에서 평가 계약의 핵심이 패치 포맷과 워크스페이스 구조라면 — 즉, 코드 파일 자체의 형태가 평가의 구조를 결정한다면 — 이는 "코드 자체가 하네스의 일부로 기능한다"는 명제를 벤치마크 영역에서 재확인하는 셈이다.

4. 병목의 오인 — 파이프라인에서 벤치마크까지

병원 응급실을 상상해 보자. 환자 대기 시간이 길어지면 사람들은 보통 "의사가 부족해서"라고 생각한다. 하지만 실제 병목이 접수 창구의 서류 양식에 있다면? 의사를 아무리 늘려도 대기 시간은 줄지 않는다. 문제를 잘못 진단한 것이다.

Tool Attention 논문은 다중 서버 MCP 배포에서 정확히 이런 '오인정의'를 발견했다. 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아니라 스키마 축적에 의해 결정된다는 것을 보인 것이다. "실제 병목 지점을 재정의"한 사례다.

그리고 Claw-SWE-Bench는 완전히 동일한 패턴의 오인을 벤치마크 영역에서 포착한다:

병렬 비교: 오인정의의 구조

파이프라인의 오인
병목 = LLM 호출 성능 → 실제 병목 = 스키마 축적 구조

벤치마크의 오인
점수 = 에이전트의 코딩 능력 → 실제 결정요인 = 평가 계약과의 아키텍처 적합성

파이프라인 서빙이 '개별 컴포넌트 최적화'가 아닌 '파이프라인 전체 구조'로 분석되어야 했던 것처럼, 벤치마크도 '외부 태스크의 난이도 조정'이 아닌 '평가 계약의 구조적 분석'으로 접근되어야 한다. Aggregate Pipeline Serving의 교훈은 파이프라인에만 국한되지 않고, 에이전트 평가 전반에 적용되는 메타 원칙임이 드러난다.

이는 실무에서 매우 중요한 시사점을 준다. 벤치마크 점수를 높이기 위해 에이전트의 추론 능력을 개선하는 것은 — 응급실에 의사를 늘리는 것과 같다. 진짜 병목인 평가 계약 적합성을 다루지 않으면, 점수 향상은 능력 향상이 아니라 계약 순응도 향상에 불과하다.

5. 하네스 스펙트럼 — 패러다임 전환의 구체적 지도

이제 전체 그림이 보인다. 평가 계약 설계를 하네스 엔지니어링에 통합한다는 것은, 하네스의 범위를 다음 스펙트럼으로 재정의하는 것이다:

애플리케이션 로직
에이전트의 추론·계획·도구 호출
원래 하네스
실행 환경
C/R 런타임, 체크포인트, 롤백
Crab의 확장
평가 계약
워크스페이스, 패치 포맷, 제출 의미론
Claw의 확장

각 단계는 독립된 층이 아니다. 셋 모두 에이전트의 의도된 동작을 구조적으로 제약하는 번역 계층이다. Agent-OS Semantic Gap이 OS 수준에서, Algorithm-System Translation Gap이 런타임 수준에서, 그리고 평가 계약이 측정 수준에서 동일한 패턴의 간극을 생성한다.

통합이 아니라 인정

중요한 것은 이것이 '확장'이 아니라는 점이다. 이미 하나의 간극이었던 것을 명시적으로 인정하는 것이다. 운전시험의 코스가 자동변속 전용이었다는 사실을 인정하고, 이를 시험 설계의 문제로 규정하는 것과 같다.

패러다임 전환: 분리에서 스펙트럼으로

기존 패러다임은 명확하게 분리되어 있었다:

  • 하네스 = 실행 인프라 (C/R 런타임, 파이프라인 서빙)
  • 벤치마크 = 외부 태스크 (능력 측정, 중립적 도구)

Claw-SWE-Bench는 이 분리가 근본적으로 잘못되었음을 보여준다. 새로운 패러다임은 다음과 같다:

  • 하네스 스펙트럼 = 애플리케이션 로직 → 실행 환경 → 평가 계약을 잇는 연속적 번역 계층
  • 벤치마크 = 스펙트럼의 최외곽에 위치한 하네스의 일종
  • 평가 계약 엔지니어링 = 하네스 엔지니어링의 필수 구성요소

이 전환의 실천적 의미는 명확하다. 벤치마크를 설계하거나 해석할 때, "이 평가 계약이 어떤 아키텍처에 유리하고 어떤 아키텍처에 불리한가?"를 필수 질문으로 삼아야 한다. 벤치마크 점수를 보고 "이 에이전트가 더 뛰어나다"고 결론내리기 전에, "이 점수가 측정한 것은 능력인가, 아니면 평가 계약에 대한 적합성인가?"를 물어야 한다.

그리고 궁극적으로 — 평가 계약을 하네스로서 의식적으로 설계해야 한다. 특정 아키텍처에 대한 편향을 최소화하는 계약 구조, 다양한 에이전트 패러다임이 공평하게 경쟁할 수 있는 제출 포맷, 워크스페이스 구조의 유연성. 이것이 하네스 엔지니어링의 새로운 프론티어다.

줄자의 눈금 간격을 설계하는 것도 측정의 일부다. 그것을 잊으면, 우리는 길이가 아니라 눈금을 재고 있는 것이다.
