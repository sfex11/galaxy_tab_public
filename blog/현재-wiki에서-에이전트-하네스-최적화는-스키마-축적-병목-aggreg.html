우리가 놓친 진짜 병목 — 인프라가 아니라 동기성 가정이었다

에이전트 하네스 최적화 논의를 관통하면 드러나는 하나의 보이지 않는 상한선, 그리고 Future-based 비동기 호출이 열어낸 패러다임 전환의 본질

📖 Concurrency without Model Changes (2025) ⏱ 12분 읽기
Section 01

요리사와 조수의 이야기 — 동기성 가정이란 무엇인가

한 명의 요리사가 레시피를 따라 요리를 합니다. 재료를 다지고, 팬을 달구고, 소스를 만들어야 하는데, 매번 특정 재료가 필요할 때마다 조수가 창고에서 가져올 때까지 요리사가 아무것도 하지 않고 가만히 서서 기다린다고 상상해 보십시오. 조수가 빠르게 뛰어다니게 해도, 창고 정리를 효율적으로 해도, 근본적으로 요리사가 '기다린다'는 사실은 변하지 않습니다.

이것이 바로 현재 LLM 에이전트 하네스가 안고 있는 동기적 실행(Synchronous Execution)의 구조입니다. LLM이 토큰을 하나씩 생성하다가 도구 호출이 필요한 순간, 디코딩이 완전히 멈추고 도구 실행이 끝날 때까지 블로킹(Blocked)됩니다. 모델 아키텍처가 바뀌지 않는 한 이건 당연한 것 아니냐고 생각할 수 있지만, 바로 이 '당연함'이 문제의 핵심입니다.

직관적 비유

이건 마치 식당 주방에서 조리대 한 대만 쓰면서, 재료 준비가 끝날 때마다 불을 끄고 아무것도 안 하고 기다리는 것과 같습니다. 주방 인프라를 아무리 개선해도 — 창고 위치를 옮기고, 조리대 높이를 바꾸고, 안전 장치를 달아도 — 요리사가 멈춰 선다는 기본 전제를 바꾸지 않으면, 식당의 속도엔 분명한 한계가 존재합니다.

지금까지의 에이전트 하네스 최적화 논의는 이 '기다림'이라는 전제를 의문 없이 수용한 채, 그 안에서 어떻게든 시간을 줄이려 했던 국소적 최적화(Local Optimization)에 불과했습니다.

Section 02

보이지 않는 상한선 아래서의 국소적 최적화들

Wiki에 축적된 기존 논의를 살펴보면, 하네스의 병목은 크게 두 가지 인프라 관점에서 분석되어 왔습니다.

스키마 축적 병목: 빠른 조수를 둔 창고 정리

📋 Aggregate Pipeline Serving

"Tool Attention Is All You Need" 논문은 다중 서버 MCP(Multi-Context Protocol) 배포 환경에서, 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출 자체가 아니라 스키마 축적(Schema Accumulation)에 의해 결정된다고 지적했습니다. 도구가 많아질수록 호출 시마다 스키마를 동기적으로 로드하고 파이프라인을 일시 정지시켜야 하는 구조적 특성 때문입니다.

하지만 이 병목은 근본 원인이 아니라 2차적 증상입니다. 도구를 부를 때마다 파이프라인이 정지해야 한다는 것 자체가 동기성 가정에서 파생된 현상이기 때문입니다. 스키마 로딩을 아무리 빠르게 해도, 매번 파이프라인이 멈춘다는 구조적 제약은 그대로입니다.

체크포인트/복원: 멈춘 상태를 찍는 안전 카메라

🔄 Agentic Harness Engineering

"Crab: A Semantics-Aware Checkpoint/Restore Runtime" 논문은 C/R(Checkpoint/Restore) 런타임을 도입해 에이전트 실행의 결함 내성, 롤백, 분기 탐색을 인프라 계층에서 직접 구현했습니다. 도구 실행이 실패하면 이전 상태로 복원하고, 다른 경로를 시도하는 식입니다.

이는 에이전트의 실패를 대비한 훌륭한 안전망입니다. 그러나 역시 근본적으로 '실행이 멈춘 상태(블로킹)'에서 상태를 스냅샷하고 복원한다는 동기적 실행의 틀 내에 있습니다. 사고가 난 도로에서 차가 완전히 멈춰 섰을 때만 블랙박스를 확인할 수 있는 구조와 같습니다.

두 접근 모두 "도구를 부르면 LLM이 멈춘다"는 전제를 의문 없이 수용한 채, 멈춰 있는 동안의 오버헤드를 줄이려 했을 뿐이다.

Section 03

Future-based 비동기 호출이 깨뜨린 것

"Concurrency without Model Changes: Future-based Asynchronous Function Calling for LLMs" 논문은 이 동기성 가정을 정면으로 깨뜨립니다. 핵심 발견을 한 문장으로 요약하면 다음과 같습니다.

기존 함수 호출은 동기적 실행 의미론(Synchronous Execution Semantics)에 구속되어 있어, LLM 디코딩이 도구 실행이 완료될 때까지 차단된다. 이는 모델 아키텍처의 한계가 아니라 실행 의미론의 한계다. — Concurrency without Model Changes, 2025

이 논문의 핵심 기법은 프로그래밍에서 이미 익숙한 Future(또는 Promise) 객체의 개념을 LLM 함수 호출에 도입하는 것입니다. 모델 구조는 그대로 두고, 도구 호출의 의미론(Semantics)만 비동기적으로 바꿉니다.

기존: 동기적 실행
LLM 디코딩
→
Blocking 도구 호출
⏸
Wait 결과 대기
→
Resume 디코딩 재개

디코딩이 도구 완료까지 완전히 멈춤

제안: Future-based 비동기 실행
LLM 디코딩
→
Non-blocking Future 발급
↗
Parallel 도구 실행
↗
LLM 계속 디코딩

디코딩이 멈추지 않고 Future 객체로 결과를 비동기 수신

요리사 비유로 돌아가면, 이건 요리사가 "3번 재료 가져와"라고 말한 뒤 조수가 오는 동안 다른 재료를 다지고, 다음 단계의 소스를 미리 준비하는 것과 같습니다. 요리사는 더 이상 멈추지 않습니다. 조수가 재료를 가져오면, 그때 그 재료를 요리에 투입하면 됩니다.

이 단 하나의 의미론적 변화가 만들어내는 파급효과는 엄청납니다. 도구가 여러 개 필요한 복잡한 에이전트 작업에서, 기존 방식은 도구 하나당 한 번씩 멈춰야 했지만, 비동기 방식은 디코딩과 도구 실행이 동시에 진행되면서 전체 레이턴시가 구조적으로 단축됩니다.

Section 04

컴펜세이션 설계에서 패러다임 전환으로

이제 거꾸로 기존 접근들을 바라보면 명확해집니다. 스키마 축적 최적화도, 체크포인트/복원 런타임도, 결국 동기성 가정이라는 한계를 인프라로 감싸안은 '컴펜세이션(Compensation) 설계'였습니다. 병을 고치는 게 아니라, 증상을 완화하는 대증요법이었던 셈입니다.

기존 패러다임

동기적 인프라 관리
병목을 '더 빠르게' 관리
하네스 = 실행의 감독자
상태 = 정적 스냅샷

새로운 패러다임

비동기적 의존성 조율
병목 자체를 '해소'
하네스 = 컨텍스트의 조율자
상태 = 흐르는 비동기 컨텍스트

Future 기반 비동기 호출이 도입되면 하네스의 역할이 근본적으로 바뀝니다. 더 이상 '멈춘 실행을 얼마나 빨리 재개할 것인가'를 관리하는 관리자가 아니라, 동시에 흘러가는 여러 비동기 작업들의 컨텍스트를 안전하게 유지하고 조율하는 역할이 됩니다.

비동기 에이전트 작업 공간의 등장

이때 비동기 에이전트 작업 공간(Asynchronous Agent Workspace)이라는 개념이 새로운 패러다임의 핵심 빌딩 블록으로 부각됩니다. 기존의 Crab 방식처럼 정지된 상태에서 스냅샷을 찍는 대신, 흘러가는 비동기 컨텍스트와 미해결 Promise(Future)들을 안전하게 보관하고, 해결된 결과를 적절한 타이밍에 디코딩 흐름에 주입하는 공간이 필요합니다.

비유 확장

기존 방식이 "요리사가 멈출 때마다 주방 전체 사진을 찍어두는 것"이었다면, 새로운 방식은 "조수들이 각자 재료를 가져오는 동안, 요리사가 계속 요리하면서 조리대 위의 반쯤 완성된 요리들과 아직 도착하지 않은 재료들의 '예약 자리'를 동시에 관리하는 주방"입니다. 이 주방이 바로 비동기 에이전트 작업 공간입니다.

이 작업 공간은 단순한 버퍼가 아닙니다. 여러 Future 객체 간의 의존성 관계를 추적하고, 어떤 결과가 도착했을 때 디코딩 흐름의 어떤 지점에 주입해야 하는지를 관리하는 능동적인 조율자입니다. 체크포인트가 '과거의 상태를 보존'하는 것이라면, 작업 공간은 '진행 중인 미래를 관리'하는 것입니다.

Section 05 — Conclusion

패러다임을 전환해야 할 때

질문자의 통찰처럼, 하네스의 근본적 병목은 인프라가 아니라 '동기성 가정 자체'에 있었습니다. 스키마 축적 병목도, 체크포인트 오버헤드도, 모두 동기적 블로킹이라는 공통 분모를 가진 하위 문제들이었습니다. 인프라를 아무리 최적화해도, 보이지 않는 상한선 아래에서의 국소적 최적화에 머물 수밖에 없었습니다.

Future-based 비동기 호출은 이 공통 분모를 제거합니다. 그리고 이 제거는 단순히 '빨라진다'는 수준의 개선이 아니라, 하네스의 설계 철학 자체를 바꿔야 한다는 것을 의미합니다.

🎯 패러다임 전환의 핵심 요약

1. 하네스의 역할을 "동기적 실행의 관리자"에서 "비동기적 컨텍스트의 조율자"로 재정의해야 한다.

2. 상태 관리를 정적 스냅샷(Checkpoint)에서 흐르는 비동기 컨텍스트와 미해결 Promise의 동적 관리로 전환해야 한다.

3. 하네스의 핵심 추상화를 비동기 에이전트 작업 공간(Asynchronous Agent Workspace)으로 재설계해야 한다.

4. 최적화의 목표를 '레이턴시 감소'에서 '동시성의 극대화와 의존성의 우아한 조율'로 확장해야 한다.

우리는 "도구를 부르면 LLM이 멈춰야 한다"는 것을 기술적 한계로 받아들였다. 하지만 그것은 한계가 아니라 선택이었고, 더 나은 선택이 가능했음을 이 논문은 보여준다. 진정한 최적화는 병목을 더 빠르게 관리하는 것이 아니라, 병목이 될 구조 자체를 없애는 것에서 시작된다.

에이전트 하네스의 다음 세대는 인프라 엔지니어링의 문제가 아니라, 실행 의미론의 재설계라는 더 근본적인 질문에서 시작되어야 합니다. 그리고 그 질문의 답은 이미 — 프로그래밍의 역사 속에서 동기에서 비동기로 넘어갔던 수많은 패러다임 전환들이 증명하듯 — 우리 앞에 놓여 있습니다.
