직관을 뒤집는 발견 — LLM이 느리다고요? 아닙니다

에이전트 파이프라인이 느리면 우리는 본능적으로 "LLM 추론이 병목이겠지"라고 생각합니다. 수십억 파라미터가 돌아가는 텐서 연산이 가장 무거울 거라고, 직관이 그렇게 말합니다.

하지만 "Tool Attention Is All You Need" 논문은 이 직관을 정면으로 뒤집습니다. 다중 서버 MCP(Model Context Protocol) 배포 환경에서 실증적으로 확인한 바에 따르면, 파이프라인 전체 레이턴시의 진짜 병목은 개별 LLM 호출이 아니라 스키마 축적(Schema Accumulation) 과정에 있습니다.

비유하자면, 이는 요리사가 요리 itself보다 냉장고에서 재료 찾는 데 더 오래 걸리는 상황과 같습니다. 불이 켜지고 프라이팬이 달궈지는 시간보다, "이 냉장고에는 뭐가 있고, 저 냉장고에는 뭐가 있는지" 하나하나 확인하는 데 시간이 더 드는 것이죠.

이 논문이 제시한 해법은 Lazy Schema Loading—필요한 순간에만 스키마를 로드하는 기법입니다. 냉장고 문을 한 번에 다 열어보는 대신, "양파 필요해" 할 때만 양파 칸을 여는 식이죠.

여기까지는 자연스러운 최적화 논리입니다. 병목을 "줄이는" 방향에 집중한 것이죠. 스키마 축적을 해결해야 할 문제로 간주한 관점입니다.

그런데, 완전히 다른 차원의 질문이 등장합니다.

두 가지 "스키마", 두 가지 운명

이 분석에서 가장 함정하기 쉬운 지점이 있습니다. 바로 "스키마"라는 단어가 사실 두 가지 전혀 다른 것을 가리키고 있다는 사실입니다. 이 구분을 명확히 하지 않으면 전체 논의가 오답으로 빠집니다.

① 도구 스키마 (Tool Schema)

함수 시그니처, 파라미터 타입, 반환값 구조 등입니다. MCP 파이프라인에서 축적되는 것이 바로 이것이죠. 예를 들어 search(query: str, limit: int) -> List[Result] 같은 정보입니다.

이것은 에이전트가 도구를 정확히 호출하기 위해 구조적으로 필수적입니다. 이를 제거하면 함수 호출 자체가 실패합니다. 마치 전화번호를 모르면 전화를 걸 수 없는 것과 같죠.

② 의미론적 메타데이터 (Semantic Metadata)

schema.org, Dublin Core 등 사람이 사전에 데이터에 부착한 주석입니다. "이 데이터셋은 기상 관측 데이터이며, 시간 범위는 2020–2023년이고, 지리적 범위는 한국이다" 같은 정보죠.

이것은 데이터의 발견가능성(Discoverability)과 상호운용성(Interoperability)을 위한 것입니다. 전통적으로는 데이터 카탈로그에서 사람이나 시스템이 데이터를 찾을 때 사용되던 경로였죠. 마치 도서관의 각 책에 붙인 분류표와 같습니다.

구분 도구 스키마 의미론적 메타데이터
예시 함수 시그니처, 파라미터 타입 schema.org 태그, Dublin Core
목적 도구를 정확히 호출하기 위해 데이터를 발견·이해하기 위해
제거 시 호출 실패 (blind invocation) 발견 경로 상실 (but 대안 가능?)
본질 구조적 필수 전통적 관행

이 둘을 섞어버리면 "스키마를 없앨 수 있다"는 주장이 "함수 호출을 스키마 없이 할 수 있다"는 터무니없는 해석으로 왜곡됩니다. 반대로, 이 둘을 분리하면 전혀 다른 해석 공간이 열립니다.

10년 신화가 흔들리는 순간 — 의미론적 메타데이터의 진실

여기서 등장하는 논문의 제목부터 도발적입니다.

"Do Agents Need Semantic Metadata? A Comparative Study in Agentic Data Retrieval" — 자율 에이전트 시대에, 의미론적 메타데이터는 정말 필요한가?

이 논문이 던지는 질문의 무게를 이해하려면 배경을 알아야 합니다. schema.org를 비롯한 의미론적 메타데이터는 FAIR 원칙(Findable, Accessible, Interoperable, Reusable)의 근간으로서 10년 넘게 데이터 생태계의 기정사실이었습니다. "데이터에는 메타데이터를 붙여야 한다"는 것은 물이 흐르는 방향처럼 당연한 전제였죠.

이는 마치 도서관에 가면 반드시 사서에게 "이 책은 무엇에 대한 책인가요?"라고 물어봐야만 책을 찾을 수 있다고 믿었던 시대와 같습니다. 그런데 어느 날, 누군가가 직접 책을 펼쳐보지 않고도 제목과 목차만으로 원하는 내용을 찾아내는 방법을 보여준 것이죠.

이 논문이 "에이전트는 schema.org 같은 의미론적 메타데이터 없이도 데이터를 검색할 수 있다"를 긍정적으로 실증한다면, 분석의 축이 근본적으로 바뀝니다.

관점 스키마 축적의 성격 대응 방향
기존 (Tool Attention) 해결해야 할 병목 Lazy loading 등으로 최적화
새로운 (Semantic Metadata) 애초에 불필요한 오버헤드 축적 자체를 회피 또는 제거

같은 "스키마 축적"이라는 말이지만, 기존 관점은 "어떻게 빠르게 모을 것인가"를 묻고, 새로운 관점은 "애초에 왜 모으는가"를 묻습니다. 질문의 레벨이 다릅니다.

병목을 고칠 것인가, 병목을 없앨 것인가

두 논문을 결합하면, 스키마 축적 문제는 이중으로 해체될 수 있습니다. 하지만 이 해체는 단순한 "제거"가 아니라, 각각 다른 논리로 진행되어야 합니다.

  1. 도구 스키마 축적: 여전히 실재하는 병목, 하지만 해법의 확장

    MCP 파이프라인에서 도구 스키마를 수집하는 과정의 병목은 Tool Attention 논문이 실증한 대로 여전히 존재합니다. 이를 무시하면 안 됩니다. 하지만 해법이 "축적을 효율화하는 것"에 머물 필요는 없습니다.

    예를 들어, 에이전트가 도구의 시그니처를 사전에 완전히 로드하지 않고도, 실행 시점에 필요한 파라미터 정보만 동적으로 파악하는 방식—adaptive-inference의 논리를 스키마 로딩으로 확장하는 것—이 가능할 수 있습니다. 정적 축적에서 동적 적응으로의 전환이죠.

    다만 여기에는 명확한 트레이드오프가 존재합니다. 스키마 축적을 과감히 줄이면 blind-tool-invocation—스키마 없이 도구를 맹목적으로 호출하여 실패하는 현상—의 위험이 증가합니다. "어디까지 스키마 없이 가능한가"의 경계선이 바로 핵심 질문입니다.

  2. 의미론적 메타데이터 축적: 완전히 제거 가능한 오버헤드

    반면, 의미론적 메타데이터의 축적은 차원이 다릅니다. 에이전트가 schema.org 없이도 데이터를 검색·이해할 수 있다면, 이것은 단순한 병목이 아니라 파이프라인에서 완전히 제거 가능한 오버헤드입니다.

    10년간 FAIR 원칙의 근간으로 여겨지던 메타데이터 부착·유지보수 비용이 에이전트 파이프라인에서는 순수한 낭비가 됩니다. 도서관의 분류표가 필요 없는 도서관이 되는 셈이죠.

    더 깊이 들어가면, 이 현상은 ai-literacy와도 연결됩니다. AI 리터러시가 "지식 습득"이 아닌 "에이전트 구축 경험"을 통해 체득된다는 관점과 결합하면, 사람 중심 메타데이터 체계가 에이전트 중심 파이프라인에서 불필요해지는 것은 단순한 기술적 최적화가 아니라 "리터러시의 주체 전환"의 인프라적 표현으로 읽힙니다.

"어떻게 빠르게 모을 것인가"에서 "무엇을 모을 것인가"로

스키마 축적은 단일한 현상이 아닙니다. "스키마"라는 하나의 단어 아래 도구 스키마와 의미론적 메타데이터가 숨어 있고, 이 둘은 완전히 다른 논리로 다루어져야 합니다.

도구 스키마의 축적은 여전히 최적화 대상입니다. 하지만 그 해법이 "더 빠르게 모으기"에서 "정적으로 모으지 않기"로 진화할 여지가 열려 있습니다. adaptive-inference가 모델 선택 차원에서 압축 상태 반응(SpecKV)까지 확장된 것처럼, 스키마 로딩도 정적 축적에서 동적 적응으로 전환 가능성이 있습니다.

의미론적 메타데이터의 축적은 더 극단적으로, 불필요한 오버헤드로 전환될 가능성이 구체적으로 열려 있습니다. 10년간의 FAIR 신화가 에이전트 파이프라인에서는 오히려 장애물이 될 수 있다는 것은 패러다임 전환을 의미합니다.

aggregate-pipeline-serving이 "병목 지점을 재정의"했다면,
의미론적 메타데이터 불필요성 연구는
"병목의 존재 자체를 재정의"하는 방향으로
분석을 확장해야 합니다.

결국 두 논문의 결합이 우리에게 요구하는 것은 이것입니다. 파이프라인 최적화의 패러다임을 "어떻게 빠르게 축적할 것인가"에서 "무엇을 축적할 것인가"로 전환하라. 냉장고 문을 더 빨리 여는 기술을 고민하기 전에, 애초에 어떤 냉장고 문을 열 필요가 있는지 다시 물어야 한다는 것입니다.
