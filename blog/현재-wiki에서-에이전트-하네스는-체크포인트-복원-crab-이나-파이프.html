에이전트 하네스 분석

코드가 하네스라면, 인프라는 무엇인가

에이전트 하네스 엔지니어링을 단순한 '코드 생성' 문제로 환원할 수 있는가? 코드의 의미론적 구조와 실행 인프라의 물리적 한계 사이에 놓인 변환 간극을 추적한다.

에이전트 하네스 엔지니어링 Code as Agent Harness 의미론적 간극

00하네스라는 말의 무게

말(harness)은 원래 말이나 개가 끄는 수레에 힘을 전달하기 위해 몸에 거는 구속구를 뜻한다. 에이전트가 수행해야 할 작업을 실제로 수행할 수 있도록 '묶어주는' 장치 — 그것이 하네스다. 이 장치가 체크포인트/복원 런타임(Crab)이거나 파이프라인 서빙 체계(Scepsy)라면, 우리는 자연스럽게 하네스를 '외부 실행 인프라'로 이해하게 된다.

그런데 최근 등장한 "Code as Agent Harness"라는 논문은 이 직관을 뒤흔든다. 코드 자체가 에이전트의 운영 기질(substrate)이자 하네스로 기능한다면, 하네스 엔지니어링은 인프라 설계가 아니라 코드 생성 자체의 문제로 환원되는 것 아닌가?

답부터 말하자면: 환원되지 않는다. 하지만 단순히 "아니다"라고 끝내기엔 이 질문이 드러내는 지도가 너무 흥미롭다. 코드와 인프라 사이의 경계가 어떻게 붕괴되고, 왜 완전히 사라지지는 않는지 — 세 가지 국면에서 추적해 보자.

01인프라의 물리적 한계: 완벽한 코드도 구원하지 못하는 것

비유: 완벽한 설계도와 현장의 벽

건축가가 구조적으로 완벽한 설계도를 그렸다고 상상해 보라. 그러나 시공 현장의 지반은 느슨하고, 콘크리트 배합 비율은 제한되어 있다. 설계도가 아무리 정교해도, 그것을 물리적 재료로 번역하는 과정에서 정보가 손실된다. 에이전트의 코드가 설계도라면, OS 수준의 실행 환경이 바로 그 현장이다.

Wiki의 Crab 논문이 보여주는 것은 하네스의 핵심 구성요소 — 결함 내성(fault tolerance), 롤백(rollback), 분기(branching) — 가 단순히 애플리케이션 로직 안에 머무르지 않는다는 사실이다. 이들은 C/R(Checkpoint/Restore) 런타임이라는 실행 환경의 층위까지 확장되어야만 제 기능을 한다.

에이전트가 가진 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS가 제공하는 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 근본적인 비효율이 발생한다.

— [[agent-os-semantic-gap]]

여기서 '의미론적 의존 그래프'와 '전체 스냅샷' 사이의 격차가 핵심이다. 에이전트는 "이전 단계에서 생성된 요약 텍스트에 의존한다"는 식으로 의미론적으로 상태를 정의하지만, OS의 체크포인트는 메모리 전체를 뭉텅이로 찍어낸다. 마치 "이 문단의 핵심 주장만 저장해 줘"라고 부탁했는데, 책 전체를 복사해서 건네주는 것과 같다.

핵심 포인트

에이전트의 의도를 완벽하게 담은 하네스 코드를 생성했다 하더라도, 이를 OS 수준에서 의미론적으로 보존하고 복원하는 인프라의 한계가 병목으로 작동한다. 이것이 하네스 문제가 단순한 코드 생성으로 환원될 수 없는 첫 번째 이유다.

02런타임 병목의 재정의: 코드 밖에서 결정되는 속도

비유: 요리사의 손도끼와 주방 설비

아무리 빠르고 정교한 요리사(에이전트 코드)가 있어도, 조리 도구가 수십 개씩 부엌에 흩어져 있고 그것을 하나하나 꺼내서 설명서를 읽어야 한다면(스키마 축적), 요리 속도는 요리사의 실력이 아니라 주방의 구조에 의해 결정된다.

"Tool Attention Is All You Need" 논문이 다중 서버 MCP(Model Context Protocol) 배포 환경에서 발견한 것은 정확히 이 지점이다. 파이프라인 전체의 레이턴시 병목은 개별 LLM 호출의 효율성도, 에이전트 코드의 알고리즘적 우수성도 아니었다. 병목은 스키마 축적(schema accumulation)에 의해 결정되었다.

에이전트가 사용할 수 있는 도구가 열 개든 백 개든, 각 도구의 스키마(이 도구가 무엇을 하고, 어떤 인자를 받는가)를 파이프라인이 취합하여 LLM에 전달하는 과정 자체가 서빙 인프라의 구조적 문제인 것이다.

에이전트 코드
최적화 가능 영역
→
스키마 축적
병목 발생 지점
→
파이프라인 서빙
인프라 구조 문제

이 발견의 의미는 명확하다. 에이전트 코드를 아무리 정교하게 생성해도, 그 코드가 호출하는 도구들의 스키마를 파이프라인이 어떻게 취합하고 전달하는가 — 이 서빙 인프라의 구조가 정체해 버리면 코드의 효율성은 무의미해진다. 하네스 엔지니어링이 코드 생성만으로 완성될 수 없음을 실험적으로 증명하는 사례다.

03코드가 곧 하네스가 되는 지점: 경계 붕괴의 시작

그렇다면 인프라가 항상 승리하는가? 그렇지 않다. 질문자의 통찰이 가리키는 방향 — 코드 자체가 하네스를 대체할 수 있다는 가능성 — 은 Wiki 내에도 명시적으로 존재한다.

논문
Code as Agent Harness

LLM의 강력한 코드 이해 및 생성 능력이 에이전트 하네스 구축의 핵심 매개체가 될 수 있음을 시사하는 연구. 인프라를 별도로 구축하는 대신, 코드 자체가 실행 환경의 역할을 흡수하는 경로를 제시한다.

이 접근의 핵심은 LLM이 생성한 코드가 단순히 '로직'만 담는 것이 아니라, 상태 관리, 오류 복구, 분기 제어 같은 전통적으로 인프라가 담당하던 기능까지 자체적으로 수행할 수 있다는 발견에 있다. 마치 자동차의 엔진이 스스로 서스펜션까지 통제하기 시작하는 것과 같은 패러다임 전환이다.

그러나 여기에도 중요한 조건이 붙는다. RunAgent 논문([[agentic-language]])이 보여주듯, 이 코드 기반 하네스는 정적인 산출물이 아니어야 한다. 자연어로 표현된 계획을 제약 기반 실행(constraint-based execution)으로 동적으로 번역하는 계층으로서 기능해야 한다.

즉, "코드가 하네스다"라는 명제는 "코드를 출력하고 끝나는 것"이 아니라 "코드가 런타임에서 계속해서 의미론적 번역을 수행하는 것"을 의미한다. 정적 코드와 동적 번역 계층의 결합 — 여기에 진짜 혁신이 있다.

04변환 간극의 재정의: 경계 붕괴가 낳은 새로운 문제

지금까지의 논의를 종합하면, 하네스 엔지니어링의 정의는 두 축 사이에서 분열되어 있는 것을 알 수 있다.

인프라 중심 코드 중심
Crab — C/R 런타임에 의존 Code as Agent Harness — 코드가 흡수
Scepsy — 파이프라인 서빙 구조 RunAgent — 동적 번역 계층
OS 수준의 체크포인트 메커니즘 제약 기반 실행으로의 번역
물리적 한계가 병목 의미론적 번역이 핵심

이 분열을 통합하기 위해, 기존 [[algorithm-system-translation-gap]]의 의미를 확장할 필요가 있다. 원래 이 개념은 '알고리즘을 시스템으로 번역하는 간극'으로 정의되어 있었다. 그러나 "Code as Agent Harness"의 등장으로 인해, 이 간극은 이제 다르게 읽혀야 한다.

생성된 에이전트 코드(알고리즘)가 스스로 실행 인프라(시스템)의 역할을 모호하게 흡수하면서 발생하는 경계 붕괴 — 이것이 번역 간극의 새로운 정의다.

경계가 붕괴된다는 것은 문제가 사라진다는 뜻이 아니다. 오히려 문제의 성격이 변한다. 코드가 인프라의 역할을 어느 정도까지 대리할 수 있는가? 그 대리의 한계선은 어디에 그어지는가? 이 물음이 하네스 엔지니어링의 새로운 중심 과제가 된다.

재정의된 변환 간극
경계 붕괴적 번역 간극

기존: 알고리즘 → 시스템 (단방향 번역의 손실)

새로움: 알고리즘 ⟷ 시스템 (코드가 시스템을 흡수하면서 발생하는 경계의 모호성과 그 모호성이 낳는 새로운 병목)

패러다임 전환: '코드냐 인프라냐'에서 '대리의 한계'로

하네스 엔지니어링을 코드 생성으로 완전히 환원할 수 없다는 결론에 도달했다. 하지만 이 부정적 결론보다 중요한 것은, 어떤 질문으로 전환해야 하는가다.

기존 패러다임은 이분법적이었다. 하네스는 인프라인가, 코드인가? Crab과 Scepsy는 전자를, Code as Agent Harness는 후자를 선택하는 것처럼 보였다.

그러나 본 분석이 보여준 것은 이 이분법 자체가 무효화되고 있다는 사실이다. 진짜 문제는 "생성된 코드가 어디까지 인프라의 의미론적 상태를 대리할 수 있는가"라는 질문에 있다. 이 질문은 [[agentic-harness-engineering]]과 [[agent-os-semantic-gap]]의 교차점에 위치하는 독자적인 문제 영역을 구성한다.

구체적으로 무엇을 바꿔야 하는가

  1. 평가 기준의 전환: 하네스의 품질을 '코드의 완성도'가 아니라 '코드가 대리하는 인프라 상태의 의미론적 보존도'로 측정해야 한다.
  2. 설계 공간의 재구성: 코드-인프라 연속체 위에서 하네스를 설계해야 한다. 어디까지 코드로, 어디서부터 외부 인프라로 — 이 경계를 의도적으로 설계하는 것이 핵심이다.
  3. 번역 간극의 조작: 알고리즘-시스템 간극을 '최소화'하는 것에서 '관리 가능한 모호성으로 유지'하는 것으로 패러다임을 전환해야 한다. 경계가 완전히 사라지는 것이 아니라, 의도적으로 느슨한 경계를 설계하는 것이 목표가 된다.

에이전트 하네스 엔지니어링은 코드 생성도, 인프라 설계도 아니다. 코드가 인프라를 어디까지 대리할 수 있는가를 탐구하는 경계 과학이다. 이 과학의 지도가 이제 막 그려지기 시작했다.
