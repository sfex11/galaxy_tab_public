ZKP 멀티에이전트 특허 안전

2026년 3월, ZKP(제로지식 증명) 기반 프라이버시 보존 자율 협상 시스템 특허를 출원하기 전 선행기술 조사를 진행했습니다. 첫 번째 조사(9개 출처)에서는 "경쟁사 제로"라는 결론을 얻었지만, 출처를 18개로 늘리자 핵심 선행기술이 발견되었습니다.

이 글에서는 이 특허 사례를 Wiki의 멀티에이전트 안전 진단 대시보드 관점에서 분석합니다: ZKP 협상이 에이전트 생태계의 어떤 취약점을 해결하고, 어떤 새로운 위험을 도입하는지.

📅 조사 타임라인

3/27 — 1차 조사

web_search 5회, 9개 출처. DAO-Agent가 가장 근접했지만 기여도 증명(Shapley value)이지 협상 프로토콜이 아님.

결론: "경쟁사 제로, 신규성 유지" ← 이 결론은 틀렸다.

3/28~30 — 스킬 보강 + 검색 엔진 차단

DuckDuckGo 5회 연속 검색 후 CAPTCHA. Brave도 PoW CAPTCHA. Google Patents는 JS 렌더링 불가.

해결: OpenAlex API — 무료, rate limit 완화, JSON 응답. "zero knowledge" + "agent negotiation" + "privacy"로 277건 탐색.

3/31 — 🔴 치명적 발견

arXiv:2601.00911 — "Device-Native Autonomous Agents for Privacy-Preserving Negotiations" (Joyjit Roy, 2026.01.01, IEEE SoutheastCon 2026)

Groth16 zk-SNARK + Paillier 암호화 + 10라운드 교대 오퍼 + Nash bargaining 근사 종료. 증명 생성 80ms.

3/31 — 비교 분석 → 살아있다

꼼꼼히 비교하니 결정적 차이 확인. 청구항 전략 수정으로 신규성 유지 가능.

등록 가능성: 70-80% (이전: "높음")

⚠️ 핵심 모순: ZKP는 "프라이버시 보호"와 "투명성/감독" 사이의 근본적 트레이드오프를 드러냅니다. 에이전트의 사적 선호를 보호하면 감독이 불가능하고, 감독을 가능하게 하면 프라이버시가 사라집니다. 이것이 멀티에이전트 거버넌스의 가장 어려운 문제입니다.

💡 교훈: 특허 조사 + Wiki 시너지

교훈 1: "경쟁사 제로"는 검색 불충분의 증거
9개 출처로 "제로"라고 확신한 게 실수였습니다. 18개로 늘리자 핵심 선행기술이 나왔습니다. Wiki의 안전 대시보드에서도 동일한 패턴: "신뢰/검증" 논문이 53편뿐인데, 이 중 실제 해결책을 제시하는 건 단 2편입니다. 대부분의 논문은 "문제가 있다"고 말하지만 "어떻게"는 침묵합니다.
교훈 2: OpenAlex API는 에이전트 도구로 강력
검색 엔진 차단 상황에서 OpenAlex API가 구세주였습니다. 무료, rate limit 완화, JSON 응답. Wiki의 [[tool-use]] 카테고리에 추가할 만한 도구입니다. 특허 조사뿐 아니라 논문 수집 파이프라인의 보조 소스로도 활용 가능합니다.
교훈 3: 한 줄 차이가 특허의 생사를 가른다
"제약 만족 증명" vs "Pareto 최적성 + 개인 합리성 증명" — 수학적으로 완전히 다른 문제지만, 표면적으로는 비슷합니다. 심사관이 구분할 수 있도록 청구항을 정밀하게 작성하는 것이 핵심입니다. 이것은 LLM 에이전트가 특허 분석을 수행할 때의 [[reasoning-integrity]] 문제와도 연결됩니다.
교훈 4: Wiki 기반 특허 분석의 가치
이 분석은 단일 특허 사례를 넘어, Wiki에 축적된 287편의 지식으로 ZKP 협상이 멀티에이전트 생태계 전체에 미치는 영향을 시스템적으로 평가했습니다. 개별 특허 분석에 문헌 기반 위험 평가를 결합하는 접근은, AI 거버넌스가 점점 중요해지는 환경에서 강력한 경쟁 우위가 될 수 있습니다.
