하나의 상자에서 여러 상자로:
에이전트 복구의 패러다임 전환

단일 샌드박스 롤백은 왜 크로스-디바이스 환경에서 무력해지는가, 그리고 복구의 기본 단위를 어떻게 재정의해야 하는가

Section 01

체크포인트의 함정: 스냅샷이 풀 수 없는 문제

에이전트 하네스(Agent Harness)의 결함 내성(Fault Tolerance)은 지금까지 단일 샌드박스 안에서의 C/R(Checkpoint/Restore)로 정의되어 왔다. Crab, DeltaBox 같은 시스템은 에이전트의 실행 상태를 어느 시점으로든 되돌릴 수 있도록 인프라 계층에서 스냅샷을 찍고, 실패 시 그 스냅샷으로 복원하는 방식을 취한다. 얼핏 완벽해 보인다.

하지만 이 접근법에는 근본적인 의미론-시스템 번역 간극(Algorithm-System Translation Gap)이 내재되어 있다. 에이전트가 "어떤 의미론적 상태에 의존하고 있는가"와 운영체제가 "어떤 전체 스냅샷을 찍는가"는 전혀 다른 질문이다.

에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 발생하는 비효율이 간극의 본질이다. — algorithm-system-translation-gap, Crab 분석 중

단일 샌드박스 내에서조차 이 불일치는 존재한다. 에이전트가 "문서의 세 번째 문단 내용"에 의존하고 있을 때, 전체 메모리 스냅샷을 찍어 되돌리는 것은 과도한 작업일 뿐 아니라, 정확히 어떤 의미론적 단위가 손상되었는지에 대한 정보를 잃어버린다. 문제는 단일 환경에서는 이 비효율이 "비용"으로만 나타나지만, 환경이 여러 개로 나뉘면 이것이 "불가능"으로 변한다는 점이다.

Section 02

부분 복구 불가능성: 여러 상자 사이의 균열

직관적 비유

주방에서 요리를 한다고 상상해 보자. 냉장고에서 재료를 꺼내고, 조리대에서 썰고, 프라이팬에 볶고, 마지막으로 접시에 담는다. 이 네 단계는 각각 다른 "장소"에서 일어난다. 만약 프라이팬에서 볶는 단계에서 실패했다고 해서, 냉장고 상태만 되돌릴 수 있다면 무슨 소용이 있겠는가? 조리대는 이미 썬 재료로 어질러져 있고, 접시도 꺼내져 있다. 하나의 장소만 되돌리면 전체 흐름이 끊어진다.

크로스-디바이스 환경에서 에이전트가 수행하는 작업은 정확히 이 구조다. 논문 "Beyond Global Replanning: Hierarchical Recovery for Cross-Device Agent Systems"가 지적하듯, 실세계 컴퓨터 사용 작업은 여러 애플리케이션과 디바이스에 걸쳐 수행된다.

여기서 단일 샌드박스의 롤백을 적용하면 부분 복구 불가능성(Partial Irrecoverability)이 발생한다. 디바이스 A의 상태만 이전 체크포인트로 되돌리면, 디바이스 B와의 의존성 그래프가 단절된다. A는 "이전 상태"이고 B는 "이후 상태"인 비일관성(Inconsistency) 상태에 빠진다.

간극의 기하급수적 증폭

단일 노드 내에서의 의미론-시스템 번역 간극은 다중 노드 토폴로지에서 "노드 간 의존성 그래프 보존 여부"라는 더 높은 차원의 간극으로 증폭된다. 단일 OS의 semantic gap은 크로스-디바이스에서 "OS 간 semantic gap의 상호 불일치"로 일반화된다.

기존 시스템이 취할 수 있는 대응은 두 가지뿐이다. 전체를 처음부터 다시 시작하는 Global Replanning이거나, 아무것도 못 하고 실패를 인정하는 것이거나. 전자는 비용이 너무 크고, 후자는 허용할 수 없다. 중간 지점이 필요하다.

Section 03

그래프-바운드 복구: 의존성을 복구 단위로 삼기

필요한 것은 복구의 기본 단위를 재정의하는 일이다. "샌드박스 상태"가 아니라 "크로스-디바이스 작업 의존성 그래프"를 복구의 원자적 단위로 삼아야 한다.

샌드박스 A
전체 스냅샷
샌드박스 B
전체 스냅샷
샌드박스 C
전체 스냅샷
↓
의존성 그래프 G
(A→B→C의 의미론적 연결)
복구 단위의 전환: 개별 스냅샷 → 그래프-바운드 상태 캡슐

논문 "Execution-State Capsules: Graph-Bound Execution-State Checkpoint and Restore"가 제시하는 방향은 정확히 이 확장의 구체적 기술적 경로다. 개별 샌드박스 스냅샷이 아니라, 실행 상태 간 의존 관계를 그래프로 명시적으로 바인딩하는 것이다.

핵심 아이디어는 간단하면서도 강력하다. 체크포인트가 "어떤 샌드박스의 전체 상태"가 아니라 "어떤 의존성 그래프의 특정 노드 집합이 일관된 상태를 유지하고 있는가"를 기록한다. 복원 시에도 개별 상태를 되돌리는 것이 아니라, 그래프 상에서 일관성이 깨진 하위 그래프(subgraph)를 원자적으로 되돌린다.

계층적 복구(Hierarchical Recovery)의 등장

"Beyond Global Replanning" 논문이 제안하는 계층적 복구는 이 그래프-바운드 접근과 자연스럽게 결합된다. 실패가 발생했을 때:

  1. 가장 낮은 계층에서 실패 노드의 로컬 상태만 복원 시도
  2. 의존성 그래프 상에서 일관성이 복원되지 않으면, 상위 계층으로 이동하여 관련 하위 그래프 전체를 복원
  3. 그래도 불가능하면 최후의 수단으로 Global Replanning

이것은 "전부 또는 무엇(All-or-Nothing)"에서 "가장 좁은 범위부터 점진적 확장"으로의 전환이다.

Section 04

새로운 연결망: Wiki 간극의 다중 노드 일반화

이 문제를 Wiki에 이미 구조화된 간극들의 관점에서 보면, 단일 진단이 다중 노드 환경에서 어떻게 일반화되는지 명확히 드러난다.

연결 설명
번역 간극 → 크로스-디바이스 토폴로지 단일 노드 내 의미론-시스템 번역 간극이 다중 노드에서는 "노드 간 의존성 그래프 보존 여부"라는 더 높은 차원의 간극으로 증폭됨
하네스 공학 ↔ 비동기 의존성 조율 하네스가 결함 내성을 인프라 계층에서 구현한다면, 크로스-디바이스에서는 이것이 필연적으로 비동기 의존성 조율 문제와 결합됨
Agent-OS Semantic Gap의 일반화 단일 OS의 semantic gap이 크로스-디바이스에서는 "OS 간 semantic gap의 상호 불일치"로 일반화됨
C/R 효율 스펙트럼의 재정의 기존 스펙트럼이 단일 샌드박스 내 효율성에 집중되어 있다면, 크로스-디바이스 의존성 그래프를 복구 단위로 삼을 때 축 자체가 확장되어야 함

특히 네 번째 연결이 중요하다. 기존의 Checkpoint-Restore Efficiency Spectrum은 "얼마나 적은 데이터로 빠르게 스냅샷을 찍고 복원하는가"라는 단일 축 위에 있었다. 하지만 크로스-디바이스 환경에서는 이 스펙트럼의 축 자체가 달라져야 한다.

스펙트럼 축의 확장

기존 축: 스냅샷 크기 ← → 복원 속도 (단일 환경)

확장 축: 복구 범위의 정밀도 ← → 그래프 일관성 보장 수준 (다중 환경)

작은 스냅샷으로 빠르게 복원해도 그래프 일관성이 깨지면 무의미하다. 반대로 그래프 일관성을 완벽히 보장해도 복구 범위가 너무 넓으면 Global Replanning과 다를 바 없다. 이 두 축 사이의 균형점이 새로운 효율성의 기준이 된다.

Section 05

패러다임 전환: "하나의 상자"에서 "상자들 사이의 관계"로

Crab과 DeltaBox가 해결한 것은 명확하다. "하나의 상자 안에서 어떻게 되돌리는가." 그리고 그것은 단일 환경에서 충분히 유효한 해결책이었다.

하지만 크로스-디바이스 환경이 요구하는 것은 "여러 상자 사이의 의존성을 어떻게 원자적으로 되돌리는가"다. 이것은 단순한 규모의 확장(Scale-up)이 아니라, 질적 전환(Paradigm Shift)이다.

복구 단위를 '샌드박스 상태'에서 '크로스-디바이스 작업 의존성 그래프'로 확장해야 한다는 주장은, Wiki의 algorithm-system-translation-gap이 단일 환경에서 이미 진단한 문제의 자연스러운 다중 노드 일반화다.

구체적으로 무엇이 바뀌어야 하는가

  • 체크포인트의 정의: "어떤 시점의 전체 상태"에서 "어떤 의존성 그래프 하위 집합의 일관된 상태"로
  • 복원의 원자성: "단일 샌드박스 내 원자성"에서 "그래프 경계 내 원자성"으로
  • 복구 전략: "전체 스냅샷 롤백 또는 전체 재계획"에서 "계층적 그래프 복구"로
  • 효율성의 기준: "스냅샷 크기와 속도"에서 "복구 범위의 정밀도와 그래프 일관성의 균형"으로

Execution-State Capsules의 그래프-바운드 접근과 Hierarchical Recovery의 계층적 구조가 만나는 지점이 바로 이 전환의 구체적 출발점이다. 에이전트 시스템이 단일 환경의 한계를 넘어 실세계의 다중 디바이스 환경에서 작동하려면, 복구의 시야를 상자 안에서 상자들 사이로 옮기는 것이 더 이상 선택이 아니라 필수가 된다.

결국 이 문제의 본질은 에이전트 공학이 "상태"에서 "관계"로 초점을 이동해야 한다는 더 넓은 방향성과 맞닿아 있다. 상자 안의 상태를 관리하는 것에서, 상자들 사이의 의존성 그래프를 관리하는 것으로. 그것이 크로스-디바이스 에이전트 시스템의 결함 내성이 나아가야 할 다음 지점이다.
