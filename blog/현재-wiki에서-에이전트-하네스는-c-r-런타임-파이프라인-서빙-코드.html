하네스의 절반만 보고 있었다

에이전트 하네스를 '실행 환경'에서 '선언적 렌즈'로 재정의해야 하는 이유

2025년 7월 · 읽기 약 12분 · 기반 논문: Probe-and-Refine Tuning of Repository Guidance
Section 01

우리가 '하네스'라고 부르는 것의 실체

에이전트 하네스(Agent Harness)라는 용어를 처음 접하면, 대부분의 사람은 에이전트가 돌아가는 그릇을 떠올립니다. Crab 논문이 제시한 C/R(Checkpoint/Restore) 런타임, 결함 내성(fault tolerance), 롤백, 분기(branching) 메커니즘—이런 것들이 하네스의 핵심이라고 알려져 왔습니다.

Tool Attention 논문은 파이프라인 서빙의 레이턴시 병목이 스키마 축적에 의해 결정된다는 사실을 보여주며, 하네스를 런타임 서빙 최적화 문제로 프레이밍했습니다. 현행 Wiki의 agentic-harness-engineering 엔트리 역시 "애플리케이션 로직에서 실행 환경까지 확장"하는 방향으로 기술되어 있습니다.

정리하면, 현재의 정의는 일관되게 하네스 = 실행 인프라라는 방정식을 따릅니다.

현행 정의의 방정식

C/R 런타임 + 파이프라인 서빙 + 코드 기질(substrate) = 하네스(실행 환경)

이 방정식에서 하네스는 에이전트가 어떻게 돌아가는지를 결정하는 인프라 계층입니다. 마치 자동차의 엔진과 변속기 같은 존재죠.

하지만 이 프레이밍에는 결정적인 맹점이 하나 있습니다. 엔진이 아무리 훌륭해도, 운전자가 지도를 읽을 줄 모르면 차는 목적지에 도달하지 못합니다. 하네스 논의에서 바로 이 '지도 읽기'에 해당하는 부분이 누락되어 있습니다.

Section 02

코드에 없는 지식이 행동을 규정한다

"Probe-and-Refine Tuning of Repository Guidance for Coding Agents"라는 논문은 이 맹점을 정면으로 찌릅니다. 이 논문의 핵심 발견을 한 문장으로 요약하면 다음과 같습니다.

코딩 에이전트는 저장소에 대한 상위 수준의 운영 지식—어떤 파일이 어떤 하위 시스템을 담고 있는지, 테스트 스위트를 어떻게 실행하는지, 역사적으로 어떤 워크플로우가 잘못된 수정으로 이어졌는지—을 필요로 한다. — Probe-and-Refine Tuning of Repository Guidance for Coding Agents

여기서 주목해야 할 단어는 "코드에 존재하지 않는다"입니다. 이 지식은 소스 코드 안에 녹아있지 않습니다. AGENTS.md, CONTRIBUTING.md, 프로젝트 고유의 암묵적 컨벤션, 과거 PR 리뷰에서 축적된 "이렇게 하면 안 된다"는 경험적 지식—이런 선언적 메타데이터로만 존재합니다.

비유: 요리책과 주방 설비

훌륭한 주방 설비—산업용 오븐, 정밀한 온도 계산기, 자동 절단기—를 갖추는 것만으로 요리가 완성되지 않습니다. 레시피가 없으면 요리사는 재료를 보고도 무엇을 만들어야 할지 결정하지 못합니다.

지금까지의 하네스 논의는 주방 설비(실행 인프라)에만 집중했습니다. 하지만 실제로 요리사의 행동을 규정하는 것은 레시피(저장소 운영 지식)입니다. 설비가 아무리 정교해도 레시피가 없으면, 요리사는 재료를 해석하지 못하고 무작위로 조작하게 됩니다.

논문이 보여주는 실험 결과는 이를 명확히 뒷받침합니다. 저장소 지침(Repository Guidance)이 없는 코딩 에이전트는 코드베이스의 구조를 파악하지 못해 관련 없는 파일을 수정하고, 잘못된 테스트 명령어를 실행하며, 이미 실패로 검증된 패턴을 반복합니다. 실행 인프라가 아무리 견고해도, 이 "해석의 부재"는 인프라 수준에서 해결할 수 없는 문제입니다.

Section 03

선언적 렌즈: 하네스의 숨겨진 반쪽

그렇다면 하네스를 어떻게 재정의해야 할까요? 제안하는 방향은 명확합니다.

실행 하네스
C/R 런타임 · 파이프라인 · 코드 기질
지금까지 "하네스"라고 불렀던 것
+
해석 하네스
AGENTS.md · 컨벤션 · 저장소 지식
지금까지 누락되어 있던 것
=
하네스 (재정의)
에이전트가 코드를 해석하는 선언적 렌즈

핵심 개념 전환이 일어나는 지점은 여기입니다. 하네스를 "에이전트가 돌아가는 그릇"에서 "에이전트가 코드를 해석하는 렌즈"로 바꾸는 것입니다.

렌즈(lens)라는 비유가 정확한 이유가 있습니다. 돋보기가 없으면 글씨는 그대로 존재하지만 읽을 수 없습니다. 마찬가지로, 코드는 그대로 존재하지만 AGENTS.md라는 렌즈가 없으면 에이전트는 코드를 해석하지 못합니다. 렌즈의 곡률(저장소 지침의 정확도)에 따라 에이전트가 보는 코드의 의미가 완전히 달라집니다.

이 재정의는 Wiki에 이미 존재하는 code-as-agent-harness라는 슬러그가 정확히 가리켜야 할 내용입니다. 현재 이 엔티티는 상세히 기술되지 않은 채 방치되어 있는데, 바로 이것이 실행 하네스와 해석 하네스를 통합하는 브릿지 개념이 되어야 합니다.

재정의된 하네스의 방정식

(실행 인프라: C/R, 파이프라인) + (해석 프레임워크: AGENTS.md, 컨벤션) = 하네스 (에이전트가 코드베이스와 상호작용할 때 사용하는 해석적 프레임워크 전체)

Section 04

간극의 지도: 세 가지 연결이 말하는 것

이 재정의는 Wiki의 기존 간극 엔티티들과 놀라운 구조적 유사성을 보여줍니다. 세 가지 연결을 따라가 보겠습니다.

첫째, 알고리즘-시스템 번역 간극의 확장

algorithm-system-translation-gap은 Crab 논문에서 보여준 문제를 담고 있습니다: 에이전트의 의미론적 의존 그래프를 OS의 체크포인트 메커니즘으로 번역할 때 발생하는 비효율. 그런데 동일한 구조가 코드-메타데이터 수준에서도 발생합니다.

저장소의 의미론적 구조("이 모듈은 인증을 담당한다")를 에이전트의 행동 결정("인증 관련 파일만 수정하라")으로 번역하는 간극. AGENTS.md는 바로 이 번역 간극을 메우는 선언적 인터페이스입니다. 즉, 이 간극이 알고리즘-시스템 수준에만 국한되지 않고 코드-메타데이터 수준에서도 동일하게 존재함을 명시해야 합니다.

둘째, 에이전트 언어와 저장소 지침의 통합

agentic-language 엔트리는 RunAgent가 자연어 계획을 제약 기반으로 해석한다는 내용을 담고 있습니다. 그런데 저장소 지침(AGENTS.md) 역시 자연어로 작성된 선언적 제약입니다.

두 개념의 본질이 동일합니다. 저장소 지침은 agentic-language의 코드베이스 도메인에 국한된 특수한 형태—즉, 도메인 특화 제약 언어입니다. 현재 두 엔티티가 분리되어 기술되어 있지만, 사실상 동일한 현상의 상보적 기술이거나 통합되어야 합니다.

셋째, OS 의미론 간극과 해석 간극의 대칭

agent-os-semantic-gap은 OS 의미론과 에이전트 의미론 사이의 간극이 실행 계층에 있다고 설명합니다. 새로운 재정의는 여기에 대칭적인 간극을 하나 더 추가합니다: 코드 의미론과 저장소 지침 의미론 사이의 간극은 해석 계층에 있다는 것입니다.

간극 유형 위치 메우는 매커니즘
알고리즘-시스템 번역 간극 실행 계층 Crab의 C/R 런타임
코드-메타데이터 번역 간극 해석 계층 AGENTS.md (저장소 지침)
OS-에이전트 의미론 간극 실행 계층 에이전트 OS 추상화
코드-지침 의미론 간극 해석 계층 선언적 렌즈 (해석 하네스)

이 표가 보여주듯, 기존 간극 지도에는 해석 계층의 간극이 완전히 누락되어 있었습니다. 하네스를 재정의한다는 것은 단순히 용어를 바꾸는 것이 아니라, 간극의 지도 자체를 갱신하는 작업입니다.

Section 05

패러다임 전환: 그릇에서 렌즈로

지금까지의 분석을 하나의 문장으로 압축하면 이렇습니다.

현행 하네스 정의는 에이전트가 "어떻게" 돌아가는지의 절반만 기술하고 있으며, 에이전트가 "무엇을" 해석하는지의另一半을 누락하고 있다.

이것은 단순한 정의 수정이 아닙니다. 엔지니어링 실천의 패러다임 전환입니다.

기존 패러다임에서 무엇이 바뀌어야 하는가

  1. 하네스 엔지니어링의 범위 확장. agentic-harness-engineering는 이제 실행 인프라 설계뿐 아니라, 저장소 지침의 설계·튜닝·검증을 포함해야 합니다. "Probe-and-Refine" 논문이 보여주듯, 지침의 품질은 인프라의 품질과 동등한 차원의 엔지니어링 문제입니다.
  2. code-as-agent-harness의 중심화. 이 슬러그를 부가적 개념에서 하네스 재정의의 중심 엔티티로 격상시켜야 합니다. "코드 자체가 하네스다"라는 명제는, 코드 + 저장소 지식이 합쳐져 에이전트의 해석적 프레임워크를 구성한다는 의미로 구체화되어야 합니다.
  3. 에이전트 언어와 저장소 지침의 통합 기술. agentic-language와 저장소 지침을 분리해서 다루는 것은 인위적입니다. "에이전트 언어의 도메인 특화 인스턴스로서 저장소 지침"이라는 통합 프레임워크가 필요합니다.
  4. 평가 지표의 이중화. 하네스의 성능을 측정할 때, 런타임 레이턴시나 결함 내성률(실행 하네스 지표)과 함께, 에이전트의 코드 해석 정확도·관련 없는 수정률·지침 준수율(해석 하네스 지표)을 동등하게 측정해야 합니다.
전환의 핵심 문장

하네스는 에이전트가 돌아가는 그릇이 아니라, 에이전트가 코드를 읽어내는 렌즈다. 렌즈의 곡률을 엔지니어링하지 않으면, 아무리 정교한 그릇도 빈 채로 남는다.

"Probe-and-Refine Tuning of Repository Guidance for Coding Agents"가 우리에게 던지는 메시지는 분명합니다. 저장소 운영 지식이 에이전트 행동을 규정하는 메커니즘은 실행 인프라와 동등한 하네스의 핵심 축이다. 이것을 인정하는 순간, 하네스에 대한 우리의 이해는 비로소 온전해집니다.

절반만 보고 있었다는 사실을 인정하는 것이 패러다임 전환의 첫 단계입니다.
