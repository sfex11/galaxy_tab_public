에이전틱 SWE에서 우리가 진정한 병목을 놓치고 있는 방식, 그리고 SpecBench가 던지는 질문

📋 SpecBench 논문 분석 ⏱ 약 12분 읽기

Section 01

우리는 이미 병목을 잘못 짚은 적이 있다

건축을 떠올려 보자. 공사장에서 벽을 세우는 속도가 느리다고 불평하는데, 알고 보니 실제 문제는 설계도에 치수가 빠져 있었다면 어떨까? 노동자가 아무리 빨라져도, 크레인이 아무리 성능이 좋아져도, 잘못 그려진 도면을 바탕으로는 제대로 된 집을 지을 수 없다.

이런 일이 실제로 AI 시스템 연구에서 일어났다. 다중 서버 MCP 배포 환경에서 연구자들은 당연히 개별 LLM 호출이 전체 파이프라인의 병목일 것이라 가정했다. 그런데 aggregate-pipeline-serving 연구의 실증 데이터는 정반대였다 — 진짜 병목은 LLM 호출 자체가 아니라 스키마 축적이었다.

📊 선례의 구조

커뮤니티의 가정: "당연히 LLM 호출이 병목일 것이다"

실제 발견: 스키마 축적이 레이턴시를 지배했다

구조적 패턴: 가시적인 것을 병목으로 오인하는 경향

이 선례는 지금 우리가 직면한 질문과 구조적으로 완벽하게 동일하다. 현재 에이전틱 소프트웨어 엔지니어링(SWE) 분야에서는 실행 파이프라인 효율 — C/R(체크포인트/리커버리) 런타임, 서빙 인프라, 파이프라인 레이턴시 — 이 병목이라고 암묵적으로 전제하고 있다. 그런데 이 전제 역시 재검토 대상이 아닐까?

Section 02

하류 인프라 최적화의 근본적 한계

하류 최적화 연구들은 확실히 가치가 있다. 하지만 그 한계를 정확히 이해해야 한다.

agentic-harness-engineering에서 분석된 Crab 런타임의 사례를 보자. 에이전트가 수행하는 작업에는 의미론적 의존 그래프라는 것이 있다 — 어떤 코드가 어떤 코드에 논리적으로 의존하는지에 대한 구조다. 그런데 이것을 실제 OS 수준에서 실행하려면, 의미론적 구조를 OS의 체크포인트 메커니즘 — 단순한 전체 상태 스냅샷 — 으로 번역해야 한다.

비유: 요리 레시피를 기계어로 번역하기

"양파를 반달 모양으로 썰어 올리브 오일과 함께 볶을 때, 투명해질 때까지 중간 불에서 저어가며 3분간 가열하라"는 지시가 있다고 치자. 이 의미론적 지시를 기계어 수준에서 번역하면 — 온도 센서 값 180°C, 회전수 60rpm, 지속 시간 180,000ms — 가 된다.

번역 자체는 가능하다. 하지만 "투명해질 때까지"라는 의미는 이미 손실되었다. 기계는 3분을 기계적으로 카운트할 뿐, 양파가 실제로 투명해졌는지는 모른다. 알고리즘-시스템 번역 간극(algorithm-system translation gap)이 바로 이것이다.

이 간극은 정보이론적으로 손실이 불가피하다. 상위 수준의 의미론을 하위 수준의 시스템 메커니즘으로 변환할 때, 의미의 일부는 필연적으로 증발한다. C/R 런타임이 아무리 정교해져도 이 한계를 넘을 수 없다.

더 근본적인 문제가 있다. 하류 인프라 최적화는 체감 레이턴시를 줄일 수는 있어도, 에이전트가 '잘못된 것을 빠르게' 실행하는 문제는 해결하지 못한다. 빠른 크레인으로 잘못된 설계도의 집을 지으면, 그저 더 빨리 무너지는 집이 만들어질 뿐이다.

Section 03

상류 결함이 하류에서 폭발하는 메커니즘

그렇다면 상류 — 요구사항 사양 설계 단계 — 에서의 결함은 구체적으로 어떻게 문제가 되는가? 여기서 compositional-incoherence(구성적 비일관성)라는 개념이 핵심 증거로 등장한다.

Multi-component LLM agents assemble probabilistic claims from components that each see only part of a joint problem; the composition can violate basic probability axioms even when every component is locally correct. — compositional-incoherence 연구에서

이 문장의 무게를 풀어보자.

지역적으로 옳지만 전역적으로 틀린

다 컴포넌트 에이전트 시스템을 상상해 보자. 컴포넌트 A는 인증 모듈을, 컴포넌트 B는 데이터베이스 스키마를, 컴포넌트 C는 API 엔드포인트를 각각 설계한다. 각 컴포넌트는 자신이 본 부분에 대해서는 정상적으로 동작한다 — A의 인증 로직은 완벽하고, B의 스키마는 정규화되어 있고, C의 엔드포인트는 RESTful하다.

그런데 이것들을 조합하면 어떻게 되는가? A는 세션 기반 인증을 가정했고, B는 토큰 기반을 전제했고, C는 둘 다 아닌 API 키 방식을 구현했다면? 각 부분은 옳았지만 전체는 기본적인 일관성을 위반한다. 확률 공리를 위반한다는 것은, 논리적으로 동시에 성립할 수 없는 상태가 시스템에 공존하게 된다는 뜻이다.

⚠️ 왜 롤백은 이 문제를 해결하지 못하는가

C/R(체크포인트/리커버리) 메커니즘은 '실패한 실행'을 되돌리는 도구다. 테스트가 깨졌을 때, 런타임 에러가 났을 때 이전 상태로 되돌아간다.

하지만 구성적 비일관성의 문제는 다르다. 각 컴포넌트의 실행 자체는 '성공'이다. 에러가 나지 않았으니 롤백할 대상이 없다. 테스트를 통과한 코드들이 조합되었을 때 비로소 일관성이 무너진다.

롤백은 '잘못 명세된 요구사항'을 교정하지 못한다. 그것은 오직 상류에서만 가능한 작업이다.

Section 04

병목의 3층 구조: 우리가 집중하는 곳과 실제 병목

Wiki에 축적된 논문들의 증거를 기반으로, 에이전틱 SWE의 병목을 세 층위로 재구성할 수 있다.

층위 병목 유형 핵심 증거 최적화 가능성
상류 요구사항 추론 품질 SpecBench, compositional-incoherence 근본적 · 난해
중류 의미론-시스템 번역 algorithm-system-translation-gap, agent-os-semantic-gap 정보손실 불가피
하류 실행 파이프라인 효율 aggregate-pipeline-serving, agentic-harness-engineering 공학적 최적화 가능

이 표가 보여주는 비대칭성이 핵심이다. 하류는 가장 '납득하기 쉬운' 엔지니어링 문제다 — 레이턴시를 줄이고, 체크포인트를 최적화하고, 파이프라인을 병렬화하면 된다. 측정도 쉽고, 성과도 직관적으로 보인다. 그래서 연구가 집중되었다.

하지만 중류는 이미 정보손실이 불가피한 영역이며, 상류는 근본적으로 난해한 추론 문제다. 그리고 상류의 결함은 중류와 하류를 거치면서 복원 불가능한 형태로 폭발한다 — 앞서 본 구성적 비일관성이 그 증거다.

비유: 3층 버그 필터

상류에서 발생한 결함은 거대한 돌멩이다. 중류 필터는 이 돌멩이를 조각내어 통과시킨다 — 의미의 손실이 여기서 발생한다. 하류 필터는 이미 조각난 파편을 처리하므로, 원래 문제가 '돌멩이였다'는 사실 자체를 알 수 없다.

하류에서 파편을 더 빠르게 처리하는 것은, 돌멩이를 필터에 통과시킨 원인을 해결하지 못한다. 오히려 더 빠르게 파편을 처리할수록, 원인을 추적하기 더 어려워진다.

benchmark-specification-gap이라는 슬러그가 Wiki에 존재한다는 사실 자체가 이 문제의 방증이다. 벤치마크가 측정하려는 것과 실제 소프트웨어 개발이 요구하는 것 사이에 간극이 있다는 인식이 이미 커뮤니티 내에 있었던 것이다. SpecBench는 이 간극을 '사양 설계'라는 구체적 축으로 좁혀, 상류 병목을 정식화하려는 첫 번째 조치다.

Section 05

패러다임 전환: '빠르게 만들기'에서 '올바르게 설계하기'로

지금까지의 분석이 하나의 명확한 결론을 향해 수렴한다.

핵심 주장

에이전틱 소프트웨어 엔지니어링의 진정한 병목은 하류 실행 효율이 아니라 상류 요구사항 추론 품질에 있다. 하류 인프라 최적화는 필수적이지만 보완적 역할에 머물며, 상류에서의 결함은 하류에서 복원 불가능한 형태로 폭발한다.

이 결론이 의미하는 패러다임 전환은 구체적으로 세 가지 방향으로 나타난다.

첫째, 평가 축의 전환

현재 SWE 에이전트 벤치마크는 "주어진 문제를 얼마나 빨리, 얼마나 정확하게 코드로 구현하는가"를 측정한다. SpecBench가 제안하는 방향은 "주어진 문제를 얼마나 정확하게 사양으로 번역하는가"를 독립적인 평가 축으로 분리하는 것이다. 코드 생성 능력과 사양 설계 능력은 별개의 능력이며, 후자가 전자의 상한을 결정한다.

둘째, 연구 투자의 재분배

하류 인프라 최적화에 쏟아지는 연구 자원의 일부를 상류 추론 품질 연구로 이동해야 한다. 구성적 비일관성을 사전에 탐지하는 메커니즘, 불완전한 요구사항을 자동으로 보완하는 추론 프레임워크, 다 컴포넌트 간 사양 일관성을 검증하는 정적 분석 기법 — 이런 방향들이 새로운 연구 프론티어가 되어야 한다.

셋째, '빠른 실패'에서 '빠른 교정'으로

현재 에이전틱 SWE의 철학은 빠른 실패(fail fast)에 가깝다 — 빠르게 코드를 생성하고, 테스트하고, 롤백하고, 재시도한다. 하지만 상류 병목의 관점에서 보면, 우리가 필요한 것은 빠른 교정(correct fast)이다 — 사양 단계에서 불일관성을 조기에 포착하고, 코드가 생성되기 전에 요구사항을 교정하는 것이다.

최종 비유: 건축의 미래

건축 산업은 이미 이 전환을 거쳤다. 과거에는 시공 속도와 재료 효율이 경쟁력의 전부였다. 하지만 지금은 BIM(건물 정보 모델링) — 시공 전에 3D 모델로 충돌을 검증하고, 구조적 일관성을 확인하는 상류 설계 프로세스 — 가 훨씬 더 중요한 경쟁력으로 자리 잡았다.

에이전틱 SWE도 같은 성숙 경로를 걷고 있다. SpecBench는 그 경로상의 이정표 — "우리는 지금 어디를 고치고 있는가?"라는 질문에 대한 체계적인 답의 시작 — 이다.

aggregate-pipeline-serving이 실행 파이프라인 내에서조차 병목의 위치를 재정의해야 했듯이, 에이전틱 SWE 전체 시스템에서도 동일한 재정의가 필요하다. 가시적인 병목을 고치는 것과, 실제 시스템의 성능을 결정하는 병목을 고치는 것은 다르다. SpecBench는 우리에게 그 차이를 직시하라고 말하고 있다.
