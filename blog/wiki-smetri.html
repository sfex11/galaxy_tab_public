요청 중심 최적화는 에이전트 서빙에서 근본적으로 잘못된 추상화 계층인가? — SMetric 논문을 축으로 세 가지 수준의 추상화 붕괴를 분석한다.

2026-05-06 · 읽기 약 12분 · SMetric · Tool Attention · Crab · SpecKV

대형 언어 모델(LLM) 서빙의 역사는 본질적으로 "하나의 요청을 얼마나 빨리 처리할 것인가"의 역사였다. vLLM의 PagedAttention도, Orca의 연속 배칭도, Splitwise의 분할 처리도 모두 이 질문에 대한 답이었다. 그런데 에이전트가 등장하면서 상황이 묘하게 어긋나기 시작했다.

SMetric 논문("Rethink LLM Scheduling for Serving Agents with Balanced Session-centric Scheduling")은 이 어긋남의 본질을 날카롭게 짚는다: 에이전트는 개별 LLM 응답 토큰이 아니라 완전한 응답, 즉 행동 완수에 대해서만 다음 결정을 내린다. 그런데 기존 스케줄러는 각 LLM 호출을 독립적인 요청으로 취급하여 개별 레이턴시를 최소화하려 한다.

비유: 응급실과 외과 수술

환자가 병원에 온다. 기존 서빙은 "각 환자를 진료실에 얼마나 빨리 들여보낼 것인가"를 최적화한다. 하지만 에이전트 환자는 수술 전 검사 → 마취 → 수술 → 회복이라는 일련의 과정이 하나의 단위로 묶여 있다. 검사 결과가 나와야 마취를 결정하고, 마취가 끝나야 수술이 시작된다.

이때 "각 진료 단계를 독립적인 환자로 취급해서 줄 세우면 전체 수술 시간이 최적화될 것"이라는 가정이 바로 요청 중심 최적화이다. 검사실 대기 시간은 줄었지만, 환자는 검사 → 마취 → 수실을 옮기는 동안 대부분의 시간을 방치된다.

결론부터 말하면, Wiki에 축적된 증거들은 이 질문에 "그렇다"에 가까운 답을 향해 모이고 있다. 하지만 단순한 "틀렸다"가 아니라, 추상화 계층의 부정합이 세 가지 수준에서 계층적으로 드러난다는 것이 더 정확한 분석이다.

01요청 내부에서의 추상화 붕괴

문제의 가장 얕은 층은 요청 내부에서 시작된다. "Tool Attention Is All You Need"(2026-04-26)라는 논문은 다중 서버 MCP(Model Context Protocol) 배포 환경에서 파이프라인 전체의 레이턴시 병목을 조사했는데, 그 결과가 당혹스러웠다.

핵심 발견

진짜 병목은 LLM 추론이 아니라 스키마 축적이었다

다중 툴 환경에서 에이전트가 하나의 요청을 처리할 때, LLM이 토큰을 생성하는 시간보다 툴 스키마를 로딩하고 축적하는 시간이 전체 레이턴시를 지배했다. 기존 서빙 최적화는 "하나의 요청 = 하나의 LLM 추론"이라는 암묵적 추상화에 기반하는데, 이 추상화가 이미 요청 내부의 실제 병목 지점을 가려버린 것이다.

이 발견이 중요한 이유는 단순히 "스키마 로딩이 느리다"가 아니다. 요청이라는 추상화 단위 자체가 무엇이 병목인지를 숨겨버린다는 점이 핵심이다. 마치 환자의 진찰 시간을 재면서 "수술 대기 시간"을 측정에서 제외하는 것과 같다.

기존 서빙 최적화는 "하나의 요청 = 하나의 LLM 추론"이라는 암묵적 추상화에 기반한다. 그러나 에이전트 파이프라인에서는 하나의 요청 내부에 툴 스키마 로딩, 여러 LLM 호출, 툴 실행 등이 연쇄적으로 존재하며, 진짜 병목은 LLM 추론이 아니라 스키마 축적이었다.

— Wiki: aggregate-pipeline-serving

SMetric의 주장은 이것의 논리적 연장선이다. 요청 내부에서조차 추상화가 붕괴하는데, 여러 요청이 연쇄적으로 연결된 세션 전체를 요청 단위로 쪼개어 스케줄링하는 것은 더욱 근본적인 오류다.

02세션 수준에서의 추상화 부정합

두 번째 층은 요청 사이의 관계로 확장된다. 에이전트가 사용자의 질문에 답하기 위해 툴을 호출하고, 결과를 보고 다시 추론하고, 추가 툴을 호출하는 과정을 상상해 보라. 이것은 여러 개의 "요청"으로 구성되지만, 에이전트의 의사결정 단위는 각 요청이 아니라 세션 전체의 행동 완수다.

비유: 레시피와 조리 단계

파스타를 만든다고 할 때, "물 끓이기", "소스 만들기", "면 삶기", "버무리기"는 각각 독립적인 작업이다. 하지만 요리사의 결정 단위는 "파스타 완성"이다. 소스가 덜 익었는데 면을 먼저 삶아놓는 것이 최적이 아닌 것처럼, 각 LLM 호출의 레이턴시를 개별적으로 최소화하는 것이 세션 전체의 완수 시간을 최소화하지 못한다.

SMetric가 제기하는 핵심은 정확히 여기에 있다. 기존 스케줄러의 원자 단위는 "요청"이지만, 에이전트의 원자 단위는 "세션"이다. 스케줄링의 원자 단위를 요청에서 세션으로 전환해야 한다는 주장은 단순한 휴리스틱이 아니라, 문제 영역의 실제 의미론적 단위에 스케줄링을 정렬하자는 구조적 제안이다.

요청 중심 스케줄링이 "잘못된" 것이 아니라, 에이전트 서빙이라는 문제 영역에 대해 충분히 낮은 추상화 계층으로 내려가지 못한 것이다.

03의미론적 단위와 시스템 단위의 간극

세 번째 층에서 문제는 더 깊은 구조적 패턴으로 드러난다. "Crab"(2026-05-03)이라는 논문은 에이전트의 의미론적 상태 의존성을 OS의 체크포인트 메커니즘으로 번역할 때 발생하는 비효율을 다루는데, 이것은 SMetric의 문제와 정확히 동일한 구조를 갖는다.

세 경우 모두 에이전트가 의미론적으로 묶여 있는 단위와 시스템이 임의로 정한 단위 사이에 불일치가 존재한다. 이것이 서로 다른 논문에서 독립적으로 발견된다는 사실 자체가, 이것이 우연이 아니라 에이전트 컴퓨팅의 구조적 특성임을 시사한다.

새로운 연결 발견

algorithm-system-translation-gap과 aggregate-pipeline-serving은 서로 다른 논문에서 독립적으로 기술되어 있으나, 본질적으로 동일한 구조 — 에이전트의 의미론적 처리 단위와 시스템의 임의적 분할 단위 사이의 불일치 — 를 각각 런타임(Crab)과 서빙(Tool Attention, SMetric) 영역에서 현현한 것이다.

이를 아우르는 상위 개념으로 추상화 계층 부정합(abstraction-layer mismatch)을 제안할 수 있다.

04미시적 적응도 요청 단위를 벗어난다

마지막 층은 훨씬 미시적인 수준에서 같은 패턴을 보여준다. SpecKV(2026-05-05)는 적응형 추론(adaptive inference)에서 모델이나 라우트 선택을 넘어 추측 길이(γ) 선택이라는 결정 차원을 확장했는데, 이 적응은 "압축이라는 내부 시스템 상태"에 반응한다.

즉, 적응의 단위 역시 요청이 아니라 토큰 생성 과정 내부의 동적 상태다. 요청 단위 최적화는 이 미시적 적응의 공간까지 포착하지 못한다. 토큰마다 KV 캐시의 압축 상태가 다른데, 요청 전체를 일괄 처리하는 것은 마치 도로의 각 구간별 교통 상황을 무시하고 "이 도로의 평균 속도는 60km/h"로 관리하는 것과 같다.

미시 수준의 비대칭

SpecKV가 보여주는 적응은 토큰 생성 과정 내부의 동적 상태에 반응한다. 이 상태는 요청 단위로는 관측 불가능하다. 요청 단위 최적화는 실제로 존재하는 미시적 적응 공간을 추상화 계층 아래에 가둬버린다.

05세 수준의 동시 붕괴: 왜 "근본적으로" 잘못된 것인가

지금까지의 분석을 하나의 그림으로 정리하면 다음과 같다.

수준 실제 단위 기존 스케줄링의 단위 관련 증거
미시 토큰 내부 압축 상태 (γ) 요청 전체 일괄 처리 SpecKV
중간 파이프라인 내 스키마 축적 개별 LLM 호출 레이턴시 Tool Attention
거시 세션 수준 행동 완수 독립적 요청 SMetric

요청 중심 최적화가 "근본적으로" 잘못된 추상화 계층인 이유는 단일 지점에서의 오류가 아니라 세 수준에서 동시에 추상화 오류가 발생하기 때문이다. 하나의 수준에서만 어긋났다면 "개선이 필요하다"고 했을 것이다. 하지만 미시·중간·거시 모든 수준에서 요청이라는 단위가 실제 병목과 결정 단위를 보이지 않게 만든다면, 이것은 개선의 여지가 아니라 패러다임 전환의 필요성을指向한다.

요청 중심 최적화는 세 수준 모두에서 실제 병목과 결정 단위를 보이지 않게 만드는 추상화로 작동한다.

∥패러다임 전환: 어디로 가야 하는가

이 분석이 시사하는 방향은 명확하다. 에이전트 서빙의 최적화는 추상화 계층을 문제 영역의 실제 의미론적 단위로 내려야 한다.

  • 스케줄링의 원자 단위를 세션으로 전환할 것 — SMetric이 제안하는 세션 중심 스케줄링은 가장 거시적인 전환이다.
  • 파이프라인 내부의 실제 병목을 추상화하지 않을 것 — Tool Attention의 발견처럼, 요청 내부의 스키마 축적, 툴 실행 대기 등을 관측 가능한 단위로 스케줄링에 반영해야 한다.
  • 미시적 적응 상태를 스케줄링 결정에 노출할 것 — SpecKV가 보여주는 토큰 수준의 동적 상태가 스케줄러의 결정에 반영될 수 있는 인터페이스가 필요하다.
  • 의미론적 단위와 시스템 단위의 정렬을 설계 원칙으로 삼을 것 — Crab의 발견까지 포괄하는 "추상화 계층 부정합"을 에이전트 시스템 설계의 핵심 진단 프레임워크로 채택해야 한다.

에이전트 시대의 서빙은 더 이상 "요청을 빨리 처리하는 기술"이 아니다. 에이전트가 의미론적으로 묶여 있는 단위를 식별하고, 시스템의 추상화 계층을 그 단위에 정렬시키는 기술이 되어야 한다. 이것이 Wiki의 논문들이 조용히, 그러나 일치하게 가리키는 방향이다.
