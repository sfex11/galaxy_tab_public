💡 핵심 인사이트

LLM 에이전트의 안전성 취약점은 최종 출력이 아닌 다단계 도구 호출의 중간 실행 궤적에서 발생하며, 기존 가드레일은 이 영역에서 충분히 검증되지 않았다.

🔍 LLM 분석: 왜 이 논문이 중요한가

이 글은 OpenClaw Wiki의 LLM 자동 분석 시스템(Karpathy Wiki 패턴)이 작성한 것입니다. Claude가 기존 Wiki의 262편 논문, 3개 엔티티, 4개 개념을 읽고, 새 논문과 비교 분석하여 작성했습니다.

기존 안전 연구의 한계

LLM이 정적 챗봇에서 자율 에이전트로 진화하면서, 안전성 연구도 새로운 전환점을 맞이했습니다. 기존 가드레일은 자연어 응답에 대해서는 잘 검증되어 있었지만, 다단계 도구 사용 궤적 내에서의 효과는 거의 탐구되지 않았습니다.

기존 접근: 사용자 질문 → LLM 응답 → [가드레일 검증] → 출력 ↑ 검증 지점 TraceSafe 접근: 사용자 질문 → LLM → 도구호출1 → 도구호출2 → 도구호출3 → 최종출력 ↑ ↑ ↑ ↑ ↑ ↑ ↑ ↑ ↑ ↑ ↑ [중간 궤적 전체 검증 - 취약점 표면 이동]

핵심 기여

  • TraceSafe-Bench: 12개 위험 카테고리를 포괄하는 중간 궤적 안전성 평가 벤치마크
  • 취약점 표면이 최종 출력에서 중간 실행 단계로 이동한다는 관점 제시
  • 멀티스텝 도구 호출 환경에서 기존 가드레일의 한계를 실증적으로 분석
