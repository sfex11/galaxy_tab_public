우리가 최적화하던 게
아예 다른 병목이었다면?

Kamera 논문이 드러낸 멀티모달 에이전트의 숨은 병목, 그리고 적응적 추론이 답해야 할 새로운 질문

논문: Kamera (Training-Free Reuse) 키워드: 적응적 추론 · KV 캐시 · 멀티모달

01오피스 빌딩의 엘리베이터를 최적화하던 사람들

한 오피스 빌딩에서 엘리베이터 대기 시간을 줄이기 위해 수억 원을 들였다고 상상해 보자. 알고리즘을 고치고, 모터를 교체하고, 호출 버튼의 우선순위 로직을 정교하게 다듬었다. 그런데 조사 결과, 직원들의 실체적 지연의 70%는 엘리베이터가 아니라 각 층에서 동일한 서류 뭉치를 처음부터 다시 정리하는 데서 발생하고 있었다면?

엘리베이터를 아무리 빠르게 만들어도 전체 업무 흐름은 거의 개선되지 않는다. 문제는 '이동 속도'가 아니라 '같은 작업의 반복'이었기 때문이다. 당연해 보이지만, 이 구조적 오인은 AI 시스템 최적화에서도 정확히 벌어지고 있다.

적응적 추론(Adaptive Inference) 분야는 줄곧 "얼마나 많은 계산을 수행할 것인가"라는 축 위에서 발전해 왔다. 더 큰 모델을 쓸지 작은 모델을 쓸지, 추측 디코딩에서 몇 토큰을 한 번에 추측할지—모든 적응의 결정이 계산량의 조절에 머물러 있었다. SpecKV가 압축 상태에 반응하는 적응을 제시하며 한 걸음 내디뎠지만, 여전히 그 질문의 골격은 "계산 예산을 어떻게 분배할까"였다.

핵심 전제
기존 패러다임의 암묵적 가정

"계산 자체가 비용이다" — 이 가정 아래에서 적응적 추론은 계산량을 줄이거나 늘리는 방향으로만 진화할 수밖에 없었다. 마치 엘리베이터 속도가 건물의 유일한 병목이라고 믿었던 것처럼.

02계산이 많아서가 아니라,
같은 것을 반복해서

Kamera 논문이 멀티모달 에이전트의 동작을 추적하면서 발견한 것은 이렇다. 에이전트가 컨텍스트 윈도우를 슬라이드하고 추론을 반복할 때마다, 동일한 비디오 프레임, UI 스크린샷, 렌더링 산출물이 다시 등장한다. 문제는 이것이 매번 처음부터 재인코딩된다는 점이다.

"멀티모달 에이전트는 컨텍스트 윈도우가 슬라이드하고 추론이 반복됨에 따라 동일한 비디오 프레임, UI 스크린샷, 렌더링 산출물을 반복적으로 재검토하며, 그러나 매번 처음부터 재인코딩한다." — Kamera 논문 핵심 발견

여기서 병목의 본질이 드러난다. "계산이 너무 많아서"가 아니라 "불필요하게 동일한 것을 반복 계산해서" 느려지는 것이다. 이것은 γ(추측 길이)를 줄이거나 모델 크기를 조절하는 것으로는 구조적으로 해결될 수 없는 문제다.

비유로 돌아가자면, 이것은 엘리베이터 속도 문제가 아니라 매 층에서 같은 서류를 처음부터 다시 정리하는 문제다. 해결책은 엘리베이터를 더 빠르게 만드는 게 아니라, 이미 정리된 서류 뭉치를 그대로 가져오는 것이어야 한다.

병목 재정의
중복이 비용이다, 계산이 비용이 아니다

Kamera는 병목의 정의 자체를 바꾼다. 계산량이 아니라 중복 계산의 존재가 병목이다. 이 인식의 전환은 단순한 기술적 디테일이 아니라, 최적화의 방향성을 근본적으로 뒤바꾼다.

03이 전환은 처음이 아니다 —
병목은 이동한다

사실 이런 종류의 '병목 재정의'는 다른 영역에서 이미 관찰된 바 있다. Tool Attention 논문이 다중 서버 MCP 배포 환경에서 보여준 것처럼, 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정된다는 사실이었다.

"다중 서버 MCP 배포에서 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정된다." — Tool Attention (Aggregate Pipeline Serving)

우리가 "적응적으로 최적화해야 할 대상"이라고 믿었던 것—개별 LLM 호출의 계산량—이 실제 병목이 아니었다. 병목은 스키마 정보가 누적되며 발생하는 간접적 비용에 있었다. Kamera는 동일한 패턴을 멀티모달 캐시 영역에서 재발견한 셈이다.

두 사례를 나란히 보면 패턴이 명확해진다.

패턴 발견
적응적 병목 이동 (Adaptive Bottleneck Migration)

파이프라인 서빙에서는 병목이 LLM 호출 → 스키마 축적으로 이동했고, 멀티모달 추론에서는 토큰 생성 계산 → 시각 프레임 재인코딩으로 이동했다. 병목은 고정되어 있지 않다. 시스템 구조가 복잡해질수록 병목은 우리가 예상하지 못한 곳으로 이동한다.

이것이 의미하는 바는 명확하다. 고정된 축 위에서의 최적화는 병목이 이동하는 순간 무력해진다. 진정한 적응적 추론은 "계산량을 조절하는 것"에 머물러서는 안 되고, "현재 시스템 상태에서 어디가 실제 병목인지를 식별하고, 그에 맞춰 적응 축 자체를 전환하는 것"까지 포괄해야 한다.

04캐시 위상 보존 —
새로운 적응의 축

그렇다면 Kamera가 제시하는 구체적 대안은 무엇인가? 바로 위치 불변(Position-Invariant) KV 재사용이다. 컨텍스트가 슬라이드되어 시각 프레임의 위치가 바뀌어도, 동일한 프레임이라면 이전에 인코딩한 KV 캐시를 그대로 재사용한다는 아이디어다.

이것은 단순한 메모리 절약 기법이 아니다. 여기에는 캐시 일관성(Cache Coherence)이라는 까다로운 제약이 따른다. 위치 정보가 바뀌었는데도 캐시를 재사용하면, attention 패턴이 왜곡될 수 있다. Kamera는 이 문제를 위치 불변 설계로 해결하려 한다—즉, 캐시의 위상(위치 정보)을 보존하면서도 일관성을 유지해야 한다.

이것을 적응적 추론의 언어로 번역하면 완전히 새로운 결정 차원이 등장한다.

차원 기존 적응적 추론 Kamera가 요구하는 적응
적응 대상 계산량 (γ, 모델 크기) 캐시 항목의 재사용 여부
상태 감지 압축률, 컨텍스트 길이 위치 불변성 (position invariance)
결정 "더/덜 계산한다" "재인코딩 없이 재사용한다 / 폐기한다"
병목 가정 계산 자체가 비용 중복이 비용

표에서 보듯, 단순히 최적화 대상이 바뀌는 게 아니다. 적응의 의미론 자체가 달라진다. "얼마나 계산할까"에서 "어떤 캐시 항목의 위상을 보존할까"로 축이 이동한다. 이를 캐시 일관성 제약 하에서의 최적화(Cache Coherence-Constrained Optimization)라고 부를 수 있다.

"이 토큰은 몇 개의 추측으로 생성할까?"
— 계산량 축
▼
"이 프레임은 이미 캐시에 있는가?
위치 불변성을 만족하는가?"
— 캐시 위상 보존 축

SpecKV가 "압축 상태에 반응하는 적응"을 제시했다면, Kamera는 "캐시 중복 상태에 반응하는 적응"이라는 새로운 유형을 추가한다. 그리고 이 두 가지는 상호보완적으로 통합될 수 있다—압축 상태를 감지해 계산량을 조절하면서 동시에, 캐시 중복을 감지해 재인코딩을 건너뛰는 식으로.

05전환이 아니라 스펙트럼의 확장

처음의 질문으로 돌아가자. "위치 불변 KV 재사용은 적응적 추론의 최적화 축을 '계산량 조절'에서 '캐시 위상 보존'으로 근본적으로 전환해야 하는가?"

대답은 "전환이라기보다는 확장이다. 하지만 그 확장이 기존 패러다임의 우위를 해체한다는 점에서 근본적이다."

'전환'이라고 하면 기존 축이 새로운 축으로 완전히 대체되는 느낌을 준다. 하지만 계산량 조절이 여전히 유효한 최적화 수단인 것은 맞다. 다만, 그것이 유일한 축이 아니게 되었고, 더욱이 특정 맥락에서는 보조적 축으로 전락한다는 점이 핵심이다.

Paradigm Shift

기존 패러다임을 어떻게 전환해야 하는가

  1. 적응적 병목 이동을 상위 개념으로 삼아라. 적응적 추론이 고정된 축(계산량) 위에서 최적화하는 것이 아니라, 병목의 위치를 추적하며 적응 축 자체를 이동시키는 능력을 포괄해야 한다.
  2. 캐시 일관성 제약을 최적화의 일차 조건으로 편입하라. 멀티모달 에이전트에서 Kamera의 위치 불변 KV 재사용은 선택적 기법이 아니라, 캐시 일관성 제약 하에서의 최적화라는 새로운 문제 공식화를 요구한다.
  3. 적응의 스펙트럼을 확장하고 계층화하라. SpecKV의 "압축 상태에 반응하는 적응"과 Kamera의 "캐시 중복 상태에 반응하는 적응"을 상호보완적으로 통합하여, 단일 축이 아닌 다차원 적응 공간에서 시스템을 운영해야 한다.

결국 Kamera가 우리에게 던지는 진짜 질문은 이런 것이다. "적응적 추론은 도대체 무엇에 대해 적응하는가?" 계산량에 적응하는 것만으로 충분했던 시대는 지났다. 이제 적응의 대상은 병목 자체이며, 병목은 이동하며, 우리는 그 이동을 추적하면서 적응 축을 동적으로 전환해야 한다.

오피스 빌딩의 비유로 끝맺자. 진정으로 똑똑한 빌딩 관리 시스템은 엘리베이터 속도만 조절하는 게 아니다. 아침에는 엘리베이터 병목에 집중하고, 점심에는 식당 혼잡도에 대응하고, 오후에는 회의실 예약 병목을 예측한다. 병목이 이동하는 만큼, 최적화의 초점도 이동해야 한다.

Kamera는 멀티모달 AI가 그 단계에 도달했음을 알려주는 신호다.
