01

보상 신호의 근본적 결핍: 끝줄만 보는 심판

비유

축구 경기에서 심판이 경기 내내 눈을 감고 있다가, 경기 종료 휘슬과 동시에 승패만 발표한다고 상상해 보세요. 선수들은 자신의 어느 패스가 실점으로 이어졌는지, 어느 태클이 동점골의 시작이었는지 전혀 알 수 없습니다. 결과적으로 "다음 경기에서 무엇을 고쳐야 할지"를 스스로 알아내야 하는데, 이것이 다단계 도구 사용 RL이 직면한 정확한 상황입니다.

다단계 도구 사용 환경에서 RLVR(Reinforcement Learning from Verifiable Rewards)의 근본적 한계는 크레딧 할당의 입자도 문제로 현현합니다. 최신 연구 "Reinforcement Learning without Ground-Truth Solutions can Improve LLMs"은 RLVR이 기존에는 ground-truth 정답에 의존해 왔음을 명시하며, 이것이 없는 영역으로의 확장을 시도합니다. 그러나 다단계 도구 사용에서는 상황이 근본적으로 다릅니다.

다단계 파이프라인에서 관측 가능한 것과 불가능한 것

개별 도구 호출이 정답인지 오답인지 평가할 수 없습니다. 전체 궤적의 성공 혹은 실패만 관측 가능하며, 중간 단계의 기여도를 역추적할 신호가 부재합니다. 이것이 단순한 최적화 난이도 문제가 아니라, 보상 신호 자체가 해당 능력 차원을 인코딩하지 못하는 구조적 결함이라면, 순수 보상 최대화는 도구 사용 능력을 발현할 수 없습니다.

cascading-distribution-shift가 암시하듯, 다단계 파이프라인에서는 분포 변이가 누적됩니다. 1단계에서 미세하게 벗어난 출력이 2단계에서 더 큰 오차를 만들고, 이것이 3단계에서는 완전히 다른 영역으로 전이되어 버립니다. 이런 환경에서 단일 단계의 보상이 전체 궤적의 품질을 proxy할 수 없게 되는 것은 최적화 알고리즘의 한계가 아니라, 신호 구조 자체의 한계입니다.

02

최적화가 닿지 못하는 구조적 병목

비유

요리사가 "어떤 재료를 언제 넣을지"를 완벽하게 결정하더라도, 주방의 냉장고 안에서 재료가 어떻게 배열되어 있는지는 요리사의 결정 능력과는 무관한 문제입니다. 아무리 좋은 레시피를 따라도, 냉장고 구조가 비효율적이면 요리 속도는 느려집니다. 도구 사용의 핵심 병목도 이와 같습니다.

가장 결정적인 증거는 "Tool Attention Is All You Need: Dynamic Tool Gating and Lazy Schema Loading"이라는 논문에서 나옵니다. 이 연구는 다중 서버 MCP(Model Context Protocol) 배포에서 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정된다는 사실을 보여줍니다.

다중 서버 MCP 배포에서 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출이 아닌 스키마 축적에 의해 결정됨 — Tool Attention Is All You Need, 2025

이것을 우리의 질문 프레임으로 번역하면 명확해집니다. 도구 사용의 핵심 병목은 "언제 호출할지"에 대한 결정 품질—보상으로 최적화 가능한 영역—이 아니라, "어떤 스키마를 어떻게 축적하고 관리할지"에 대한 구조적 문제입니다.

보상 최대화는 LLM의 출력 분포를 조정할 수 있습니다. 하지만 스키마 축적은 파이프라인 아키텍처 자체의 속성입니다. "To Call or Not to Call" 같은 연구가 도구 호출을 평가·최적화의 대상으로 삼으면서도, 그 프레임워크가 다루지 못하는 인프라 계층의 의사결정이 별도로 존재한다는 것이 이 증거의 핵심입니다.

요약: 최적화 영역과 구조적 영역의 분리

보상 최대화가 조정할 수 있는 것은 "모델이 내뱉는 토큰의 확률 분포"입니다. 하지만 스키마가 어떻게 로드되고, 파이프라인이 어떻게 구성되며, 도구 간 의존성이 어떻게 관리되는지는 파이프라인 아키텍처의 속성이지, 토큰 확률의 문제가 아닙니다. 이 두 영역이 혼재되어 있기 때문에 "보상만으로 도구 사용을 배우게 하려는 시도"가 구조적으로 불가능에 가깝습니다.

03

보상이 역효과를 낳는 역설

비유

자전거를 탈 때 "페달을 밟을 때마다 100원을 줍니다"라고 하면, 아이는 페달만 밟고 방향은 전혀 신경 쓰지 않을 것입니다. 반대로 "페달을 밟지 않을 때마다 100원을 줍니다"라고 하면, 아이는 멈춰서서 보상만 받으려 할 것입니다. 페달링이라는 행위 자체가 아니라, 페달링과 방향 전환의 균형이 자전거 타기의 핵심인데, 이 균형을 단일 스칼라 보상으로 인코딩하는 것은 근본적으로 어렵습니다.

Wiki에 blind-tool-invocation이라는 엔티티가 존재한다는 사실 자체가 시사적입니다. 이것은 도구를 판단 없이 호출하는 현상을 지칭하는데, 문제는 이것이 보상 최대화의 부산물로 발생할 수 있다는 점입니다.

  • 도구 호출 자체에 긍정적 보상이 부여되면 → 과도한 호출이 발생합니다
  • 호출 억제에 보상을 부여하면 → 필요한 호출까지 억제됩니다
  • 즉, 보상 함수의 형태에 무관하게 다단계 맥락에서 적절한 호출 균형을 보상만으로 인코딩하는 것은 불가능에 가깝습니다

capability-cooperation-paradox는 이 현상을 더 넓은 프레임으로 포착합니다. 개별 능력—언어 이해, 추론, 도구 스키마 파싱—이 각각 향상되어도, 이들이 협력하여 다단계 궤적을 구성하는 능력은 개별 향상에서 emergent하지 않는다는 역설입니다.

이것은 마치 오케스트라의 각 악기 연주자가 개인적으로 완벽한 기량을 갖추더라도, 지휘자 없이 악보 없이 합주하려면 혼란이 불가피한 것과 같습니다. 개별 기량의 합이 합주의 질로 이어지려면, 악기 간의 시간적 조율, 음량 균형, 악장 전환의 타이밍 등 "협력 메커니즘"이 별도로 존재해야 합니다. 도구 사용도 마찬가지입니다.

04

기하급수적 상한: 단계가 늘어날수록 무너지는 수학

비유

10개의 연속적인 문을 통과해야 하는 미로를 생각해 보세요. 각 문을 통과할 확률이 90%라면, 10개를 모두 통과할 확률은 약 35%에 불과합니다. 각 문의 통과 확률을 95%로 끌어올려도, 10개를 모두 통과할 확률은 약 60%입니다. 문 하나하나의 성능을 크게 높여도, 연속 구조 때문에 전체 성공률은 파이프가 길어질수록 수학적으로 억제됩니다.

"When Does Combining Language Models Help?"에서 도출된 co-failure-ceiling 개념은 다중 모델 시스템의 이득이 공동 실패율에 의해 상한이 정해진다는 것을 보여줍니다. 다단계 도구 사용 RL도 정확히 동일한 수학적 구조를 가집니다.

다단계 궤적의 성공 확률
P전체 = pN
p: 각 단계의 독립적 성공 확률  |  N: 전체 단계 수
구체적 수치 시뮬레이션

각 단계의 성공 확률 p를 0.9에서 0.95로 5%p 향상시켰다고 가정해 봅시다. 보상 최적화가 개별 단계의 성능을 상당히 끌어올린 것입니다. 하지만:

N = 5일 때: 0.59 → 0.77 (18%p 향상)
N = 10일 때: 0.35 → 0.60 (25%p 향상)
N = 20일 때: 0.12 → 0.36 (24%p 향상)
N = 50일 때: 0.005 → 0.077 (7.2%p 향상)

단계 수가 늘어날수록 동일한 최적화 노력의 효과가 기하급수적으로 감소합니다. 이것은 최적화 알고리즘의 효율성 문제가 아니라, 다단계 합성의 수학적 구조가 보상 기반 학습의 적용 가능 영역 자체를 제한한다는 것을 의미합니다.

다시 말해, 아무리 정교한 보상 함수를 설계하고, 아무리 강력한 최적화 알고리즘을 사용하더라도, 다단계 구조가 만들어내는 지수적 감쇠 앞에서는 보상 최대화 접근법이 본질적인 성능 상한에 부딪힙니다. 이 상한은 최적화의 질이 아니라 파이프라인의 길이에 의해 결정됩니다.

05

패러다임 전환: 별도 차원으로서의 도구 사용

지금까지 살펴본 네 가지 증거—크레딧 할당의 결핍, 구조적 병목, 보상의 역효과, 기하급수적 상한—는 각각 독립적인 문제가 아닙니다. 이것들은 하나의 통일된 결론을 향해 수렴합니다.

보상 최대화로 획득 가능한 차원
  • 단일 호출의 적절성 판단 — 이 도구를 써야 할지 말지
  • 개별 도구의 결과 해석 — 반환값의 의미 파악
  • 단일 단계 내에서의 파라미터 구성
보상 최대화로 획득 불가능한 차원
  • 스키마 축적 구조 — 어떤 도구 목록을 어떻게 관리할지
  • 호출/미호출 균형 — blind invocation의 회피
  • 다단계 협력 — 능력 간 조율 메커니즘
  • 기하급수적 상한 극복 — 파이프라인 길이 대응
  • 미세 크레딧 할당 — 중간 단계의 기여도 추적

왼쪽 차원은 "단일 시점, 단일 결정"의 영역입니다. 보상 신호가 이 결정의 옳고 그름을 비교적 명확하게 전달할 수 있습니다. 하지만 오른쪽 차원은 "다단계, 다능력, 다인프라"의 교차 영역입니다. 이 영역에서의 의사결정은 단일 스칼라 보상으로 인코딩될 수 없는 구조적 복잡성을 가집니다.

핵심 결론과 패러다임 전환

도구 사용 능력은 단일 차원이 아니라 두 개의 독립적인 차원으로 구성됩니다. 하나는 보상 최대화로 획득 가능한 "단일 결정의 질"이고, 다른 하나는 보상 최대화로 획득 불가능한 "다단계 협력의 구조"입니다.

기존 패러다임은 이 두 차원을 구분하지 않고, "더 나은 보상 함수 + 더 강력한 최적화 = 더 나은 도구 사용"이라는 암묵적 가정 아래 움직여 왔습니다. 하지만 논문 "Why Multi-Step Tool-Use RL Collapses"가 보여주는 붕괴 현상은 이 가정이 근본적으로 잘못되었음을 증명합니다.

필요한 전환은 다음과 같습니다:

  • 보상 설계 문제가 아니라 학습 파이프라인의 구조적 재설계 문제로 프레이밍할 것
  • 감독 신호(supervisory signal)를 "보상의 대체품"이 아니라 별도 차원에 대한 전용 학습 채널로 conceptualize할 것
  • 도구 사용 벤치마크에서 단계 수 N을 독립 변수로 통제하고, N에 대한 성능 감쇠 곡선을 핵심 평가 지표로 삼을 것
  • 스키마 관리, 호출 균형, 능력 조율을 각각 독립적인 모듈로 분리하여 설계할 것

결론적으로, 도구 사용 능력은 순수 보상 최대화로부터 emergent하게 얻을 수 없는 별도의 능력 차원입니다. 이를 인정하는 것이 붕괴를 멈추고, 진정한 다단계 도구 사용 에이전트로 나아가는 첫걸음입니다.
