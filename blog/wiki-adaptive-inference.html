모델은 이미 알고 있다
계산량과 컨텍스트, 분리된 줄 알았던 두 축이 하나였다는 증거

적응적 추론과 컨텍스트 관리는 독립적 차원으로 분류되어 왔다. 하지만 SWE-Pruner Pro와 SpecKV가 각각 독립적으로 밝혀낸 사실은 놀랍도록 같은 방향을 가리키고 있다.

대상 논문: SWE-Pruner Pro 관련: SpecKV, CADENCE

수도꼭지와 물줄기: 왜 우리는 두 개로 나누어 보았는가

일상적 비유

한겨울, 집의 온도를 맞추고 싶습니다. 방법은 크게 두 가지로 보입니다. 수도꼭지를 조절해 물의 양을 줄이는 것과 보일러 출력을 낮춰 물의 온도를 조절하는 것. 오랫동안 우리는 이 둘을 독립적인 제어 차원으로 여겼습니다 — 하나는 "량"의 문제, 다른 하나는 "질"의 문제.

하지만 만약 보일러가 이미 어느 방의 물이 필요 없는지 알고 있어서, 수도꼭지를 돌리기도 전에 그 방의 파이프로 물을 보내지 않고 있었다면 어떨까요? 그때 수도꼭지 조절과 보일러 출력 조절은 사실상 하나의 판단에서 나온 두 가지 행동에 불과합니다.

LLM 연구에서도 동일한 구조가 반복되어 왔다. 적응적 추론(adaptive inference)은 "모델 내부의 계산량을 상황에 맞게 조절하는 문제"로, 컨텍스트 관리(context management)는 "외부 인프라에서 불필요한 정보를 압축·축출하는 문제"로 각각 정의되었다. 하나는 안쪽, 하나는 바깥쪽 — 마치 수도꼭지와 보일러처럼.

그런데 SWE-Pruner Pro라는 논문이 이 분할에 균열을 냈다. 논문의 제목 자체가 선언한다:

"The Coder LLM Already Knows What to Prune"

— 코딩 LLM은 이미 무엇을 가지치기해야 할지 알고 있다.

이 문장이 던지는 파문은 단순하지 않다. "이미 알고 있다"는 것은, 컨텍스트 축소라는 외부 작업의 근거가 사실 모델 내부에 이미 존재한다는 뜻이다. 그렇다면 적응적 추론이 읽어내는 "내부 상태"와 컨텍스트 관리가 결정하는 "축소 대상"은 과연 다른 것일까?

SpecKV가 흘린 첫 번째 단서: 압축은 관련성의 거울이다

이 가설을 검증하려면 먼저 적응적 추론 쪽의 증거를 살펴야 한다. SpecKV는 기존 적응적 추론의 결정 차원을 "어떤 모델을 쓸까"에서 "추측 길이(γ)를 얼마로 할까"로 확장한 논문이다. CADENCE 같은 선행 연구들이 외부 환경(입력의 난이도 등)에 반응해 적응했다면, SpecKV는 한 단계 더 들어간다.

SpecKV의 핵심 발견

SpecKV는 압축(compression)이라는 내부 시스템 상태에 반응하는 적응의 새로운 유형을 제시했다. 여기서 "압축 상태"란 무엇인가? 그것은 본질적으로 어떤 정보가 보존되고 어떤 정보가 손실되었는가에 대한 모델 내부의 기록이다.

여기서 잠시 멈춰서 생각해보자. "어떤 정보가 보존되고 손실되는가" — 이것은 무엇에 대한 설명인가? 바로 컨텍스트 관련성(context relevance)에 대한 설명이다. 특정 토큰이 압축 과정에서 보존된다는 것은, 모델이 그 토큰을 "관련 있다"고 판단했다는 것의 다른 표현에 불과하다.

즉, SpecKV가 적응적 추론의 입력으로 사용하는 "압축 상태" 신호는 — 이름만 다를 뿐 — 컨텍스트 관리가 외부에서 찾으려 애쓰는 "관련성 신호"와 동일한 신호다. 경계가 흔들리기 시작한다.

핵심 포인트: 압축 상태는 관련성의 역함수가 아니다. 압축 상태 그 자체가 관련성의 인코딩이다. "이 정보는 압축해도 괜찮다"는 내부 판단과 "이 정보는 관련이 없다"는 판단은 동전의 양면이다.

SWE-Pruner Pro: 반대편에서 도착한 같은 결론

이제 컨텍스트 관리 쪽으로 시선을 돌리자. SWE-Pruner 시리즈는 소프트웨어 엔지니어링 작업에서 방대한 코드베이스를 LLM에 제공할 때, 어떤 부분을 컨텍스트에서 제외할지 결정하는 문제를 다룬다.

초대 SWE-Pruner는 이를 위해 별도의 코드 분석 모듈을 부착했다. 외부에서 관련성을 판단하고, 그 판단 결과를 바탕으로 컨텍스트를 축소하는 방식이었다. 이는 기존 패러다임의 정석적인 접근이었다 — "컨텍스트 관리는 외부 인프라의 문제"라는 전제 위에 서 있었다.

그런데 SWE-Pruner Pro가 이 구조를 뒤집었다.

별도의 분석 모듈 없이, 모델 내부 표현 자체가 이미 컨텍스트 관련성을 인코딩하고 있음을 보여준다. 모델은 외부 판단자가 없어도 무엇을 가지치기해야 하는지 알고 있다.

이 발견의 구조적 의미를 파악하려면, 두 논문을 나란히 배치해야 한다.

축 기존 분류 SpecKV가 보여준 것 SWE-Pruner Pro가 보여준 것
적응적 추론 계산량을 내부 상태에 맞게 조절 내부 상태 = 압축 상태 —
컨텍스트 관리 외부 인프라의 압축·축출 — 축소 대상 = 모델 내부 표현이 이미 결정

표가 말해주는 것은 명확하다. SpecKV 쪽에서는 적응적 추론의 입력이 압축 상태라는 이름의 관련성 신호였고, SWE-Pruner Pro 쪽에서는 컨텍스트 축소의 근거가 모델 내부 표현에 이미 존재하는 관련성 신호였다. 두 팀이 각자의 축에서 독립적으로 도달한 결론이 서로를 향해 손가락질하고 있다.

"압축 상태"와 "컨텍스트 관련성 표현"이 동일한 내부 신호의 두 가지 읽기 방식이라면, 두 축은 분리 불가능하다. 하나의 신호에서 두 가지 행동 — 계산량 조절과 컨텍스트 축소 — 이 나온다.

병목 오귀인: 분리해서 보면 안 보이는 것이 보인다

이 통합적 시각이 왜 중요한지, 다른 영역에서의 유사 패턴을 통해 확인할 수 있다.

파이프라인 서빙에서의 병례

다중 서버 MCP 배포를 다룬 연구에서는, 파이프라인 전체의 레이턴시 병목이 개별 LLM 호출(내부)이 아닌 스키마 축적(외부 인프라)에 의해 결정됨을 보여주었다. 여기서도 "LLM 호출 최적화"와 "스키마 관리"가 독립적 차원으로 가정되었지만, 실제 병목은 둘의 결합점에서 발생했다.

알고리즘-시스템 번역 간극

더 근본적으로, algorithm-system-translation-gap이라는 개념은 이 간극의 구조를 명확히 설명해준다. Crab 에이전트 사례에서 드러났듯, 에이전트의 의미론적 의존 그래프(어떤 상태에 의존하는가)를 OS의 체크포인트 메커니즘(전체 스냅샷)으로 번역할 때 비효율이 발생한다.

완전히 동일한 구조가 적응적 추론과 컨텍스트 관리 사이에 존재한다.

번역 간극의 정확한 구조

"모델이 내부적으로 이미 알고 있는 관련성 신호"를 → "외부 컨텍스트 윈도우 관리"라는 독립적 메커니즘으로 번역하는 데서 비효율이 발생한다.

SWE-Pruner Pro는 이 번역이 불필요함을 보여주고, SpecKV는 이 번역 없이 직접 내부 상태에 접근할 수 있음을 각각 보여준다.

이를 병목 오귀인(bottleneck misattribution)의 렌즈로 읽으면 더 선명해진다. 우리는 컨텍스트 관리를 "외부 인프라 문제"로, 적응적 추론을 "내부 계산 문제"로 각각 귀인해왔다. 하지만 이 분류 자체가 병목의 본질을 흐리게 만든다. 실제 병목은 어디에 있는가?

실제 병목의 위치: 내부 표현에서 관련성 신호를 어떻게 읽어내어 행동(계산량 조절 vs. 컨텍스트 축소)으로 변환하는가.

즉, 문제는 "어떤 축을 최적화할까"가 아니라 "하나의 신호를 어떻게 행동으로 번역할까"로 재정의되어야 한다.

SpecKV가 보여주듯, 적응의 결정 지점이 "모델/라우트 선택"에서 "추측 길이 선택"으로, 나아가 "컨텍스트 보존 결정"으로 이동하는 것은 병목이 다른 곳으로 옮겨간 것이 아니다. 그것은 동일한 내부 신호에 대한 독출 방식의 확장이다. 하나의 광맥에서 채굴 방향만 바꾼 것과 같다.

패러다임 전환: 두 축에서 하나의 신호로

지금까지의 논의를 하나의 문장으로 압축하면 이렇게 된다.

"계산량 조절"과 "컨텍스트 축소"는 동일한 내부 관련성 신호에 대한 두 가지 행동 출력이다.

SpecKV는 적응적 추론이 압축 상태 — 곧 관련성의 인코딩 — 에 반응한다는 것을 보였고, SWE-Pruner Pro는 컨텍스트 축소 대상이 모델 내부 표현에 이미 인코딩되어 있다는 것을 보였다. 두 논문이 독립적으로 도달한 결론이 서로를 보완하여, 위의 통합적 이해를 지지한다.

그렇다면 우리는 무엇을 바꾸어야 하는가?

  1. 엔티티 경계의 재검토. Wiki에서 적응적 추론과 컨텍스트 관리를 분리된 엔티티로 분석하던 방식을 재고해야 한다. 최소한 두 영역의 교차점을 명시적으로 표현하는 통합 엔티티가 필요하다.
  2. 최적화 대상의 재정의. "컨텍스트 윈도우를 얼마로 줄일까"와 "계산량을 얼마로 줄일까"를 따로 최적화하는 대신, "내부 관련성 신호를 어떻게 가장 효율적으로 행동으로 변환할까"를 최적화 대상으로 삼아야 한다.
  3. 외부 모듈 부착에 대한 회의. SWE-Pruner 초대가 보여준 "별도 분석 모듈 부착" 방식은, 모델이 이미 알고 있는 것을 외부에서 다시 판단하는 이중 노동일 가능성이 높다. 내부 표현에서 직접 신호를 읽어내는 방향으로 설계 패러다임이 이동해야 한다.
  4. 병목 진단 프레임워크의 수정. 시스템 병목을 "내부 vs. 외부"로 나누어 진단하는 기존 프레임워크는, 두 축이 실제로는 하나라는 사실을 간과하게 만든다. "신호-행동 번역 지점"을 중심으로 병목을 진단하는 새로운 프레임워크가 필요하다.

다시 수도꼭지 비유로 돌아가자. 보일러가 이미 어느 방의 물이 불필요한지 알고 있다면, 우리가 설계해야 할 것은 수도꼭지와 보일러를 각각 최적화하는 독립 시스템이 아니다. 보일러의 판단을 수도꼭지에 직접 연결하는 하나의 파이프다.

모델은 이미 알고 있다. 우리가 해야 할 일은 그 지식에 귀를 기울이는 것이다.
